Section 6~12.5%

Solution & Operations Excellence

Operate reliably in production: observability, release management, and resilience testing — the operational-excellence pillar in practice.

The observability stack

Cloud Monitoring (metrics, dashboards, alerting policies), Cloud Logging (logs, log-based metrics, sinks to BigQuery/Pub/Sub/GCS), and Cloud Trace / Profiler (latency and CPU/heap profiling). Define SLIs and SLOs, and let error budgets govern how fast you ship.

Release management & resilience

Prefer progressive delivery — canary, blue/green, rolling — with automated rollback on SLO breach. Prove reliability with chaos engineering, load testing, and penetration testing rather than assuming it.

⚠️ Exam Trap: Operational-excellence answers favour automation and measurable SLOs over heroic manual intervention.

SRE: SLIs, SLOs & error budgets

Operational excellence is measured, not assumed. Pick SLIs (e.g. request latency, availability) that reflect user happiness, set SLOs (targets) on them, and run on the resulting error budget: while budget remains, ship features; when it's spent, freeze risky changes and invest in reliability. Reduce toil with automation, and run blameless postmortems after incidents.

⚠️ Exam Trap: Chasing 100% availability is the wrong answer — SRE deliberately targets an SLO below 100% so an error budget exists to spend on velocity.
SLO Service Level Objective = the line you promise; the gap to 100% is your error budget to spend on shipping.

🧠 Warm-Up Quiz

Q1. You want to release to 5% of users, watch error rates, then ramp up with instant rollback. Which strategy?

Reveal answer

Canary deployment with progressive traffic shifting.

Gradual exposure + metric-gated promotion + fast rollback.

Q2. Which tool profiles production CPU and memory usage to find hotspots?

Reveal answer

Cloud Profiler.

Continuous profiling of CPU/heap in production with low overhead.

Q3. A team has burned its monthly error budget. What should SRE practice dictate?

Reveal answer

Freeze risky feature releases and prioritise reliability work.

The error budget governs velocity: empty budget means slow down and stabilise.

Q4. Where do you route logs for long-term analytics and compliance retention?

Reveal answer

A Cloud Logging sink to BigQuery (or GCS for archive).

Sinks export logs to BigQuery for SQL analysis or to GCS for cheap long-term retention.

📝 Exam-Style Practice

Q1. A product team wants to ship features quickly, but leadership worries about reliability. What SRE construct balances release velocity against stability?

  • A) A change-advisory board approving every release manually.
  • B) An SLO with an error budget that gates risky releases when exhausted.
  • C) A target of 100% availability enforced on every service.
  • D) Freezing all releases for the last week of each month.
Reveal answer & explanation

Correct: B. The error budget is the SRE mechanism: while budget remains, ship; when it's spent, slow down and fix reliability. It balances both goals automatically.

A: A manual CAB is a bottleneck, not a measured balance. C: 100% is unrealistic and leaves no budget for change. D: A blanket freeze ignores actual reliability data.

Q2. You must release a new version to a small fraction of live users, monitor error rate and latency, then ramp or roll back fast. Which release approach should the architecture use?

  • A) Big-bang deploy to all users during a maintenance window.
  • B) Canary deployment with progressive traffic shifting and automated rollback on SLO breach.
  • C) Recreate strategy that tears down old pods before starting new ones.
  • D) Manual deploy with a rollback runbook for on-call.
Reveal answer & explanation

Correct: B. Canary exposes a small slice, gates promotion on metrics, and rolls back automatically — minimal blast radius with fast recovery.

A/C: Big-bang and recreate expose everyone at once. D: Manual rollback is slower and error-prone versus automated SLO-gated promotion.

Q3. Compliance requires keeping all audit logs for seven years and also running ad-hoc SQL analysis on recent logs. How should you configure log routing?

  • A) Leave everything in the default _Default log bucket and query there.
  • B) Create Cloud Logging sinks: one to BigQuery for analysis and one to GCS (with retention) for long-term archive.
  • C) Export logs manually with gcloud once a month.
  • D) Increase the default log bucket retention to seven years and query it directly.
Reveal answer & explanation

Correct: B. Sinks route logs to the right place: BigQuery for SQL analysis and GCS for cheap, durable long-term retention — meeting both needs.

A/D: The default bucket isn't built for seven-year archive or rich SQL analytics economically. C: Manual monthly exports are fragile and miss data.
