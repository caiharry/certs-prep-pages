Section 1.1

Designing for security and compliance

The exam wants least privilege, key-control nuance, sensitive-data handling, sovereignty, governance structure, and clean dev/prod separation.

WeightSection 1: ~22%
IAM and org policies Encryption and key management Privacy and PII Regional compliance Governance architecture Dev vs prod

Official guide bullets

  • Cloud IAM and organization policies.
  • Data security, encryption, and key management.
  • Privacy strategies for personally identifiable information.
  • Regional/data sovereignty and legal compliance.
  • Project, dataset, and table architecture for governance.
  • Multi-environment use cases.

What to know

  • CMEK: customer controls Cloud KMS keys; supported by BigQuery. Good default for auditability and control.
  • CSEK: customer supplies raw key material per request; not supported by BigQuery. Mainly Cloud Storage and Compute Engine.
  • EKM: key material stays in external HSM/KMS; choose when regulations require keys never leave your infrastructure.
  • Cloud DLP / Sensitive Data Protection: discover, classify, mask, tokenize, or de-identify sensitive data.
  • VPC Service Controls: service perimeter for supported Google-managed APIs such as BigQuery, Cloud Storage, and Pub/Sub.
  • Separate dev/test/prod with projects, datasets, service accounts, IAM, and environment-specific CMEK/config.
Exam traps: CSEK sounds strongest but is too limited for BigQuery. For a compromised CMEK on Cloud Storage, rotation only affects new encryption operations; rewrite objects with a new key/bucket to re-encrypt existing data.

Section 1.2

Designing for reliability and fidelity

Pick the data-quality, orchestration, recovery, and consistency patterns that keep pipelines correct, restartable, and observable.

WeightSection 1: ~22%
Data cleaning Pipeline monitoring Orchestration DR and fault tolerance ACID and availability Validation

Official guide bullets

  • Preparing and cleaning data with Dataform, Dataflow, Cloud Data Fusion, and LLM-assisted query generation.
  • Monitoring and orchestration of data pipelines.
  • Disaster recovery and fault tolerance.
  • ACID compliance and availability decisions.
  • Data validation.

What to know

  • Dataform: SQL ELT in BigQuery with dependencies, versioned workflow logic, and assertions for data quality.
  • Dataflow: custom Beam pipelines for unified batch/stream, cleansing, windowing, enrichment, and late data handling.
  • Cloud Data Fusion: low-code visual ETL when non-engineers need to build pipelines.
  • Validate with row counts, uniqueness, null checks, reconciliation queries, schema checks, and business-report samples.
  • Make loads idempotent and replayable; use retries and alerts for transient failures.
Exam signal: "ELT in BigQuery plus data quality checks inside the workflow" -> Dataform assertions. "Complex scheduled data pipeline" -> Cloud Composer.

Section 1.3

Designing for flexibility and portability

Map business constraints to architecture without locking data into a shape that blocks future residency, multi-cloud, discovery, or governance needs.

WeightSection 1: ~22%
Future requirements Portability Residency Staging Cataloging Profiling

Official guide bullets

  • Map current and future business requirements to the architecture.
  • Design for data and application portability, including multi-cloud and data residency requirements.
  • Plan data staging, cataloging, profiling, and discovery.

What to know

  • BigLake: query Cloud Storage, S3, or Azure data with BigQuery and centralized governance without loading every file into BigQuery.
  • Dataplex: govern lakes and warehouses with discovery, catalog, lineage, quality, and access management.
  • Dataplex Catalog: search and understand data assets.
  • Cloud Storage is the raw landing zone; BigQuery is the curated warehouse; Dataplex ties distributed data domains together.
Exam heuristic: if the question says governed lake files or cross-cloud lake access, think BigLake. If it says enterprise governance across lakes and warehouses, think Dataplex.

Section 1.4

Designing data migrations

Choose migration tooling from source, volume, downtime tolerance, and whether the need is one-time transfer, scheduled ingest, database migration, or CDC.

WeightSection 1: ~22%
Stakeholder analysis Migration plan Validation BQDTS DMS Transfer Appliance Datastream

Official guide bullets

  • Analyze current stakeholder needs, users, processes, and technologies.
  • Plan migration and validation to Google Cloud using BigQuery Data Transfer Service, Database Migration Service, Transfer Appliance, networking, and Datastream.

What to know

  • Storage Transfer Service: network transfer from S3, Azure Blob, GCS, or on-prem HTTP/HTTPS sources.
  • Transfer Appliance: physical device for hundreds of TB to PB when bandwidth is limited.
  • Datastream: ongoing CDC replication from operational databases to BigQuery or Cloud Storage.
  • BigQuery Data Transfer Service: scheduled transfers from Google SaaS and supported external sources into BigQuery.
  • Database Migration Service: lift-and-shift relational DB migration to Cloud SQL or AlloyDB.
Exam signal: "large data + slow/shared link" -> Transfer Appliance. "Live replication / CDC" -> Datastream. "Google SaaS to BigQuery" -> BigQuery Data Transfer Service.

Section 2.1

Planning the data pipelines

Before building, define sources, sinks, transformation logic, orchestration, network boundaries, and encryption requirements.

WeightSection 2: ~25%
Sources and sinks Transformations Orchestration Networking Encryption

Official guide bullets

  • Define data sources and sinks.
  • Define transformation and orchestration logic.
  • Apply networking fundamentals.
  • Plan data encryption.

What to know

  • Use Pub/Sub for spiky native event ingestion before processing.
  • Use Dataflow when you need one Beam model for batch and streaming.
  • Use BigQuery as the analytics sink; use Bigtable for low-latency key/range serving; use Cloud Storage for raw landing.
  • Use VPC Service Controls for managed-service exfiltration boundaries, not broad firewall/CIDR thinking.
  • Use CMEK/EKM when key-control requirements are explicit.
Planning shortcut: source -> buffer -> processor -> sink -> orchestrator -> observability -> IAM/CMEK/perimeter.

Section 2.2

Building the pipelines

This is the highest-value drilling area: service selection, Beam windowing, late data, Pub/Sub semantics, batch/streaming transforms, and enrichment.

WeightSection 2: ~25%
Data cleansing Dataflow / Beam Dataproc / Spark Cloud Data Fusion BigQuery Pub/Sub Kafka Windowing

Official guide bullets

  • Data cleansing and transformations.
  • Identify services: Dataflow, Apache Beam, Dataproc, Cloud Data Fusion, BigQuery, Pub/Sub, Spark, Hadoop, Kafka.
  • Batch, streaming, windowing, late-arriving data, processing logic, and AI enrichment.
  • Data acquisition, import, and new data-source integration.

What to know

  • Dataflow / Beam: new unified batch/stream pipeline, especially streaming.
  • Dataproc: existing Spark, Hadoop, Hive, or Pig workloads. Serverless is Spark-only with no cluster management.
  • Cloud Data Fusion: low-code visual ETL.
  • Pub/Sub: at-least-once delivery; duplicates are possible; ordering is per ordering key.
  • Windowing: fixed/tumbling for non-overlapping buckets, sliding/hopping for rolling windows, session for inactivity gaps.
  • Allowed lateness: keeps windows open after watermark; accumulating panes include previous results, discarding panes include only new records.
Watermark

Dataflow's estimate that events up to event-time T have probably arrived. It is not a guarantee.

Trigger

AfterWatermark is the standard on-time trigger. Processing-time triggers are for speculative early output.

Debugging

Use Reshuffle to break fusion and isolate bottlenecks; do not add noisy logging inside ParDo.

Exam patterns: "5-minute windows + late data + emit at watermark" -> FixedWindows + AfterWatermark + allowed lateness. "Rolling average" -> sliding. "Per-user activity bursts" -> session.

Section 2.3

Deploying and operationalising pipelines

Make pipelines repeatable: schedule them, deploy them through CI/CD, and pick Composer vs Workflows correctly.

WeightSection 2: ~25%
Cloud Composer Workflows CI/CD Job automation

Official guide bullets

  • Job automation and orchestration with Cloud Composer and Workflows.
  • Continuous Integration and Continuous Deployment.

What to know

  • Cloud Composer: managed Airflow for complex scheduled data DAGs with operators for BigQuery, Dataproc, GCS, and notifications.
  • Workflows: lightweight serverless API/microservice orchestration.
  • Eventarc: event trigger source, not a full data pipeline orchestrator.
  • Version Composer DAGs, Dataflow code/templates, Dataform workflows, SQL, and schemas.
  • Use dedicated service accounts, least privilege IAM, and environment-specific config.
Composer trap: SQL transformation/query job -> BigQueryInsertJobOperator. BigQueryUpsertTableOperator is for table metadata/properties, not transformations.

Section 3.1

Selecting storage systems

Storage questions are access-pattern questions. Choose from query shape, consistency, scale, latency, cost, lifecycle, and operational model.

WeightSection 3: ~20%
Access patterns Managed services Cost and performance Lifecycle management

Official guide bullets

  • Analyze data access patterns.
  • Choose managed services: BigQuery, BigLake, AlloyDB, Bigtable, Spanner, Cloud SQL, Cloud Storage, Firestore, Memorystore.
  • Plan for storage costs, performance, and lifecycle management.

What to know

  • BigQuery: OLAP, analytics, SQL scans, joins, aggregations.
  • Bigtable: massive low-latency key/range reads; row-key design controls performance.
  • Spanner: global ACID SQL, horizontal relational scale, strong consistency.
  • Cloud SQL: standard managed relational app workloads.
  • AlloyDB: high-performance PostgreSQL-compatible workloads.
  • Firestore: document/mobile/serverless app data.
  • Cloud Storage: object/blob storage, raw lake, backups.
  • Memorystore: in-memory cache/session store.
Exam traps: Bigtable is not for joins or ad hoc analytics. Do not pick Spanner just because HA is mentioned; look for global relational scale and strict consistency.

Section 3.2

Planning for a data warehouse

BigQuery depth: model shape, partitioning, clustering, BI Engine, materialized views, pricing model, and query diagnosis.

WeightSection 3: ~20%
Data model Normalization Business requirements Access patterns

Official guide bullets

  • Design the data model.
  • Decide the degree of data normalization.
  • Map business requirements.
  • Define architecture to support data access patterns.

What to know

  • BigQuery often favors denormalized or star-schema models for analytics performance.
  • Partitioning: split by column, usually date; filter on partition column to scan less data.
  • Clustering: sort within partitions by common high-cardinality filters such as user_id or device_id.
  • Materialized views: repeated expensive aggregations used by many users.
  • BI Engine: in-memory acceleration for dashboard latency; not a pricing model.
  • Slots: BigQuery compute units; Editions/reservations provide predictable capacity.
Exam heuristic: partition by time/date, cluster by high-cardinality columns frequently used in filters. Use both when queries filter on date and another selective column.

Section 3.3

Using a data lake

Know how raw object storage becomes useful: discovery, governance, cost controls, lake query, processing, and monitoring.

WeightSection 3: ~20%
Discovery Access controls Cost controls Processing Monitoring

Official guide bullets

  • Manage the lake with data discovery, access, and cost controls.
  • Process data in the lake.
  • Monitor the data lake.

What to know

  • Cloud Storage: raw lake storage and lifecycle policies.
  • BigLake: BigQuery engine and governance over lake files without loading all data into BigQuery.
  • Dataplex: discovery, cataloging, lineage, data quality, and governance across lake/warehouse assets.
  • Autoclass: automatically moves objects to colder classes based on access; useful for unpredictable access patterns.
  • Dual-region + Turbo Replication: 15-minute replication RPO target for Cloud Storage.
Autoclass trap: lifecycle rules are time-based and normal retrieval fees apply. Autoclass is the answer for unpredictable access plus no retrieval/early deletion charges after movement.

Section 3.4

Designing for a data platform

The data-platform answer is about governed self-service across distributed assets, not just one lake or one warehouse.

WeightSection 3: ~20%
Dataplex Dataplex Catalog BigQuery Cloud Storage Federated governance

Official guide bullets

  • Build a data platform with Google Cloud tools such as Dataplex, Dataplex Catalog, BigQuery, and Cloud Storage.
  • Build a federated governance model for distributed data systems.

What to know

  • Data lake: raw data in native formats, usually Cloud Storage plus BigLake.
  • Data warehouse: curated structured data, usually BigQuery.
  • Data platform: governed operating model across lakes, warehouses, teams, and domains.
  • Federated governance: teams own domains while central controls define policy, discovery, access, and quality expectations.
Exam signal: "catalog, lineage, data quality, access management across distributed lakes and warehouses" -> Dataplex.

Section 4.1

Preparing data for visualization

Prepare BigQuery for BI users: precalculate where useful, accelerate dashboards, troubleshoot slow queries, and protect sensitive columns.

WeightSection 4: ~15%
Connecting tools Precalculated fields BI Engine Materialized views Query troubleshooting Data masking / DLP

Official guide bullets

  • Connect to visualization tools.
  • Precalculate fields.
  • Use BigQuery BI features such as BI Engine and materialized views.
  • Troubleshoot poor-performing queries.
  • Apply security, data masking, IAM, and Cloud DLP.

What to know

  • BI Engine: in-memory acceleration for dashboards and repeated queries; reduces latency.
  • Materialized views: precompute repeated expensive aggregations.
  • Query tuning: avoid SELECT *, prune partitions, cluster on common high-cardinality filters, inspect execution details/shuffle/slots.
  • Protect sensitive data with IAM, policy tags/masking patterns, and Sensitive Data Protection when discovery/de-identification is needed.
Fast recognition: repeated dashboard aggregation -> materialized view. Low-latency dashboard acceleration -> BI Engine. High scan cost -> partition pruning and column selection.

Section 4.2

Preparing data for AI and ML

For this exam, AI/ML prep is mostly data readiness, BigQuery ML, feature engineering, and unstructured-data preparation for embeddings/RAG.

WeightSection 4: ~15%
Feature engineering Training and serving BigQuery ML Embeddings RAG

Official guide bullets

  • Prepare data for feature engineering, training, and serving ML models.
  • Use BigQuery ML where appropriate.
  • Prepare unstructured data for embeddings and retrieval-augmented generation.

What to know

  • BigQuery ML: train and run ML models in BigQuery using SQL.
  • For features: validate labels, handle nulls/outliers, avoid leakage, keep training/serving transformations consistent.
  • For embeddings/RAG: clean documents, chunk them, generate/store embeddings, retrieve relevant chunks, and ground model output.
  • For LLM-generated SQL: validate generated SQL, constrain accessible datasets, and keep IAM/governance boundaries in place.
Exam angle: you already know the ML concepts; look for the Google Cloud data-prep service choice and governance boundary.

Section 4.3

Sharing data

Know the BigQuery-native sharing path and the governance controls around published datasets, reports, and visualizations.

WeightSection 4: ~15%
Sharing rules Publishing datasets Reports and visualizations Analytics Hub

Official guide bullets

  • Define rules to share data.
  • Publish datasets.
  • Publish reports and visualizations.
  • Use BigQuery sharing with Analytics Hub.

What to know

  • Analytics Hub: publish and subscribe to shared BigQuery datasets.
  • Use IAM/governance controls to define who can discover, subscribe, query, or administer shared assets.
  • For sensitive data, pair sharing with masking/de-identification/policy controls rather than copying raw PII everywhere.
  • Dataplex helps users discover governed assets; Analytics Hub is the BigQuery sharing marketplace pattern.
Exam signal: "share curated BigQuery datasets with consumers" -> Analytics Hub.

Section 5.1

Optimizing resources

Balance cost and capacity: BigQuery pricing, slots, cluster lifecycle, storage tiering, and only paying for what the business needs.

WeightSection 5: ~18%
Cost minimization Critical capacity Persistent vs job-based clusters

Official guide bullets

  • Minimize costs per required business need for data.
  • Ensure enough resources for business-critical data processes.
  • Choose persistent or job-based clusters, such as Dataproc.

What to know

  • BigQuery on-demand: pay by bytes scanned; best for irregular/low query volume.
  • Editions + reservations: predictable slot capacity for high, consistent volume or critical workloads.
  • Dataproc Serverless: Spark only, no cluster management.
  • Ephemeral Dataproc cluster: custom config or non-Spark Hadoop ecosystem jobs; delete after use.
  • Autoclass: use when Cloud Storage access patterns are unpredictable.
Dataproc signal: "Spark only + no special config" -> Serverless. "Hive/custom config/non-Spark" -> ephemeral cluster.

Section 5.2

Designing automation and repeatability

Airflow fundamentals matter: DAGs, operators, sensors, retries, catchup, and passing only small metadata between tasks.

WeightSection 5: ~18%
DAGs Scheduling Repeatable jobs Cloud Composer

Official guide bullets

  • Create directed acyclic graphs for Cloud Composer.
  • Schedule and orchestrate jobs in a repeatable way.

What to know

  • DAG: Python workflow with tasks, dependencies, and schedule.
  • Operator: task type such as BigQuery job, Dataproc job, Python function, or GCS copy.
  • Sensor: waits for a condition such as a file in GCS; deferrable sensors/operators avoid tying up workers.
  • XCom: small cross-task metadata only, not datasets.
  • catchup=False: prevents backfilling every missed schedule since start_date.
  • Use retries, retry delays, and failure callbacks for recoverable failures and alerting.
Exam signal: complex BigQuery/Dataproc/GCS pipeline with branching and schedules -> Composer, not Workflows.

Section 5.3

Organizing workloads based on business requirements

Separate interactive and batch needs, reserve capacity for critical workloads, and diagnose slot contention.

WeightSection 5: ~18%
BigQuery Editions Reservations Interactive queries Batch jobs

Official guide bullets

  • Capacity management with BigQuery Editions and reservations.
  • Interactive or batch query jobs.

What to know

  • Slots: BigQuery compute units.
  • Reservations: assign projects/folders/orgs to a slot pool so workloads share predictable capacity.
  • On-demand: cost varies with bytes scanned; simple but less predictable.
  • Reservations/Editions: better when usage is high, steady, business-critical, or needs isolation from noisy neighbours.
  • Use BigQuery admin panel/job history, slot utilization, and execution details for capacity diagnosis.
Exam signal: slot utilization near 100% plus slow jobs -> capacity constrained. Use reservations/slot management, not BI Engine.

Section 5.4

Monitoring and troubleshooting processes

Know which signal belongs where: Monitoring, Logging, BigQuery job history, Dataflow lag/watermarks, Pub/Sub backlog, quotas, and billing.

WeightSection 5: ~18%
Cloud Monitoring Cloud Logging BigQuery admin panel Billing Quotas

Official guide bullets

  • Observability of data processes with Cloud Monitoring, Cloud Logging, and the BigQuery admin panel.
  • Monitor planned usage.
  • Troubleshoot error messages, billing issues, and quotas.
  • Manage jobs, queries, compute capacity, and reservations.

What to know

  • Cloud Monitoring: metrics, alerts, dashboards.
  • Cloud Logging: error details and audit trails.
  • BigQuery admin panel / job history: jobs, active queries, slots, reservations.
  • Dataflow falling behind: check system lag, watermark progress, worker autoscaling, and fused stages.
  • Pub/Sub consumer lag: backlog size and oldest unacked message age.
  • BigQuery capacity issue: slot utilization near 100%.
Dataflow debugging trap: to isolate bottlenecks between fused stages, insert Reshuffle. Heavy logging inside ParDo adds noise and overhead.

Section 5.5

Maintaining awareness of failures and mitigating impact

Design for replay, restart, regional resilience, backup/restore, corruption handling, and service-specific replication/failover.

WeightSection 5: ~18%
Fault tolerance Restarts Multi-region / zones Corruption and missing data Replication and failover

Official guide bullets

  • Design systems for fault tolerance and manage restarts.
  • Run jobs in multiple regions or zones where needed.
  • Prepare for data corruption and missing data.
  • Plan data replication and failover such as Cloud SQL and Redis clusters.

What to know

  • Make pipelines idempotent so retries and replays do not duplicate outputs.
  • For corruption/missing data: validate, stop bad publishes, replay from source where possible, and restore from backups where necessary.
  • Cloud Storage strict 15-minute replication RPO -> dual-region bucket with Turbo Replication.
  • Cloud SQL uses HA/failover replicas and read replicas depending on requirement.
  • Memorystore/Redis cluster patterns provide replication/failover for cache/session workloads.
  • BigQuery multi-region datasets replicate within the multi-region; capacity is handled separately with reservations/editions.
RPO trap: "strict 15-minute Cloud Storage replication RPO" is not generic multi-region. It is dual-region plus Turbo Replication.
