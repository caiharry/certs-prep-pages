4.1 Pressing the Button — Scalable & HA Deployment

We've made it to the final section of our study guide! Now it's time to actually construct our databases. But wait... you aren't going to build these production instances by clicking random buttons in the GCP Console, are you? No! That is strictly for amateurs! 🖱️

🏗️ Infrastructure as Code (IaC) — The Terraform Way

Real-world Cloud Architects write, review, and version-control their database infrastructure using Terraform. This ensures total environment consistency (your staging database will match production perfectly) and prevents fat-finger manual mistakes. Here is how you declare a Cloud SQL instance with High Availability enabled:

resource "google_sql_database_instance" "primary" {
  name             = "example-prod-db"
  database_version = "POSTGRES_15"
  region           = "europe-west1"

  settings {
    tier = "db-custom-2-7680" # 2 vCPU, 7.5 GB RAM

    # The magic high availability standby flag!
    availability_type = "REGIONAL" 

    backup_configuration {
      enabled    = true
      start_time = "03:00"
      point_in_time_recovery_enabled = true # Near-zero RPO!
    }
  }
}
Watch out for manually created databases! If some rogue developer deployed a database manually and you need to bring it under Terraform control, don't just write the resource block and run terraform apply (that will try to recreate or overwrite the database, causing a total production outage!).

Always use terraform import to safely link the existing live resource to your Terraform state file first! After importing, run terraform plan like a hawk to catch any configuration drift. Reviewing drift is the difference between a clean deployment and an emergency coffee run at 2 AM! ☕

📚 Scaling Reads Without Melting the Primary

Read scaling is the classic "same data, too many eyeballs" problem. Make sure you pick the replica type that matches the engine:

  • Cloud SQL: Add Read Replicas to offload read-heavy traffic, run reports, provide regional locality, or act as DR building blocks. Note: Enterprise Plus read pools can raise availability when configured with enough nodes.
  • AlloyDB: Use Read Pools to scale read and analytical traffic away from the primary, and combine with the columnar engine for HTAP reporting.
  • Spanner: Choose your instance configuration carefully. Use read-only replicas or multi-region configurations to scale reads globally, but ensure your schema is designed well to avoid hot spots!

🐳 Game-Day Failover & DR Testing

A Disaster Recovery (DR) plan that has never been tested is like an uninflated lifejacket — you can only pray it actually works when the ship starts sinking! 🚢

Examiners love to test your operational confidence here. To prove your setup works, schedule regular "Game-Day" failover drills:

  1. Trigger a Manual Failover: In the Cloud SQL or AlloyDB console (or via the gcloud CLI), trigger a manual failover event to verify that the standby instance takes over successfully, and that your application connection pools survive the blip!
  2. Drill Replica Promotion: Disconnect a Cross-Region Read Replica and promote it to be a standalone primary database. Verify your application config updates correctly to handle regional outages.
  3. Test Spanner Differently: Multi-region Spanner is platform-managed, so focus your drills on application retry logic, leader-region latency, monitoring, backup/restore runbooks, and simulated client failures. Don't go looking for a generic regional failover button in Spanner — it doesn't exist!
🧠 Memory Tip — Multi-Region Spanner:

For the highest-tier databases that require multi-regional replication with 99.999% availability and strong consistency, head straight for Cloud Spanner. Spanner uses TrueTime for globally consistent transactions, but remember: you still need excellent schema design because bad keys can still cause hotspots and make your database slow and grumpy!

📈 HA Monitoring: Trust, But Verify

After deploying your HA database, monitor these four metrics to prove HA is actually doing its job:

  • Failover Events: Did a failover occur, and how long did the application take to reconnect?
  • Replica Health: Monitor replication lag, replica availability, promotion readiness, and read pool node health.
  • Backup Health: Track backup success, PITR coverage, restore-test results, and retention compliance.
  • SLOs & Saturation: Track availability, latency, error budgets, connection errors, and saturation signals.

🧠 Quick-Fire Warm-Up Quiz!

Test your recall before checking out the final practice exam. You've got this!

Q1: Which Terraform block setting establishes a high-availability standby for Cloud SQL?

Setting availability_type = "REGIONAL". (Setting it to ZONAL deploys a cheaper single-instance with zero standby failover).

Q2: How should you bring an existing, manually-created GCP database under Terraform management safely?

Use the terraform import command! This links the existing live resource to your state file, avoiding accidental destruction or duplicate provisioning.

Q3: What is the best way to verify that your disaster recovery replica promotion procedure behaves as expected?

Schedule regular Game-Day drills! For Cloud SQL/AlloyDB, manually trigger failovers and replica promotions. For Spanner, test app resilience, monitoring, restore workflows, and dependency failures.

📝 Section 4 Practice Exam Questions

Finish your database study preparation strong with these scenario questions!

Question 1: An infrastructure engineer wants to automate the provisioning of all company databases on GCP. They write a Terraform module to spin up five new Cloud SQL PostgreSQL instances. However, they discover that one database instance was already created manually in the console last month by a separate team. They need to manage this existing instance alongside the five new instances under Terraform, without causing database downtime or recreating the manual database. What should they do?

  • A) Write the Terraform configuration for the existing instance, run terraform apply with the force-replace flag, and let Terraform overwrite the live database state.
  • B) Define the existing instance in code, run terraform import to bind the live resource, and execute terraform plan to confirm alignments.
  • C) Delete the manually created database instance in the console during off-hours, and let the scheduled Terraform pipeline recreate the database from scratch.
  • D) Create a read replica of the manually created database in Terraform, promote the replica to primary, and delete the original database in the console.

Correct Answer: B

Why Option B is correct: When bringing legacy or manually created infrastructure under Terraform control, running terraform apply without importing first causes errors (since the resource name already exists). The terraform import command binds the live GCP resource state directly to your Terraform state file. Once imported, you can safely deploy any setting adjustments declaratively with zero downtime.

  • A: Applying before import can fail or attempt unwanted changes because Terraform does not know the live resource yet.
  • C: Deleting a production database to satisfy Terraform is the sort of plan that makes auditors reach for biscuits.
  • D: Read replicas solve read scaling/DR patterns, not IaC state adoption.

Question 2: You have deployed a multi-region Cloud Spanner instance to support a global mobile game backend. You want a quarterly disaster recovery drill that validates the application and operational runbook. What should you do?

  • A) Deploy a script to only restore yesterday's backup to a temporary zonal instance, verify basic query connectivity, and call the DR drill complete.
  • B) Run a scheduled Game-Day that tests client retry/reconnection logic, leader-region latency assumptions, monitoring/alerts, and backup/restore runbooks without assuming a generic manual regional-failover command.
  • C) Contact Google Cloud Support to physically disconnect the network fiber to one of your active regions for 30 minutes to trigger automatic database failover.
  • D) Migrate the gaming backend database to Cloud SQL, enable Regional High Availability (HA), and let the application automatically switch to the passive replica.

Correct Answer: B

Why Option B is correct: Multi-region Spanner is platform-managed, so the useful customer drill is validating the application and operations around it: retries, latency expectations, monitoring, backups/restores, and dependency failure handling. For dual-region Spanner there are quorum-management commands, but that is not the same as a generic multi-region failover button.

  • A: Restore testing is valuable, but alone it does not test live app resilience or monitoring.
  • C: Please do not ask anyone to unplug the internet. Also, this is not a normal customer DR test.
  • D: Cloud SQL Regional HA is not a replacement for a multi-region Spanner architecture.

Question 3: Your Cloud SQL primary is healthy, but global reporting traffic is causing high read latency and CPU pressure. Writes are not the bottleneck. What should you deploy first?

  • A) Add read replicas or read pools and route reporting/read traffic to them.
  • B) Disable all automated daily database backups and PITR binary logging to free up underlying VM CPU and I/O resources on the primary instance.
  • C) Migrate the database to a single-zone instance to reduce cross-zone networking overhead and improve local reporting cache hits.
  • D) Configure the application database client connection pooler to increase retry loops indefinitely and keep all analytical reads on the primary.

Correct Answer: A

Why Option A is correct: Read replicas/read pools are designed to scale read-heavy traffic away from the primary. This preserves write capacity and can improve locality for readers.

  • B: Backups are a recovery control, not a performance tuning knob to casually throw out the window.
  • C: Single-zone reduces availability and does not solve read scaling well.
  • D: Retries help transient failures, but they do not add database read capacity.

Question 4: A junior database administrator manually enables an external public IP on a production Cloud SQL instance in the GCP Console to perform some quick troubleshooting. This action violates company security policy, as the database is declared in Terraform to only have a Private IP address. What will happen during the next scheduled Terraform deployment pipeline, and how is this drift resolved safely?

  • A) Terraform will ignore the manual change entirely because manual GCP console updates have a higher priority than state files.
  • B) Terraform's drift detection will identify the unexpected public IP configuration during the terraform plan phase, and the subsequent terraform apply will automatically disable the public IP to reconcile the database instance back to its declared private-only state.
  • C) Terraform will flag the configuration drift as a critical error, automatically destroy the active database instance, and recreate a new one from scratch.
  • D) The Terraform pipeline will freeze, lock the state file, and prevent any future deployments until the public IP is manually disabled in the GCP Console first.

Correct Answer: B

Why Option B is correct: This highlights the magic of declarative Infrastructure as Code (IaC)!

  • Drift Detection & Reconciliation: When you run a Terraform pipeline, Terraform compares your local code files with the actual live GCP resources. It detects that a public IP was manually enabled.
  • Safe Correction: During terraform apply, Terraform will safely modify the instance settings to disable the public IP — reconciling the live environment back to your secure, declared code configurations without destroying or recreating the database! 🛠️🌟

Why the others are wrong:

  • A: Terraform's core purpose is to enforce the declared state. It will not ignore changes made manually outside of Terraform.
  • C: Modifying a network parameter like disabling a public IP is an in-place modification on Cloud SQL and does not trigger a destructive recreation of the entire instance.
  • D: Terraform does not freeze or lock you out of deployments because of configuration drift; it simply plans to undo the drift.
