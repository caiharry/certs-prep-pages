Exam Revision Style

GCP Cloud Developer

High-yield revision deck ordered by the official exam guide. Platforms, auth, storage, and deployment traps.

💻 The big picture: The PCD exam is practical. Default to the most managed option that meets the constraint, prefer event-driven integration over polling, and never use long-lived service-account keys when Workload Identity Federation or ADC will do. Know the storage decision tree and Cloud Run vs GKE cold.
Section 1 ~32%

Designing Scalable, Secure, Reliable Apps

Platform choice, secure design, and picking the right data store — a third of the exam.

Platform choice

  • Cloud Run: stateless containers, scale-to-zero, concurrency per instance, fastest path.
  • GKE: stateful/complex, sidecars, mesh, fine-grained control.
  • Compute Engine: lift-and-shift, licensing, GPUs.

API & eventing

  • REST for broad clients; gRPC for low-latency internal service-to-service.
  • Pub/Sub: decoupled async messaging.
  • Eventarc: route events (incl. Cloud Audit Logs) to Cloud Run.
  • Cloud Tasks: per-task rate/retry control.

Secrets & auth

  • Secret Manager: versioned secrets, IAM-controlled, rotation.
  • ADC then service accounts for service identity.
  • Workload Identity Federation: keyless access for external/CI workloads.
  • Binary Authorization: only signed images deploy.

Section 2 ~23%

Building & Testing Applications

Dev environment, container builds, and automated testing.

Dev environment

  • Local emulators (Firestore, Pub/Sub, Spanner, Bigtable) via gcloud for offline unit tests.
  • Cloud Workstations: managed, secure cloud IDE.
  • Cloud Code + Gemini Cloud Assist: in-IDE GCP tooling.

Build pipeline

  • Cloud Build: CI; build steps in containers.
  • Artifact Registry: store images & language packages.
  • Generate build provenance → verify with Binary Authorization.
⚠️ Exam Trap: For local unit tests against Firestore/Pub/Sub, use the emulator — don't hit real billable services in CI.
Section 3 ~24%

Configuring Apps for Deployment

Cloud Run and GKE deployment, triggers, and health.

Cloud Run

  • Deploy from source (buildpacks) or a container.
  • Each revision is immutable; split traffic for canary.
  • Triggers: Eventarc / Pub/Sub push.

GKE health & scaling

  • Liveness probe restarts a hung container; readiness probe gates traffic.
  • HPA scales pods on CPU/custom metrics.
  • Set requests/limits so HPA and the scheduler behave.
Probes Readiness = Receive traffic; Liveness = restart if Lifeless. Mixing them up is a classic exam slip.
⚠️ Exam Trap: A failing readiness probe removes the pod from the Service endpoints (no restart); only a failing liveness probe restarts the container.
Section 4 ~21%

Integrating with Google Cloud Services

Datastore connections, efficient API consumption, and observability.

Efficient API use

  • Exponential backoff (+ jitter) on retryable errors (429/5xx).
  • Paginate large lists; request only needed fields.
  • Batch requests; cache idempotent reads.

Observability

  • Instrument metrics, logs, traces (OpenTelemetry).
  • Error Reporting groups exceptions.
  • Propagate trace IDs to correlate spans across services.
⚠️ Exam Trap: On HTTP 429/503, the correct client behaviour is retry with exponential backoff, not immediate tight-loop retries (which worsen the overload).
