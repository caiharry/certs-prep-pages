Section 1~32%

Designing Scalable, Secure, Reliable Apps

The heavyweight third of the exam: choose the right platform, design secure applications, and pick the correct data store and schema.

Platform selection

Match the runtime to the workload:

  • Cloud Run — stateless containers, scale-to-zero, per-instance concurrency, fastest path.
  • GKE — complex or stateful microservices, sidecars, service mesh, fine control.
  • Compute Engine — lift-and-shift, licensing, GPUs.

Add caching with Memorystore, decouple with Pub/Sub, and route events with Eventarc. Orchestrate multi-step flows with Workflows, schedule with Cloud Scheduler, and queue per-task work with Cloud Tasks.

⚠️ Exam Trap: When the scenario stresses minimal operational overhead or fastest delivery, the answer is almost always the most managed option — usually Cloud Run.

The authentication ladder

Authenticate to Google APIs the recommended way, in order of preference:

  • Application Default Credentials (ADC) with an attached service account — no keys to manage.
  • Workload Identity Federation for external/CI workloads — keyless.
  • Long-lived service-account keys only as a last resort (rotate, store in Secret Manager).

Store secrets in Secret Manager, protect images with Binary Authorization, and run every service with least privilege.

⚠️ Exam Trap: Distributing service-account keys to clients or embedding API keys for IAM-protected services is a guaranteed wrong answer.

Storing & accessing data

Pick the data store by consistency and shape — the exam tests the boundaries:

  • Spanner — global, horizontally scalable, strong consistency (relational at planet scale).
  • AlloyDB — Postgres-compatible, high-performance transactional + analytical (HTAP).
  • Cloud SQL — regional managed MySQL/PostgreSQL/SQL Server for classic apps.
  • Bigtable — massive-throughput, low-latency wide-column (time series, IoT).
  • Firestore — serverless document store with mobile/web sync and offline support.

Grant time-limited object access with signed URLs rather than making a bucket public.

⚠️ Exam Trap: Let a client upload/download one object for a limited time with no IAM change = a signed URL. Making the bucket public is the wrong answer.

Async & messaging patterns

Decouple components with the right primitive — they are not interchangeable:

  • Pub/Sub — fan-out, many-subscriber, high-throughput event streaming.
  • Cloud Tasks — per-task queues with rate limiting and retries; dispatch work to a specific worker/endpoint.
  • Workflows — orchestrate a defined sequence of service calls with branching and error handling.
  • Eventarc — route events from Google sources (e.g. a GCS finalize) to a service via CloudEvents.
⚠️ Exam Trap: "Throttle and retry individual outbound jobs to one endpoint" = Cloud Tasks. "Broadcast an event to many independent subscribers" = Pub/Sub. Don't swap them.
PTWE Pub/Sub broadcasts, Tasks throttle, Workflows orchestrate, Eventarc routes.

Statelessness & resilience

Design services stateless (push session/state to Memorystore, Firestore, or a token) so any instance can serve any request and autoscaling works. Make write operations idempotent so safe retries don't double-charge or duplicate. Degrade gracefully (cache, fallbacks, circuit breakers) when a dependency is slow or down, and cap blast radius with timeouts.

⚠️ Exam Trap: Storing session state in local instance memory breaks autoscaling and rolling deploys — externalise it (e.g. Memorystore).

🧠 Warm-Up Quiz

Q1. A stateless API container must scale to zero and need no node management. Which platform?

Reveal answer

Cloud Run.

Managed, scale-to-zero, request autoscaling.

Q2. Your Cloud Run service must call Cloud Storage with the least credential burden. What do you use?

Reveal answer

An attached service account via Application Default Credentials.

No keys to manage or rotate; library picks up ADC automatically.

Q3. You need globally consistent, horizontally scalable relational data across continents. Which database?

Reveal answer

Cloud Spanner.

It's the only relational store offering global strong consistency with horizontal scale.

Q4. One event must fan out to several independent subscribers at high volume. Which service?

Reveal answer

Pub/Sub.

Built for high-throughput, many-subscriber fan-out; Cloud Tasks targets a single worker.

📝 Exam-Style Practice

Q1. A team must ship a stateless REST API fast, scale it to zero between bursts, and avoid managing nodes or clusters. Which platform best fits?

  • A) GKE Standard with a Horizontal Pod Autoscaler and cluster autoscaler.
  • B) Cloud Run, deploying the container with request-based autoscaling.
  • C) A managed instance group of Compute Engine VMs behind a load balancer.
  • D) App Engine flexible environment with custom runtimes.
Reveal answer & explanation

Correct: B. Cloud Run runs stateless containers, scales to zero, and needs no node management — the fastest, most managed fit.

A: GKE Standard means managing nodes and won't scale to zero cheaply. C: MIGs need VM and image management. D: App Engine flex keeps at least one instance running and is heavier than Cloud Run.

Q2. A CI job on GitHub Actions must call Google Cloud APIs. Security policy forbids long-lived service-account keys. What is the recommended authentication method?

  • A) Create a service-account JSON key and store it as a CI secret.
  • B) Use Workload Identity Federation so the external CI exchanges its token for short-lived credentials.
  • C) Embed an API key in the pipeline configuration.
  • D) Use a user's personal OAuth refresh token stored in CI.
Reveal answer & explanation

Correct: B. Workload Identity Federation lets the external workload exchange its own identity token for short-lived Google credentials — keyless, exactly what the policy requires.

A: A JSON key is the long-lived credential the policy bans. C: API keys don't authenticate IAM-protected APIs and leak easily. D: Personal tokens tie automation to an individual and are discouraged.

Q3. A payment endpoint occasionally retries on transient failures and you must ensure a retry never charges a customer twice. What design property must the endpoint have?

  • A) It must store each session in the serving instance's local memory.
  • B) It must be idempotent, e.g. de-duplicating on a client-supplied idempotency key.
  • C) It must disable all retries on the client side.
  • D) It must run only on a single non-autoscaling instance.
Reveal answer & explanation

Correct: B. Idempotency (e.g. keyed de-duplication) makes a repeated request safe, so retries can't double-charge.

A: Local session state breaks autoscaling and doesn't address double charges. C: Disabling retries sacrifices reliability and still doesn't make the operation safe. D: A single instance hurts availability and doesn't guarantee exactly-once.
