Section 2~25%

Building & Implementing CI/CD Pipelines

Joint-largest section: design and run CI/CD for apps, infra, and ML, with a secured software supply chain.

CI/CD building blocks

Cloud Build for CI (build, test, containerise); Cloud Deploy for CD to GKE/Cloud Run using Skaffold and Kustomize; Artifact Registry to store images and packages. Add approval gates and pipeline triggers; integrate third-party tools (Git, Jenkins, Argo CD) where needed.

Deployment strategies

Choose by risk tolerance: canary (small % first, metric-gated ramp), blue/green (instant cutover with fast rollback), rolling, and traffic splitting / feature flags. Define success metrics from app/ML telemetry to drive promotion.

⚠️ Exam Trap: Need gradual exposure with metric-gated promotion and quick rollback? That's canary, not blue/green (which flips 100% at once).

Securing the pipeline

Run Artifact Analysis vulnerability scanning, enforce Binary Authorization and the SLSA framework for supply-chain integrity, and scope IAM by environment. Inject secrets at build vs runtime via Secret Manager, Certificate Manager, and KMS.

Supply chain Binary Authorization + SLSA = only verified, attested artifacts reach production.

Trigger patterns & CI/CD for infra and ML

Wire Cloud Build triggers to git events (push, tag, PR) so builds are automatic and traceable. CD isn't just for apps: apply the same pipeline discipline to infrastructure (Terraform plan/apply gated by review) and ML (train → evaluate → register, with an eval gate). Cache dependencies and run steps in parallel to keep pipelines fast.

⚠️ Exam Trap: "Apply Terraform only after review and approval" = a CI/CD pipeline running plan on PR and gated apply on merge — not engineers running apply locally.

🧠 Warm-Up Quiz

Q1. Release to 5% of users, watch metrics, then ramp with rollback. Which strategy?

Reveal answer

Canary deployment.

Gradual, metric-gated, fast rollback.

Q2. Continuous delivery to GKE/Cloud Run with Skaffold and approvals. Which service?

Reveal answer

Cloud Deploy.

Managed CD with release promotion and approval gates.

Q3. Only artifacts your pipeline built and signed may run in prod. Which two controls?

Reveal answer

Binary Authorization + build attestations (SLSA).

Binary Authorization admits only images with valid attestations from your build process.

Q4. You want an instant 100% cutover with the old version kept ready for fast rollback. Which strategy?

Reveal answer

Blue/green deployment.

Two full environments; flip traffic at once and flip back instantly if needed.

📝 Exam-Style Practice

Q1. A team needs managed continuous delivery to GKE and Cloud Run with promotion through dev→staging→prod, approval gates, and one-click rollback, using Skaffold. Which service anchors this?

  • A) Cloud Build alone, with custom deploy scripts per environment.
  • B) Cloud Deploy delivery pipelines with targets and approvals.
  • C) A Cloud Scheduler job running gcloud deploy commands.
  • D) Argo CD self-managed on a dedicated cluster.
Reveal answer & explanation

Correct: B. Cloud Deploy is the managed CD service: Skaffold-based pipelines, multi-target promotion, approvals, and rollback out of the box.

A: Cloud Build is CI; promotion/approvals/rollback are Cloud Deploy's role. C: Scheduled gcloud lacks promotion and approvals. D: Argo CD works but is self-managed overhead versus the native managed option.

Q2. Compliance requires that only container images produced and attested by the official pipeline can run in production clusters. Which combination enforces this at admission?

  • A) Artifact Registry IAM limiting who can push images.
  • B) Binary Authorization requiring build attestations (SLSA) before a pod admits.
  • C) A naming convention for production image tags.
  • D) Vulnerability scanning that emails the security team.
Reveal answer & explanation

Correct: B. Binary Authorization checks cryptographic attestations at admission, so only images the pipeline built and signed can run.

A: Push permissions don't control what actually runs. C: Tags are spoofable. D: Scanning detects issues but doesn't block deployment.

Q3. A new release must reach a small fraction of users first, with promotion gated on live error-rate and latency metrics and fast rollback if they regress. Which deployment strategy fits?

  • A) Blue/green with an immediate 100% cutover.
  • B) Canary with progressive, metric-gated traffic shifting.
  • C) Recreate, replacing all instances at once.
  • D) A big-bang deploy during a maintenance window.
Reveal answer & explanation

Correct: B. Canary exposes a small slice, gates promotion on metrics, and rolls back quickly — exactly the requirement.

A: Blue/green flips everyone at once (no gradual exposure). C/D: Recreate and big-bang expose all users simultaneously.
