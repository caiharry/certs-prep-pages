Section 4~25%

Observability & Troubleshooting

Joint-largest section: instrument telemetry, analyse logs/metrics/traces, and find root cause.

Collecting telemetry

Collect logs and metrics with the Ops Agent and OpenTelemetry; scrape Prometheus metrics via Google Cloud Managed Service for Prometheus. Optimise log cost with filtering, sampling, and exclusions. Create synthetic monitors to probe endpoints proactively and log-based metrics for custom signals.

Logs, metrics, dashboards & alerts

Query with the Logs Explorer and Logging query language; export/retain via sinks to BigQuery/Pub/Sub/GCS. Analyse metrics in Metrics Explorer (PromQL supported), build dashboards, and configure alerting policies on SLIs/SLOs. Capture distributed traces (OpenTelemetry), analyse span waterfalls, and correlate trace IDs with structured logs.

⚠️ Exam Trap: To detect VM shutdowns quickly, create a log-based alert on the audit-log method (e.g., compute.instances.stop) — not an hourly scheduled BigQuery query.

Root-cause workflow

Debug systematically, top-down from the user symptom: confirm the SLO impact in Cloud Monitoring, narrow the time/scope, then drill into Cloud Trace to find the slow span and into Cloud Logging (correlated by trace ID) for the error. Use Cloud Profiler for CPU/memory hotspots and Error Reporting to group recurring exceptions. The three pillars — metrics, logs, traces — answer "how bad", "what happened", and "where".

⚠️ Exam Trap: "Which microservice adds the latency in a request that crosses ten services?" = distributed tracing (Cloud Trace), not reading each service's logs by hand.

🧠 Warm-Up Quiz

Q1. Get notified quickly when a VM stops. Best approach?

Reveal answer

A log-based alert on the v1.compute.instances.stop method in Cloud Logging.

Real-time, no polling; scheduled BigQuery queries lag and cost more.

Q2. Ingest existing Prometheus metrics at scale on GCP. Which service?

Reveal answer

Google Cloud Managed Service for Prometheus.

Managed, scalable Prometheus-compatible metrics.

Q3. A request crossing 10 microservices is slow. Which signal finds the offending hop?

Reveal answer

Distributed tracing (Cloud Trace) span waterfall.

Traces show per-span latency across services, pinpointing the slow hop — logs alone can't.

Q4. Logging bills are too high from verbose debug logs. Two cheap levers?

Reveal answer

Log exclusions/filters and sampling.

Drop or sample low-value logs before ingestion to cut cost without losing key signals.

📝 Exam-Style Practice

Q1. The on-call team must be alerted within seconds whenever a production VM is stopped, with no polling cost. What is the best approach?

  • A) An hourly scheduled BigQuery query over exported audit logs.
  • B) A Cloud Logging log-based alert on the compute.instances.stop audit-log method.
  • C) A cron job that lists instances and compares state every minute.
  • D) A weekly review of the activity log.
Reveal answer & explanation

Correct: B. A log-based alert fires in near-real-time off the audit log with no polling, exactly matching the requirement.

A/C: Hourly queries and per-minute polling lag and cost more. D: Weekly review is far too slow.

Q2. A user request that traverses ten microservices is intermittently slow, and you must identify which service contributes the latency. Which observability signal is the right tool?

  • A) Aggregate CPU metrics for the whole cluster.
  • B) Distributed tracing (Cloud Trace) with correlated trace IDs.
  • C) Reading each service's logs in sequence by hand.
  • D) Error Reporting exception counts.
Reveal answer & explanation

Correct: B. A trace's span waterfall shows per-service latency across the whole request, pinpointing the slow hop.

A: Cluster CPU doesn't isolate one service's request latency. C: Manual log reading across ten services is slow and error-prone. D: Error Reporting counts exceptions, not latency.

Q3. An existing Kubernetes estate already exposes Prometheus metrics, and the team wants them collected at scale on Google Cloud without running and scaling their own Prometheus servers. What should they adopt?

  • A) Google Cloud Managed Service for Prometheus.
  • B) A self-hosted Prometheus per cluster with manual federation.
  • C) Rewrite all metrics as Cloud Monitoring custom metrics by hand.
  • D) Ship metrics as logs and build log-based metrics from them.
Reveal answer & explanation

Correct: A. Managed Service for Prometheus ingests existing Prometheus metrics at scale with no servers to run, keeping PromQL compatibility.

B: Self-hosting is the operational burden they want to avoid. C: Hand-porting metrics is wasteful and error-prone. D: Logs-as-metrics is a workaround, not native Prometheus ingestion.
