Section 2~14%

Managing Data & Models Across Teams

Explore and preprocess org-wide data, prototype in notebooks, and track experiments.

Preprocessing at scale

Match the transform tool to the data and the team:

  • BigQuery SQL — the data is already in BigQuery and transforms express cleanly as SQL.
  • Dataflow (Apache Beam) — large-scale, streaming or batch, with the same code for both; ideal for heavy custom transforms.
  • TFX / tf.transform — preprocessing that must run identically at training and serving (the transform graph travels with the model).
  • Dataproc (Spark) — you already live in the Spark/Hadoop ecosystem.
⚠️ Exam Trap: "Transformation logic must be the same offline and online" hints at TFX/tf.transform or a shared Feature Store — not an ad-hoc pandas script that only runs at training time.

Feature management & training-serving skew

Centralise reusable features in Vertex AI Feature Store. It serves the same feature values offline for training and online (low-latency) for serving — the canonical defence against training-serving skew (when the features a model sees in production differ from those it trained on). It also gives point-in-time correct lookups so you don't leak future data into training.

SKEW Same features, Keyed by entity, Everywhere (offline + online) = no skew. One Feature Store, two serving modes.

Notebook backends

Pick the prototyping environment by the data and collaboration need:

  • Vertex AI Workbench — managed JupyterLab with GCP integration; the default single-user choice.
  • Colab Enterprise — collaborative, shareable notebooks with enterprise governance.
  • Notebooks on Dataproc — Spark kernels for terabyte-scale data already in a Hadoop/Spark world.
⚠️ Exam Trap: Workbench + pandas chokes on TB-scale data. "Prototype on terabytes in an existing Spark ecosystem" = Dataproc notebooks.

Experiments, metadata & lineage

Track and compare runs with Vertex AI Experiments (parameters, metrics) and visualise with TensorBoard. Vertex ML Metadata records the lineage of datasets, executions, and models so you can answer "which data and code produced this model?" — essential for reproducibility and audits.

EXP Experiments compare runs; Metadata remembers lineage; the Model Registry versions the winners.

Governance & sensitive data

Manage labelled datasets in Vertex AI managed datasets. For PII/PHI, run Cloud DLP / Sensitive Data Protection to inspect and de-identify before features reach the store, and control access with IAM and (where needed) VPC Service Controls. Treat the Feature Store and training data as governed assets, not scratch files.

⚠️ Exam Trap: "Remove or mask PII before training" points to Sensitive Data Protection (DLP), not a manual SQL REPLACE or dropping columns by hand.

🧠 Warm-Up Quiz

Q1. Prototype on terabytes already in a Spark/Hadoop ecosystem with familiar kernels. Which notebook backend?

Reveal answer

Notebooks on Dataproc with Spark kernels.

Managed Spark at scale; Workbench+pandas can't handle TB-scale Spark data.

Q2. How do you ensure identical features at training and serving time?

Reveal answer

Use Vertex AI Feature Store for both.

Centralised offline+online features prevent skew.

Q3. An auditor asks which dataset version and training run produced a deployed model. Where do you look?

Reveal answer

Vertex ML Metadata (lineage).

It records datasets, executions, and artifacts and links them, giving full reproducible lineage.

Q4. Preprocessing must run byte-for-byte identically during training and online serving. What approach?

Reveal answer

TFX / tf.transform (transform graph shipped with the model), or shared Feature Store values.

The transform is baked into the serving graph, so train and serve apply the exact same logic.

📝 Exam-Style Practice

Q1. A model performs well in offline evaluation but its accuracy drops sharply in production. Investigation shows the serving pipeline computes a key feature slightly differently from the training pipeline. What is the most robust fix?

  • A) Retrain the model more frequently so it adapts to the production feature values.
  • B) Serve training and serving features from a shared Vertex AI Feature Store so both read identical values.
  • C) Add the production feature distribution as a new input column at serving time only.
  • D) Increase the model's capacity so it is robust to the feature discrepancy.
Reveal answer & explanation

Correct: B. This is classic training-serving skew. A shared Feature Store guarantees both paths read the same computed values, eliminating the discrepancy at its source.

A: Retraining on skewed features doesn't fix the mismatch. C: Adding a serving-only column widens the skew. D: A bigger model can't compensate for systematically wrong inputs.

Q2. A data scientist must prototype interactively over 5 TB of logs that already sit in a Dataproc Spark cluster, using familiar Spark APIs. Which environment fits with the least data movement?

  • A) A Vertex AI Workbench instance loading the logs into a pandas DataFrame.
  • B) A BigQuery notebook after exporting all logs to a BigQuery table.
  • C) A Dataproc-backed notebook using Spark kernels against the existing cluster.
  • D) Colab Enterprise downloading the logs to local session storage.
Reveal answer & explanation

Correct: C. A Dataproc notebook runs Spark on the existing cluster, so 5 TB stays put and the scientist keeps Spark APIs.

A: pandas can't hold 5 TB in memory. B: Exporting everything to BigQuery is unnecessary data movement. D: A Colab session can't download or hold terabytes locally.

Q3. Before training a model on customer records, the security team requires that names, emails, and card numbers are detected and masked across a large, messy dataset, repeatably. What should you use?

  • A) Cloud DLP / Sensitive Data Protection to inspect and de-identify the data in a pipeline.
  • B) A one-off SQL query that drops any column whose name contains "email" or "card".
  • C) Vertex AI Feature Store access controls to hide sensitive features from trainers.
  • D) IAM conditions restricting who can read the training bucket.
Reveal answer & explanation

Correct: A. Sensitive Data Protection detects many PII types (even in free text) and de-identifies them repeatably as part of a pipeline — exactly the requirement.

B: Name-based column dropping misses PII buried in text and is brittle. C/D: Access controls limit who reads the data but don't mask the PII that still ends up in the model.
