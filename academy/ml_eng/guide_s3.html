Section 3~18%

Scaling Prototypes into ML Models

Build models, train at scale, tune hyperparameters, and choose the right accelerators.

Training at scale

Vertex AI custom training runs any framework packaged in a container. Use prebuilt containers for common frameworks (TF, PyTorch, scikit-learn, XGBoost) and a custom container when you need specific dependencies. Point training at a Vertex managed dataset or Cloud Storage, and write checkpoints to GCS so interrupted jobs can resume.

PACK Prebuilt container if it fits; otherwise build your own and Push to Artifact Registry.

Distributed training

When one machine isn't enough:

  • Data parallelism — replicate the model across workers, split the batch (e.g. MultiWorkerMirroredStrategy). The common case.
  • Model parallelism — split a model too big for one device across devices. Needed for very large models.
  • Reduction Server — a Vertex feature that speeds the all-reduce gradient step in multi-GPU data-parallel jobs, cutting training time.
⚠️ Exam Trap: "Model is too large to fit on a single GPU" = model parallelism. "Speed up multi-GPU all-reduce" = Reduction Server. Don't swap them.

Choosing accelerators

  • CPU — small or classical models, or cheap experimentation.
  • GPU — most deep learning; flexible, supports custom ops and PyTorch.
  • TPU — very large dense matrix-multiply graphs in TensorFlow/JAX.
  • Reduction Server — accelerates multi-GPU gradient all-reduce.
⚠️ Exam Trap: TPUs win for large TF/JAX matmul workloads; for smaller models, custom ops, or PyTorch, reach for GPUs first.
TPU Tensor-heavy, Predictable, hUge matmuls.

Hyperparameter tuning

The Vertex AI hyperparameter tuning service (powered by Vizier) runs many trials to optimise a metric you report. Choose a search algorithm (Bayesian by default, or grid/random), set parallel trials to trade speed for cost, and enable early stopping to kill unpromising trials. Define the metric and search space; let the service do the sweep.

⚠️ Exam Trap: More parallel trials finish faster but explore less efficiently than sequential Bayesian search. The exam may pit "fastest wall-clock" against "best metric for the budget".

Fine-tune, don't reinvent

When a strong base model exists, fine-tune it (transfer learning, or supervised tuning of a foundation model in Model Garden) instead of training from scratch. You get a good model with far less data, compute, and time. From-scratch training is reserved for genuinely novel architectures or domains with no suitable base.

REUSE A pretrained base + a little of your data beats burning a GPU farm to relearn the basics.

🧠 Warm-Up Quiz

Q1. Very large TensorFlow model, massive matrix multiplications, best price-performance accelerator?

Reveal answer

Cloud TPUs.

Purpose-built for large dense matmul graphs.

Q2. You have a strong base LLM and a small domain dataset. Cheapest path to a specialised model?

Reveal answer

Fine-tune the foundation model.

Avoids costly from-scratch training.

Q3. Your model is too large to fit on a single GPU. Which distribution strategy do you need?

Reveal answer

Model parallelism.

It splits the model itself across devices; data parallelism only splits the batch and still needs the whole model per device.

Q4. You need a custom training job in PyTorch with a niche C++ dependency. Prebuilt or custom container?

Reveal answer

Custom container.

Prebuilt containers cover common stacks; a niche native dependency means building and pushing your own image.

📝 Exam-Style Practice

Q1. A team trains a large TensorFlow recommendation model dominated by dense matrix multiplications and wants the best price-performance at scale. Which accelerator should they choose?

  • A) High-memory CPU instances with many vCPUs for parallel throughput.
  • B) A single large GPU with model simplification to fit the graph.
  • C) Cloud TPUs, which are purpose-built for large dense matmul graphs.
  • D) A fleet of small GPUs without a Reduction Server.
Reveal answer & explanation

Correct: C. TPUs are designed for exactly this — large, dense TensorFlow matmul workloads — and give the best price-performance at scale.

A: CPUs are far too slow for large deep models. B: Forcing the model onto one GPU sacrifices accuracy or scale. D: Many small GPUs without all-reduce acceleration is slower and pricier than TPUs here.

Q2. You must specialise a strong foundation LLM for a niche legal domain. You have a few thousand high-quality examples, a tight budget, and two weeks. What is the best approach?

  • A) Pre-train a new transformer from scratch on the legal corpus.
  • B) Fine-tune (supervised tuning) the foundation model on the labelled examples.
  • C) Run a large hyperparameter sweep over a small model trained from scratch.
  • D) Increase the prompt with all examples on every request instead of training.
Reveal answer & explanation

Correct: B. Fine-tuning a capable base model on a few thousand examples gives a specialised model cheaply and quickly — the textbook fit for limited data and budget.

A: From-scratch pre-training needs far more data, compute, and time. C: A small from-scratch model won't match a tuned foundation model. D: Few-shot prompting may help but won't reliably bake in domain behaviour and grows cost per call.

Q3. A hyperparameter-tuning job must find a strong configuration within a fixed compute budget, exploring the space efficiently rather than just fastest wall-clock. How should you configure it?

  • A) Maximise parallel trials so all configurations run at once.
  • B) Use Bayesian optimisation with modest parallelism and early stopping.
  • C) Use exhaustive grid search across every combination.
  • D) Run a single trial with hand-picked values to save budget.
Reveal answer & explanation

Correct: B. Bayesian search learns from completed trials, so modest parallelism plus early stopping spends the budget where it matters and finds strong configs efficiently.

A: Full parallelism is fastest but each trial can't learn from the others, wasting budget. C: Grid search explodes combinatorially. D: One hand-tuned trial isn't a search at all.
