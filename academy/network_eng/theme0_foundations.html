Laying down the absolute basics so you can navigate VPCs, BGP, and GKE like a cloud native networking wizard!

Theme 0 🟣

0.1 IPv4 & CIDR Notation Demystified

Welcome to Lesson Zero! If you've ever looked at an IP address like 10.128.0.0/20 and immediately felt a cold sweat coming on, don't worry. Networking people love to use scary-sounding acronyms to keep their salaries high, but we're about to bust that door down. 🚪💥

An IPv4 Address is just a 32-bit digital address written in four decimal parts (called octets) separated by dots (e.g., 192.168.1.50). Each octet is 8 bits, meaning its value can range from 0 to 255.

🌍 RFC 1918: The Private Playground

Computers on the public internet must have globally unique IP addresses. But inside our own private networks (like your home Wi-Fi or a GCP VPC), we use a special set of "un-routable" IP ranges reserved by the internet gods under a document called RFC 1918. These are completely free for anyone to use internally:

  • 10.0.0.0/8: The heavy lifter. Huge range (16.7 million IPs). GCP defaults to using subnets from this space.
  • 172.16.0.0/12: The middle-child range (1.04 million IPs). Often used to avoid overlaps with on-prem networks.
  • 192.168.0.0/16: The home-classic (65,536 IPs). Most home routers live here.

📏 CIDR Math: How to Calculate Subnet Sizes in Your Head

How do we carve up these private playgrounds into smaller chunks? We use Classless Inter-Domain Routing (CIDR). The slash notation (e.g., /24) tells us how many bits are locked in for the "network" part of the address. The remaining bits are left free for "hosts" (like VMs or Kubernetes pods).

Since an IPv4 address is 32 bits, the number of free bits is 32 - [slash number]. The total number of IPs in a CIDR block is 2(32 - slash).

But you don't need a calculator under exam pressure! Just remember these three anchor points and the Doubling Rule:

CIDR Slash Total IP Addresses Typical GCP Use Case
/32 1 IP (232-32 = 20 = 1) A single Virtual Machine's external IP address.
/24 256 IPs (232-24 = 28 = 256) A standard regional subnet for standard microservices.
/16 65,536 IPs (232-16 = 216 = 65.5k) A large VPC-wide address space or heavy-duty GKE cluster.

💡 The Doubling Rule (CIDR Cheat Code!)

Subtracting 1 from the CIDR slash number DOUBLES the host capacity. Adding 1 cuts it in HALF!

  • If /24 is 256 IPs...
  • Then /23 is 512 IPs (256 × 2)
  • And /22 is 1,024 IPs (512 × 2)
  • And /21 is 2,048 IPs (1,024 × 2)
  • And /25 is 128 IPs (256 ÷ 2)
The "GCP 5-IP Subtraction" Examiner Favorite!
In traditional physical networking, you lose 2 IP addresses per subnet (the first for the Network ID, the last for the Broadcast address). But Google Cloud isn't traditional! Google reserves 5 IP addresses in every subnet. If you deploy a 10.0.0.0/24 subnet (256 total IPs), the following are strictly reserved:
  1. 10.0.0.0: Network ID
  2. 10.0.0.1: Default Gateway (for VMs to talk outside the subnet)
  3. 10.0.0.2: Google DNS server
  4. 10.0.0.3: Reserved by Google for future infrastructure wizardry
  5. 10.0.0.255: Subnet Broadcast address
This leaves you with 251 usable IPs! If the examiner asks you to size a subnet for 252 database nodes and you choose a /24, you've fallen right into their trap! 🕳️

0.2 Layer 3 vs Layer 4: Packets, Connections & Ports

To master firewalls and load balancing, you have to understand where they operate. We're zooming in on two layers of the famous OSI Model:

📬 Layer 3 (Network Layer) — Packets & Postcards

Layer 3 is all about IP addresses and packets. It is completely stateless and doesn't care if the destination computer is actually online, listening, or if the packet gets wet and ruined on the way. It just slaps the destination IP on the envelope and drops it in the mailbox.

🤝 Layer 4 (Transport Layer) — TCP vs. UDP

Layer 4 sits on top of Layer 3 and handles the actual "conversation." It offers two completely different personalities:

🤝 TCP (Transmission Control Protocol) — The Reliable Handshaker

TCP is the polite, organized, and reliable protocol. It is connection-oriented, meaning it won't send a single byte of actual data until both computers perform a friendly 3-way handshake:

  1. SYN: "Hello! I'd like to open a connection."
  2. SYN-ACK: "Hi there! I hear you, and I am ready to connect too!"
  3. ACK: "Excellent, let the data flow!"

TCP guarantees that every single packet arrives safely, in the exact correct order, and will re-transmit any that get lost. It's like a telephone call. 📞

📣 UDP (User Datagram Protocol) — The Fast-and-Furious Screamer

UDP is the chaotic, high-speed speedster. It is completely connectionless. It doesn't perform handshakes, doesn't verify if anyone is listening, and doesn't care if packets arrive out of order or get lost in a routing storm. It just screams packets out of a megaphone. 📣

Why use it? It has zero setup overhead, making it incredibly fast. Perfect for live streaming, gaming, and VoIP calls where a lost packet is just a tiny blip but a delay is fatal.

🏢 Port Numbers: The Apartment Numbers

An IP address gets a packet to the right building (the server), but how does it get to the right door (the application)? We use Port Numbers! Ports range from 1 to 65,535. You must memorize these high-yield common ports for the exam:

  • Port 80 (HTTP) & Port 443 (HTTPS): Web traffic. HTTPS is encrypted!
  • Port 22 (SSH): Secure Shell (remotely managing Linux VMs).
  • Port 3389 (RDP): Remote Desktop Protocol (remotely managing Windows VMs).
  • Port 53 (DNS): Domain Name System (resolving domain names to IP addresses).

0.3 The GCP Networking Mindset (The SDN Shift)

Forget everything you know about hardware rack cabinets and yellow ethernet cables. In Google Cloud, networking is entirely virtual! It is a Software-Defined Network (SDN) called Andromeda.

This SDN architecture gives GCP some incredible characteristics that are major exam topics:

🚀 The Global VPC

In other clouds (like AWS or Azure), virtual networks are bound to a single region. If you want resources in Virginia to talk to resources in Dublin, you have to configure peering links.

In GCP, the VPC is Global! A single VPC spans the entire planet. While subnets inside that VPC are regional (meaning subnet-us lives in Iowa and subnet-eu lives in Belgium), they can communicate out-of-the-box over Google's ultra-fast private fiber backplane without any complex configuration. 🌎✨

🛡️ Stateful Firewalls Demystified

GCP Firewall Rules (and NGFW policies) are stateful. This means once a packet is allowed through a rule in one direction, the returning response traffic is automatically allowed in the opposite direction, regardless of other rules!

If you allow ingress (incoming) traffic on Port 443 so users can load your website, you do NOT need to write an egress (outgoing) rule to let the website content travel back to the user's browser. Andromeda keeps track of the connection state and waves the return traffic right through!

0.4 The Foundational Jargon Buster

Keep this handy glossary in your back pocket. These terms pop up in almost every exam question:

Term What it actually means Why it matters for GCP
MTU (Max Transmission Unit) The maximum size (in bytes) of a single packet. If a packet exceeds this, it must be chopped up (fragmented) or dropped. GCP networks support standard 1500 bytes or jumbo 8896 bytes. HA VPN tunnels require exactly 1460 bytes.
Next Hop The routing term for "where do we send this packet next to reach its destination?" GCP routes traffic based on Next Hops (like a default internet gateway, a VPN tunnel, or an Internal Load Balancer).
BGP (Border Gateway Protocol) The postal system of the internet. Routers use BGP to dynamically advertise which IP addresses live on their side of the fence. Used by GCP's Cloud Router to exchange routing paths dynamically with your on-premises routers.
DNS (Domain Name System) The system that translates human-readable names (e.g., google.com) into computer-readable IPs (e.g., 142.250.190.46). Crucial for private GKE endpoints and hybrid setups. Managed via Cloud DNS.
Latency vs. Bandwidth Latency is the delay (time) for a packet to travel round-trip. Bandwidth is the volume of data that can be sent per second. Think of latency as how fast the delivery truck drives, and bandwidth as how wide the highway is. Gcp Premium Tier minimizes latency!

🧠 Quick-Fire Warm-Up Quiz!

Test your recall before diving into the proper practice exam. No peeking!

Q1: If you configure a subnet range with a /24 CIDR block in Google Cloud, how many VMs can you assign a private IP address to?

252 VMs! A /24 contains 256 total IP addresses, but Google Cloud reserves 4 IP addresses in every primary IPv4 range (network address, default gateway, second-to-last, and broadcast), leaving 252 usable host IPs.

Q2: You have a subnet with CIDR block 10.0.0.0/22. You need to double the number of available IP addresses. What should your new CIDR block slash notation be?

10.0.0.0/21! According to the Doubling Rule, subtracting 1 from the slash number doubles the host capacity (from 1,024 IPs in a /22 to 2,048 IPs in a /21).

Q3: If you write an ingress firewall rule in GCP to block all incoming traffic on Port 80, will this prevent an established outbound web request (initiated by a VM inside the network) from receiving its response on Port 80?

No! GCP firewall rules are stateful. Since the outbound request was allowed egress, the return traffic is automatically tracked and allowed back in, bypassing the ingress block!

Q4: Which transport protocol should you use for real-time video streaming where speed is critical and a few missing pixels are preferable to buffering delays?

UDP (User Datagram Protocol)! UDP has no handshake overhead or connection-tracking delay, making it the perfect choice for fire-and-forget, low-latency streaming workloads.

📝 Theme 0 Practice Exam Questions

Let's see if you can handle these foundational networking and GCP mindset brain busters!

Question 1: You are assigning the primary IPv4 range for a new subnet in a custom-mode VPC. The service will grow to roughly 500 instances, each needing an internal IP, and the network team wants the smallest range that still fits so address space isn't wasted. Accounting for Google Cloud's reserved addresses, which range should you choose?

  • A) 10.8.0.0/24
  • B) 10.8.0.0/23
  • C) 10.8.0.0/22
  • D) 10.8.0.0/25

Correct Answer: B

Why Option B is correct: A /23 holds 512 addresses. Google Cloud reserves 4 addresses in every subnet's primary IPv4 range (network address, default gateway, second-to-last, and broadcast), leaving 512 - 4 = 508 usable. That is the smallest standard block that still covers ~500 instances, so it fits without wasting space.

Why the other options are wrong:
Option A (/24): 256 total minus 4 reserved leaves only 252 usable — well short of 500.
Option C (/22): 1,024 total leaves 1,020 usable. It works, but it wastes roughly 500 addresses, breaking the "smallest range that fits" requirement.
Option D (/25): 128 total leaves only 124 usable — far too small.

Question 2: A VM with no external IP has an ingress firewall rule that denies all inbound TCP 443, yet it still receives the responses to HTTPS requests it makes to an external API. A colleague wants to add an ingress allow rule so the replies keep working. What is the correct explanation?

  • A) The VPC's implied allow-ingress rule already permits established TCP traffic on ephemeral source ports, so the reply packets match it.
  • B) Cloud NAT rewrites the reply packets onto an internal flow, which lets them skip firewall evaluation on the way back in.
  • C) Egress rules automatically generate a matching temporary ingress rule for every new outbound connection a VM opens.
  • D) VPC firewall rules are stateful, so replies to a connection the VM initiated are tracked and allowed without any ingress rule.

Correct Answer: D

Why Option D is correct: Google Cloud VPC firewall rules are stateful. Once egress for a new connection is allowed, the return traffic for that connection is tracked and permitted automatically — the ingress deny rule never applies to it. No extra ingress rule is needed.

Why the other options are wrong:
Option A: The only implied rules are allow-egress and deny-ingress; there is no "established on ephemeral ports" allow rule. Statefulness, not a port match, is why replies arrive.
Option B: Cloud NAT provides outbound translation for VMs without external IPs; it does not exempt packets from firewall evaluation.
Option C: Egress rules never spawn ingress rules. Connection tracking, not rule generation, is what lets replies through.

Question 3: Architects from another cloud ask how much routing you must configure for a VM in us-central1 to reach a VM in europe-west1 over internal IPs, where both VMs live in the same custom-mode VPC. What should you tell them?

  • A) Nothing extra — a VPC is a global resource, so subnet routes for every region exist automatically and internal traffic crosses Google's backbone.
  • B) Enable global dynamic routing and run a Cloud Router in each region so that each one advertises the other region's subnets over BGP.
  • C) Create VPC Network Peering between the two regions and add static routes that point at each region's internal gateway address.
  • D) Build an HA VPN tunnel mesh between the regions, because subnets in different regions are isolated from one another by default.

Correct Answer: A

Why Option A is correct: In Google Cloud a VPC spans all regions. Subnets are regional, but their subnet routes are global by default, so a VM in one region reaches a VM in another over internal IPs with no extra routing — the traffic simply rides Google's private backbone.

Why the other options are wrong:
Option B: Global dynamic routing and Cloud Routers control how BGP-learned (on-prem) routes propagate across regions, not intra-VPC reachability, which is already automatic.
Option C: VPC Network Peering connects separate VPCs; subnets that already share one VPC don't peer with themselves.
Option D: HA VPN is for hybrid or cross-cloud links. Subnets in the same VPC are not region-isolated, so a tunnel mesh is unnecessary.
