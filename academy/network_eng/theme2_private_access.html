Theme 2: Private Connectivity, NCC & VPC-SC

Private Google Access, PSA vs PSC, Network Connectivity Center, and VPC Service Controls

Theme 2 🟢

2.1 Private Google Access & VIPs

Right, let's talk about accessing Google's own APIs (like Cloud Storage, BigQuery, or Pub/Sub) privately. If your VM doesn't have a public IP, how does it talk to Google APIs? It can't route to the public internet! The answer is Private Google Access (PGA). 🌟

📡 Private Google Access (PGA)

PGA is a simple subnet-level checkbox. When enabled, VMs in that subnet with only private IPs can send traffic to the external IP addresses of Google APIs. The traffic travels entirely over Google's internal network, never touching the public internet!

But wait! What if you are on-premises and want to reach Google APIs privately over VPN or Interconnect? On-prem networks can't use PGA directly. Instead, you must use one of Google's Virtual IPs (VIPs):

  • Private VIP (private.googleapis.com): Resolves to 199.36.153.8/30. Supports most Google APIs, but excludes consumer-facing services like Gmail or YouTube. Extremely secure!
  • Restricted VIP (restricted.googleapis.com): Resolves to 199.36.153.4/30. Supports only APIs that can be secured using VPC Service Controls. This is the gold standard for compliance because it completely blocks access to non-permissible Google services! 🔒
The DNS Forwarding VIP Trap! To make on-premises resources use these VIPs, you must configure your on-prem DNS servers to resolve *.googleapis.com as a CNAME to private.googleapis.com or restricted.googleapis.com, and set up DNS forwarding policies in Cloud DNS. If you forget the DNS overrides, on-prem traffic will try to resolve public IPs and fail!

2.2 PSA vs. PSC vs. Serverless VPC Access

This is the absolute core architectural decision of the exam! How do you connect your VPC to managed services (like Cloud SQL, AlloyDB, or third-party SaaS) and serverless environments (Cloud Run, Cloud Functions)? Let's break down the three giants:

Service Type Under the Hood Mechanism Transitive Routing Friendliness Best Use Cases
Private Services Access (PSA) VPC Network Peering to a Google-managed tenant VPC. Requires allocating an IP range block (e.g., /24 or /20). No! VPC Peering is non-transitive. On-prem networks cannot reach it without complex "Export Custom Routes" and Cloud Router BGP hacks. 🧱 Classic Cloud SQL, AlloyDB, Memorystore, and managed GKE control planes.
Private Service Connect (PSC) Places a local endpoint (Private IP) directly inside one of your consumer VPC subnets. Uses a forwarding rule. No peering! Yes! Since it's a local subnet IP, on-premises networks and peered VPCs can route to it natively without any peering updates. 🌟 Modern Cloud SQL, third-party SaaS (Snowflake, MongoDB), and publishing your own internal services to other VPCs.
Serverless VPC Access Uses a pool of connector VM instances (Serverless VPC Access Connector) that bridge your serverless environment into your VPC. N/A (Outbound only) Allowing Cloud Run, Cloud Functions, or App Engine to access private resources (VMs, databases) inside your VPC.
🧠 Crucial Memory Tip: PSA vs. PSC

Remember: PSA is a Peering, PSC is a Point!

  • PSA peers entire networks together. It's rigid, non-transitive, and gobbles up large IP blocks.
  • PSC is just an endpoint (a single IP address) in your local subnet. It's incredibly flexible, transitive-friendly, and uses only one IP!
"Why did the network engineer break up with VPC Peering? Because they felt the relationship was non-transitive and had too many boundaries!" 💔

2.3 Network Connectivity Center (NCC)

As your cloud footprint grows, managing pairwise VPC peerings becomes a nightmare. Network Connectivity Center (NCC) is Google's hub-and-spoke network connectivity model. It acts as a single centralized logical control plane to connect VPCs, on-premises offices, and other clouds!

🌐 NCC Topologies & Spokes

An NCC configuration consists of a central Hub and multiple attached Spokes:

  • Spoke Types:
    • VPC Spoke: Connects a VPC network to the hub.
    • Hybrid Spoke: Connects on-premises networks using HA VPN tunnels or Cloud Interconnect VLAN attachments.
    • Router Appliance Spoke: Connects third-party SD-WAN or firewall VMs (Router Appliances) to exchange routes dynamically.
  • Preset Topologies (Configured at Hub creation, cannot be changed!):
    • Mesh Topology (Default): All spokes belong to a single group. Full inter-spoke connectivity is established automatically. All VPC spokes can talk to all other VPC spokes and hybrid spokes.
    • Star Topology: Spokes belong to either the Center or Edge group. Edge spokes can talk to Center spokes, but Edge spokes cannot talk to other Edge spokes! This is perfect for multi-tenant isolation or retail branch networks where branches should only talk to the central hub, not each other. 🌟
The NCC Star Topology Hybrid Trap! If you configure an NCC hub with a Star topology, hybrid spokes with site-to-site data transfer enabled (which allows on-prem sites to transit through Google's network to talk to other on-prem sites) are only supported in the Center spoke group! Edge spoke groups only support hybrid spokes with site-to-site data transfer disabled.

2.4 VPC Service Controls (VPC-SC)

Firewall rules block network access, but they don't prevent data exfiltration (where an insider with valid credentials copies sensitive database data to an external, unauthorized GCP project). To stop this, you need VPC Service Controls (VPC-SC)! 🛡️

🔒 Service Perimeters & Bridges

VPC-SC allows you to define a Service Perimeter around your Google Cloud resources (like Cloud Storage, BigQuery, or Cloud SQL). This perimeter acts as a giant security wall. APIs inside the perimeter cannot talk to resources outside, even if they have valid IAM credentials!

  • Perimeter Bridges: What if two separate secure perimeters need to share data? You don't merge them! Instead, you create a Perimeter Bridge to allow specific projects in Perimeter A to talk to specific projects in Perimeter B.
  • Ingress/Egress Policies: Highly granular rules that allow traffic to cross the perimeter boundary based on identity, project, and device state (using Access Context Manager).
  • Dry-Run Mode: Always use dry-run mode first! It logs any violations without actually blocking traffic. You can inspect the dry-run logs in Cloud Logging to identify and fix any missing permissions before enforcing the perimeter and breaking production! 🛠️

🧠 Quick-Fire Warm-Up Quiz!

Test your recall before diving into the proper practice exam. No peeking!

Q1: Which Google API VIP should you use if your security policy demands that VMs must only access APIs that support VPC Service Controls?

restricted.googleapis.com (Restricted VIP). It resolves to 199.36.153.4/30 and strictly limits access to VPC-SC supported services, blocking all others (like consumer Gmail).

Q2: True or False: You can change an NCC hub's preset topology from Mesh to Star after it has been created.

False! The preset topology (Mesh or Star) must be specified at hub creation and cannot be changed later. You would have to delete the hub and recreate it.

Q3: Why is Private Service Connect (PSC) preferred over Private Services Access (PSA) for complex hub-and-spoke networks?

Because PSC uses a local IP endpoint in a consumer subnet, making it fully compatible with transitive routing. PSA uses VPC Peering, which is non-transitive and blocks on-premises or cross-VPC routing natively.

Q4: What is the recommended first step when deploying a new VPC Service Controls perimeter to prevent breaking existing applications?

Deploy the perimeter in Dry-Run Mode! This logs all policy violations in Cloud Logging without actually blocking any API requests, allowing you to safely troubleshoot and write ingress/egress policies.

📝 Theme 2 Practice Exam Questions

Let's see if you can handle these actual exam-style private access and security perimeter brain busters!

Question 1: A secure data warehouse in BigQuery sits inside a VPC Service Controls perimeter. An automated pipeline running on-premises loads data into BigQuery over Dedicated Interconnect and fails with "Request blocked by VPC Service Controls". How should you resolve this securely?

  • A) Resolve the BigQuery API on-premises to the public IP of a transit VM in the hub VPC, and add a hierarchical firewall rule tagged to bypass the perimeter check.
  • B) Resolve the BigQuery API to private.googleapis.com (199.36.153.8/30), advertise it over BGP, and leave the perimeter in dry-run mode so the pipeline is auto-allowed.
  • C) Resolve the BigQuery API to restricted.googleapis.com (199.36.153.4/30), advertise that range over BGP, and add a VPC-SC ingress policy authorizing the pipeline's identity.
  • D) Create a VPC-SC perimeter bridge to the on-premises gateway project, and use Cloud NAT to SNAT the API requests so their on-premises origin is hidden.

Correct Answer: C

Why Option C is correct: Reaching VPC-SC–protected APIs from on-premises requires routing to the restricted VIP (restricted.googleapis.com) over the Interconnect, plus an ingress policy that explicitly authorizes the pipeline's service-account identity and source. That combination crosses the perimeter without weakening it. 🔒

  • A: Firewall policies and transit VMs can't bypass VPC-SC — the perimeter is enforced at the Google API layer, not by routing or firewall tags.
  • B: private.googleapis.com does not enforce VPC-SC, and dry-run mode only logs violations; it never auto-allows traffic once the perimeter is enforced.
  • D: Perimeter bridges join projects in different perimeters, not on-premises networks, and SNAT changes the source IP without satisfying any ingress rule.

Question 2: You run a multi-tenant SaaS platform. A central producer VPC hosts a shared analytics service that must be exposed to hundreds of independent customer VPCs. Customers must not reach each other, and there must be no direct peering to your VPC so that overlapping customer IP ranges never matter. What architecture fits?

  • A) Use Private Service Connect: publish a service attachment behind an internal load balancer in the producer VPC, and have each customer create a PSC endpoint (a private IP) in their own VPC.
  • B) Establish pairwise VPC peering from the producer VPC to every customer VPC, and use Cloud NAT to translate any overlapping customer CIDR ranges on the links.
  • C) Build an NCC hub with a star topology, putting the producer VPC in the center group and every customer VPC in the edge group with BGP route leaking.
  • D) Terminate a Cloud VPN tunnel from each customer VPC and run a pool of proxy VMs in the producer VPC to destination-NAT the overlapping traffic.

Correct Answer: A

Why Option A is correct: Private Service Connect is built for this. Each consumer reaches the published service through a private endpoint IP in its own subnet, which sidesteps peering quotas and IP overlaps and keeps consumers fully isolated from one another. 🌟

  • B: Peering caps the number of connections, forbids overlapping CIDRs, and Cloud NAT can't dynamically de-overlap peered ranges.
  • C: A star topology isolates edge spokes, but spokes still need non-overlapping IP space — impossible to guarantee across hundreds of independent customers.
  • D: Per-customer VPN plus proxy-VM NAT is heavy to operate and a scaling bottleneck compared with native PSC.

Question 3: You're using a Network Connectivity Center (NCC) hub to connect several retail branch offices (over HA VPN) to your central Google Cloud VPC. Branches must reach the central VPC directly, but must never route to each other through Google's network. How should you configure the hub?

  • A) Create a mesh-topology hub, attach all branch VPNs as hybrid spokes, and add VPC firewall rules in the central VPC to drop branch-to-branch traffic.
  • B) Create a star-topology hub, put the central VPC spoke in the center group, and place all branch hybrid spokes in the edge group.
  • C) Skip NCC and connect each branch with VPC peering, importing custom routes but restricting BGP advertisements at the host-project level.
  • D) Create a star-topology hub, put the branch hybrid spokes in the center group and the central VPC in the edge group, then disable BGP on the branch routers.

Correct Answer: B

Why Option B is correct: In an NCC star topology, edge spokes can reach center spokes but not one another. Branches in the edge group and the central VPC in the center group gives exactly "branches reach the center, never each other," enforced natively by NCC. 🌟

  • A: Mesh lets every spoke reach every other spoke, so branches could transit to each other; central-VPC firewall rules don't stop spoke-to-spoke transit at the NCC layer.
  • C: You can't terminate branch VPN tunnels inside a peering connection, and peering doesn't scale to many dynamically-routed branches.
  • D: Putting branches in the center group lets them reach each other (the very thing you must prevent), and disabling BGP would break their connectivity.

Question 4: After deploying a VPC Service Controls perimeter, developers report their Cloud Run apps can no longer read a secure Cloud Storage bucket. What is the most efficient way to pinpoint the exact cause and fix it without disrupting other services?

  • A) Delete the perimeter, recreate it with Cloud Run removed from the protected services, and read the audit logs to rebuild it correctly.
  • B) Disable VPC-SC enforcement, add an IP-based access level in Access Context Manager for the Cloud Run identity, and rely on IAM.
  • C) Grant roles/storage.admin to the relevant identities and set a bucket policy that ignores IP-based restriction checks.
  • D) Search Cloud Logging for the VPC-SC violationReason, read the callerIp, serviceName, and authenticationInfo fields, then write a precise ingress policy.

Correct Answer: D

Why Option D is correct: VPC-SC logs every violation with the caller identity, source IP, and target service. Reading those fields tells you exactly what to authorize, so you can add a tight ingress policy for the legitimate traffic without touching anything else. 🛠️

  • A: Deleting and recreating the perimeter is disruptive, weakens security, and still doesn't pinpoint the original cause.
  • B: Disabling VPC-SC removes exfiltration protection, and ACM IP levels don't cleanly apply to serverless Cloud Run identities without egress routing.
  • C: IAM roles never override a VPC-SC perimeter — even a project Owner is blocked crossing the boundary without an ingress rule.
