Theme 4: GKE Networking

VPC-native clusters, Dataplane V2, network policies, and GKE load balancing

Theme 4 🟢

4.1 VPC-Native vs. Route-Based Clusters

When you deploy Google Kubernetes Engine (GKE), you are faced with an immediate fork in the road: VPC-Native or Route-Based networking. For the exam, there is only one correct answer: always choose VPC-Native! 🚢

🐳 VPC-Native (Alias IP) Networking

VPC-Native clusters use Alias IP ranges allocated from your VPC subnets. Pod IP addresses are native, fully routable IPs inside your VPC. This has massive benefits:

  • No Route Table Limits: Route-based clusters rely on static routes in your VPC route table to locate Pods. This hits a hard limit of 100 routes (meaning your cluster can never scale beyond 100 nodes!). VPC-Native bypasses this entirely.
  • Direct Routing: Traffic to Pods is routed natively by the VPC control plane, avoiding extra hops.
  • Security Integration: You can apply VPC firewall rules directly to Pod IP ranges!

When planning a VPC-Native cluster, you must allocate three IP ranges:

  1. Primary Subnet Range: Used for GKE Nodes (the VMs themselves).
  2. Secondary Range A (Pods): A large block used for Kubernetes Pods (e.g., /14 or /16).
  3. Secondary Range B (Services): A block used for Kubernetes ClusterIP Services (e.g., /20).
The GKE IP Exhaustion Trap! If you configure your Pod secondary range too small, your GKE cluster will fail to scale up node pools even if you have plenty of Compute Engine quota! GKE allocates a fixed /24 block (256 IPs) per node for Pods by default. If your secondary range runs out of IPs, GKE cannot provision new nodes. Plan your IPAM space carefully!

🧮 Alias-IP Sizing: A Worked Example

Exam questions love to make you size the two secondary ranges for a cluster's projected growth while "minimizing address consumption". Work it from the per-node Pod block (default /24 = 256 Pod IPs per node):

  • Pods range = nodes × 256 IPs. For 100 nodes that's 25,600 IPs → round up to a power of two → a /17 (32,768 IPs) fits; a /18 (16,384) would not.
  • Services range = number of ClusterIP Services. For ~1,500 services, a /21 (2,048 IPs) is comfortable while a /24 (256) is far too small.
  • Nodes (primary) range just needs one IP per node (plus headroom) — a small /25 covers 100+ nodes.

So a 100-node / 1,500-service target maps cleanly to a /25 primary subnet with secondary ranges of /17 for Pods and /21 for Services.

Secondary Ranges Are Immutable! You cannot resize a cluster's Pod or Service secondary range after the cluster is created (the expand-ip-range command applies to primary subnet ranges, not secondaries). That's why you size for the 2-year projection up front — "start small and resize later" is always a trap answer for secondary ranges.

4.2 Control Plane Isolation & DNS-Based Endpoints

Securing access to your Kubernetes API server (the control plane) is critical. GKE offers several layers of isolation:

🔒 Public vs. Private Clusters & DNS-Based Endpoints

  • Private Cluster: Nodes only have private IPs. The control plane endpoint can be configured as Private (only accessible inside the VPC or via hybrid connections) or Public (accessible over the internet but locked down using Master Authorized Networks).
  • Master Authorized Networks: A static list of CIDR blocks (like your corporate office IP) allowed to talk to the control plane. Useful, but rigid and hard to manage dynamically.
  • ✨ DNS-Based Endpoint (The Modern Game-Changer!): Enabled via --enable-dns-access. Instead of accessing the control plane via a direct IP address, GKE creates a unique fully qualified domain name (FQDN) (e.g., *.gke.goog) for your cluster.
    • How it works: The FQDN resolves to a secure Google Cloud API frontend, not the cluster's direct internal IP!
    • The Bastion-Free Advantage: Because it's a standard Google Cloud API, authorized users can connect to a private cluster's control plane from anywhere (including home or other clouds) without needing a bastion host or VPN, as long as they have GCP IAM permissions (roles/container.developer)!
    • Force DNS Only: You can completely disable IP-based endpoints using --no-enable-ip-access to force all traffic through the identity-gated DNS endpoint.
The Hub-and-Spoke / On-Prem Control-Plane Trap! When a scenario needs control-plane access from other spoke projects or on-premises, the DNS-based endpoint is the modern answer — and it sidesteps the classic VPC Network Peering is non-transitive problem entirely, because the FQDN resolves to Google's API frontend rather than a peered internal IP. Access is gated by IAM (container.clusters.connect) plus VPC Service Controls, not IP authorized networks. For a private-only path (no public internet), reach that endpoint over Private Service Connect, Cloud VPN, or Cloud Interconnect (with a DNS record for the gke.goog domain). The old answers — "deploy a proxy/bastion" or "export peering routes + authorized networks" — are exactly what this replaces.
"Why did the GKE cluster throw away its bastion host? Because it got a DNS-based endpoint and realized it didn't need a middleman to manage its relations!" 🤾‍♂️

4.3 Dataplane V2: eBPF to the Rescue!

By default, older GKE clusters used standard Linux iptables to manage routing and Kubernetes Network Policies. However, iptables scales terribly—as you add hundreds of services and network policies, the kernel has to evaluate thousands of sequential rules, causing CPU spikes and latency! 🐌

🐝 GKE Dataplane V2 (eBPF / Cilium)

Modern GKE clusters use Dataplane V2 (enabled by default in Autopilot and newer Standard clusters). It replaces iptables with eBPF (Extended Berkeley Packet Filter), running Cilium under the hood:

  • Blistering Performance: eBPF executes sandboxed code directly inside the Linux kernel, bypassing the slow iptables evaluation chain. Routing and packet filtering happen at hardware-like speeds!
  • Native Network Policies: Kubernetes Network Policies are enforced natively in the kernel, scaling effortlessly to thousands of rules without performance degradation.
  • Enhanced Observability: Provides deep, real-time logging of container-level network flows (VPC Flow Logs for GKE).

4.4 GKE Load Balancing & DNS

How does external traffic reach your Pods, and how do Pods resolve domain names?

🌐 GKE Gateway vs. Ingress Controller

  • GKE Ingress Controller: The classic way. It automatically provisions a Google Cloud Application Load Balancer (L7) based on Kubernetes Ingress resources. It routes traffic to your nodes, which then route it to Pods.
  • GKE Gateway Controller: The modern, multi-tenant standard (conforming to the Kubernetes Gateway API). It splits responsibilities between the Platform Admin (who manages the physical Load Balancer infrastructure) and the Application Developer (who manages the routing rules). Much more flexible!
  • Container-Native Load Balancing (NEGs): Whether using Ingress or Gateway, always use Network Endpoint Groups (NEGs)! Instead of routing traffic to GKE Nodes (which then use kube-proxy to hop to a Pod, causing extra latency), the Load Balancer routes traffic directly to the Pod IP addresses. This eliminates extra network hops and ensures perfect load distribution! 🎯
🕵️ GKE DNS Performance Trio:

To keep DNS resolution fast and prevent overloading Cloud DNS, GKE uses three layers:

  • kube-dns: The standard cluster-internal DNS service running as Pods.
  • NodeLocal DNSCache: Runs a tiny DNS caching agent as a DaemonSet on every single node. Pods query their local node cache first, reducing DNS latency by up to 90% and preventing kube-dns bottlenecks! ⚡
  • Cloud DNS for GKE: Replaces kube-dns entirely with Google's fully-managed Cloud DNS, offering massive scale and seamless cross-VPC resolution.

🧠 Quick-Fire Warm-Up Quiz!

Test your recall before diving into the proper practice exam. No peeking!

Q1: Why does a route-based GKE cluster fail to scale beyond 100 nodes?

Because route-based clusters rely on static routes in the VPC route table to locate Pods. GCP VPC route tables have a hard quota limit of 100 custom routes, capping the cluster at 100 nodes. VPC-Native uses Alias IPs and has no such limit!

Q2: True or False: GKE DNS-based endpoints require a VPN or Interconnect connection to access a private cluster externally.

False! DNS-based endpoints resolve to a global Google Cloud API frontend. As long as you have internet access and the correct GCP IAM permissions, you can securely connect to a private cluster externally without any VPN, Interconnect, or bastions!

Q3: What underlying technology does GKE Dataplane V2 use to bypass the slow iptables packet evaluation chain?

eBPF (Extended Berkeley Packet Filter), running Cilium. It executes routing and network policy rules directly inside the Linux kernel for blistering speed.

Q4: What GKE feature enables Google Cloud Load Balancers to bypass GKE nodes and route traffic directly to Pod IP addresses?

Container-Native Load Balancing via NEGs (Network Endpoint Groups). This eliminates the extra kube-proxy hop on the node, reducing latency.

📝 Theme 4 Practice Exam Questions

Let's see if you can handle these actual exam-style GKE networking brain busters!

Question 1: A private GKE Standard cluster will host a banking API. The control plane must never expose a public IP, no bastion or jump box is allowed in the VPC, yet developers working from home must deploy to the cluster securely. Which architecture fits?

  • A) Create a public cluster, disable public node IPs, and restrict control-plane access with Master Authorized Networks set to the developers' home IP ranges.
  • B) Create a private cluster and front the control plane with a public internal Application Load Balancer behind Cloud CDN to proxy developer connections.
  • C) Create a private cluster, enable the DNS-based control-plane endpoint and disable the IP-based endpoint, and have developers authenticate with gcloud to the FQDN.
  • D) Create a route-based cluster with Private Google Access and add VPC routes that map the developers' local clients to the node-pool gateways.

Correct Answer: C

Why Option C is correct: The DNS-based control-plane endpoint removes the IP endpoint entirely. Access flows through Google's API frontend, which enforces IAM, so remote developers reach the cluster by FQDN with no public IP and no bastion. 🛡️

  • A: A public cluster exposes a control-plane public IP (violating the constraint), and home IPs are dynamic, making Master Authorized Networks brittle.
  • B: You can't put an internal ALB or Cloud CDN in front of GKE's managed control plane — it lives in a Google-owned tenant project.
  • D: Route-based clusters are legacy and don't solve control-plane exposure; you can't route external clients to nodes without a public interface.

Question 2: A 500-pod GKE app makes constant DNS lookups. The kube-dns pods are pegged at high CPU and application pods report intermittent "DNS lookup timeout" errors. What resolves this with the lowest added latency?

  • A) Enable NodeLocal DNSCache so each node runs a local caching agent that pods query over a loopback address.
  • B) Scale kube-dns up to 100 replicas and add pod anti-affinity so they spread evenly across zones.
  • C) Switch from VPC-native to route-based networking to bypass alias-IP lookups in the pod network.
  • D) Point the application pods at Google public DNS (8.8.8.8 / 8.8.4.4) instead of the in-cluster resolver.

Correct Answer: A

Why Option A is correct: NodeLocal DNSCache runs a caching DaemonSet on every node; pods hit the local cache over loopback, removing the network hop to kube-dns and the conntrack overhead — lower latency and no more timeouts. ⚡

  • B: More kube-dns replicas burns resources and still pays the cross-node hop and connection-tracking cost.
  • C: Route-based vs VPC-native doesn't change in-cluster DNS resolution paths.
  • D: Public DNS can't resolve internal service names (e.g. my-svc.default.svc.cluster.local), breaking service discovery.

Question 3: You deploy a VPC-native GKE cluster in your service project using subnets the central team allocated in the host project. Creation fails with "Permission denied on subnet secondary ranges." What is the most likely cause?

  • A) Your personal user identity lacks the Owner or Editor role in the host project to create the subnets and routes.
  • B) The network team disabled Shared VPC support for GKE node pools, so the cluster must run over a Dedicated Interconnect.
  • C) The assigned subnets have Private Google Access turned off, which blocks the nodes from reaching Google APIs.
  • D) The service project's GKE service agent hasn't been granted compute.networkUser on the host subnets and their secondary ranges.

Correct Answer: D

Why Option D is correct: GKE provisions VPC-native clusters using the service project's GKE service agent (...@container-engine-robot.iam.gserviceaccount.com). It needs roles/compute.networkUser on the host subnets so it can claim the secondary ranges for Pods and Services. Missing it produces exactly this error. 🤖

  • A: Your personal permissions aren't used by GKE's robot account to provision the network.
  • B: GKE on Shared VPC is the standard enterprise model and can't be globally disabled like this.
  • C: PGA is best practice, but its absence doesn't raise an IAM permission error at cluster creation.

Question 4: A GKE cluster with 200 microservices uses Kubernetes Network Policies for namespace isolation. Node CPU is high and pod-to-pod latency has climbed; you trace it to nodes burning cycles evaluating long Linux iptables chains. How should you optimize this?

  • A) Delete the Network Policies and replace them with VPC firewall rules that target pod network tags instead.
  • B) Enable GKE Dataplane V2 so policy and routing run in the kernel via eBPF instead of iptables.
  • C) Switch from VPC-native to route-based networking to force IP-in-IP encapsulation for pod traffic.
  • D) Roll the node pools onto larger machine types to give more vCPU for kernel-level packet processing.

Correct Answer: B

Why Option B is correct: GKE Dataplane V2 replaces iptables with eBPF (Cilium), enforcing policy and routing inside the kernel. It scales to hundreds of policies with far less CPU and latency than sequential iptables chains. 🐝

  • A: Dropping policies breaks isolation and compliance, and VPC firewall rules can't natively target Kubernetes namespaces or pods.
  • C: Route-based clusters still rely on iptables for policy, so the bottleneck remains.
  • D: Bigger nodes is a costly band-aid that doesn't fix the iptables scaling limit as services grow.
