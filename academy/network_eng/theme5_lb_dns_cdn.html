Theme 5: Load Balancing, Cloud DNS & Cloud CDN

Application, proxy, and passthrough load balancers, Cloud DNS routing policies, and Cloud CDN

Theme 5 🟢

5.1 Load Balancing Taxonomy: The Speed Demons

Google Cloud Load Balancing is built on Google's software-defined network virtualization stack (Maglev and Andromeda). It is incredibly fast, highly scalable, and doesn't require pre-warming! To pass this exam, you must memorize the current load balancer taxonomy and their distinct behaviors: ⚖️

Load Balancer Type Layer Proxy vs. Passthrough Deployment Modes Key Features & Best For
Application Load Balancer (ALB) Layer 7 (HTTP/S) Proxy (terminates client connection, starts new one to backend) Global external, Regional external, Regional internal, Cross-region internal Content-based routing (URL maps), traffic splitting, URL rewrites, SSL offloading. Best for web apps! 🌐
Proxy Network Load Balancer (NLB) Layer 4 (TCP/SSL) Proxy (terminates client connection) Global external, Regional external, Regional internal, Cross-region internal Reverse proxy for non-HTTP TCP traffic. Supports SSL offloading and single global IP. Best for global databases or mail servers.
Passthrough Network Load Balancer (NLB) Layer 4 (TCP/UDP/ESP/ICMP) Passthrough (does not terminate connection; packets pass through unchanged) Regional external, Regional internal Preserves client source IP address, Direct Server Return (DSR). Blistering speed! Best for high-performance gaming, UDP, or tunneling. ⚡
The Client IP Preservation Trap! If an exam question asks you to design a load-balanced architecture where the backend servers MUST see the original client source IP address natively in the packet header, you cannot use an Application Load Balancer or Proxy Network Load Balancer (they terminate the connection, replacing the source IP with the LB's proxy IP). You MUST use a Passthrough Network Load Balancer!

That taxonomy tells you what each load balancer is. This next matrix is the one that wins exam questions — it lines up the discriminators examiners use to separate the right answer from the plausible-looking wrong ones. Burn it into memory: 🔥

Capability Application LB (proxy, L7) Proxy NLB (proxy, L4) Passthrough NLB (L4)
IPv6 / dual-stack ✅ Yes ✅ Yes ✅ Yes (modern backend-service based)
Port translation (frontend port ≠ backend port) ✅ Yes ✅ Yes ❌ No — same port end to end
Content-based routing (URL maps, host/path) ✅ Yes ❌ No ❌ No
TLS / SSL passthrough (no termination at LB) ❌ No (terminates) ❌ No (terminates) ✅ Yes
Preserves client source IP ❌ No (X-Forwarded-For instead) ❌ No ✅ Yes
Global access (reach from any region / on-prem) ✅ Yes (global & cross-region modes) ✅ Yes (global & cross-region modes) ✅ Internal, with global access enabled
Usable as a route next hop (for NVAs) ❌ No ❌ No ✅ Internal passthrough NLB only
Proxy-only subnet required ✅ Yes (internal/regional ALB) ✅ Yes (internal/regional proxy NLB) ❌ No
The Port-Translation Trap! A passthrough load balancer delivers packets to the backend on the same port the forwarding rule received them — it can never map, say, frontend port 700 to backend port 800. The moment a scenario needs a different frontend and backend port, you need a proxy load balancer. (Don't fall for the old "passthrough NLBs don't support IPv6" reasoning — modern backend-service-based passthrough NLBs do support IPv6; port translation is the real discriminator.)
The Proxy-Only Subnet Prerequisite! Before you can create an internal (or regional) Application Load Balancer, the region must already have a proxy-only subnet (purpose REGIONAL_MANAGED_PROXY). It's a prerequisite, not an afterthought — the Envoy proxies live there. Firewall rules and a reserved IP, by contrast, can be added after the LB exists.
The NVA High-Availability Pattern! To send traffic through two network virtual appliances simultaneously with automatic failover, put them behind an internal passthrough Network Load Balancer and point a custom static route (e.g. 10.0.0.0/8) at that LB as its next hop. Only the internal passthrough NLB can be a route next hop — proxy LBs cannot, and two priority routes only give active/standby (not both at once, and no health checking).

5.2 Traffic Management & Backends

Load balancers distribute traffic to backends. GCP supports several backend types:

  • Instance Groups: Managed (MIGs) or Unmanaged. MIGs support auto-scaling based on CPU or load balancer capacity.
  • Network Endpoint Groups (NEGs):
    • Zonal NEGs: Direct routing to GKE Pod IPs (container-native load balancing).
    • Internet NEGs: Routes traffic to external backends located on-premises or in other clouds.
    • Serverless NEGs: Integrates the load balancer with Cloud Run, Cloud Functions, or App Engine.

🚗 Advanced Traffic Management on Application Load Balancers

Global External Application Load Balancers support advanced Layer 7 routing rules configured in URL Maps:

  • Traffic Splitting (Canary Deployments): Route 95% of traffic to production and 5% to a new "canary" release backend service.
  • Traffic Mirroring: Copy 100% of live production traffic and send it to a test environment in the background to profile performance without affecting users!
  • URL Rewrites & Redirects: Automatically rewrite incoming request paths (e.g., /images/logo.png to /static/logo.png) or redirect HTTP to HTTPS at the edge.

5.3 Cloud CDN: Speeding Up the Edge

Cloud CDN (Content Delivery Network) caches static content at Google's global edge co-location facilities (Points of Presence - PoPs), bringing content closer to users and reducing load on your backends.

  • Supported Origins: Managed Instance Groups, Cloud Storage buckets, Cloud Run, and external backends (via Internet NEGs).
  • Cache Invalidation: If you update an image or file, you don't want users seeing the old cached version. You can trigger a Cache Invalidation command to purge the cached content globally. Warning: Purging the entire cache (/) can overload your origin databases and servers as they get hit with a flood of new requests!*

5.4 Cloud DNS: The Global Address Book

Cloud DNS is a highly available, global managed DNS service. Let's look at the advanced routing policies that examiners absolutely love to test:

🗺️ Advanced DNS Routing Policies

Cloud DNS allows you to configure dynamic traffic steering on your resource record sets:

  • Weighted Round Robin (WRR): Distributes traffic across multiple IP addresses based on custom weights (e.g., sending 80% of queries to IP A and 20% to IP B). Excellent for canary testing or capacity balancing.
  • Geolocation (GEO): Directs users to the nearest healthy region.
    • Geofencing (--enable-geo-fencing): If enabled, traffic is strictly restricted to that region. If all endpoints in that region are unhealthy, queries will not failover to other regions (useful for strict data residency compliance). If disabled (default false), Cloud DNS will failover and route queries to the next nearest healthy region! 🌍
  • Failover: Configures a primary IP (like an active load balancer) and a backup IP (like a DR site).
    • Trickle Ratio (--backup-data-trickle-ratio): A value between 0 and 1. It sends a tiny, continuous percentage of live traffic to the backup target even when the primary is completely healthy! This is a massive best practice because it ensures your standby/DR environment is actually warm and working under live conditions, rather than failing silently on game day! 💧
"Why did the DNS record set apply a trickle ratio to its failover policy? Because it wanted to keep its backup environment warm, just in case the primary got a cold!" 🥶

5.5 Hybrid DNS Architecture: Resolving Across the Boundary

Routing policies steer traffic within a zone. Hybrid DNS is a different beast: it's about which resolver answers a query when names live on-premises, in another VPC, or in Google Cloud. Three building blocks do all the work — get the direction right and these questions become easy. 🧭

🧭 The Three Building Blocks (and which way they point)

  • Forwarding zone (outbound): A Cloud DNS private zone whose queries are forwarded out to a specific set of on-premises (or external) DNS servers. Scope it to the exact private zones you want resolved on-prem — not everything. This is how Google Cloud VMs resolve on-premises names.
  • DNS peering (VPC → VPC): Lets one VPC hand its queries to another VPC's Cloud DNS configuration. In a hub-and-spoke design, the spokes peer to the hub so they inherit the hub's forwarding zone — you configure forwarding once on the hub instead of in every spoke.
  • Server policy (inbound / outbound): Attached to a VPC.
    • Inbound: creates an inbound forwarder IP so on-premises servers can resolve Google Cloud private names (the reverse direction).
    • Outbound (alternative name servers): sends all queries from the VPC to the listed servers — a blunt instrument, not zone-scoped.

🛰️ The Hub-and-Spoke Forwarding Pattern

Want VMs in every spoke VPC to resolve a specific set of private zones via on-premises DNS? Don't apply a forwarding zone to each spoke (that advertises the same DNS proxy range from every VPC and your on-prem router can't tell who asked → dropped queries). Instead:

  • Create the forwarding zone on the hub VPC only, pointing at the on-prem DNS servers.
  • From each spoke, create a DNS peering zone to the hub.

One forwarding config, scoped to the zones you choose, inherited by every spoke.

The 35.199.192.0/19 Trap! For outbound resolution (Google Cloud → on-prem), forwarded DNS queries leave Google from the source range 35.199.192.0/19. You must advertise that range to on-premises via Cloud Router and allow it through on-prem firewalls, or the queries never arrive. The lookalike 169.254.169.254 is the metadata DNS server — wrong answer. (And remember: an inbound server policy is for the opposite direction, on-prem resolving Google Cloud names.)

🧠 Quick-Fire Warm-Up Quiz!

Test your recall before diving into the proper practice exam. No peeking!

Q1: Which load balancer type should you use to preserve the client's original source IP address natively?

Passthrough Network Load Balancer (NLB). Because it is not a proxy, packets are received by backend VMs with unchanged source and destination IP addresses.

Q2: What happens in a Geolocation DNS routing policy if geofencing is enabled (--enable-geo-fencing) and all endpoints in the nearest region fail health checks?

Queries will still be resolved to that region! Geofencing prevents traffic from failing over to other regions, prioritizing data residency compliance over availability. If disabled (default), it would fail over to the next nearest healthy region.

Q3: What is the purpose of the --backup-data-trickle-ratio flag in a Cloud DNS Failover routing policy?

It sends a tiny percentage of live traffic to the backup/standby target while the primary is healthy. This continuously tests and warms the DR environment so you know it works before an actual disaster occurs.

Q4: Can you use a Global External Application Load Balancer with Standard Tier network routing?

No! Standard Tier only supports regional load balancers. Global load balancing strictly requires Premium Tier routing.

📝 Theme 5 Practice Exam Questions

Let's see if you can handle these actual exam-style load balancing and DNS routing brain busters!

Question 1: A global web app needs path-based routing (/api to a backend pool, /static to Cloud Storage), TLS termination at the edge, and traffic mirroring for background testing. Which load balancer should you deploy?

  • A) Global External Proxy Network Load Balancer with TCP SSL offloading and a URL-map header-routing rule for the path splits.
  • B) Internal passthrough Network Load Balancer paired with Cloud NAT doing SNAT to provide the public path and traffic mirroring.
  • C) Global External Application Load Balancer, which provides URL-map path routing, SSL offload, and traffic mirroring at the edge.
  • D) Regional External Passthrough Network Load Balancer fronting an internal Application Load Balancer transit layer in each region.

Correct Answer: C

Why Option C is correct: The Global External Application Load Balancer is the L7 product. URL maps give path-based routing, it terminates TLS (SSL offload), and it supports traffic mirroring natively at Google's edge. 🌐

  • A: Proxy NLB is Layer 4 (TCP) — it can't parse HTTP paths or URLs, so no path routing or URL maps.
  • B: Internal passthrough NLB is private and L4; Cloud NAT doesn't turn it into a public global L7 with SSL termination.
  • D: Passthrough NLB is L4 and regional; stacking an internal ALB behind it is needlessly complex and breaks single-point SSL offload.

Question 2: You run active-passive DR with a primary Application Load Balancer in us-east1 and a standby in europe-west1. Cloud DNS must auto-route users to the standby if the primary fails, and you also want a steady 1% of live traffic hitting the standby so it stays warm and verified. How should you configure Cloud DNS?

  • A) A FAILOVER routing policy with the primary on the us-east1 LB and the backup a GEO policy to europe-west1, health checks on, and --backup-data-trickle-ratio set to 0.01.
  • B) A Geolocation policy with geofencing on, the European LB as a dynamic fallback, and a Cloud Run script polling the primary region's health.
  • C) A Geolocation policy whose primary and passive weights a Cloud Scheduler job rewrites hourly, plus a separate weighted CNAME that sends 1% of queries over.
  • D) A Weighted Round Robin policy with equal weights on both LBs and DNS health checks disabled on the passive region to avoid traffic shifting.

Correct Answer: A

Why Option A is correct: The FAILOVER policy auto-switches to the backup on health-check failure, and --backup-data-trickle-ratio 0.01 continuously sends exactly 1% of queries to the standby — keeping it warm and verified under live load. 💧

  • B: Geolocation doesn't do active-passive failover with a trickle ratio, and an external Cloud Run poller isn't wired into Cloud DNS's automatic failover.
  • C: Hourly Cloud Scheduler reweighting is slow and lacks sub-minute, health-based failover.
  • D: Equal weights with no health checks is active-active — it sends ~50% to the passive site and keeps sending even when it's unhealthy.

Question 3: A multiplayer game backend on Compute Engine talks UDP and needs very low latency, and the servers must see each player's original public IP for matchmaking. Which load balancer should you choose?

  • A) Global External Application Load Balancer with a Proxy Protocol v2 header to carry the client IP to the instance group.
  • B) Regional External Passthrough Network Load Balancer, which forwards UDP unchanged so backends see the real client IP.
  • C) Global External Proxy Network Load Balancer with SSL session-ticket resumption to keep UDP latency low.
  • D) Regional Internal Application Load Balancer with Private Service Connect endpoints in each regional subnet.

Correct Answer: B

Why Option B is correct: The passthrough Network Load Balancer is L4 and doesn't terminate the connection; packets reach backends with the original source IP intact, and it supports UDP — exactly what low-latency matchmaking needs. ⚡

  • A: ALBs handle HTTP/S only (not raw UDP) and terminate the connection; Proxy Protocol is a proxy-only feature.
  • C: Proxy NLB terminates the connection and doesn't natively pass raw UDP or the unmodified client IP.
  • D: Internal LBs aren't reachable from the public internet, and PSC is for private service consumption, not public game clients.

Question 4: A Geolocation DNS policy sends users to the nearest of three regional Application Load Balancers. For GDPR, European users must always be routed to the European LB and must never fail over to a US LB — even if the European LB is fully down. How should you configure this?

  • A) A standard Geolocation policy with no extra flags, plus IAM rules to stop US instances from resolving the European IP ranges.
  • B) A Weighted Round Robin policy with 100% weight on Europe and 0% on the US, backed by a Cloud DNS forwarding zone.
  • C) Disable health checks on the European LB and enable global Cloud CDN so US edges serve European responses during an outage.
  • D) A Geolocation policy with geofencing enabled (--enable-geo-fencing) on the European record set.

Correct Answer: D

Why Option D is correct: Geofencing pins queries from a region to that region's endpoints even when they're all unhealthy — no cross-jurisdiction failover, which is exactly what the data-residency rule requires. 🌍

  • A: Without geofencing, Geolocation fails European users over to the US on an outage, and IAM can't control public DNS resolution.
  • B: WRR ignores the user's location, so it can't keep European users in Europe or US users out.
  • C: Disabling health checks doesn't enforce a geofence, and caching European data in US edges breaks residency.
