Section 2: Communications & Boundary Protection

Perimeter controls, network segmentation, and private connectivity (~22%). Keep the good traffic flowing and the bad traffic — and your data — firmly on the right side of the fence.

Section 2 🔵 ~22%

Section 1 decided who gets in. Section 2 decides where they can go once inside and how data is allowed to move. This domain is a tale of two perimeters: the classic network perimeter (firewalls, load balancers, private links) and the modern API data perimeter (VPC Service Controls). Mix those two up and you'll lose easy marks — so let's draw the fence lines clearly. 🚧

2.1 The Edge Defenders

Several services guard the front door of your applications. Each owns a distinct job, and the exam tests the boundaries between them.

Defender Operates At Its One Job
Cloud Armor L7, in front of the external load balancer WAF rules, DDoS protection, IP/geo allow/deny, rate limiting. Inbound web traffic only.
Identity-Aware Proxy (IAP) L7, identity layer Identity- and context-aware access to apps/SSH/RDP — no VPN.
Cloud NGFW L3–L7 network firewall Stateful rules, FQDN/geo objects, and (Enterprise) L7 intrusion prevention.
Secure Web Proxy Egress, L7 Filters and logs outbound internet traffic by URL/host.
Certificate Authority Service PKI Run a private CA to issue and manage internal certificates.
Cloud Armor is inbound; Secure Web Proxy is outbound. If the stem says "filter what our VMs can reach on the internet and log every request," that's Secure Web Proxy egress filtering — not Cloud Armor, which only protects inbound HTTP(S) at the load-balancer edge.

2.2 The Three Firewall Layers

GCP firewalling comes in three layers, and a favourite exam discriminator is which one can a project owner not delete.

Layer Attaches To Who Controls It
VPC firewall rules A single VPC network; scoped by tags/service accounts Anyone with network-admin on that project — editable, not a guardrail.
Hierarchical firewall policies Organization or folder node; inherit downward Central security team. Project owners cannot remove them. Use goto_next to delegate the rest.
Cloud NGFW Enterprise Network firewall policies + L7 inspection Adds intrusion prevention, TLS inspection, and threat intelligence.

💡 Hierarchy = Handcuffs

A hierarchical firewall policy at the org/folder is a handcuff the project owner can't pick. A plain VPC firewall rule is a sticky note — anyone with network-admin can peel it off. "Owners cannot remove it" ⇒ go hierarchical.

2.3 VPC Service Controls — The Anti-Exfiltration Perimeter

This is the single most-tested service in the whole domain, so internalise it. VPC Service Controls (VPC-SC) draws a perimeter around Google-managed, API-based services (Cloud Storage, BigQuery, Pub/Sub, and friends). Inside the perimeter, data can move freely; crossing the boundary is blocked — even for an identity with perfectly valid IAM permissions.

🛡️ Why VPC-SC Is Special

IAM answers "are you allowed?" VPC-SC answers "are you inside the box?" A stolen credential with legitimate Storage access still can't copy a bucket to a project outside the perimeter, because the request fails at the API layer, not the IAM layer. That's the defence against the classic exfiltration attack: valid creds, wrong destination.

Firewalls don't stop serverless exfiltration. BigQuery and Cloud Storage calls go to Google's API frontends, not arbitrary internet egress, so a VPC firewall or subnet egress rule won't reliably block an authorised API-to-API copy. To stop data leaving a serverless / API product, you need VPC Service Controls. Full stop.
"Our data tried to escape the perimeter. VPC Service Controls gave it a stern look and said: 'You're not on the list.'" 📋🚪

2.4 Private Connectivity & Controlled Egress

These options all keep traffic off the public internet, but they solve different problems. The exam rewards you for telling the pipes apart.

Option Problem It Solves Tell-Tale Phrase
Private Google Access A no-external-IP VM reaching Google APIs over internal routes. "VM has no public IP but must call the Storage/BigQuery API."
Private Service Connect (PSC) Consuming a Google or published service via a private endpoint IP in your VPC. "Reach a managed service through a private endpoint inside our network."
HA VPN / Cloud Interconnect Private connectivity between on-prem and GCP. "Connect the data centre to the VPC privately / with an SLA."
Cloud NAT Outbound internet for private VMs with no inbound exposure. "Private VMs need to download updates but accept no inbound."
Secure Web Proxy Policy-based, logged egress filtered by hostname/URL. "Allow only approved domains and log every outbound request."

💡 PERIMETER vs PIPE

VPC Service Controls is a perimeter around API data. PSC, Private Google Access, VPN, and NAT are private pipes for connectivity. When the question is about exfiltration, reach for the perimeter; when it's about reaching something privately, reach for a pipe. Don't plumb a pipe at an exfiltration problem.

🧠 Quick-Fire Warm-Up Quiz!

Retrieve before you review. Cover the answers!

Q1: Prevent a user who has legitimate BigQuery access from exporting a dataset to an outside project. Which control?

A VPC Service Controls perimeter. It blocks data egress from in-scope services across the perimeter boundary regardless of IAM.

Q2: Give employees identity-based access to an internal web app from anywhere, with no VPN. What do you use?

Identity-Aware Proxy (IAP) — context-aware, identity-based access to apps without a VPN.

Q3: Security must enforce a deny-all baseline firewall rule across every project that owners cannot delete. Which feature?

A hierarchical firewall policy at the organization or folder node. It inherits downward and is managed centrally, so project network admins cannot remove it.

Q4: A VM with no external IP must reach the Cloud Storage API over Google's network. What do you enable on its subnet?

Private Google Access. It routes requests to Google APIs internally, so no public IP is required.

📝 Section 2 Practice Exam Questions

Left-click to select, right-click to strike out a distractor, then Submit for the full diagnostic.

Question 1: A compromised service account inside a project has valid IAM permissions on a Cloud Storage bucket and is attempting to copy the bucket's contents to a personal project in another organization. You must stop the copy even though the IAM grant is legitimate. What blocks it?

  • A) A VPC firewall egress rule denying outbound traffic to the internet from the affected subnet.
  • B) Removing the Storage Object Viewer role from the service account on the bucket.
  • C) A VPC Service Controls perimeter around the project that restricts Cloud Storage egress.
  • D) A Cloud Armor security policy applied in front of the storage endpoint.

Correct Answer: C

Why C is correct: VPC Service Controls enforces a perimeter at the API layer, so a copy from in-perimeter Cloud Storage to a project outside the perimeter is denied regardless of the identity's valid IAM permissions.

Why the others are wrong:
A: Storage API calls go to Google's API frontends, not arbitrary internet egress, so a subnet egress rule doesn't reliably stop an authorised API-to-API copy.
B: The grant is described as legitimate and needed; revoking it breaks normal use and an attacker with IAM-edit could restore it.
D: Cloud Armor protects inbound HTTP(S) load-balanced applications, not outbound storage API exfiltration.

Question 2: An internal HR application runs on Compute Engine and must be reachable by employees from anywhere, but only after their identity and device posture are verified, and you want to retire the corporate VPN. Which design meets this?

  • A) Put the app behind Identity-Aware Proxy with an access level that checks device posture and group membership.
  • B) Publish the app on an external load balancer protected by a Cloud Armor IP allowlist of office ranges.
  • C) Keep the VPN but add a hierarchical firewall policy restricting the app's port to internal ranges.
  • D) Expose the app through Private Service Connect so only endpoints inside the VPC can reach it.

Correct Answer: A

Why A is correct: IAP combined with an Access Context Manager access level gives identity-aware, device-aware, VPN-less access from any network — the BeyondCorp model the question describes.

Why the others are wrong:
B: An IP allowlist binds access to office networks, which fails the "from anywhere" requirement and checks neither identity nor device.
C: This keeps the VPN you were asked to retire and does no identity/device verification.
D: PSC restricts reachability to inside-VPC endpoints, the opposite of letting verified employees connect from anywhere.

Question 3: VMs in a locked-down subnet have no external IPs but must download OS patches from a specific set of vendor domains on the public internet, and security wants every outbound request logged and filtered by hostname. Which control fits best?

  • A) Cloud NAT on the subnet so the VMs can reach the patch servers via a managed egress IP.
  • B) Private Google Access so the VMs can resolve and reach the external vendor endpoints.
  • C) Secure Web Proxy with a policy allowing only the approved vendor hostnames and logging all egress.
  • D) A VPC Service Controls perimeter listing the vendor domains as allowed egress destinations.

Correct Answer: C

Why C is correct: Secure Web Proxy applies URL/host-based egress policy and logs every request, exactly matching "filtered by hostname and logged."

Why the others are wrong:
A: Cloud NAT provides egress but applies no hostname filtering or per-request application logging.
B: Private Google Access only reaches Google APIs, not arbitrary external vendor domains.
D: VPC-SC governs access to Google-managed API services, not generic internet hostnames for patch downloads.
