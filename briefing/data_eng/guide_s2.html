Section 2~25%

Ingesting & Processing the Data

The largest section: plan pipelines, build them with the right services, and operationalise them.

Choosing the processing service

  • Dataflow (Apache Beam) — unified batch + streaming, autoscaling, the default for new pipelines.
  • Dataproc — managed Spark/Hadoop, best for lift-and-shift of existing Spark jobs.
  • Cloud Data Fusion — GUI-based, code-free ETL.
  • Pub/Sub — ingestion and decoupling for streaming.
⚠️ Exam Trap: Existing Spark/Hadoop jobs → Dataproc. New unified batch+streaming → Dataflow. A code-free GUI requirement → Data Fusion.

Streaming windows & late data

In streaming, group events with windows: tumbling (fixed, non-overlapping), hopping/sliding (overlapping, e.g., last 30s every 2s), and session (activity-gap based). Handle late-arriving data with watermarks and triggers. Orchestrate jobs with Cloud Composer (Airflow) or Workflows; add CI/CD.

⚠️ Exam Trap: "Aggregate the last 30 seconds, every 2 seconds" describes a hopping (sliding) window, not tumbling or session.

Pub/Sub ingestion & delivery semantics

Pub/Sub decouples producers from consumers and absorbs spikes. Know its guarantees and knobs:

  • At-least-once delivery by default — design consumers to be idempotent (or enable exactly-once delivery where supported).
  • Ordering keys preserve order per key when you actually need it (it costs throughput).
  • Dead-letter topics capture messages that repeatedly fail, so a poison message can't block the stream.
  • BigQuery / Cloud Storage subscriptions land data directly with no custom code.
  • Pub/Sub Lite trades features for lower cost at very high, predictable throughput.
⚠️ Exam Trap: A few repeatedly-failing messages stalling a subscription = configure a dead-letter topic, not infinite retries.

🧠 Warm-Up Quiz

Q1. Migrate existing on-prem Spark jobs with minimal changes. Which service?

Reveal answer

Dataproc.

Managed Spark/Hadoop; lift-and-shift friendly.

Q2. Real-time supply/demand aggregated over the last 30s computed every 2s. Which window?

Reveal answer

Hopping (sliding) window.

Overlapping windows of fixed size at a fixed period.

Q3. A handful of malformed messages keep failing and blocking a Pub/Sub subscription. What do you configure?

Reveal answer

A dead-letter topic.

After N delivery attempts, the message is moved aside so the stream keeps flowing.

Q4. A new pipeline must run the same logic for both nightly batch and real-time streams. Which engine?

Reveal answer

Dataflow (Apache Beam).

Beam's unified model runs the same pipeline in batch and streaming with autoscaling.

📝 Exam-Style Practice

Q1. A team has years of on-prem Apache Spark jobs and wants to move them to Google Cloud with the least rewriting. Which managed service should they target?

  • A) Dataflow, rewriting the jobs as Apache Beam pipelines.
  • B) Dataproc, running the existing Spark jobs as-is on a managed cluster.
  • C) Cloud Data Fusion, rebuilding the jobs in its visual designer.
  • D) BigQuery, porting the transformations to SQL.
Reveal answer & explanation

Correct: B. Dataproc is managed Spark/Hadoop, so existing Spark jobs lift-and-shift with minimal change.

A: Dataflow would require rewriting to Beam. C: Data Fusion means rebuilding visually. D: Porting Spark to SQL is a full rewrite.

Q2. A dashboard must show a metric aggregated over the last 30 seconds, recomputed every 2 seconds. Which streaming window type does this require?

  • A) Tumbling (fixed, non-overlapping) windows.
  • B) Session windows keyed on user activity.
  • C) Hopping (sliding) windows of 30s every 2s.
  • D) Global windows with a single final trigger.
Reveal answer & explanation

Correct: C. Overlapping windows of fixed size (30s) emitted at a fixed period (2s) are hopping/sliding windows.

A: Tumbling windows don't overlap, so they can't recompute every 2s over a 30s span. B: Session windows are gap-based, not periodic. D: A global window doesn't produce rolling 30s aggregates.

Q3. A streaming pipeline occasionally receives malformed events that fail processing and are retried forever, stalling the subscription. What is the best fix?

  • A) Increase the acknowledgement deadline so retries have more time.
  • B) Configure a dead-letter topic after a maximum number of delivery attempts.
  • C) Disable retries entirely so bad messages are dropped silently.
  • D) Switch the subscription from pull to push delivery.
Reveal answer & explanation

Correct: B. A dead-letter topic moves repeatedly-failing messages aside after N attempts, so the stream keeps flowing and the bad data can be inspected.

A: A longer deadline doesn't fix permanently bad messages. C: Silently dropping loses data with no record. D: Push vs pull doesn't address poison messages.
