Section 3~20%

Storing the Data

Select the right store, model the warehouse, and govern the lake.

Storage selection

Match access pattern and consistency: BigQuery (analytics warehouse), Bigtable (high-throughput wide-column/time-series), Spanner (global strongly-consistent relational), AlloyDB/Cloud SQL (OLTP), Firestore (document/mobile), Cloud Storage (objects/lake), Memorystore (in-memory cache).

⚠️ Exam Trap: Billions of time-series rows with high write QPS and millisecond key lookups = Bigtable, not BigQuery (which is for scans/aggregations).

Warehouse modelling & the lake

Model BigQuery for access patterns — denormalise/nest for analytics, use partitioning and clustering. Govern a lake/lakehouse with Dataplex (discovery, lineage, quality) and BigLake, building a federated governance model across distributed data.

BigQuery cost & performance

BigQuery bills mostly on bytes scanned (on-demand) or reserved slots (capacity). Cut both cost and latency:

  • Partition tables (by date/ingestion time) so queries prune to the needed days.
  • Cluster on the columns you filter/sort by to skip blocks.
  • Select only the columns you need — SELECT * scans everything.
  • Use materialized views / BI Engine for repeated aggregations.
  • Choose editions/slot reservations for steady, heavy workloads; on-demand for spiky/ad-hoc.
⚠️ Exam Trap: "Reduce cost of a frequently-run dashboard query that scans a huge table" = partition + cluster (and maybe a materialized view), not a bigger slot reservation.

🧠 Warm-Up Quiz

Q1. Serve billions of IoT readings with very high writes and ms lookups by device+time. Which store?

Reveal answer

Bigtable (with a good row key).

Built for high-throughput low-latency wide-column time-series.

Q2. Govern a decentralised data ecosystem with discovery, lineage, and quality. Which service?

Reveal answer

Dataplex.

Unified data management/governance across distributed data.

Q3. A dashboard query always filters by event date but scans the whole multi-TB table. Cheapest fix?

Reveal answer

Partition the table by date (and cluster on common filters).

Partition pruning means the query reads only the relevant days, slashing bytes scanned.

Q4. A device row key is the current timestamp, causing one node to take all writes. What's wrong?

Reveal answer

Hotspotting from a monotonic row key.

Sequential keys send all writes to one tablet; prefix with device ID (field promotion) to spread load.

📝 Exam-Style Practice

Q1. A fleet of sensors writes billions of readings per day and clients need millisecond lookups by device and time range, at very high write throughput. Which store fits best?

  • A) BigQuery, querying the readings table per lookup.
  • B) Bigtable with a row key combining device ID and reversed timestamp.
  • C) Cloud SQL with an index on device and timestamp.
  • D) Firestore documents per reading.
Reveal answer & explanation

Correct: B. Bigtable is built for high-throughput, low-latency wide-column time series; a well-designed row key gives millisecond device+time lookups.

A: BigQuery excels at scans/aggregations, not per-key millisecond lookups at high QPS. C: Cloud SQL won't sustain that write throughput or scale. D: Firestore isn't sized for billions of high-rate time-series writes.

Q2. A heavily-used BigQuery dashboard query filters on event_date but repeatedly scans an entire 8 TB table, driving up on-demand cost. What change reduces cost most directly?

  • A) Switch the query to SELECT * to leverage caching.
  • B) Partition the table by event_date and cluster on the common filter columns.
  • C) Buy a much larger slot reservation.
  • D) Export the table to Cloud Storage and query it as an external table.
Reveal answer & explanation

Correct: B. Partitioning lets the query prune to the relevant dates and clustering skips blocks — both cut bytes scanned, which is what on-demand bills.

A: SELECT * scans more, not less. C: A bigger reservation adds cost without reducing bytes scanned. D: External tables on GCS are usually slower and don't inherently prune like native partitioning.

Q3. An organisation has data spread across many BigQuery datasets and Cloud Storage buckets owned by different teams and needs unified discovery, lineage, and data-quality governance. Which service should anchor this?

  • A) Dataplex for unified governance across the distributed data.
  • B) A spreadsheet catalogue maintained by each team.
  • C) IAM groups per dataset with no central catalogue.
  • D) Cloud Logging sinks aggregating access logs.
Reveal answer & explanation

Correct: A. Dataplex provides discovery, lineage, and quality across distributed lakes and warehouses — federated governance without centralising the data.

B: Manual spreadsheets don't scale or stay current. C: IAM controls access but isn't a catalogue/lineage tool. D: Logging tracks access events, not data discovery or quality.
