Section 2~23%

Building & Testing Applications

Set up a productive dev loop, build containers reproducibly, and automate testing.

Local development & emulators

Use local emulators (Firestore, Pub/Sub, Spanner, Bigtable) via the gcloud CLI for offline unit tests — never hit billable services in CI. Cloud Workstations give a managed, secure cloud IDE; Cloud Code and Gemini Cloud Assist bring GCP tooling and AI assistance into your editor.

⚠️ Exam Trap: For unit tests against Firestore/Pub/Sub, the recommended approach is the local emulator, not a real or shared project.

Build pipeline & provenance

Cloud Build runs CI steps in containers; store images and language packages in Artifact Registry (the successor to Container Registry). Generate build provenance (a signed attestation of how an image was built) in Cloud Build, and verify it with Binary Authorization so only trusted, attested images can deploy.

⚠️ Exam Trap: "Only images built by our pipeline may run in prod" = Binary Authorization checking attestations — not a naming convention or a manual review.

Reproducible container builds

Build images reproducibly with a Dockerfile, Cloud Native Buildpacks (no Dockerfile needed), or Jib for Java. Pin base images and dependency versions, keep images small and non-root, and scan for vulnerabilities (Artifact Analysis) in the pipeline. Tag by immutable digest, not a moving latest.

BUILD Buildpacks for no-Dockerfile, Jib for Java, Dockerfile for full control — all land in Artifact Registry.

Testing strategy

Follow the test pyramid: many fast unit tests (with emulators for data services), fewer integration tests against real-ish dependencies as Cloud Build steps, and targeted load tests before launch. Gate merges on tests; fail the build on regressions rather than catching them in production.

⚠️ Exam Trap: Running unit tests against a shared real project is wrong — it's slow, flaky, billable, and pollutes data. Use emulators.

🧠 Warm-Up Quiz

Q1. How should CI unit tests exercise Firestore logic without cost or touching prod?

Reveal answer

Use the Firestore emulator from the gcloud CLI.

Local, free, faithful Firestore semantics.

Q2. Where do you store built container images and verify their provenance?

Reveal answer

Artifact Registry, with Binary Authorization checking attestations.

Registry stores images; Binary Authorization gates deployment on trusted provenance.

Q3. A Java team wants reproducible container images without writing or maintaining a Dockerfile. What do you suggest?

Reveal answer

Jib (or Cloud Native Buildpacks).

Both produce optimized, reproducible images straight from source with no Dockerfile.

Q4. You want a secure, consistent cloud development environment for the whole team. Which service?

Reveal answer

Cloud Workstations.

Managed, policy-controlled cloud IDEs that standardise the dev environment.

📝 Exam-Style Practice

Q1. Compliance requires that production can run only container images produced by the official Cloud Build pipeline. What enforces this at deploy time?

  • A) A naming convention requiring all prod images to start with a fixed prefix.
  • B) Binary Authorization policy that requires a Cloud Build attestation before a pod runs.
  • C) IAM restricting who can push to Artifact Registry.
  • D) A manual code-review checklist before each release.
Reveal answer & explanation

Correct: B. Binary Authorization checks a cryptographic attestation at admission, so only images the pipeline built and signed can deploy.

A: Names are trivially spoofed. C: Push permissions limit who uploads but not what runs. D: Manual checklists aren't enforced by the platform.

Q2. Unit tests that hit a shared development Firestore database are slow, flaky, and occasionally corrupt each other's data in CI. What is the recommended fix?

  • A) Give each test its own real Firestore project, created and destroyed per run.
  • B) Run the tests against the Firestore emulator started by the gcloud CLI.
  • C) Add sleeps between tests to avoid contention on the shared database.
  • D) Run the suite only nightly to reduce contention.
Reveal answer & explanation

Correct: B. The local emulator is fast, free, isolated per run, and faithful to Firestore semantics — the standard way to unit-test data logic.

A: Spinning real projects per run is slow and costly. C: Sleeps hide races without fixing isolation. D: Nightly runs delay feedback and don't fix flakiness.

Q3. Deployments sometimes pull a different image than was tested because the team deploys the latest tag. How do you guarantee the exact tested artifact ships?

  • A) Re-tag the image as latest more carefully on each build.
  • B) Deploy by immutable image digest (sha256) rather than a moving tag.
  • C) Rebuild the image at deploy time from the same source.
  • D) Keep only one image in the registry at a time.
Reveal answer & explanation

Correct: B. A digest references one exact image forever, so what you tested is exactly what deploys.

A: latest still moves. C: Rebuilding can produce a different image than the one tested. D: Pruning images doesn't pin which one deploys.
