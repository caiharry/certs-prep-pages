Integrating with Google Cloud Services

Connect to data stores, consume APIs efficiently, and instrument observability.

Efficient API consumption

On retryable errors (HTTP 429/503), retry with exponential backoff and jitter — never a tight loop. Paginate large list results, request only the fields you need, batch requests where supported, and cache idempotent reads. Authenticate API calls with service accounts.

⚠️ Exam Trap: The correct response to 429/503 is exponential backoff; immediate retries amplify the overload.

Connecting to managed data stores

Reach private backends securely. Connect to Cloud SQL via the Cloud SQL Auth Proxy / language connectors (IAM auth, encrypted, no public IP). Let serverless services (Cloud Run, Cloud Functions) reach VPC resources through a Serverless VPC Access connector or Direct VPC egress. Pool connections so you don't exhaust the database under autoscaling.

⚠️ Exam Trap: A serverless service can't reach a private Cloud SQL/VPC resource without VPC egress (connector / Direct VPC) — giving the database a public IP is the wrong fix.

Observability & troubleshooting

Instrument metrics, logs, and traces (OpenTelemetry) into Google Cloud Observability. Cloud Logging centralises structured logs, Cloud Monitoring drives dashboards and alerts on SLOs, Error Reporting groups exceptions, and Cloud Trace stitches spans. Propagate trace IDs so a single request can be followed across services.

LMT Logs (what happened), Metrics (how much/SLOs), Traces (where the time went). The three pillars.

Diagnosing latency & cost in production

Use Cloud Profiler (continuous CPU/heap profiling) to find hot code paths and memory bloat in live services with negligible overhead, and Cloud Trace latency reports to spot slow spans. Set up log-based metrics and alerting policies tied to user-facing SLOs so you're paged on symptoms, not noise.

⚠️ Exam Trap: "Find which function consumes the most CPU/memory in production, continuously" = Cloud Profiler, not adding manual timing logs.

🧠 Warm-Up Quiz

Q1. Your service gets intermittent 429s from a Google API during spikes. Correct client behaviour?

Reveal answer

Retry with exponential backoff and jitter.

Spaces out retries and relieves the overload.

Q2. How do you follow a single request across multiple microservices in Cloud Trace?

Reveal answer

Propagate and correlate by trace ID.

Trace IDs stitch spans across services.

Q3. A Cloud Run service must reach a Cloud SQL instance that has no public IP. What enables this?

Reveal answer

Serverless VPC Access (or Direct VPC egress) plus the Cloud SQL connector.

It gives the serverless service a route into the VPC to reach the private instance.

Q4. You need to find the hottest CPU function in a live service without redeploying instrumentation. Which tool?

Reveal answer

Cloud Profiler.

Continuous, low-overhead production profiling of CPU and memory.

📝 Exam-Style Practice

Q1. During traffic spikes a client receives sporadic HTTP 429 and 503 responses from a Google API. What client behaviour best handles this without making the overload worse?

  • A) Retry immediately in a tight loop until the call succeeds.
  • B) Retry with exponential backoff and jitter, up to a sensible cap.
  • C) Switch to a second project's quota for the same API.
  • D) Cache the failed response and return it to users.
Reveal answer & explanation

Correct: B. Exponential backoff with jitter spaces out retries, relieving the overload and avoiding synchronized retry storms.

A: Tight retries amplify the overload. C: Splitting quota across projects to dodge limits is an abuse pattern, not a fix. D: Caching errors serves users broken data.

Q2. A Cloud Run service must connect to a Cloud SQL instance that has only a private IP, with credentials managed by IAM and no public exposure. What should you configure?

  • A) Give the Cloud SQL instance a public IP and allowlist Cloud Run's egress ranges.
  • B) Use Serverless VPC Access (or Direct VPC egress) with the Cloud SQL connector for IAM auth.
  • C) Open the database to 0.0.0.0/0 and rely on strong passwords.
  • D) Tunnel through a bastion VM that the service SSHes into per request.
Reveal answer & explanation

Correct: B. VPC egress lets the serverless service reach the private instance, and the Cloud SQL connector provides encrypted, IAM-based access with no public IP.

A/C: Public IPs and open firewalls violate the no-exposure requirement. D: Per-request SSH bastions are fragile and not how serverless connects.

Q3. A production service is intermittently slow and you must identify which code path burns the most CPU, continuously and with minimal overhead, without shipping custom timing instrumentation. Which tool fits?

  • A) Cloud Profiler statistical CPU/heap profiling.
  • B) Adding manual stopwatch logs around suspect functions.
  • C) Error Reporting exception grouping.
  • D) A load test that reproduces the spike in staging.
Reveal answer & explanation

Correct: A. Cloud Profiler continuously samples CPU and heap in production at negligible overhead and points directly at the hot functions.

B: Manual logs are exactly the instrumentation you want to avoid and add overhead. C: Error Reporting groups exceptions, not CPU usage. D: A staging load test may not reproduce the real production hotspot.
