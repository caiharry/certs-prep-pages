Section 1.1

Designing the resource hierarchy

The exam wants policy inheritance done right: organize resources, centralize networking, and enforce guardrails that prevent bad resources rather than detecting them after the fact.

WeightSection 1: ~20%
Org / folders / projects Shared VPC & PSC IAM & org policies Service accounts Multi-project observability Data residency

Official guide bullets

  • Organizing resources (application-centric, projects, folders).
  • Shared networking (Shared VPC, VPC Network Peering, Private Service Connect).
  • Multi-project monitoring and logging.
  • IAM roles and organization-level policies.
  • Creating and managing service accounts.
  • Data residency.

What to know

  • Hierarchy: Org → Folders → Projects → resources. IAM and org policies inherit downward, so apply them at the highest level that avoids repetition.
  • Organization Policy Service: constraints such as Resource Location Restriction (residency), domain-restricted sharing, and disabling SA key creation. They block non-compliant creation up front.
  • Shared VPC: a central host project owns the network; service projects attach to it — the answer when one team owns networking.
  • Prefer Workload Identity Federation or attached service accounts over downloaded SA keys.
  • Multi-project observability: use a metrics scope to view metrics across projects, and route logs with aggregated sinks when teams need centralized retention or analysis.
  • Use a separate (sandbox) organization to trial policy changes before touching prod.

Section 1.2

Managing infrastructure as code

Repeatable, identical environments come from declarative IaC — not console clicks, snapshots, or bash scripts. Terraform is the Google-recommended default.

WeightSection 1: ~20%
Terraform Infrastructure Manager Config Connector Cloud Foundation Toolkit GitOps & Helm

Official guide bullets

  • IaC tooling and managed services (Infrastructure Manager, Cloud Foundation Toolkit, Config Connector, GitOps, Terraform, Helm).
  • Making infrastructure changes using Google-recommended practices and blueprints.
  • Automation with scripting (Python, Go).

What to know

  • Terraform is the Google-recommended IaC: declarative, idempotent, repeatable — the answer for identical dev/UAT/prod. Keep state remote (GCS backend); never hand-edit or recreate via the console.
  • GitOps: Git is the source of truth; reconcilers (Config Sync, Argo CD) continuously drive actual state to match.
  • Infrastructure Manager runs Terraform as a managed Google Cloud service. Config Connector makes Google Cloud resources Kubernetes objects. Cloud Foundation Toolkit gives opinionated bootstrap blueprints.

Section 1.3

Designing the CI/CD architecture stack

Know the managed building blocks — Cloud Build for CI, Cloud Deploy for CD, Artifact Registry for artifacts — and where third-party tools slot in.

WeightSection 1: ~20%
Cloud Build (CI) Cloud Deploy (CD) Artifact Registry Skaffold & Kustomize Jenkins / Argo CD / kpt / Packer

Official guide bullets

  • Continuous integration with Cloud Build.
  • Continuous delivery with Cloud Deploy, including Kustomize and Skaffold.
  • Artifact Registry configuration.
  • Widely used third-party tooling (Git, Jenkins, Argo CD, Packer, kpt).
  • Security of CI/CD tooling.

What to know

  • Cloud Build: serverless CI; steps live in cloudbuild.yaml; triggers fire from repos and webhooks. Use private pools + VPC Service Controls for builds with no public IP and exfiltration protection.
  • Cloud Deploy: managed CD. A delivery pipeline has ordered targets (test → staging → prod); you promote one release through them. It uses Skaffold under the hood for render and deploy.
  • Artifact Registry: stores container images and language packages — the successor to the now-shut-down Container Registry. Container Registry shut down Mar 2025
  • Skaffold: build/test/deploy workflow for Kubernetes. Kustomize: template-free manifest overlays per environment. Packer: build VM images. kpt: package and transform Kubernetes config.
  • Cloud Deploy vs GitOps: Cloud Deploy is the managed promotion workflow; Argo CD / Config Sync are reconcilers that keep cluster state aligned with Git.

Section 1.4

Managing multiple environments

Fleets manage GKE at scale; release channels and maintenance controls govern when upgrades happen; canary the risky patches.

WeightSection 1: ~20%
GKE fleets Release channels Maintenance windows / exclusions Ephemeral environments Safe patching

Official guide bullets

  • Managing ephemeral environments.
  • Managing configuration and policy.
  • Managing GKE clusters across an enterprise (fleets).
  • Safe and secure patching and upgrading practices.

What to know

  • Fleets: group GKE clusters for unified config, policy, mesh, and identity (namespace sameness).
  • Connect gateway: lets Google Cloud users and services reach registered fleet clusters, including hybrid and multi-cloud GKE, without exposing the Kubernetes API publicly.
  • Release channels (Rapid / Regular / Stable) trade newness for stability — but the channel alone does not stop node upgrades.
  • Upgrade strategies: surge vs blue/green node-pool upgrades; protect Pods with PodDisruptionBudgets.
  • VM Manager patches fleets of VMs; for a critical patch, canary to a small group with a short duration first.
  • Ephemeral environments: spin up per-PR via IaC, validate, then tear down.
Maintenance exclusion scopeBlocks node upgrades?Use for
No upgrades (default)Yes (all)Freeze everything briefly (e.g. holiday peak)
No minor upgradesNo (patches still apply)Pin the minor version, still get patches
No minor or node upgradesYes (no node evictions)Protect disruption-sensitive workloads from Pod eviction
Exam traps: a legacy GKE workload that can't tolerate evictions → maintenance exclusion (“No minor or node upgrades”). Stable channel alone, max-surge=0, or a 100% PDB do not reliably prevent upgrade-driven evictions. (Note: scopes other than “No upgrades” require the cluster to be enrolled in a release channel.) A critical VM patch that must be fast and cheap → canary rollout, short duration — not all-at-once, not a full A/B copy.

Section 1.5

Enabling secure dev environments

Keeping developer environments consistent and centrally patched is a managed-service problem — not a “trust everyone to update their laptop” problem.

WeightSection 1: ~20%
Cloud Workstations Cloud Shell Custom images / IDE / SDK Gemini Code & Cloud Assist

Official guide bullets

  • Configuring and managing cloud dev environments (Cloud Workstations, Cloud Shell).
  • Bootstrapping environments with required tooling (custom images, IDE, Cloud SDK).
  • Leveraging AI for development and operations (Gemini Code Assist, Gemini Cloud Assist, Gemini CLI).

What to know

  • Cloud Workstations: managed, templated dev environments. A configuration defines the machine type, disk, container image, and tools; changes take effect on the next workstation start — central patching for a whole team.
  • Cloud Shell: an ephemeral browser shell with the SDK preinstalled; great for ad-hoc work, not for standardized team environments.
  • Gemini Code Assist: in-IDE coding help. Gemini Cloud Assist: ops and troubleshooting across Google Cloud (logs, metrics, traces).
Exam trap: “consistent templated dev env + keep security patches current across a team” → Cloud Workstations configurations. Golden snapshots, laptop images, and update checklists all drift because they depend on developers patching themselves.

  • Cloud Shell

Section 2.1

Designing pipelines

Define triggers, artifact flow, deployment targets, and approval gates before building. Approvals are a Cloud Deploy feature; security gating is separate.

WeightSection 2: ~25%
Pipeline triggers Artifact Registry Hybrid / multi-cloud (GKE) Approval flows

Official guide bullets

  • CI/CD of applications and infrastructure.
  • Artifact management with Artifact Registry.
  • Deployment to hybrid and multi-cloud environments (GKE).
  • CI/CD pipeline triggers.
  • Configuring deployment processes (approval flows).

What to know

  • Triggers: push, pull-request, tag, webhook, manual, Pub/Sub, and scheduled events all drive builds and releases.
  • Cloud Deploy approvals: require a human to approve promotion into a sensitive target (e.g. prod). Approval emits a Pub/Sub message ticketing systems can subscribe to.
  • Artifact Registry: one source of truth for images and packages; pin by digest, not the floating latest tag.
  • Hybrid / multi-cloud: deploy to GKE anywhere via fleets and the Connect gateway; use GitOps controllers when the requirement is continuous reconciliation from Git.
Exam signal: “require human approval before production” → Cloud Deploy approval gate. Binary Authorization is a security policy check (what's allowed to run), not the approval/deployment mechanism itself.

Section 2.2

Implementing & managing pipelines

Pick the rollout strategy that matches the risk, prove it in production safely, and keep an audit trail. This is a high-yield drilling area.

WeightSection 2: ~25%
Canary Blue/green Rolling Traffic splitting Feature flags Cloud Audit Logs

Official guide bullets

  • Auditing and tracking deployments (Artifact Registry, Cloud Build, Cloud Deploy, Cloud Audit Logs).
  • Deployment strategies (canary, blue/green, rolling, traffic splitting, feature flags) and defining success metrics from telemetry.
  • Troubleshooting and mitigating deployment issues.

What to know

  • Cloud Run revision tag + --no-traffic: deploy a revision that serves at a tagged URL but receives 0% of user traffic — the clean way to let beta testers hit prod.
  • Tie promotion and rollback to SLO / telemetry success metrics, not gut feel.
  • Cloud Audit Logs are your record of who deployed what, and when.

Section 2.3

Managing config and secrets

Secrets never live in the repo. Secret Manager + IAM is the answer; Workload Identity Federation kills long-lived keys.

WeightSection 2: ~25%
Secret Manager Cloud KMS Workload Identity Federation Certificate / Parameter Manager Build vs runtime injection

Official guide bullets

  • Key management (Cloud Key Management Service).
  • Configuration and secret management (Secret Manager, Certificate Manager, Parameter Manager, Workload Identity Federation).
  • Build versus runtime secret injection.

What to know

  • Secret Manager: versioned secrets, IAM-scoped for least privilege. Cloud Build references them via parameter substitution in build steps.
  • Cloud KMS: manage encryption keys (CMEK, rotation).
  • Workload Identity Federation: let external identities (GitHub Actions, GitLab, Azure DevOps, Terraform Cloud, other clouds, on-prem) impersonate a service account without downloaded keys.
  • Certificate Manager manages TLS certificates; Parameter Manager stores non-secret runtime configuration. Use Secret Manager for credentials and sensitive values.
  • Build vs runtime: prefer injecting secrets at runtime so they aren't baked into image layers.
Exam trap: builds triggered both manually and by webhook need confidential data → Secret Manager + IAM, referenced in the build. Encrypted files in the repo break least privilege; CLI parameters don't work for webhook-triggered builds.

  • Secrets in Cloud Build

  • Cloud KMS

Section 2.4

Securing the deployment pipeline

Scan artifacts, gate deploys to only trusted and attested images, and use Google-curated dependencies instead of scanning the whole world yourself.

WeightSection 2: ~25%
Artifact Analysis Binary Authorization SLSA Assured OSS Per-environment IAM

Official guide bullets

  • Artifact Analysis and vulnerability scanning.
  • Software supply chain security (Binary Authorization, SLSA framework).
  • IAM policies based on environment.

What to know

  • Artifact Analysis: automatic vulnerability scanning of images in Artifact Registry.
  • Binary Authorization: a deploy-time policy — only images with the required attestations (signed and verified) may run. Supports GKE, Cloud Run, Cloud Service Mesh, and GDC. Use dry-run to audit policy impact before enforcing blocks.
  • SLSA: supply-chain integrity levels backed by build provenance.
  • Private pool nuance: Cloud Build private pools need provenance/verification explicitly enabled when you rely on generated attestations for Binary Authorization.
  • Assured OSS: open-source packages built in Google's secured pipelines and continuously scanned and fuzzed (SLSA-3) — the cost-effective way to trust dependencies. Free tier available
  • Tighten IAM per environment; prod is stricter than dev.

Section 3.1

Balancing change, velocity & reliability

SLIs, SLOs, SLAs and the error budget decide whether you ship. No budget left means reliability wins — you don't renegotiate the contract to ship.

WeightSection 3: ~18%
SLI SLO SLA Error budget Nines & opportunity cost

Official guide bullets

  • Defining SLIs (availability, latency), SLOs, and SLAs.
  • Error budgets (Cloud Service Mesh definitions).
  • Opportunity cost of risk and reliability (number of “nines”).

What to know

  • Error budget = 100% − SLO. Budget remaining → ship features. Budget exhausted → freeze features, focus on reliability.
  • Burn-rate alerts: alert on how quickly the service is spending error budget, not just whether an SLI crossed one raw threshold.
  • Cloud Service Mesh: can define service-level telemetry and SLO views for mesh traffic, useful when reliability is measured across service-to-service calls.
  • Don't move the SLA or SLO just to unblock a release.
  • Each extra “nine” costs exponentially more — choose the target by business need, not by instinct.
TermWhat it isAudienceRule of thumb
SLIThe measured metric (e.g. % successful requests)InternalWhat you actually measure
SLOThe internal target (e.g. 99.9%)InternalSet stricter than the SLA
SLAThe external contract, with consequencesCustomerSet looser than the SLO
Exam trap: a 99% SLA already at exactly 99.00% (budget gone) → defer the release to the next period. Never lower the SLA to ship, and remember SLOs are internal — they aren't the customer agreement.

Section 3.2

Managing service lifecycle

Plan capacity with quotas and reservations, autoscale per platform, and pick the cheapest compute that fits the workload's fault tolerance.

WeightSection 3: ~18%
Capacity planning Quotas / reservations Dynamic Workload Scheduler Autoscaling Spot VMs

Official guide bullets

  • Service management (planning, deployment, maintenance, retirement).
  • Capacity planning (quotas, limits, reservations, Dynamic Workload Scheduler).
  • Autoscaling (managed instance groups, Cloud Run, GKE).

What to know

  • Quotas cap usage; reservations guarantee capacity; CUDs commit for discounts; Dynamic Workload Scheduler obtains scarce GPUs/TPUs for flexible, calendar-aware workloads.
  • Autoscaling: MIG autoscaler (CPU/LB/metric), Cloud Run (per-request, scale-to-zero), GKE (HPA/VPA + cluster autoscaler + node auto-provisioning).
  • MIG autohealing repairs unhealthy VMs — a reliability feature, not a cost optimization.
  • Spot VMs: deeply discounted (up to ~91% off) and preemptible — ideal for fault-tolerant, parallel batch.
Exam trap: fault-tolerant, large-scale parallel batch, cheapest → Spot VMs. Sole-tenant nodes, accelerator-optimized + autoscaling, and autohealing MIGs are all more expensive and miss the “cost-effective + fault-tolerant” cue.

Section 3.3

Mitigating incident impact

During an incident: drain or redirect, add capacity, roll back. Afterwards: a blameless postmortem that fixes the system, never the person.

WeightSection 3: ~18%
Drain / redirect traffic Add capacity Rollback Blameless postmortem

Official guide bullets

  • Draining/redirecting traffic.
  • Adding capacity.
  • Rollback strategies.

What to know

  • Drain: stop new traffic to a bad backend at the load balancer and let in-flight requests finish; redirect to a healthy region.
  • Add capacity: scale out the MIG, raise a quota, or shift traffic to spare regions.
  • Rollback: revert to the last known-good revision (Cloud Deploy rollback, or Cloud Run traffic to a prior revision).
  • Blameless postmortem: objectively find the technical root cause and document learnings for everyone.
Exam trap: after an on-call change caused customer impact → blameless postmortem documenting the technical root cause. Policies that blame the individual, add a personal second-reviewer, or ban night-time changes are anti-patterns.

Further reading

  • SRE Book: blameless postmortems

  • Cloud Deploy rollback

Section 4.1

Instrumenting & collecting telemetry

Collect with the Ops Agent and OpenTelemetry; control cost with filters and exclusions; create custom and log-based metrics; probe proactively with synthetic monitors.

WeightSection 4: ~25%
Ops Agent OpenTelemetry Managed Service for Prometheus Log-based metrics Synthetic monitors Log cost controls

Official guide bullets

  • Collecting/importing logs (Ops Agent, OpenTelemetry, Cloud Audit Logs, VPC Flow Logs, Cloud Service Mesh).
  • Optimizing logs (filtering, sampling, exclusions, cost management).
  • Collecting metrics (apps, platform, Managed Service for Prometheus, hybrid/multi-cloud).
  • Creating synthetic monitors to probe endpoints/workflows.
  • Creating custom metrics, including log-based metrics.

What to know

  • Ops Agent: a single agent collecting logs + metrics (and OTLP traces) from Compute Engine — the replacement for the legacy Logging/Monitoring agents.
  • Managed Service for Prometheus: Prometheus metrics at scale, queried with PromQL.
  • OpenTelemetry: vendor-neutral instrumentation for traces, metrics, and logs; use it when you need portable application telemetry rather than VM-level agent collection only.
  • VPC Flow Logs: network telemetry for flows, useful for traffic analysis and security investigations; they are not application logs.
  • Log-based metrics: counter or distribution metrics derived from log entries.
  • Exclusion / sampling filters: cut Logging ingestion cost while still routing what you keep.
  • Synthetic monitors: proactively probe endpoints and user journeys, and can collect log/trace evidence for the probe path.
Exam signal: “reduce Logging cost” → exclusion / sampling filters, not disabling logging. Need to alert on a specific event → log-based metric or alert (see 4.2).

Section 4.2

Managing & analyzing logs

Logs Explorer for ad-hoc queries, Log Analytics for SQL, sinks for export and retention, and log-based alerts for fast, event-driven notifications.

WeightSection 4: ~25%
Logs Explorer Log Analytics Log sinks (BQ / GCS / Pub/Sub) Log-based alerts PII / PHI redaction

Official guide bullets

  • Analyzing logs using Logs Explorer and the Logging query language.
  • Exporting and retaining logs (routing to BigQuery, Pub/Sub, Cloud Storage).
  • Handling sensitive data (log processors to redact PII/PHI).
  • Using Gemini Cloud Assist for AI-powered log analysis.

What to know

  • Log Analytics now branded Observability Analytics: upgrade a log bucket, then run SQL over your logs in-place. Link a BigQuery dataset when you need to join with other BQ data or alert on SQL results — the linked dataset gives read access with no extra ingestion cost.
  • Log-based alerts: alert directly off a log entry (e.g. a v1.compute.instances.stop event) for fast notification.
  • Redact PII/PHI with log processors or Sensitive Data Protection before logs become broadly queryable.
  • Gemini Cloud Assist helps summarize and interpret log evidence; it does not replace retention, routing, access control, or alerting design.

Section 4.3

Metrics, dashboards & alerts

Build alerting policies on SLIs/SLOs, route notifications to the right channel, and watch error-budget burn rate rather than raw single-point thresholds.

WeightSection 4: ~25%
Metrics Explorer Dashboards & PromQL Alerting policies Notification channels PagerDuty / webhooks

Official guide bullets

  • Analyzing metrics using Metrics Explorer.
  • Managing dashboards (creating, filtering, sharing, playbooks, PromQL).
  • Configuring alerting and alerting policies (SLIs, SLOs, cost control).
  • Integrating with third-party alerting tools (webhooks, PagerDuty, Rootly).
  • Leveraging Gemini Cloud Assist for metrics interpretation.

What to know

  • Alerting policy = condition (threshold / metric absence / forecast) + notification channels.
  • Notification channels: email, SMS, Pub/Sub, webhook, Slack, PagerDuty, and incident tools such as Rootly via webhook/integration paths.
  • SLO monitoring: define SLOs in Cloud Monitoring and alert on burn rate to catch budget spend early. Pair a fast-burn alert (short lookback, high threshold, e.g. 10× baseline) with a slow-burn alert (24h lookback, ~2× baseline).
  • Dashboards: PromQL/MQL, filters, sharing, and playbooks attached to alerts for responders.
Exam signal: “page the on-call team in PagerDuty/Slack” → configure a notification channel / webhook on the alerting policy. “Catch budget exhaustion early” → burn-rate alert.

Section 4.4

Capturing & analyzing distributed traces

Trace tells you where time goes across services; the waterfall finds the slow hop; trace IDs link straight back to structured logs.

WeightSection 4: ~25%
Cloud Trace OpenTelemetry Spans & waterfalls Trace ↔ log correlation

Official guide bullets

  • Utilizing tracing frameworks (OpenTelemetry).
  • Analyzing trace waterfalls and spans.
  • Correlating trace IDs with structured logs.
  • Employing Gemini Cloud Assist for trace analysis.

What to know

  • Cloud Trace: distributed latency across services; each span is one hop, and the waterfall exposes the slow segment.
  • Trace ≠ Profiler: Trace = cross-service latency; Profiler = CPU/heap hotspots within a process.
  • Put trace_id in structured logs to jump between a span and its logs.
  • OpenTelemetry is the recommended instrumentation framework.

What to know

  • Network tiers: Standard (cheaper, regional egress) vs Premium (Google's global backbone, lower latency).
  • Recommenders / Active Assist: rightsizing, idle-resource cleanup, CUD recommendations.
  • Reservations vs CUDs: reservations reserve capacity; CUDs buy committed spend/usage discounts. You might need both for predictable high-demand workloads.
  • Logging cost: exclusion / sampling filters.
  • Per-workload: GKE cluster autoscaler + Spot node pools + bin-packing; Cloud Run scale-to-zero and concurrency tuning; Compute Engine rightsizing and idle-resource cleanup.
