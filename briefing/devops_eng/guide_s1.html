Section 1~20%

Bootstrapping & Maintaining an Organization

Lay the foundation: resource hierarchy, shared networking, infrastructure as code, and secure cloud dev environments.

Hierarchy, IaC & blueprints

Organise resources with folders/projects and org-level policies. Manage infrastructure declaratively with Infrastructure Manager, Config Connector, Terraform, and the Cloud Foundation Toolkit blueprints, following GitOps. Automate with scripting (Python, Go).

Fleets & secure dev environments

Manage many GKE clusters as a fleet for consistent config and policy. Provide secure, reproducible developer environments with Cloud Workstations and Cloud Shell, bootstrapped with required tooling and accelerated by Gemini Code Assist / Cloud Assist.

Policy guardrails & config sync

Set guardrails high in the hierarchy and let them inherit. The Organization Policy Service enforces constraints (restrict resource locations, disable service-account key creation, require OS Login, restrict external IPs). For fleets, Config Sync applies a Git repo of config to every cluster, and Policy Controller (OPA Gatekeeper) blocks non-compliant Kubernetes resources at admission — consistent policy as code.

⚠️ Exam Trap: "Stop anyone in the org from creating VMs with external IPs" = an organization policy constraint, not a per-project firewall or manual review.
GUARD Org Policy sets the rules, Config Sync distributes them, Policy Controller enforces them at the cluster door.

🧠 Warm-Up Quiz

Q1. Apply consistent configuration and policy across dozens of GKE clusters. Which concept?

Reveal answer

GKE fleets.

Fleets manage multi-cluster config/policy at enterprise scale.

Q2. Provision repeatable infrastructure declaratively on Google Cloud. Which tools?

Reveal answer

Terraform / Infrastructure Manager (IaC).

Declarative, version-controlled, reproducible.

Q3. Org-wide, no one may create resources outside EU regions. What enforces this?

Reveal answer

An organization policy restricting resource locations.

Org policies inherit down folders/projects and constrain where resources can be created.

Q4. Keep dozens of clusters' Kubernetes config in sync from one Git repo. Which tool?

Reveal answer

Config Sync (with Policy Controller for guardrails).

GitOps reconciliation applies the repo to every fleet cluster continuously.

📝 Exam-Style Practice

Q1. Security mandates that, across the entire organization, no Compute Engine VM may be created with an external IP address, with no per-team exceptions to maintain by hand. What enforces this centrally?

  • A) A firewall rule in each project denying egress to the internet.
  • B) An organization policy constraint disabling external IPs, set at the org node.
  • C) A weekly script that deletes VMs found with external IPs.
  • D) IAM that prevents developers from creating VMs at all.
Reveal answer & explanation

Correct: B. An org policy constraint inherits to every folder and project and blocks creation of VMs with external IPs — central, preventive, no manual upkeep.

A: Firewall rules govern traffic, not whether an external IP is assigned, and would be per-project. C: Deleting after the fact is reactive and disruptive. D: Blocking all VM creation overshoots the requirement.

Q2. A platform team runs 40 GKE clusters and needs identical configuration and admission policies applied to all of them from a single source of truth. What should they use?

  • A) Manually kubectl apply the same manifests to each cluster.
  • B) A fleet with Config Sync distributing a Git repo and Policy Controller enforcing rules.
  • C) A separate Terraform workspace per cluster with copied config.
  • D) One giant cluster instead of 40 to simplify config.
Reveal answer & explanation

Correct: B. A fleet with Config Sync continuously reconciles every cluster to the Git repo, and Policy Controller enforces guardrails at admission — consistent config-as-code at scale.

A: Manual apply drifts and doesn't scale to 40 clusters. C: Copied Terraform per cluster duplicates and drifts. D: Collapsing to one cluster harms isolation and blast radius.

Q3. After repeated outages from inconsistent, hand-built environments, leadership wants every environment provisioned the same way, reviewed before changes apply. What practice should the team adopt?

  • A) A documented console runbook each engineer follows.
  • B) Infrastructure as Code (Terraform / Infrastructure Manager) under GitOps with pull-request review.
  • C) Granting only senior engineers console access.
  • D) Taking project snapshots to roll back after mistakes.
Reveal answer & explanation

Correct: B. IaC under GitOps makes provisioning declarative, reviewed, and repeatable, eliminating drift between environments.

A/C: Manual steps and access limits still allow inconsistent, unreviewed changes. D: Snapshots recover state but don't make provisioning consistent.
