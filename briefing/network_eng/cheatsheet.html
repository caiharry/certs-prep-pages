Section 1: Designing & Planning a Google Cloud VPC Network

Weight: ~21%

Official Guide Objectives

  • Differentiating between Premium and Standard network service tiers.
  • Designing for high availability, failover, disaster recovery, and scale.
  • Designing DNS topologies and choosing appropriate load balancers.
  • Planning GKE networking (secondary ranges, scale potential, control plane access).
  • Planning connectivity to managed services (PSA, PSC, Serverless VPC Access).

High-Yield Facts to Memorize

  • Network Tiers: Premium Tier uses Google's global fiber network with cold-potato routing. Standard Tier uses the public internet with regional hot-potato routing. Global load balancing is strictly Premium Tier only!
  • VPC MTU: VPC MTU can be configured from 1300 to 8896 bytes (Jumbo frames). Default is 1460 (legacy) or 1500 (standard). Every subnet inherits the VPC-level MTU.
  • GKE IPAM: GKE VPC-Native clusters require three IP ranges: Primary (Nodes), Secondary A (Pods), and Secondary B (Services). GKE allocates a fixed /24 block (256 IPs) per node for Pods by default.
  • Shared VPC IAM: Grant roles/compute.networkUser to the GKE Service Agent or VM deployment service accounts in the Host Project (either at the project level or bound to specific shared subnets).

⚠️ The GKE IP Exhaustion Trap

If you configure your Pod secondary range too small, your GKE cluster will fail to scale up node pools even if you have plenty of Compute Engine VM quota! Always calculate your Pod range based on the maximum anticipated nodes × 256 IPs per node.

Alias-IP Sizing & Immutability

  • Pods range = nodes × 256 → round up to a power of two. 100 nodes = 25,600 IPs → /17 (32,768).
  • Services range = number of ClusterIP services. ~1,500 services → /21 (2,048).
  • Immutable: Pod/Service secondary ranges cannot be resized after cluster creation — size for projected growth up front. "Start small, resize later" is always a trap.

Section 2: Implementing a VPC Network

Weight: ~20%

Official Guide Objectives

  • Creating VPC resources, subnets, and configuring VPC Network Peering.
  • Configuring Private Google Access (PGA) and VPC Service Controls (VPC-SC).
  • Setting up static, dynamic, and policy-based routing (PBR).
  • Configuring Network Connectivity Center (NCC) spokes and topologies.
  • Enabling GKE Dataplane V2, network policies, and DNS-based endpoints.

High-Yield Facts to Memorize

  • VPC Peering: Peerings are strictly non-transitive. To share hybrid routes (VPN/Interconnect) across peerings, you must explicitly enable "Export Custom Routes" on the hosting peering link and "Import Custom Routes" on the peer.
  • DNS-Based Endpoint: Enabled via --enable-dns-access. Creates a unique FQDN (e.g. HASH.gke.goog) resolving to Google's secure global API frontend. Bastion-free control-plane access to private clusters from other spokes / on-prem / other clouds — sidesteps non-transitive peering, gated by IAM (container.clusters.connect) + VPC Service Controls (not IP authorized networks). For a private-only path, reach it via PSC / VPN / Interconnect with a gke.goog DNS record.
  • Dataplane V2: eBPF-based (Cilium). Enforces Kubernetes Network Policies natively in the Linux kernel, bypassing slow, sequential iptables rules for blistering speed and massive scale.
  • Policy-Based Routing: Routes traffic based on source IP, destination IP, protocol, and network tags. Can target an Internal Load Balancer (ILB) as the next hop to distribute traffic across a pool of NVA firewalls.

⚠️ The Non-Transitive Peering Wall

Traffic coming from on-premises via VPN or Interconnect cannot automatically cross a VPC Peering boundary to talk to a peered VPC or a PSA-provisioned database. You must configure the peering to export/import custom routes and advertise the peer ranges via BGP on your Cloud Router!

VPC Peering Routing Exchanging custom routes across peered VPC boundaries. GKE DNS-Based Endpoints Secure, bastion-free control plane access for private GKE clusters.

Section 3: Configuring Managed Network Services

Weight: ~16%

Official Guide Objectives

  • Determining the correct load balancing solution (L4 vs L7, proxy vs passthrough, internal vs external).
  • Configuring backend services, health checks, session affinity, and NEGs.
  • Setting up traffic management (traffic splitting, mirroring, URL rewrites).
  • Configuring Cloud CDN origins, external backends, and cache invalidation.
  • Managing Cloud DNS zones, records, and routing policies (geolocation, failover, weighted).

High-Yield Facts to Memorize

  • Load Balancer Taxonomy:
    • Application Load Balancers (L7): HTTP/S only. Supports URL maps, traffic splitting, mirroring, and CDN.
    • Proxy Network Load Balancers (L4): TCP/SSL. Terminates connections. Supports single global IP and SSL offloading.
    • Passthrough Network Load Balancers (L4): TCP/UDP. Does not terminate connections. Preserves client source IP address natively.
  • Cloud DNS Failover Policy: Supports primary/backup IPs. Backup must use GEO. Use --backup-data-trickle-ratio (0 to 1) to continuously route a tiny percentage of live traffic to the DR site to ensure it is warm and functioning!
  • Cloud DNS Geofencing: Enabling --enable-geo-fencing on a Geolocation policy restricts traffic to that region even if all endpoints there are unhealthy (guarantees data residency compliance like GDPR).
  • Container-Native LB: Always use Network Endpoint Groups (NEGs) in GKE to route traffic from the load balancer directly to Pod IPs, bypassing the slow kube-proxy node hop.

Load Balancer Capability Discriminators

  • Port translation (frontend port ≠ backend port): proxy LBs only. A passthrough NLB delivers to backends on the same port — it cannot map 700→800. (IPv6 is not the discriminator anymore: modern backend-service passthrough NLBs support IPv6/dual-stack.)
  • Route next hop (for NVA high-availability): internal passthrough NLB only. Point a static route (e.g. 10.0.0.0/8) at the ILB to use both appliances actively with health-checked failover.
  • Proxy-only subnet (REGIONAL_MANAGED_PROXY): a prerequisite for internal/regional Application LBs and proxy NLBs. Firewall rules and reserved IPs can come after.
  • TLS passthrough + global access (latency-sensitive, multi-region private): internal passthrough NLB — the internal ALB terminates TLS and can't pass it through.

Hybrid DNS Architecture Picker

  • Forwarding zone (outbound): Google Cloud → on-prem resolution, scoped to specific private zones.
  • DNS peering (VPC → VPC): hub-and-spoke pattern — put the forwarding zone on the hub only, peer each spoke to it (don't repeat forwarding in every VPC).
  • Server policy: inbound = forwarder IP so on-prem resolves Google Cloud names; outbound alternative name servers = send all VPC queries out (not zone-scoped).
  • Advertise 35.199.192.0/19 to on-prem (via Cloud Router + firewall) for outbound forwarded queries. 169.254.169.254 is the metadata DNS server — a trap.

⚠️ The Client IP Preservation Trap

If backend servers must see the original public IP address of the client natively in the packet header, you cannot use proxy-based load balancers (ALB or Proxy NLB). You must deploy a Passthrough Network Load Balancer!

Section 4: Configuring Hybrid & Multicloud Network Interconnectivity

Weight: ~16%

Official Guide Objectives

  • Configuring Dedicated, Partner, and Cross-Cloud Interconnect.
  • Implementing 99.9% and 99.99% SLAs for Interconnect topologies.
  • Configuring HA VPN toward on-premises gateways and other VPCs.
  • Implementing BGP attributes (ASN, MED, AS-path prepend, link-local, authentication).
  • Configuring Bidirectional Forwarding Detection (BFD) and custom routes.

High-Yield Facts to Memorize

  • 99.99% Interconnect SLA: Requires a multi-metro topology (two physical connections in Metro A, and two physical connections in Metro B) with four VLAN attachments routed to two separate Cloud Routers in your VPC.
  • HA VPN: Provides a 99.99% SLA by default using a single gateway with two external interfaces. Requires dynamic routing (BGP). Has a strict MTU ceiling of 1460 bytes due to IPSec overhead!
  • MACsec: Media Access Control Security. Provides Layer 2 hardware-level encryption directly over physical Dedicated Interconnect links.
  • BGP Path Selection:
    • Legacy Mode (Default): Compares AS-path lengths only within a single BGP task, leading to potential routing inconsistencies across separate routers.
    • Standard Mode: Adheres strictly to RFC 4271, consistently comparing AS-path lengths network-wide. Supports advanced features like bgpAlwaysCompareMed.

⚠️ The HA VPN UDP Black-Hole Trap

If your VPC MTU is configured for 1500 or 8896 bytes and you route traffic over HA VPN (MTU 1460), TCP traffic will survive due to MSS clamping. However, UDP traffic (like DNS or Syslog) will be silently black-holed if PMTUD fails! Always set the VPC MTU to match the lowest common denominator on the hybrid path, or clamp MSS.

Section 5: Managing, Monitoring, & Troubleshooting Network Operations

Weight: ~13%

Official Guide Objectives

  • Analyzing network traffic using VPC Flow Logs, Firewall Logs, and Cloud Logging.
  • Using Network Intelligence Center (NIC) modules for network diagnostics.
  • Troubleshooting VPN, Interconnect, and Cloud Router BGP peering issues.
  • Implementing connection draining (deregistration delay) on load balancers.

High-Yield Facts to Memorize

  • Connectivity Tests: Performs a logical configuration simulation of the packet path (evaluating routes, firewalls, and perimeters in the control plane) to check if two endpoints can communicate. Does not send live traffic!
  • Network Analyzer: Continuously and automatically scans VPC configurations in the background to detect misconfigurations, shadowed routes, invalid next hops, and IP address exhaustion.
  • Firewall Insights: Analyzes firewall logs and configuration metrics to identify shadowed, redundant, or unused firewall rules that can be safely deleted.
  • BGP Troubleshooting: BGP session stuck in Active or Idle indicates a physical link or IPSec tunnel failure preventing TCP communication on port 179. Check link-local IP configurations, ASNs, pre-shared keys, and local firewalls.

⚠️ Proactive vs. Reactive Troubleshooting

If an exam question asks how to automatically detect that a route's next-hop VM has been deleted, the answer is Network Analyzer. If a question asks how to verify if a specific VM can talk to an external database, the answer is Connectivity Tests.

Network Analyzer Automated VPC configuration monitoring and failure detection. Connectivity Tests Simulating packet paths to diagnose firewalls, routes, and perimeters.

Section 6: Configuring Network Security

Weight: ~14%

Official Guide Objectives

  • Configuring Cloud Armor WAF rules, rate limiting, bot management, and Adaptive Protection.
  • Configuring Cloud NGFW Essentials, Standard, and Enterprise tiers.
  • Implementing hierarchical firewall policies and effective rules evaluation.
  • Configuring Public Cloud NAT, Private NAT, and Dynamic Port Allocation (DPA).
  • Deploying Secure Web Proxy (SWP) for egress filtering.

High-Yield Facts to Memorize

  • Cloud Armor Adaptive Protection: Uses machine learning to continuously analyze traffic patterns, automatically drafting custom blocking rules to mitigate Layer 7 HTTP flood DDoS attacks in real-time.
  • Cloud NGFW Enterprise: The only tier that supports Layer 7 Packet Inspection (Intrusion Prevention System - IPS) to detect malware and spyware, which it enforces via managed Firewall Endpoints.
  • Dynamic Port Allocation (DPA): Automatically doubles a VM's NAT port allocation (e.g., 32 -> 64 -> 128 -> 256) up to a configured maximum based on demand, preventing SNAT port exhaustion while maximizing NAT IP efficiency. Min and Max ports must be powers of 2!
  • Secure Web Proxy (SWP): Fully managed, serverless egress proxy that filters outbound HTTP/S traffic based on FQDNs, URLs, and web categories without needing a NAT gateway or proxy VMs.

⚠️ Secure Tags vs. Network Tags

Avoid network tags for critical security boundaries! Network tags are easily modified by VM administrators. Instead, use Secure Tags (governed by IAM) or Service Accounts to define secure micro-segments that cannot be bypassed.

Google Cloud Armor WAF rules, DDoS protection, and Adaptive Protection. Cloud NAT Port Allocation Static vs. Dynamic Port Allocation and preventing SNAT exhaustion.
