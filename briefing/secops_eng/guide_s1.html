Section 1~14%

Platform Operations

Choose telemetry sources, integrate tools, and configure access across Google SecOps and Security Command Center.

Know the platform split

Half the wrong answers swap these tools — learn the split cold:

  • Security Command Center (SCC) — cloud posture/CSPM, misconfiguration, Security Health Analytics, Event Threat Detection.
  • Google SecOps SIEM — ingest telemetry, write detections, threat-hunt.
  • Google SecOps SOAR — playbooks, case management, automation.
  • Google Threat Intelligence (GTI) — IOCs, actor and campaign intel.
⚠️ Exam Trap: Posture/misconfiguration → SCC. Custom log-based detection → SecOps SIEM + YARA-L. Don't swap them.

Access & telemetry prioritisation

Grant IAM for feature access vs data access separately — analysts can use a tool without seeing all raw data. Use service accounts/API keys for automation and Workforce Identity Federation for external analysts. Prioritise telemetry sources (SCC, SecOps, GTI, Cloud IDS) and integrate them to close coverage gaps.

⚠️ Exam Trap: To restrict which analysts can read sensitive raw logs, use data-access IAM — feature access alone does not gate the data.

Feeds, forwarders & getting telemetry in

Knowing the tools is only half of it — the exam loves the plumbing that gets logs into Google SecOps. Pick the right intake for the source:

  • Direct ingestion / native GCP feeds — Cloud Audit Logs, VPC Flow, DNS, and SCC findings flow in through the Google Cloud integration with minimal setup; prefer this for first-party telemetry.
  • Feeds — pull/push connectors for third-party SaaS and APIs (Okta, AWS CloudTrail, Microsoft 365, threat feeds) on a schedule.
  • Forwarder — a lightweight on-prem/VM agent that ships syslog, packet, and file-based logs from inside your network to SecOps.
  • Ingestion API / Bindplane agent — for endpoints and custom apps that emit their own structured events.
⚠️ Exam Trap: On-prem firewall or appliance emitting syslog → deploy a forwarder inside the network. A cloud SaaS API → a feed. Don't reach for a forwarder when a managed feed already exists.

Scoping access cleanly

SecOps access has two dials and the exam tests both. Feature access (via IAM roles) controls which parts of the product an analyst can open — SIEM, SOAR, dashboards, rule editor. Data access (data RBAC, scoped by label) controls which slices of the normalised logs they can actually read. A tier-1 analyst might open SIEM search yet be blocked from reading HR or finance log labels.

  • Use data RBAC labels to fence sensitive log scopes from broad analyst groups.
  • Use Workforce Identity Federation for external/contractor analysts so no Google accounts are provisioned.
  • Use service accounts / API keys for automation and SOAR integrations, never a human's credentials.
2 DIALS Feature = which doors open; Data = which rooms you may enter. Granting the door does not grant the room.

🧠 Warm-Up Quiz

Q1. You need to find Google Cloud misconfigurations like public buckets, no custom logic. Which tool?

Reveal answer

SCC Security Health Analytics.

Continuous posture/CSPM scanning against benchmarks.

Q2. External SOC contractors need access via their own IdP without Google accounts. What do you configure?

Reveal answer

Workforce Identity Federation.

Federates external identities without provisioning Google accounts.

Q3. An on-prem firewall emits syslog that must reach Google SecOps. Which intake mechanism fits?

Reveal answer

A SecOps forwarder deployed inside the network.

Forwarders collect syslog/packet/file logs from on-prem and ship them to SecOps; feeds are for cloud/SaaS APIs.

Q4. A tier-1 analyst can open SIEM search but must not read finance-team logs. What enforces that?

Reveal answer

Data RBAC (data-access scoping by label).

Feature access opens the tool; data RBAC restricts which log scopes are readable.

📝 Exam-Style Practice

Q1. Your SOC already ingests Cloud Audit Logs and VPC Flow natively. The team now needs Okta sign-in events and AWS CloudTrail in Google SecOps, with the least operational overhead and no agents to maintain. What do you set up?

  • A) Configure managed feeds for the Okta and AWS CloudTrail connectors so SecOps pulls the events via API
  • B) Deploy a SecOps forwarder on a VM in each cloud and point the providers' syslog output at it
  • C) Export the logs to a Cloud Storage bucket and import them into Security Command Center for analysis
  • D) Write a YARA-L rule that references the Okta and CloudTrail APIs directly at detection time
Reveal answer & explanation

Correct: A. Okta and AWS CloudTrail are cloud/SaaS sources with first-party connectors, so managed feeds pull them over API with no agents to run — exactly the low-overhead intake for SaaS telemetry.

B: A forwarder is for on-prem syslog/file/packet logs and adds the VM-agent maintenance the scenario explicitly wants to avoid. C: SCC is a posture/CSPM tool, not the SIEM where these logs are searched and detected on, and bucket round-trips add latency. D: YARA-L runs over normalised UDM events already ingested; it cannot pull from external APIs and is a detection construct, not an ingestion one.

Q2. Compliance requires that only the fraud team can read payment-system logs, even though all 40 analysts use the same SecOps SIEM for hunting. You must not stop the wider team from using search. What do you implement?

  • A) Remove the SIEM feature-access role from everyone except the fraud team so others cannot open search
  • B) Apply data RBAC so payment-system log labels are readable only by the fraud team's scope
  • C) Move the payment-system logs into a separate Google Cloud project with its own IAM bindings
  • D) Create a YARA-L reference list of payment hostnames and exclude other analysts from matching it
Reveal answer & explanation

Correct: B. Data RBAC scopes which log labels each group can read while leaving feature access intact, so the wider team keeps searching everything except the fenced payment logs.

A: Stripping feature access blocks the 40 analysts from search entirely, breaking the requirement that they keep hunting. C: Splitting projects fragments the SIEM tenant and does not map to how SecOps data-access controls work. D: Reference lists tune detection logic, not who is permitted to read raw log data during search.

Q3. Leadership asks for one finding that proves a bucket is publicly exposed AND chained to an over-privileged service account that can reach customer data. They want it surfaced as a posture risk, not as a log-based detection. Where does this belong?

  • A) A multi-event YARA-L correlation rule joining IAM and storage UDM events over a time window
  • B) A scheduled UDM search in the SIEM that pivots from bucket access to service-account activity
  • C) Security Command Center, which evaluates posture and surfaces toxic-combination attack paths
  • D) A SOAR playbook that periodically queries storage and IAM APIs and opens a case on a match
Reveal answer & explanation

Correct: C. Chaining individually-minor misconfigurations into a path to sensitive data is a posture/attack-path problem, which SCC's toxic-combination analysis is purpose-built to surface.

A: YARA-L correlates activity events for detections; static posture exposure is not an event stream it watches. B: A UDM search inspects logged activity, not the standing configuration state of buckets and IAM. D: A playbook can react to findings but is response automation, not the engine that evaluates posture risk in the first place.
