Section 4~22%

Detection Engineering

The largest section: write YARA-L detection rules and tune them with threat intelligence.

Anatomy of a YARA-L rule

A YARA-L rule has: meta (author, severity), events (UDM match conditions), match (group-by variables + time window), condition, and outcome (e.g., a risk score). Reference lists hold reusable allow/deny values (known-good service accounts, watchlisted domains) referenced across rules.

YARA-L "My Events Match Conditions, Output risk" — meta → events → match → condition → outcome.

Tuning & reducing false positives

Cut noise without losing coverage: add known-good entities to a reference list and exclude them in the rule, and use entity-graph context to raise accuracy. Score alerts by IOC risk and suppress repetitive benign alerts. Detect posture changes with SCC Security Health Analytics and posture management.

⚠️ Exam Trap: Reduce false positives with reference lists and entity context — never by deleting the rule (which removes coverage).

Single-event vs multi-event rules

The presence (or absence) of a match section changes a YARA-L rule's whole personality — and the exam tests the distinction:

  • Single-event rule — no match section; fires on each individual event meeting the condition (e.g., one connection to a watchlisted IP). Great for IoC matching.
  • Multi-event rule — has a match with group-by variables and a time window; correlates several events (e.g., 20 auth failures from one user in 5 minutes, or failure-then-success = brute-force success).
  • Placeholders vs match variables — placeholder variables ($ip) join conditions within an event; match variables (in match) group across events.
MATCH = MANY See a time window and a "happens N times / A then B" scenario → you need a match section (multi-event). One bad event on its own → single-event, no match.

Curated detections, the rule lifecycle & testing

You don't write everything from scratch. Curated detections are Google-authored, continuously-updated rule sets (cloud threats, Applied Threat Intelligence, etc.) you enable and tune rather than build. Around your own rules, run a disciplined lifecycle:

  • Test before deploy — run a rule against historical data (a test/retrohunt run) to gauge true- vs false-positive volume before it goes live.
  • Tune, don't delete — cut noise with reference-list exclusions, entity context, and tighter conditions; deleting a noisy rule throws away coverage.
  • Manage as code — version, review, and stage rules; promote from a dev/test state to production deliberately.
  • Alerting vs not — a rule can run for context/risk-scoring without raising an alert every time; reserve alerts for what needs a human.
⚠️ Exam Trap: If Google already ships a maintained rule for a common threat, enable the curated detection and tune it — don't reinvent it as a hand-written YARA-L rule you now have to maintain.

🧠 Warm-Up Quiz

Q1. Detect when one user fails auth >20 times in 5 minutes and assign a risk score. What do you write?

Reveal answer

A YARA-L rule in SecOps SIEM (events + match window + outcome).

YARA-L matches UDM events, groups by entity over a window, emits risk.

Q2. A rule keeps alerting on known automation accounts. Best fix?

Reveal answer

Exclude them via a reference list in the rule logic.

Keeps the detection live while suppressing benign cases.

Q3. You must correlate "auth failure then success from the same user inside 10 minutes." Which YARA-L construct is required?

Reveal answer

A multi-event rule with a match section (group-by + time window).

Correlating several events over a window needs match variables and a window; a single-event rule can't.

Q4. Google already maintains a rule set for a common cloud threat you need to cover. What's the efficient move?

Reveal answer

Enable and tune the curated detection.

Curated detections are Google-authored and continuously updated — no need to hand-build and maintain your own.

📝 Exam-Style Practice

Q1. You need a detection that fires only when a single user account produces more than 15 failed logins followed by a successful login within a 10-minute window, then assigns a risk score. How should the YARA-L rule be structured?

  • A) A single-event rule whose condition counts failed logins, since each login is one event
  • B) A multi-event rule with a match section grouping by user over a time window, plus an outcome score
  • C) A reference-list lookup that flags users whose failed-login count exceeds a static threshold
  • D) A curated detection enabled as-is, because brute-force is a standard built-in scenario
Reveal answer & explanation

Correct: B. Correlating many failures then a success for the same user inside a window requires a match section with a group-by user variable and a time window, and the risk score comes from the outcome block.

A: A single-event rule sees one event at a time and cannot count or sequence events across a window. C: Reference lists store reusable values; they don't count events or evaluate time-windowed sequences. D: Even if a curated rule exists, the question specifies bespoke thresholds and a custom outcome, which means you author or tune a rule, and the structure still needs a match section.

Q2. A newly written rule looks correct, but you're worried it will bury analysts in false positives once live. Before deploying it to production, what is the best validation step in SecOps?

  • A) Run the rule against historical telemetry in a test run to measure true- vs false-positive volume
  • B) Deploy it straight to production at low severity and watch the alert queue for a week
  • C) Add every conceivable benign entity to a reference list up front so nothing can false-positive
  • D) Lower the rule's outcome risk score so any false positives are automatically de-prioritised
Reveal answer & explanation

Correct: A. Testing the rule against historical data shows exactly what it would have matched, letting you measure and tune false-positive volume before it ever touches the live queue.

B: Shipping to production first is precisely the risk you're trying to avoid; low severity still floods the queue. C: Pre-loading every imaginable benign entity is guesswork that risks over-suppression and blind spots without any measurement. D: Lowering the score hides noise rather than validating accuracy, and you still don't know the real false-positive rate.

Q3. A high-value rule is accurate but fires constantly on three known CI/CD service accounts performing legitimate automated deployments. You must keep the detection's coverage for everyone else. What's the correct tuning?

  • A) Delete the rule and rely on the entity risk score to surface the same activity when it matters
  • B) Reduce the rule's severity so the CI/CD alerts sink below the analysts' triage threshold
  • C) Exclude the three service accounts via a reference list referenced in the rule condition
  • D) Shorten the rule's match window so the automated bursts no longer satisfy the condition
Reveal answer & explanation

Correct: C. A reference list of the known-good service accounts, excluded in the condition, suppresses the benign cases while the rule keeps full coverage against every other entity.

A: Deleting the rule removes coverage for everyone, the classic trap. B: Lowering severity hides all alerts from the rule, not just the three benign accounts, weakening real detections. D: Shrinking the window degrades detection logic globally and may cause the rule to miss genuine slow attacks.
