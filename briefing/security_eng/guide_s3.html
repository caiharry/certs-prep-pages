Section 3: Ensuring Data Protection

Protect data at rest, in transit, and in use; tame secrets, keys, and modern AI workloads (~23%). The crown jewels live here — encrypt them, hide them, and keep a kill-switch handy.

Section 3 🟢 ~23%

This domain is all about the data itself — not who reaches it (Section 1) or how it travels (Section 2), but how it's encrypted, de-identified, and secret-managed. The exam leans hard on the four encryption-key models and on Sensitive Data Protection, so we'll spend most of our time there. Grab your cryptographic hard hat. ⛏️🔐

3.1 The Four Encryption Key Models

Everything in Google Cloud is encrypted at rest by default. The exam question is always who holds and controls the key. Memorise these four rungs — the distinctions win you marks.

Model Where The Key Lives You Reach For It When...
Google-managed (default) Entirely in Google's infrastructure; you never see it. No special key-control requirement exists.
CMEK (Cloud KMS) In Cloud KMS; you own rotation, disable, destroy. Supports import. You must control the key lifecycle and have a cryptographic kill-switch.
Cloud EKM In an external/on-prem HSM or KMS — never persists in Google Cloud. A regulator says "keys must never reside in the cloud."
Confidential Computing N/A — encrypts data in use, in memory, via the CPU. Data must stay encrypted even while being processed.
EKM vs CMEK-import is the classic trap. "Keys must never leave / never reside in Google Cloud" = Cloud EKM. "We generate the key ourselves then bring it in" = CMEK import into Cloud KMS (the material now lives in KMS). "Encrypted while being processed in memory" = Confidential Computing, not a key model at all.

💡 The 3-in-a-row: REST / TRANSIT / USE

Data at rest → CMEK/EKM. Data in transit → TLS (automatic on Google's network). Data in use → Confidential Computing. If a stem stresses "while being processed," the answer is almost always Confidential VMs/GKE Nodes.

3.2 CMEK in Practice: Rotation, Kill-Switch & Separation of Duties

Choose CMEK and you own the whole key lifecycle in Cloud KMS: create key rings, set automatic rotation schedules, and disable, destroy, or import key material. Each Google service reaches your key through a per-service service agent that needs roles/cloudkms.cryptoKeyEncrypterDecrypter on it.

🔌 The Cryptographic Kill-Switch

Need to make encrypted data instantly unreadable without touching the data itself? Disable (or destroy) the CMEK key version, or revoke the service agent's key role. With no key, the ciphertext can't be decrypted — an immediate, surgical kill-switch. Re-running DLP or editing bucket IAM does not achieve this.

🧑‍⚖️ Separation of Duties

The person who manages keys should not also be able to read the data those keys protect. Split it: the key admin holds only KMS roles in the key project and holds no data-read role in the data project. Now no single human can both unlock and read — satisfying the audit finding examiners love to set.

"Instantly revoke access to encrypted data" = disable the key version. Don't pick "re-run DLP," "change bucket IAM," or "delete the data." Only the cryptographic move gives an instant, reversible-by-re-enabling kill-switch without destroying the underlying objects.

3.3 Sensitive Data Protection (formerly Cloud DLP)

Sensitive Data Protection does three distinct jobs the exam carefully separates:

Job What It Does Trigger Phrase
Discovery / Profiling Finds where sensitive data lives across BigQuery, Cloud Storage, Cloud SQL at scale. "Find out which tables contain PII."
Inspection Matches infoTypes (credit cards, national IDs, emails) in content. "Detect credit-card numbers in this dataset."
De-identification Redaction, masking, or format-preserving tokenisation/pseudonymisation. "Hide the IDs but keep analytics working."

💡 FIND, then FUZZ

Sensitive Data Protection first finds the PII (discovery + inspection), then fuzzes it (redact / mask / tokenise). If the stem says analytics or joins must still work, the answer is deterministic tokenisation (the same input always maps to the same token) — not redaction, which deletes the value and breaks joins.

"I asked Cloud KMS to keep a secret. It said: 'Sure — but if you ever want it back, you'll need the key.' Brutal, but honest." 🔑🤐

3.4 Secret Manager

Passwords, API keys, and certificates do not belong in code, config files, or container images. Secret Manager gives you central, versioned secrets with IAM-scoped access, CMEK encryption, rotation, and audit logging — fetched at runtime.

Secret Manager stores secrets, not KMS keys. If a question is about separation of key/data duties, the answer involves Cloud KMS roles — not granting Secret Manager Accessor. Secret Manager holds the database password; Cloud KMS holds the encryption key. Don't swap them.

3.5 Securing AI / ML Workloads

The current exam adds security controls for ML systems. Protect training data and models from exfiltration (wrap the project in VPC Service Controls, use CMEK on the buckets and Vertex AI resources), lock down Vertex AI access with least-privilege IAM and private endpoints, and screen generative AI inputs/outputs with Model Armor and Sensitive Data Protection (catch prompt injection, jailbreaks, and PII leakage).

🤖 Model Armor in One Line

Think of Model Armor as a WAF for LLM prompts and responses: it inspects generative AI traffic for prompt-injection, harmful content, and sensitive-data leakage, independent of which model you run.

🧠 Quick-Fire Warm-Up Quiz!

Retrieve before you review. No peeking!

Q1: Discover and redact credit-card numbers in a Cloud Storage dataset before analysis. Which service?

Sensitive Data Protection (Cloud DLP). It detects, classifies, and de-identifies sensitive data.

Q2: A regulator requires that encryption keys are held only in your on-prem HSM and never reside in Google Cloud. Which option?

Cloud EKM. External keys never persist in Google Cloud; Google receives only a short-lived wrapped reference.

Q3: An auditor asks you to instantly render a CMEK-encrypted dataset unreadable without deleting the data. What do you do?

Disable (or destroy) the CMEK key version in Cloud KMS. Without the key, the ciphertext can't be decrypted — an immediate cryptographic kill-switch.

Q4: A database password is currently baked into a container image. Where should it live instead?

Secret Manager. Versioned, IAM-controlled, encrypted storage with rotation, retrieved at runtime.

📝 Section 3 Practice Exam Questions

Left-click to select, right-click to strike out a distractor, then Submit for the full diagnostic.

Question 1: A financial regulator requires that the encryption keys protecting your BigQuery datasets are generated and stored exclusively in your own on-premises HSM and never reside in Google Cloud, even briefly. Which key option satisfies this?

  • A) CMEK with key material imported into Cloud KMS from the on-prem HSM.
  • B) Google-managed encryption keys with Access Approval enabled on the project.
  • C) Cloud External Key Manager backed by your on-prem HSM through a supported partner.
  • D) CMEK in Cloud HSM with a strict automatic rotation policy and audit logging.

Correct Answer: C

Why C is correct: Cloud EKM keeps the key in your external HSM and Google only receives a short-lived wrapped reference, so the key material never resides in Google Cloud — exactly the regulator's constraint.

Why the others are wrong:
A: Importing key material places that material into Cloud KMS, violating "never resides in Google Cloud."
B: Google-managed keys live entirely in Google's infrastructure; Access Approval gates support access but not key location.
D: Cloud HSM keys are generated and held in Google's HSMs, not your on-prem HSM.

Question 2: Analysts need to run aggregate queries over a customer table that contains national ID numbers. The IDs must stay non-readable, yet rows for the same customer must still join consistently across tables. What is the right transformation?

  • A) Apply Sensitive Data Protection redaction so the ID values are removed from the output entirely.
  • B) Apply Sensitive Data Protection deterministic tokenisation (format-preserving encryption) to the ID column.
  • C) Encrypt the whole table with a CMEK key and grant analysts the key decrypt role.
  • D) Move the table behind a VPC Service Controls perimeter and grant analysts BigQuery access.

Correct Answer: B

Why B is correct: Deterministic tokenisation replaces each ID with a stable surrogate, so the value is unreadable but the same input always yields the same token, preserving joins across tables.

Why the others are wrong:
A: Redaction removes the values, breaking the cross-table joins the analysts require.
C: Granting decrypt access makes the raw IDs readable, defeating the requirement.
D: A perimeter controls where data can go but does nothing to de-identify the ID values for the analysts who are allowed in.

Question 3: Your security team must ensure that the engineer who manages encryption keys cannot also grant themselves read access to the data those keys protect, satisfying a separation-of-duties audit finding. How do you structure this?

  • A) Give the key engineer Owner on the project so they can manage both keys and data consistently.
  • B) Store all keys as Google-managed so no human holds key permissions at all.
  • C) Put the keys in Secret Manager and grant the key engineer the Secret Accessor role.
  • D) Grant the key engineer Cloud KMS Admin on the key project and ensure they hold no data-read role on the data project.

Correct Answer: D

Why D is correct: Splitting duties means the key admin holds only KMS roles in the key project and has no data-read role on the data resources, so they can manage keys but cannot read the protected data.

Why the others are wrong:
A: Owner grants both key and data control to one person, which is the exact concentration the audit flagged.
B: Google-managed keys remove the need to manage keys at all, but the question requires a human key admin who is fenced off from data.
C: Secret Manager stores secrets, not the KMS keys that encrypt the datasets, so it doesn't implement key/data separation.
