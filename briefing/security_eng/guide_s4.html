Section 4: Managing Operations

Automate security, detect threats, and respond — the SecOps backbone of the exam (~19%). When the alarm goes off, this is the domain that tells you how to see it, prove it, and contain it.

Section 4 🟠 ~19%

Sections 1–3 build the walls. Section 4 is the security operations centre: hardening the software supply chain, capturing the right logs, spotting threats, and responding without trampling the evidence. Two things get tested to death here — which audit logs are on by default and what Security Command Center actually does. Let's wire up the alarm system. 🚨

4.1 Supply-Chain & Deploy-Time Security

Stop bad software before it ever runs. Binary Authorization is a deploy-time gate for GKE and Cloud Run that admits only container images signed by an approved attestor; everything else is rejected. Feed it by scanning images for CVEs in CI/CD (Artifact Registry / Artifact Analysis), automating image hardening and patching, and catching configuration drift with custom organization policies and Security Health Analytics custom modules.

✍️ Binary Authorization in One Line

"Only signed, attested images may deploy" ⇒ Binary Authorization. It's the bouncer at the cluster door checking each image's signature against your policy before admission.

4.2 The Four Cloud Audit Log Types

This is the most reliably tested fact in the whole domain. Cloud Audit Logs has four streams, and the killer detail is which are on by default.

Log Type Records Default State
Admin Activity Configuration changes (who changed what). Always on, free, cannot disable.
System Event Google-initiated system actions. Always on, free.
Data Access Reads/writes of data (who read which object/row). Off by default (except BigQuery) — enable per service in the IAM audit config.
Policy Denied Requests denied by a security policy (e.g. VPC-SC). On when the relevant service is used.
"We had no record of who read the data." This almost always means Data Access audit logs were never enabled — they're off by default. Don't blame the sink or buy a SIEM first; fix the IAM audit config so the events exist, then route them. A sink can only ship logs that are already being generated.

💡 "Admins are Always Audited; Data is a Decision"

Admin Activity = Always on. Data Access = a Decision you must switch on. If the missing evidence is a read of data, the cause is unenabled Data Access logs.

4.3 Routing Logs: Sinks & Aggregated Sinks

Logs land in the _Default and _Required buckets, but for long retention, analysis, or a SIEM you create log sinks that export to BigQuery, Pub/Sub, Cloud Storage, or an external destination.

🧲 Aggregated Sinks = One Sink to Rule Them All

An aggregated sink at the organization or folder level captures logs from every current and future child project in one place, with no per-project setup. When a question says "central SIEM, least ongoing maintenance, all projects," that's an aggregated sink.

4.4 Threat Detection: Security Command Center

Security Command Center (SCC) is the single pane of glass for posture and threats. Know which built-in service does which job.

SCC Service Detects Example Finding
Security Health Analytics (SHA) Misconfigurations & posture drift. Public bucket, open firewall, no MFA.
Event Threat Detection (ETD) Threats in logs. Malware C2 callout, brute force, anomalous IAM grant.
Container / VM Threat Detection Runtime threats in workloads. Crypto-mining binary, reverse shell, added bad library.

💡 SCC = "SEE": SHA sees config, ETD sees threats

Security Command Center is where you SEE it all. SHA = misconfig. ETD = log threats. CTD/VMTD = workload threats. Posture, vulnerabilities, and active attacks in one console.

4.5 Network Forensics

Three network tools, three depths of visibility — a favourite discriminator.

Tool What You See Use When...
VPC Flow Logs Connection metadata (5-tuple, bytes) — not payloads. You need who-talked-to-whom and volume, cheaply.
Packet Mirroring Full packet payloads copied to a collector. You need deep inspection of actual traffic content.
Cloud IDS Signature-based intrusion detection on mirrored packets. You want managed (Palo Alto) threat signatures, no rule-writing.
Flow Logs ≠ payload inspection. If the stem wants signature-based, full-payload intrusion detection on east-west traffic with no rule authoring, that's Packet Mirroring feeding Cloud IDS. VPC Flow Logs only carry metadata, and Cloud Armor only guards inbound HTTP(S) at the edge — neither inspects east-west payloads.

4.6 Incident Response Without Destroying Evidence

When a host is compromised, the order of operations matters. Contain, then preserve, then investigate:

  • Isolate the VM with a deny-all firewall rule via a network tag — cuts the command-and-control channel immediately.
  • Snapshot the disks for forensic evidence before any further change.
  • Rotate / disable compromised credentials and keys.
  • Then analyse — never delete the instance first, or you destroy the evidence.
"First rule of incident response: don't rm the crime scene. The forensics team gets very grumpy." 🕵️‍♀️💾

Question 1: A compliance team needs a tamper-resistant record of every read against objects in a set of regulated Cloud Storage buckets, retained for seven years in an external SIEM. Currently nothing is recorded. What is the correct first step before any export will help?

  • A) Create an aggregated log sink to the SIEM and assume the read events will start flowing.
  • B) Enable Data Access audit logs for Cloud Storage in the IAM audit configuration.
  • C) Turn on VPC Flow Logs on the subnets that host the storage clients.
  • D) Add a Security Health Analytics custom module that watches the buckets for reads.

Correct Answer: B

Why B is correct: Object reads are Data Access events, which are off by default; until you enable Data Access audit logs for Cloud Storage there is nothing for any sink to export.

Why the others are wrong:
A: A sink only routes logs that already exist — with Data Access logging off there's nothing to route.
C: Flow Logs record network connection metadata, not who read which storage object.
D: Security Health Analytics reports misconfigurations, not per-object read events.

Question 2: Event Threat Detection raises a finding that a Compute Engine instance is communicating with a known crypto-mining command-and-control IP. You must contain the host immediately while preserving evidence for later analysis. What is the best response?

  • A) Delete the instance right away so the mining process can no longer run anywhere.
  • B) Remove the service account from the instance and wait to see if the traffic stops on its own.
  • C) Apply a deny-all firewall rule via a network tag to isolate the VM, then snapshot its disks.
  • D) Lower the instance's machine type to starve the miner of CPU while you investigate.

Correct Answer: C

Why C is correct: Isolating the VM with a deny-all firewall tag cuts the C2 channel immediately, and snapshotting the disks preserves forensic evidence before any further change.

Why the others are wrong:
A: Deleting the instance destroys exactly the evidence you're told to preserve.
B: Removing the service account doesn't stop outbound network traffic, and "wait and see" isn't containment.
D: Resizing slows the miner but leaves the C2 connection open, so the host isn't contained.

Question 3: Security wants signature-based intrusion detection on east-west traffic between application tiers in a VPC, with full-payload inspection and managed threat signatures, without writing detection rules themselves. Which combination fits?

  • A) Packet Mirroring feeding Cloud IDS, which applies Google-managed threat signatures.
  • B) VPC Flow Logs exported to BigQuery with scheduled queries flagging suspicious patterns.
  • C) Cloud Armor rules inspecting the traffic between the application tiers.
  • D) A custom Security Health Analytics module scanning the tier configurations for drift.

Correct Answer: A

Why A is correct: Cloud IDS consumes mirrored packets and applies Google-managed (Palo Alto) threat signatures for full-payload, signature-based detection with no rule authoring required.

Why the others are wrong:
B: Flow Logs carry only connection metadata, not payloads, and require you to write the detection queries yourself.
C: Cloud Armor protects inbound HTTP(S) at the load balancer edge, not east-west VPC traffic.
D: Security Health Analytics checks configuration posture, not live payload-level intrusions.
