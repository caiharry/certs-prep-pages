Section 3: Migrating Data Solutions

Continuous replication pipelines, zero-downtime homogeneous moves, heterogeneous schema mappings, and fallback loops!

~23% of the Exam! 🎯

3.1 Shifting the Data — Migration & Replication

Let's face it: packing up a physical server database and moving it to the cloud is not like moving house. You can't just throw your data tables into cardboard boxes, load them into a rusty transit van, and hope none of the primary keys fall out on the motorway! 🚛

🚀 Homogeneous vs. Heterogeneous — Choose Your Battle!

The type of migration you perform determines exactly how much hair your engineering team will pull out during the process:

  • Homogeneous (Same engine, same vibes): MySQL to Cloud SQL MySQL, or PostgreSQL to AlloyDB. Since the database engine is the same, this is usually an "easy peasy" job. You still need to validate extension compatibilities, matching version numbers, engine flags, database drivers, and run performance baselines, but the schema mapping is a breeze! 🍃
  • Heterogeneous (Different engine, absolute chaos): Oracle to AlloyDB, or SQL Server to AlloyDB. Because the database dialects and feature sets are completely different, you have to convert DDL (table schemas), custom data types, stored procedures, triggers, and operational habits. This is a massive job that requires proper tools and plenty of tea! ☕

🧠 Memory Tip: Reverse Replication is Your Parachute!

If you cut over to your brand new cloud database and it suddenly crashes or behaves weirdly under load, you'll need to fail back to your old on-premises system. Always set up reverse replication so all new writes in GCP are synced back to the on-prem source! BUT watch out: CDC is asynchronous, so always check replica lag and document the RPO risk before promising the business absolute zero data loss!

⚙️ The Big Serverless Players: DMS vs. Datastream

GCP has two primary serverless change capture tools. They sound similar, but their exam jobs are completely different. Don't mix them up!

  • Database Migration Service (DMS): This is your primary database relocator. It handles initial loading and continuous CDC for migrating databases into Cloud SQL, AlloyDB, and Spanner. If you are moving a database lock-stock-and-barrel to GCP with near-zero downtime, DMS is your weapon of choice! 🛠️
  • Datastream: This is an ongoing live data stream. It captures changes from databases (MySQL, PostgreSQL, Oracle, SQL Server, Spanner, etc.) and streams them in near-real-time to BigQuery (for analytics) or Cloud Storage (for data lakes). If you want ongoing data pipelines, reach for Datastream!
DMS Conversion Workspaces! When doing a heterogeneous migration (e.g., Oracle to AlloyDB), DMS provides built-in Conversion Workspaces. They give you automated schema translations, highlight code compatibility errors, and even offer Gemini-assisted code fixes! But remember: the tool is clever, not clairvoyant — human review and testing of stored procedures are still absolutely mandatory! 🕵️

🐌 Extended-Outage Migration: Simple, If You Can Stand It!

Not every exam scenario demands near-zero downtime. If the business can tolerate a planned weekend outage, keep it simple and skip the complex live CDC pipelines:

  • Freeze Writes: Put the source on-prem application into maintenance/read-only mode so no new data is written. 🛑
  • Export & Import: Perform a native dump (like pg_dump or mysqldump), upload the file to Cloud Storage, and restore it directly into your target GCP instance.
  • Validate: Verify row counts, check table checksums, run app smoke tests, and double-check user permissions.
  • Go Live: Point your applications to the new GCP database and celebrate! It's low-risk, low-stress, and highly reliable when RTO allows.

🧠 Quick-Fire Warm-Up Quiz!

Test your recall before taking on the practice questions. No peeking!

Q1: What is the main difference between Database Migration Service (DMS) and Datastream?

DMS migrates instances (brings databases into Cloud SQL/AlloyDB/Spanner). Datastream streams CDC continuously to BigQuery/GCS for real-time analytical pipelines.

Q2: Why is continuous Change Data Capture (CDC) essential for near-zero downtime migrations?

It captures transactions happening on the source database while the historical data is being copied, ensuring the new target is perfectly up-to-date at cutover time.

Q3: Why should you enable reverse replication back to your source system after cutover?

As a fallback safety net! But only promise what the tooling supports, and always watch replication lag. Reverse replication is a parachute, not a time machine.

📝 Section 3 Practice Exam Questions

Test your migration logic with these exam-style questions!

Question 1: Your team is planning to migrate a business-critical 8 TB PostgreSQL database hosted on-premises to AlloyDB in Google Cloud. The business cannot tolerate more than 5 minutes of total downtime during cutover. What is the most appropriate migration strategy and toolset?

  • A) Perform a logical pg_dump on-premises, compress and upload the dump to Cloud Storage, and restore it on AlloyDB during a weekend window using automated Cloud Data Flow pipelines.
  • B) Configure Google Cloud Database Migration Service (DMS) to perform an initial load followed by continuous replication (CDC), and cut over during a low-traffic window.
  • C) Establish an ongoing Datastream pipeline to stream database tables directly from PostgreSQL to AlloyDB with dynamic schema evolution enabled.
  • D) Backup PostgreSQL directly to Google Cloud Storage using Storage Transfer Service, and mount Cloud Storage on the AlloyDB instance using gcsfuse.

Correct Answer: B

Why Option B is correct: An 8 TB database is far too large to migrate using offline export/import (like pg_dump), which would cause hours of downtime. Database Migration Service (DMS) solves this by taking a baseline snapshot (initial load) and then continuously streaming live transactions (CDC) from on-prem to GCP. When you are ready to cut over, replication lag is near-zero, enabling you to point your apps to AlloyDB in seconds.

  • A: Offline dump/restore is fine when the outage window allows it, but 5 minutes is not that window.
  • C: Datastream is CDC for analytics/custom pipelines, not the primary managed AlloyDB migration workflow.
  • D: Storage Transfer Service moves files and objects; it does not replay live database transactions into AlloyDB.

Question 2: You are tasked with migrating a mission-critical legacy Oracle database to AlloyDB for PostgreSQL. The schema includes packages, triggers, and stored procedures. What should you do to prepare the heterogeneous migration properly?

  • A) Use DMS Conversion Workspaces, review/fix converted objects, apply the schema to AlloyDB, then run a migration job with validation and a documented rollback plan.
  • B) Assume DMS Conversion Workspaces automatically translate every PL/SQL object perfectly and execute the heterogeneous schema migration in-place without human review.
  • C) Set up a Bare Metal Solution (BMS) instance running Oracle to bypass schema conversion entirely, and configure a Cloud Router loopback BGP interface.
  • D) Use Storage Transfer Service to copy Oracle raw data files directly into Cloud Storage and point AlloyDB at the bucket using external tables.

Correct Answer: A

Why Option A is correct: Oracle to AlloyDB is heterogeneous, so you need conversion workspaces, human review, testing, data migration, validation, and rollback planning. This is the sensible grown-up answer with a clipboard and a cup of tea.

  • B: Conversion Workspaces help, but they do not guarantee every object converts perfectly without review.
  • C: Bare Metal Solution keeps Oracle as Oracle; it does not meet the PostgreSQL-compatible target requirement.
  • D: Storage Transfer Service moves files, not a live compatible AlloyDB database state.

Question 3: A small internal application can be offline for a whole weekend. Its 200 GB MySQL database is simple, low-write, and the team wants the least complex migration path. What strategy is most appropriate?

  • A) Use an offline export/import during the planned outage, validate the target, then cut over.
  • B) Build a complex multi-week CDC pipeline using Database Migration Service (DMS) because every database migration must prioritize near-zero downtime.
  • C) Use Datastream directly into the Cloud SQL primary instance as the only migration and dynamic schema replication mechanism.
  • D) Manually write application-level dual-write logic inside the client applications to sync writes to both databases during the weekend.

Correct Answer: A

Why Option A is correct: If the outage window is generous and the database is modest, offline export/import is simpler, cheaper, and easier to validate. The exam rewards matching the strategy to requirements, not always choosing the fanciest tool.

  • B: CDC is great for tight downtime windows, but it adds unnecessary complexity here.
  • C: Datastream is not the default direct Cloud SQL primary migration path.
  • D: Manual dual writes are risky and unnecessary during an approved outage.

Question 4: Your analytics team wants to build a real-time data warehouse in BigQuery that mirrors active operational transactions occurring in a production Oracle database on-premises. The synchronization must capture insert, update, and delete actions continuously, and the database operations team mandates that the extraction process must not perform heavy periodic queries or full-table scans. Which tool and pattern should you use?

  • A) Write a custom Dataflow python script that polls the Oracle tables using SELECT statements every 10 seconds and writes them directly into BigQuery.
  • B) Use Database Migration Service (DMS) to continuously sync Oracle to a Cloud SQL MySQL database.
  • C) Deploy Datastream to capture Change Data Capture (CDC) events serverlessly from the Oracle transaction logs, and stream them directly into BigQuery.
  • D) Set up Storage Transfer Service (STS) to export Oracle daily dumps to Cloud Storage, and set up a scheduled load job into BigQuery.

Correct Answer: C

Why Option C is correct: This is Datastream's core use case!

  • Ongoing CDC Pipeline: Datastream is an ongoing serverless CDC engine that reads changes directly from the database's transaction/redo logs (low overhead, no heavy SQL polls!).
  • Direct Integration: It streams those operational inserts, updates, and deletes directly into BigQuery or Cloud Storage in real time, making it the perfect and most cost-effective choice for real-time analytics. 🚀

Why the others are wrong:

  • A: Polling tables using SELECT statements consumes massive database CPU, misses intermediate changes, and violates the "no heavy queries" requirement.
  • B: Database Migration Service (DMS) is for migrating databases into Cloud SQL or AlloyDB, not for streaming ongoing analytics directly to BigQuery.
  • D: Daily exports are batch-oriented, not real-time, resulting in up to 24 hours of data latency.

Question 5: Your team is performing a near-zero downtime homogeneous migration of a critical 12 TB MySQL database to Cloud SQL MySQL using DMS. As part of a robust cutover playbook, the business demands an immediate rollback path: if the application encounters any unresolvable bugs on Cloud SQL during the first 48 hours of live traffic, you must be able to redirect traffic back to the on-premises database with less than 5 minutes of downtime and absolutely no data loss for writes made in the cloud. How should you design this fallback path?

  • A) Run logical manual database exports on Cloud SQL every 10 minutes, and keep them ready to be manually imported on-premises.
  • B) After initial migration, configure and enable reverse replication (syncing write changes from the Cloud SQL primary back to the on-premises database) using CDC. Monitor replication lag closely before cutting back if needed.
  • C) Re-platform the database to Cloud Spanner since it supports native global fallback triggers.
  • D) Rely on daily automated backups and Point-In-Time Recovery (PITR) on the on-premises server to reconstruct missing writes manually.

Correct Answer: B

Why Option B is correct: Reverse replication is your migration parachute! By setting up a continuous CDC replication loop from the newly promoted Cloud SQL instance back to the on-premises database, you ensure the legacy system remains perfectly synchronized with any writes made in the cloud. If you need to roll back, you can redirect traffic back to on-prem with near-zero RPO and RTO! 🪂

Why the others are wrong:

  • A: Exporting and importing a 12 TB database takes hours, violating the 5-minute RTO constraint, and 10-minute exports would still cause up to 10 minutes of data loss (RPO).
  • C: Moving to Spanner is a heterogeneous migration that requires extensive schema design and code rewrites, which is out of scope for a MySQL homogeneous rollback plan.
  • D: Backups and PITR on-premises can reconstruct the legacy state before migration, but they cannot magically fetch new write transactions that occurred in* the cloud after cutover!
