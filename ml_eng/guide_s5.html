Section 5~22%

Automating & Orchestrating Pipelines

The largest section: build reproducible end-to-end pipelines, automate retraining, and track metadata.

Orchestration choice

Pick the orchestrator by the shape of the workflow:

  • Vertex AI Pipelines (KFP) — the default for ML: serverless, ML-native, reproducible, with built-in lineage via Vertex ML Metadata.
  • Cloud Composer (Airflow) — broad, cross-system workflow orchestration (data + ML + ops) when you already run Airflow.
  • TFX — standard components (ExampleGen, Transform, Trainer, Evaluator, Pusher) for TensorFlow pipelines, runnable on Vertex Pipelines.
⚠️ Exam Trap: ML-specific, serverless, reproducible pipeline = Vertex AI Pipelines. Composer is for general workflow orchestration across many systems.

Reusable, parameterised components

Build pipelines from versioned, containerised components with typed inputs/outputs, so steps are reusable and cacheable (unchanged steps are skipped on re-runs). Parameterise data paths, hyperparameters, and thresholds so the same pipeline serves dev, staging, and prod.

REUSE One pipeline, many parameters — promote the definition, change only the inputs.

Automated retraining triggers

Kick off retraining from: a schedule (Cloud Scheduler), a drift/skew signal from Model Monitoring, or new data landing (Pub/Sub / Eventarc on a GCS drop). The pipeline should retrain, evaluate, and only promote if the new model beats the incumbent.

⚠️ Exam Trap: "Retrain when production data drifts" wires Model Monitoring alerts to trigger the pipeline — not a fixed nightly cron that retrains regardless.

CI/CD for ML (a validation gate)

Use Cloud Build to test pipeline code and trigger runs. Add a model-evaluation gate: the pipeline compares candidate metrics against the current production model and a minimum bar, and the Pusher/registration step deploys only if the candidate passes. This is "CD for models," not just code.

GATE Train → Evaluate → only then Deploy. A failing eval blocks the rollout automatically.

Metadata & reproducibility

Every run logs artifacts, parameters, and executions to Vertex ML Metadata, giving end-to-end lineage: which data + code + hyperparameters produced which model. That makes models auditable and lets you reproduce or roll back a specific version with confidence.

Reveal answer

Vertex AI Pipelines.

ML-native, serverless, integrates with Vertex ML Metadata.

Q2. What records model artifact versions and data lineage on Vertex AI?

Reveal answer

Vertex ML Metadata.

Tracks lineage and versions for reproducibility.

Q3. You must retrain only when production input distributions drift, not on a fixed timer. How do you trigger it?

Reveal answer

Wire a Model Monitoring drift alert to trigger the pipeline.

Retraining fires on the signal that actually matters, saving compute when nothing changed.

An evaluation/validation gate before the deploy step.

The Pusher/registration step runs only when the candidate clears the metric bar.

📝 Exam-Style Practice

Q1. A team needs a serverless, reproducible training-and-deployment pipeline that records full artifact lineage and skips unchanged steps on re-runs, with the least operational overhead. What should they use?

  • A) Cloud Composer (Airflow) DAGs running each ML step as a task.
  • B) Vertex AI Pipelines with containerised KFP components and caching.
  • C) A Cloud Run service that chains the steps with custom code.
  • D) Cron-triggered scripts on a Compute Engine VM.
Reveal answer & explanation

Correct: B. Vertex AI Pipelines is serverless and ML-native, gives lineage through ML Metadata, and caches unchanged steps — minimal ops for an ML workflow.

A: Composer is for broad cross-system orchestration and adds a managed Airflow to run. C/D: Hand-rolled services or cron scripts give no lineage, caching, or reproducibility.

Q2. Retraining is currently a nightly cron that runs whether or not anything changed, wasting compute. The team wants retraining to happen only when it's actually needed. What is the best trigger?

  • A) Keep the nightly cron but on a larger machine to finish faster.
  • B) Trigger the pipeline from Vertex AI Model Monitoring drift/skew alerts.
  • C) Retrain manually whenever someone notices accuracy drop.
  • D) Retrain on every single new row via a streaming trigger.
Reveal answer & explanation

Correct: B. Drift/skew alerts fire when the data actually shifts, so retraining runs when it adds value and is skipped otherwise.

A: A faster cron still wastes runs. C: Manual noticing is slow and unreliable. D: Per-row retraining is wildly expensive and unstable.

Q3. Leadership requires that a retrained model is promoted to production only if it outperforms the incumbent on a holdout set. How do you enforce this automatically?

  • A) Always deploy the latest model and roll back later if metrics look bad.
  • B) Add an evaluation gate that compares candidate vs production metrics, deploying only on pass.
  • C) Email the metrics to a reviewer who approves deployments by hand.
  • D) Deploy both versions permanently and let users pick.
Reveal answer & explanation

Correct: B. A built-in evaluation/validation step gates the Pusher: the candidate is registered and deployed only when it clears the bar, enforcing the policy automatically.

A: Deploy-then-rollback risks shipping a worse model. C: Manual approval doesn't scale and isn't automatic. D: Keeping both forever isn't a promotion policy.
