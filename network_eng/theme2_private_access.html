Theme 2: Private Connectivity, NCC & VPC-SC

Private Google Access, PSA vs PSC, Network Connectivity Center, and VPC Service Controls

Theme 2 🟢

2.1 Private Google Access & VIPs

Right, let's talk about accessing Google's own APIs (like Cloud Storage, BigQuery, or Pub/Sub) privately. If your VM doesn't have a public IP, how does it talk to Google APIs? It can't route to the public internet! The answer is Private Google Access (PGA). 🌟

📡 Private Google Access (PGA)

PGA is a simple subnet-level checkbox. When enabled, VMs in that subnet with only private IPs can send traffic to the external IP addresses of Google APIs. The traffic travels entirely over Google's internal network, never touching the public internet!

But wait! What if you are on-premises and want to reach Google APIs privately over VPN or Interconnect? On-prem networks can't use PGA directly. Instead, you must use one of Google's Virtual IPs (VIPs):

  • Private VIP (private.googleapis.com): Resolves to 199.36.153.8/30. Supports most Google APIs, but excludes consumer-facing services like Gmail or YouTube. Extremely secure!
  • Restricted VIP (restricted.googleapis.com): Resolves to 199.36.153.4/30. Supports only APIs that can be secured using VPC Service Controls. This is the gold standard for compliance because it completely blocks access to non-permissible Google services! 🔒
The DNS Forwarding VIP Trap! To make on-premises resources use these VIPs, you must configure your on-prem DNS servers to resolve .googleapis.com as a CNAME to private.googleapis.com or restricted.googleapis.com, and set up DNS forwarding policies in Cloud DNS. If you forget the DNS overrides, on-prem traffic will try to resolve public IPs and fail!

2.2 PSA vs. PSC vs. Serverless VPC Access

This is the absolute core architectural decision of the exam! How do you connect your VPC to managed services (like Cloud SQL, AlloyDB, or third-party SaaS) and serverless environments (Cloud Run, Cloud Functions)? Let's break down the three giants:

Service Type Under the Hood Mechanism Transitive Routing Friendliness Best Use Cases
Private Services Access (PSA) VPC Network Peering to a Google-managed tenant VPC. Requires allocating an IP range block (e.g., /24 or /20). No! VPC Peering is non-transitive. On-prem networks cannot reach it without complex "Export Custom Routes" and Cloud Router BGP hacks. 🧱 Classic Cloud SQL, AlloyDB, Memorystore, and managed GKE control planes.
Private Service Connect (PSC) Places a local endpoint (Private IP) directly inside one of your consumer VPC subnets. Uses a forwarding rule. No peering! Yes! Since it's a local subnet IP, on-premises networks and peered VPCs can route to it natively without any peering updates. 🌟 Modern Cloud SQL, third-party SaaS (Snowflake, MongoDB), and publishing your own internal services to other VPCs.
Serverless VPC Access Uses a pool of connector VM instances (Serverless VPC Access Connector) that bridge your serverless environment into your VPC. N/A (Outbound only) Allowing Cloud Run, Cloud Functions, or App Engine to access private resources (VMs, databases) inside your VPC.
🧠 Crucial Memory Tip: PSA vs. PSC

Remember: PSA is a Peering, PSC is a Point!

  • PSA peers entire networks together. It's rigid, non-transitive, and gobbles up large IP blocks.
  • PSC is just an endpoint (a single IP address) in your local subnet. It's incredibly flexible, transitive-friendly, and uses only one IP!
"Why did the network engineer break up with VPC Peering? Because they felt the relationship was non-transitive and had too many boundaries!" 💔

2.3 Network Connectivity Center (NCC)

As your cloud footprint grows, managing pairwise VPC peerings becomes a nightmare. Network Connectivity Center (NCC) is Google's hub-and-spoke network connectivity model. It acts as a single centralized logical control plane to connect VPCs, on-premises offices, and other clouds!

🌐 NCC Topologies & Spokes

An NCC configuration consists of a central Hub and multiple attached Spokes:

  • Spoke Types:
    • VPC Spoke: Connects a VPC network to the hub.
    • Hybrid Spoke: Connects on-premises networks using HA VPN tunnels or Cloud Interconnect VLAN attachments.
    • Router Appliance Spoke: Connects third-party SD-WAN or firewall VMs (Router Appliances) to exchange routes dynamically.
  • Preset Topologies (Configured at Hub creation, cannot be changed!):
    • Mesh Topology (Default): All spokes belong to a single group. Full inter-spoke connectivity is established automatically. All VPC spokes can talk to all other VPC spokes and hybrid spokes.
    • Star Topology: Spokes belong to either the Center or Edge group. Edge spokes can talk to Center spokes, but Edge spokes cannot talk to other Edge spokes! This is perfect for multi-tenant isolation or retail branch networks where branches should only talk to the central hub, not each other. 🌟
The NCC Star Topology Hybrid Trap! If you configure an NCC hub with a Star topology, hybrid spokes with site-to-site data transfer enabled (which allows on-prem sites to transit through Google's network to talk to other on-prem sites) are only supported in the Center spoke group! Edge spoke groups only support hybrid spokes with site-to-site data transfer disabled*.

2.4 VPC Service Controls (VPC-SC)

Firewall rules block network access, but they don't prevent data exfiltration (where an insider with valid credentials copies sensitive database data to an external, unauthorized GCP project). To stop this, you need VPC Service Controls (VPC-SC)! 🛡️

🔒 Service Perimeters & Bridges

VPC-SC allows you to define a Service Perimeter around your Google Cloud resources (like Cloud Storage, BigQuery, or Cloud SQL). This perimeter acts as a giant security wall. APIs inside the perimeter cannot talk to resources outside, even if they have valid IAM credentials!

  • Perimeter Bridges: What if two separate secure perimeters need to share data? You don't merge them! Instead, you create a Perimeter Bridge to allow specific projects in Perimeter A to talk to specific projects in Perimeter B.
  • Ingress/Egress Policies: Highly granular rules that allow traffic to cross the perimeter boundary based on identity, project, and device state (using Access Context Manager).
  • Dry-Run Mode: Always use dry-run mode first! It logs any violations without actually blocking traffic. You can inspect the dry-run logs in Cloud Logging to identify and fix any missing permissions before enforcing the perimeter and breaking production! 🛠️

🧠 Quick-Fire Warm-Up Quiz!

Test your recall before diving into the proper practice exam. No peeking!

Q1: Which Google API VIP should you use if your security policy demands that VMs must only access APIs that support VPC Service Controls?

restricted.googleapis.com (Restricted VIP). It resolves to 199.36.153.4/30 and strictly limits access to VPC-SC supported services, blocking all others (like consumer Gmail).

Q2: True or False: You can change an NCC hub's preset topology from Mesh to Star after it has been created.

False! The preset topology (Mesh or Star) must be specified at hub creation and cannot be changed later. You would have to delete the hub and recreate it.

Q3: Why is Private Service Connect (PSC) preferred over Private Services Access (PSA) for complex hub-and-spoke networks?

Because PSC uses a local IP endpoint in a consumer subnet, making it fully compatible with transitive routing. PSA uses VPC Peering, which is non-transitive and blocks on-premises or cross-VPC routing natively.

Q4: What is the recommended first step when deploying a new VPC Service Controls perimeter to prevent breaking existing applications?

Deploy the perimeter in Dry-Run Mode! This logs all policy violations in Cloud Logging without actually blocking any API requests, allowing you to safely troubleshoot and write ingress/egress policies.

📝 Theme 2 Practice Exam Questions

Let's see if you can handle these actual exam-style private access and security perimeter brain busters!

Question 1: Your company has deployed a secure data warehouse in BigQuery inside a VPC Service Controls perimeter. An automated pipeline running on-premises needs to load data into BigQuery over a Dedicated Interconnect connection. The pipeline is failing with "Request blocked by VPC Service Controls" errors. How should you resolve this securely?

  • A) Configure on-premises DNS to resolve BigQuery APIs to the public IP of a transit gateway VM in the Hub VPC, and create a custom route that bypasses VPC Service Controls checks by using a hierarchical firewall policy with a "target-service" tag.
  • B) Configure on-premises DNS to resolve BigQuery APIs to restricted.googleapis.com (199.36.153.4/30), advertise this range over BGP, and write a VPC-SC Ingress Policy allowing access from the on-premises pipeline's public IP or service account identity.
  • C) Configure on-premises DNS to resolve BigQuery APIs to private.googleapis.com (199.36.153.8/30), advertise this range over dynamic BGP, and configure the service perimeter in dry-run mode to permanently auto-approve incoming pipeline connections.
  • D) Create a VPC-SC Perimeter Bridge between the on-premises network gateway project and the BigQuery project, and configure a Cloud Router NAT gateway to perform SNAT on all incoming API requests to hide their on-premises origin.

Correct Answer: B

Why Option B is correct: To access VPC-SC protected APIs from on-premises, you must route traffic through the Restricted VIP (restricted.googleapis.com). You must also configure an Ingress Policy on the service perimeter to explicitly authorize the on-premises pipeline's identity (service account) and network origin to cross the perimeter boundary. 🔒

  • A: Hierarchical firewall policies and transit gateway VMs cannot bypass or override VPC Service Controls perimeter checks, which are enforced at the Google API gateway layer.
  • C: private.googleapis.com does not support VPC-SC perimeter enforcement, and dry-run mode only logs violations without enforcing them; it does not "auto-approve" or resolve blocking errors once enforced.
  • D: You cannot create a VPC-SC Perimeter Bridge with an on-premises network; bridges are strictly for connecting projects within different perimeters. SNAT does not bypass VPC-SC boundary rules.

Question 2: You are designing a multi-tenant SaaS platform on Google Cloud. You have a central Producer VPC containing a shared analytics service. You want to expose this service to hundreds of independent Customer VPCs. The customer networks must not be able to communicate with each other, and they must not have any direct VPC peering connections to your Producer VPC to avoid IP address space overlaps. What is the recommended architecture?

  • A) Establish pairwise VPC Network Peering links between the Producer VPC and every Customer VPC, and configure custom route exchange properties to dynamically re-translate overlapping CIDR subnets using Cloud NAT.
  • B) Deploy Private Service Connect (PSC). Create a PSC Service Attachment in the Producer VPC behind an Internal Load Balancer, and have each customer create a local PSC Endpoint (Private IP) in their own VPC pointing to your Service Attachment.
  • C) Configure a Network Connectivity Center (NCC) hub with a Star topology, placing the Producer VPC in the Center group and all Customer VPCs in the Edge group, then enable dynamic BGP route leaking across all spokes.
  • D) Use Cloud VPN to establish secure IPSec tunnels between the Producer VPC and every Customer VPC, and deploy a pool of third-party proxy VMs in the Producer VPC to perform destination NAT on overlapping traffic.

Correct Answer: B

Why Option B is correct: This is the textbook use case for Private Service Connect (PSC)! By using PSC, customers can access the shared producer service via a local IP endpoint in their own subnet. This completely avoids VPC Peering limits, prevents any IP address conflicts (since NAT is handled natively by PSC), and ensures customers are completely isolated from each other. 🌟

  • A: VPC Peering has a strict limit on the number of active peerings, requires non-overlapping IP address spaces, and Cloud NAT cannot translate peered traffic dynamically to resolve IP overlaps.
  • C: While NCC Star topology isolates Edge spokes from each other, it still requires coordinating non-overlapping IP address allocations across all spokes, which is impossible to guarantee with hundreds of independent customers.
  • D: Cloud VPN has massive configuration and cost overhead, and deploying proxy VMs to manage hundreds of overlapping NAT configurations introduces a massive bottleneck and operational nightmare.

Question 3: You are configuring a Network Connectivity Center (NCC) hub to connect several regional retail branch offices to your central Google Cloud VPC. The branch offices are connected via HA VPN. You want to ensure that branch offices can communicate directly with the central VPC, but branch offices must strictly be blocked from routing traffic to each other through Google's network. How should you configure the NCC hub?

  • A) Create the NCC hub with a Mesh topology and attach all branch VPNs as hybrid spokes, then configure separate VPC firewall rules in the central VPC to drop inter-branch traffic at the egress layer.
  • B) Create the NCC hub with a Star topology. Place the central VPC spoke in the Center group, and place all branch office hybrid spokes in the Edge group.
  • C) Use dynamic VPC Network Peering instead of NCC to connect all branches, and configure the peering links to import custom routes but restrict BGP dynamic route advertisements at the Host Project level.
  • D) Create the NCC hub with a Star topology. Place all branch office hybrid spokes in the Center group, and the central VPC in the Edge group, then disable BGP route advertisements on all branch Cloud Routers.

Correct Answer: B

Why Option B is correct: NCC's Star topology is designed exactly for this! In a Star topology, spokes in the Edge group can communicate with spokes in the Center group, but Edge spokes are strictly blocked from communicating with other Edge spokes. Placing the branches in the Edge group and the central VPC in the Center group enforces this isolation natively. 🌟

  • A: Mesh topology allows all spokes to communicate with each other, meaning branches would be able to route traffic to other branches over your GCP network. Egress firewall rules do not prevent transit routing at the NCC layer.
  • C: You cannot terminate customer VPN tunnels directly inside a VPC Peering connection, and peering does not scale to support dynamic branch routing.
  • D: Placing branches in the Center group would allow them to communicate directly with each other, violating the security requirements. Disabling BGP advertisements would break basic cloud connectivity.

Question 4: You are troubleshooting a newly deployed VPC Service Controls perimeter. Developers report that their Cloud Run applications can no longer read files from a secure Cloud Storage bucket. You inspect the Cloud Logging logs. What is the most efficient way to identify the exact cause of the block without disrupting other services?

  • A) Delete the service perimeter, recreate it with Cloud Run excluded from the protected services list, and inspect the Cloud Audit Logs for permission-denied events to rebuild the perimeter.
  • B) Search Cloud Logging for protoPayload.metadata.violationReason and inspect the callerIp, serviceName, and authenticationInfo fields of the blocked requests to write an appropriate Ingress Policy.
  • C) Disable VPC Service Controls enforcement entirely, use the Access Context Manager to set up an IP-based access level for the Cloud Run service account, and rely on IAM permissions.
  • D) Grant the roles/storage.admin role to all user identities and the GKE service agents in the project, and configure a Cloud Storage bucket policy that bypasses IP-based restriction checks.

Correct Answer: B

Why Option B is correct: Cloud Logging captures detailed metadata for every VPC Service Controls violation. By searching for the specific violation logs, you can extract the exact caller identity, IP address, and target service, allowing you to write a precise, secure Ingress Policy to allow the legitimate traffic through. 🛠️

  • A: Deleting and recreating the perimeter is highly disruptive, breaches security compliance, and does not provide precise diagnostic details for the original configuration.
  • C: Disabling VPC-SC violates security compliance and exposes the system to data exfiltration. Access Context Manager IP levels do not apply directly to standard serverless Cloud Run identities without PGA or egress routing.
  • D: IAM roles do not override VPC Service Controls perimeters. Even a project Owner or Storage Admin will be blocked if they try to cross a perimeter boundary without an Ingress Policy!
