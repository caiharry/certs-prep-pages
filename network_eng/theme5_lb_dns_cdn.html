Theme 5: Load Balancing, Cloud DNS & Cloud CDN

Application, proxy, and passthrough load balancers, Cloud DNS routing policies, and Cloud CDN

Theme 5 🟢

5.1 Load Balancing Taxonomy: The Speed Demons

Google Cloud Load Balancing is built on Google's software-defined network virtualization stack (Maglev and Andromeda). It is incredibly fast, highly scalable, and doesn't require pre-warming! To pass this exam, you must memorize the current load balancer taxonomy and their distinct behaviors: ⚖️

Load Balancer Type Layer Proxy vs. Passthrough Deployment Modes Key Features & Best For
Application Load Balancer (ALB) Layer 7 (HTTP/S) Proxy (terminates client connection, starts new one to backend) Global external, Regional external, Regional internal, Cross-region internal Content-based routing (URL maps), traffic splitting, URL rewrites, SSL offloading. Best for web apps! 🌐
Proxy Network Load Balancer (NLB) Layer 4 (TCP/SSL) Proxy (terminates client connection) Global external, Regional external, Regional internal, Cross-region internal Reverse proxy for non-HTTP TCP traffic. Supports SSL offloading and single global IP. Best for global databases or mail servers.
Passthrough Network Load Balancer (NLB) Layer 4 (TCP/UDP/ESP/ICMP) Passthrough (does not terminate connection; packets pass through unchanged) Regional external, Regional internal Preserves client source IP address, Direct Server Return (DSR). Blistering speed! Best for high-performance gaming, UDP, or tunneling. ⚡
The Client IP Preservation Trap! If an exam question asks you to design a load-balanced architecture where the backend servers MUST see the original client source IP address natively in the packet header, you cannot use an Application Load Balancer or Proxy Network Load Balancer (they terminate the connection, replacing the source IP with the LB's proxy IP). You MUST use a Passthrough Network Load Balancer!

5.2 Traffic Management & Backends

Load balancers distribute traffic to backends. GCP supports several backend types:

  • Instance Groups: Managed (MIGs) or Unmanaged. MIGs support auto-scaling based on CPU or load balancer capacity.
  • Network Endpoint Groups (NEGs):
    • Zonal NEGs: Direct routing to GKE Pod IPs (container-native load balancing).
    • Internet NEGs: Routes traffic to external backends located on-premises or in other clouds.
    • Serverless NEGs: Integrates the load balancer with Cloud Run, Cloud Functions, or App Engine.

🚗 Advanced Traffic Management on Application Load Balancers

Global External Application Load Balancers support advanced Layer 7 routing rules configured in URL Maps:

  • Traffic Splitting (Canary Deployments): Route 95% of traffic to production and 5% to a new "canary" release backend service.
  • Traffic Mirroring: Copy 100% of live production traffic and send it to a test environment in the background to profile performance without affecting users!
  • URL Rewrites & Redirects: Automatically rewrite incoming request paths (e.g., /images/logo.png to /static/logo.png) or redirect HTTP to HTTPS at the edge.

5.3 Cloud CDN: Speeding Up the Edge

Cloud CDN (Content Delivery Network) caches static content at Google's global edge co-location facilities (Points of Presence - PoPs), bringing content closer to users and reducing load on your backends.

  • Supported Origins: Managed Instance Groups, Cloud Storage buckets, Cloud Run, and external backends (via Internet NEGs).
  • Cache Invalidation: If you update an image or file, you don't want users seeing the old cached version. You can trigger a Cache Invalidation command to purge the cached content globally. Warning: Purging the entire cache (/) can overload your origin databases and servers as they get hit with a flood of new requests!*

5.4 Cloud DNS: The Global Address Book

Cloud DNS is a highly available, global managed DNS service. Let's look at the advanced routing policies that examiners absolutely love to test:

🗺️ Advanced DNS Routing Policies

Cloud DNS allows you to configure dynamic traffic steering on your resource record sets:

  • Weighted Round Robin (WRR): Distributes traffic across multiple IP addresses based on custom weights (e.g., sending 80% of queries to IP A and 20% to IP B). Excellent for canary testing or capacity balancing.
  • Geolocation (GEO): Directs users to the nearest healthy region.
    • Geofencing (--enable-geo-fencing): If enabled, traffic is strictly restricted to that region. If all endpoints in that region are unhealthy, queries will not failover to other regions (useful for strict data residency compliance). If disabled (default false), Cloud DNS will failover and route queries to the next nearest healthy region! 🌍
  • Failover: Configures a primary IP (like an active load balancer) and a backup IP (like a DR site).
    • Trickle Ratio (--backup-data-trickle-ratio): A value between 0 and 1. It sends a tiny, continuous percentage of live traffic to the backup target even when the primary is completely healthy! This is a massive best practice because it ensures your standby/DR environment is actually warm and working under live conditions, rather than failing silently on game day! 💧
"Why did the DNS record set apply a trickle ratio to its failover policy? Because it wanted to keep its backup environment warm, just in case the primary got a cold!" 🥶

🧠 Quick-Fire Warm-Up Quiz!

Test your recall before diving into the proper practice exam. No peeking!

Q1: Which load balancer type should you use to preserve the client's original source IP address natively?

Passthrough Network Load Balancer (NLB). Because it is not a proxy, packets are received by backend VMs with unchanged source and destination IP addresses.

Q2: What happens in a Geolocation DNS routing policy if geofencing is enabled (--enable-geo-fencing) and all endpoints in the nearest region fail health checks?

Queries will still be resolved to that region! Geofencing prevents traffic from failing over to other regions, prioritizing data residency compliance over availability. If disabled (default), it would fail over to the next nearest healthy region.

Q3: What is the purpose of the --backup-data-trickle-ratio flag in a Cloud DNS Failover routing policy?

It sends a tiny percentage of live traffic to the backup/standby target while the primary is healthy. This continuously tests and warms the DR environment so you know it works before an actual disaster occurs.

Q4: Can you use a Global External Application Load Balancer with Standard Tier network routing?

No! Standard Tier only supports regional load balancers. Global load balancing strictly requires Premium Tier routing.

📝 Theme 5 Practice Exam Questions

Let's see if you can handle these actual exam-style load balancing and DNS routing brain busters!

Question 1: You are architecting a global, high-performance web application. The application must support path-based routing (e.g., /api to backend pool A, /static to Cloud Storage), automatic SSL offloading, and traffic mirroring for background testing. Which load balancer should you deploy?

  • A) Global External Proxy Network Load Balancer configured with TCP SSL offloading and a custom URL map header-routing rule.
  • B) Global External Application Load Balancer
  • C) Regional External Passthrough Network Load Balancer backed by an Internal Application Load Balancer transit layer in each regional subnet.
  • D) Internal TCP/UDP Load Balancer integrated with an external Cloud NAT gateway running SNAT to handle traffic mirroring.

Correct Answer: B

Why Option B is correct: The Global External Application Load Balancer (L7) is the only load balancer that supports advanced HTTP/S traffic management features like path-based routing (URL maps), SSL offloading, and traffic mirroring natively at the edge. 🌐

  • A: Proxy NLB operates at Layer 4 (TCP) and does not understand HTTP paths or URLs, so it cannot perform path-based routing or URL mapping.
  • C: Passthrough NLB is a Layer 4 regional load balancer with no L7 routing, SSL offloading, or native traffic mirroring capabilities; backing it with an internal load balancer is unnecessarily complex and breaks SSL offloading.
  • D: Internal load balancers are for private VPC traffic only, and integrating them with Cloud NAT does not allow public-facing global routing, SSL termination, or native L7 traffic mirroring.

Question 2: You are designing a multi-region active-passive disaster recovery architecture. You have a primary Application Load Balancer in us-east1 and a standby Application Load Balancer in europe-west1. You want to configure Cloud DNS to automatically route users to the standby region if the primary region goes offline. However, you want to continuously send 1% of live production traffic to the standby region to ensure the standby servers are warm and functioning correctly. How should you configure this?

  • A) Configure a standard Geolocation DNS policy with geofencing enabled, set the European load balancer target as a dynamic fallback, and write a Cloud Run script to ping and check the primary region's availability.
  • B) Configure a Geolocation DNS policy, manually adjust the primary and passive target weights every hour using a Cloud Scheduler job, and route 1% of queries via a separate weighted CNAME record.
  • C) Configure a FAILOVER DNS routing policy with the primary set to the us-east1 LB, the backup set to a GEO policy pointing to europe-west1, enable health checking, and set the --backup-data-trickle-ratio to 0.01.
  • D) Configure a Weighted Round Robin (WRR) routing policy with equal weights assigned to both load balancers, and disable DNS health checks on the passive region to prevent automatic traffic shifting.

Correct Answer: C

Why Option C is correct: This is the exact operational best practice for active-passive DR! The FAILOVER routing policy automatically switches traffic to the backup region if the primary fails health checks. Setting the --backup-data-trickle-ratio to 0.01 forces Cloud DNS to continuously route exactly 1% of queries to the standby region, keeping the standby environment warm and verified under live conditions! 💧

  • A: Geolocation policies do not support active-passive failover with custom trickle ratios natively, and external Cloud Run scripts are not integrated with Cloud DNS automatic dynamic failover.
  • B: Manual weight adjustment via Cloud Scheduler is slow, prone to delays, and does not support sub-minute automatic health-check-based failover.
  • D: Equal weights without health checks represents an active-active setup and will route 50% of traffic to the passive site constantly, violating the active-passive constraint and sending traffic to unhealthy nodes if they fail.

Question 3: You are deploying a high-performance multiplayer gaming backend on Compute Engine. The game client communicates over UDP and requires extremely low latency. The backend servers must be able to see the original public IP address of the players to perform geolocation-based matchmaking. Which load balancer should you choose?

  • A) Global External Application Load Balancer configured with a Proxy Protocol v2 header to carry the client IP down to the backend Instance Group.
  • B) Global External Proxy Network Load Balancer with SSL Session Ticket Resumption enabled to handle low-latency UDP encapsulation.
  • C) Regional External Passthrough Network Load Balancer
  • D) Regional Internal Application Load Balancer configured with Private Service Connect (PSC) endpoints in each regional subnet.

Correct Answer: C

Why Option C is correct: The Regional External Passthrough Network Load Balancer operates at Layer 4 and does not terminate the connection. Packets are forwarded directly to the backend VMs with their source IP addresses unchanged, allowing the game servers to see the players' original public IPs natively. It also supports UDP, which is critical for gaming! ⚡

  • A: Application Load Balancers only support HTTP/S traffic, not raw UDP, and they terminate the connection. Proxy Protocol v2 is only supported on certain Layer 4 proxies.
  • B: Proxy NLBs terminate the connection, do not natively support UDP, and do not forward the original raw IP address natively without proxy headers.
  • D: Internal load balancers are not accessible from the public internet, and PSC endpoints are for private consumption, not public game servers.

Question 4: You have configured a Geolocation DNS routing policy to route users to the nearest of three regional Application Load Balancers. To comply with strict European data privacy regulations (GDPR), you want to ensure that users located in Europe are strictly routed to the European load balancer, and their queries must never failover to US-based load balancers even if the European load balancer experiences an outage. How should you configure this?

  • A) Configure a standard Geolocation policy with no extra flags, and use GCP IAM permissions to block US-based Compute Engine instances from resolving European IP ranges.
  • B) Configure a Geolocation policy and enable Geofencing (--enable-geo-fencing) on the European record set.
  • C) Switch the entire DNS record architecture to a Weighted Round Robin policy with 100% weight on Europe and 0% on the US, and configure a Cloud DNS forwarding zone.
  • D) Disable health checking on the European load balancer and enable Cloud CDN caching globally to store European responses in US edge caches during local outages.

Correct Answer: B

Why Option B is correct: Enabling Geofencing (--enable-geo-fencing) on a Geolocation routing policy forces Cloud DNS to always route queries from that geographic region to the designated regional endpoints, even if all endpoints in that region are unhealthy. This guarantees compliance with strict data residency mandates (like GDPR) by preventing automatic failover to other jurisdictions! 🌍

  • A: Standard Geolocation policies (without geofencing) will automatically failover and route European users to US load balancers if the European load balancer goes offline. IAM permissions cannot block or control public DNS resolution behavior.
  • C: Weighted Round Robin does not evaluate the user's geographic location, so it cannot restrict US users from hitting Europe or vice versa.
  • D: Disabling health checking prevents the system from detecting outages but does not guarantee geofenced routing boundaries, and caching in US edge locations violates the strict data residency requirements.
