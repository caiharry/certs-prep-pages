6.1 Google Cloud Armor: The Shield of GCP

Leaving your web applications exposed to the raw public internet without a Web Application Firewall (WAF) is like inviting a group of rowdy vikings to your tea party! Google Cloud Armor is your global shield, protecting applications behind external HTTP/S load balancers from DDoS attacks and application-level exploits: 🛡️

⚔️ Cloud Armor Core Capabilities

  • Preconfigured WAF Rules (OWASP Top 10): Out-of-the-box protection against SQL Injection (SQLi), Cross-Site Scripting (XSS), Remote File Inclusion (RFI), and Local File Inclusion (LFI).
  • Adaptive Protection (ML Anomaly Detection): Uses machine learning to continuously analyze traffic patterns. If it detects a sudden, anomalous spike in traffic (like a Layer 7 DDoS attack), it automatically drafts a custom blocking rule for you to enforce with a single click! 🧠
  • Rate Limiting: Prevents brute-force login attempts or API scraping by capping the number of requests a single client IP can make within a time window.
  • Bot Management: Integrates natively with reCAPTCHA Enterprise to detect and block malicious scraping bots, credential stuffers, and automated spam.
  • Google Threat Intelligence: Automatically blocks traffic from known malicious IPs, Tor exit nodes, and active botnets using Google's global threat database.
Edge vs. Backend Security Policies!
  • Backend Security Policies: Attached to backend services. They support full WAF rules, L7 inspection, and rate limiting.
  • Edge Security Policies: Enforced at Google's global edge (Points of Presence) before traffic even reaches your load balancer. They only support basic IP-based filtering and geography-based blocking. Use Edge policies to block entire countries or blacklisted CIDR blocks at the absolute perimeter to save backend processing power!

6.2 Cloud Next-Generation Firewall (NGFW)

Google Cloud has modernized its firewall architecture. The legacy "VPC firewall rules" are being replaced by Cloud NGFW policies. Let's memorize the tiers and the evaluation order: 🧱

NGFW Tier Features Supported How It's Enforced
Essentials (Free) Basic L4 rules (IP, port, protocol), network tags, service accounts, and global/regional network firewall policies. Enforced natively in the Andromeda virtual network stack.
Standard Adds FQDN filtering (e.g., allow egress only to .github.com), Geo-IP matching, and Google Threat Intelligence lists. Enforced natively in Andromeda.
Enterprise Adds Layer 7 Packet Inspection (Intrusion Prevention System - IPS) to detect malware, spyware, and active exploits inside packets. Routes traffic through managed Firewall Endpoints in your subnets for deep packet inspection. 🔍

🚦 Firewall Rule Evaluation Order

When a packet enters your VPC, Google evaluates firewall rules in a strict top-down order. Memorize this sequence:

  1. Hierarchical Firewall Policies: Attached to the Organization or Folder level. They override everything below!
  2. Global Network Firewall Policies: Attached to the VPC network globally.
  3. Regional Network Firewall Policies: Attached to specific regions in the VPC.
  4. Legacy VPC Firewall Rules: The classic per-VPC rules.

Micro-segmentation Best Practice: Avoid network tags for critical security boundaries! Network tags are easily modified by VM administrators. Instead, use Secure Tags (governed by IAM) or Service Accounts to define secure micro-segments that cannot be bypassed!

6.3 Public Cloud NAT & Secure Web Proxy

How do private VMs safely initiate outbound connections to the internet (e.g., to download software updates) without exposing themselves to inbound attacks?

💧 Cloud NAT & Dynamic Port Allocation (DPA)

Cloud NAT provides managed Source NAT (SNAT) for private instances. The #1 production issue with Cloud NAT is SNAT Port Exhaustion (where VMs open too many outbound connections and run out of allocated source ports, causing connection drops and OUT_OF_RESOURCES errors).

🛠️ The Dynamic Port Allocation (DPA) Fix: Instead of statically assigning a fixed number of ports (like 64) to every VM regardless of usage, enable DPA!

  • How it works: Each VM starts with a minimum number of ports (e.g., 32). If a VM is close to exhausting its ports, Cloud NAT doubles its allocation (32 -> 64 -> 128 -> 256 -> 512 -> 1024) up to a configured maximum (e.g., 4096). When connection demand drops, ports are released back to the pool!
  • The Powers of 2 Rule: Both --min-ports-per-vm and --max-ports-per-vm MUST be powers of 2! (Min: 32 to 32768, Max: 64 to 65536).
🧠 Crucial Memory Tip: Secure Web Proxy (SWP)

If you need to secure and inspect outbound HTTP/HTTPS traffic (egress) from your VPC based on domain names, URLs, or web categories, don't use Cloud NAT (which only filters by IP/port) or complex NVA proxy VMs. Reach for Secure Web Proxy (SWP)! It is a fully managed, serverless egress proxy that doesn't require a NAT gateway or VM maintenance. 🌐

"Why did the Cloud NAT gateway get a power of 2 tattoo? Because it wanted to remind everyone that it only doubles port allocations dynamically!" 🔢

6.4 Self-Managed NVAs & Packet Mirroring

For some compliance frameworks, you must route all inter-VPC traffic through a pool of self-managed third-party firewall virtual machines (Network Virtual Appliances - NVAs) like Palo Alto or Fortinet.

To do this in a highly available, scalable manner, deploy the NVAs as multi-network interface card (multi-NIC) VMs behind an Internal Load Balancer (ILB). You configure Policy-Based Routing (PBR) in your VPC to intercept traffic and set the next hop to the ILB's forwarding rule IP address. This ensures traffic is distributed across healthy firewalls automatically! 🛡️

If you need out-of-band security analysis (like an Intrusion Detection System - IDS) without intercepting or slowing down live traffic, use Packet Mirroring. It clones a copy of all traffic from selected VM instances or subnets and forwards it to a pool of collector VMs behind an ILB for passive analysis.

🧠 Quick-Fire Warm-Up Quiz!

Test your recall before diving into the proper practice exam. No peeking!

Q1: Which Cloud NGFW tier is required to perform deep packet inspection (Layer 7 IPS) to detect active malware exploits?

Cloud NGFW Enterprise. It is the only tier that supports L7 packet inspection, which it enforces by routing traffic through managed Firewall Endpoints.

Q2: True or False: Hierarchical firewall policies can override rules defined in global or regional VPC network firewall policies.

True! Hierarchical policies (attached at the Org or Folder level) are evaluated first and take absolute precedence over VPC-level and regional-level policies.

Q3: What is the mathematical constraint on the minimum and maximum ports per VM when configuring Cloud NAT Dynamic Port Allocation?

Both values MUST be powers of 2 (e.g., 32, 64, 128, 256, 512, 1024, etc.).

Q4: What managed GCP service should you deploy to restrict outbound VM traffic to specific external domains (like .github.com) without managing proxy VMs?

Secure Web Proxy (SWP). It is a fully managed egress proxy that supports FQDN, URL, and category-based filtering for HTTP/S traffic.

Question 1: You are managing a high-traffic e-commerce website behind a Global External Application Load Balancer. During a major sales event, your backend servers experience a massive surge in CPU utilization. You inspect the logs and discover a distributed Layer 7 HTTP flood attack originating from thousands of rotating IP addresses. You need to mitigate this attack automatically without blocking legitimate shoppers. What should you do?

  • A) Create an Edge Security Policy in Cloud Armor to block the top 10 countries originating the traffic.
  • B) Enable Cloud Armor Adaptive Protection on your backend security policy. Allow it to analyze the traffic anomaly, draft a custom blocking rule, and apply it to block the attack signature.
  • C) Configure a rate-limiting rule in Cloud Armor to limit all users to 1 request per minute.
  • D) Switch the load balancer to a Passthrough Network Load Balancer to bypass Layer 7 parsing.

Correct Answer: B

Why Option B is correct: Cloud Armor Adaptive Protection uses machine learning to detect anomalous Layer 7 HTTP floods. It automatically identifies the unique signature of the attack traffic (across thousands of rotating IPs) and generates a precise blocking rule. This mitigates the attack instantly while allowing legitimate shoppers to continue browsing unaffected! 🛡️

  • A: Blocking entire countries will block thousands of legitimate shoppers, causing massive revenue loss.
  • C: A rate limit of 1 request per minute is far too restrictive and will break the website experience for legitimate users.
  • D: Passthrough NLB does not support Cloud Armor or L7 security policies, leaving your backends completely unprotected from the HTTP flood.

Question 2: Your company's security policy dictates that all outbound internet traffic from private Compute Engine VMs must be restricted. VMs must only be allowed to download software updates from .ubuntu.com and .github.com. You want to implement this egress filtering with minimal operational overhead and no VM-level proxy configurations. What should you deploy?

  • A) Deploy a Public Cloud NAT gateway and configure static routes.
  • B) Deploy Secure Web Proxy (SWP) in your VPC, configure an egress policy allowing .ubuntu.com and .github.com, and route outbound HTTP/S traffic through it.
  • C) Create VPC firewall rules blocking all egress traffic to 0.0.0.0/0.
  • D) Deploy a pool of squid proxy VMs in an auto-scaling group behind an Internal Load Balancer.

Correct Answer: B

Why Option B is correct: Secure Web Proxy (SWP) is a fully managed, serverless egress proxy designed exactly for this! It allows you to write granular egress policies based on FQDNs (like .ubuntu.com) for HTTP/S traffic with zero infrastructure to manage, representing the lowest operational overhead. 🌐

  • A: Cloud NAT only filters traffic at Layer 4 (IP/port). It cannot perform FQDN or wildcard domain filtering (like .github.com).
  • C: Blocking 0.0.0.0/0 via standard firewall rules blocks all internet traffic, preventing VMs from downloading updates entirely.
  • D: Managing a pool of squid proxy VMs introduces significant operational overhead (patching, scaling, configuration management) compared to the serverless Secure Web Proxy.

Question 3: You are managing a large GKE cluster with 1,000 microservice pods that frequently make outbound API calls to external payment gateways. During peak traffic hours, application logs report intermittent connection timeouts with OUT_OF_RESOURCES errors pointing to your Cloud NAT gateway. You inspect metrics and confirm SNAT port exhaustion. You want to resolve this immediately while minimizing the number of public IP addresses consumed by the gateway. What should you do?

  • A) Allocate 50 additional static public IP addresses to the Cloud NAT gateway.
  • B) Enable Dynamic Port Allocation (DPA) on the Cloud NAT gateway, setting --min-ports-per-vm=64 and --max-ports-per-vm=1024.
  • C) Increase the static minimum ports per VM to 4096.
  • D) Disable Cloud NAT and assign public IPs to all GKE nodes.

Correct Answer: B

Why Option B is correct: Enabling Dynamic Port Allocation (DPA) allows GKE nodes (VMs) that are experiencing heavy outbound connection spikes to dynamically double their port allocation (up to 1024), while keeping idle nodes at the minimum of 64 ports. This prevents SNAT port exhaustion (OUT_OF_RESOURCES errors) while maximizing the efficiency of your existing NAT IP pool! 💧

  • A: Adding 50 static IPs is expensive, wastes public IP resources, and represents a brute-force approach compared to dynamic scaling.
  • C: Increasing the static minimum to 4096 forces Cloud NAT to allocate 4,096 ports to every VM, even idle ones, which will immediately exhaust your NAT IP pool and require buying dozens of additional public IPs.
  • D: Assigning public IPs to GKE nodes exposes them directly to the internet, violating core security best practices.

Question 4: You are designing a secure VPC architecture for a government agency. The security team demands that all traffic entering or leaving your production VPC must undergo deep packet inspection to detect active malware exploits and spyware. The inspection must happen inline, and any detected threats must be blocked automatically. What is the recommended GCP native solution?

  • A) Enable Cloud Armor on all external load balancers.
  • B) Configure Cloud NGFW Enterprise and enable Layer 7 packet inspection with Intrusion Prevention System (IPS) security profiles.
  • C) Deploy Packet Mirroring to clone traffic to an IDS VM.
  • D) Enable Cloud NGFW Essentials and write hierarchical firewall rules.

Correct Answer: B

Why Option B is correct: Cloud NGFW Enterprise is Google's native solution for inline Layer 7 packet inspection. It integrates with an Intrusion Prevention System (IPS) powered by Palo Alto Networks threat signatures to detect and automatically block malware, spyware, and exploits inside packet payloads. 🔍

  • A: Cloud Armor is a WAF that protects incoming web traffic at the load balancer; it does not perform inline bidirectional packet inspection for standard VPC VM traffic.
  • C: Packet Mirroring is out-of-band (passive). It can detect threats but cannot block them inline because it only inspects a clone of the traffic.
  • D: Cloud NGFW Essentials only supports basic Layer 4 port/protocol rules and does not have Layer 7 packet inspection capabilities.
