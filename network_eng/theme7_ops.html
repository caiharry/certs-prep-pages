Theme 7 🟢

7.1 Logging & Monitoring: The Eyes and Ears

In a complex cloud network, you can't fix what you can't see! Google Cloud Observability provides deep logging and metrics for every network component. Let's look at the high-yield logs and metrics you must know for the exam: 📊

🔍 Core Network Logs

  • VPC Flow Logs: Captures samples of network traffic entering and leaving VM network interfaces. It records the standard 5-tuple (source IP, destination IP, source port, destination port, protocol) plus packet counts and latency. Essential for traffic auditing, security forensics, and cost optimization!
  • Firewall Rules Logging: Logs every packet that matches a firewall rule (both allowed and denied). Always enable this when debugging connectivity issues to see if a firewall is silently blocking your traffic!
  • Cloud NAT Logs: Logs connection translations and, crucially, OUT_OF_RESOURCES errors which flag SNAT port exhaustion!
  • VPC Service Controls Logs: Logs any API requests blocked by a security perimeter, detailing the exact violationReason and caller identity.

7.2 Network Intelligence Center (NIC)

Instead of digging through raw text logs for hours, Google Cloud provides Network Intelligence Center (NIC)—a single unified console for proactive network monitoring, visualization, and automated troubleshooting. Examiners love to test these modules: 🧠

NIC Module What It Does Best Use Case
Network Topology Generates an interactive, visual graph of your global VPC network, showing traffic flows, throughput, and packet loss between regions, subnets, and VMs. Visualizing global traffic patterns and identifying high-cost cross-region data transfers. 🌍
Connectivity Tests Simulates a packet's path from a source (VM, GKE, internet IP) to a destination, evaluating routing tables and firewalls along the way. Proactive troubleshooting! Checking if a VM can talk to a database before deploying, or verifying if a firewall change blocked a route. 🎯
Performance Dashboard Provides real-time packet loss and latency metrics across Google's global network, both project-scoped and Google-wide. Determining if a network slowdown is caused by a Google Cloud infrastructure issue or your own application.
Firewall Insights Analyzes firewall rule usage metrics. Identifies shadowed rules (rules that can never be hit because a higher-priority rule matches first) and unused rules. Firewall cleanup and hardening! Safely deleting redundant rules to optimize performance and security. 🛡️
Network Analyzer Continuously and automatically scans VPC configurations to detect suboptimal settings, shadowed routes, invalid next hops, and IP address exhaustion. Receiving proactive alerts for network failures (e.g., a route pointing to a deleted VM next hop) before users report outages! 🚨
Flow Analyzer Enables deep, interactive analysis of VPC Flow Logs stored in BigQuery without needing to write complex SQL queries. Drilling down into specific traffic spikes or auditing security compliance.
Proactive vs. Reactive Troubleshooting! If a question asks how to automatically detect that a route's next-hop VM has been deleted, or that a subnet is running out of IP addresses, the answer is Network Analyzer (it continuously monitors configurations). If a question asks how to verify if a specific VM can talk to a database across a peered VPC, the answer is Connectivity Tests (it runs a simulated packet test).

7.3 Maintaining & Troubleshooting Connectivity

Let's look at the classic real-world break-fix scenarios that examiners love to throw at you:

🚨 Troubleshooting BGP Peering Issues

If your hybrid connection is down, check your Cloud Router BGP session state in Cloud Logging:

  • Active or Idle State: The BGP session is down! This usually means there is a physical link failure (Interconnect) or an IPSec tunnel failure (VPN) preventing the routers from talking. Check local firewall rules (UDP ports 500 and 4500 must be open for IPSec!) and verify BGP pre-shared keys.
  • Established State: The BGP session is healthy! If traffic is still failing, check if you are advertising the correct subnets or if a VPC firewall rule is blocking the data plane.
  • Bidirectional Forwarding Detection (BFD): Always enable BFD on your BGP sessions! It detects sub-second link failures, triggering instant failover to backup paths much faster than standard BGP keepalive timers.
🧠 Crucial Memory Tip: ALB Connection Draining

If you need to perform maintenance on a backend VM behind an Application Load Balancer (ALB) without dropping active user sessions, enable Connection Draining (also known as Deregistration Delay). The load balancer will stop sending new requests to that VM, but will allow existing, active connections to complete gracefully within a configured timeout window (e.g., 300 seconds) before shutting down the VM. No dropped checkouts! 🛒

"Why did the network engineer love Connectivity Tests? Because they could simulate their packet's journey and prove to the developers that the firewall wasn't the one blocking the connection!" 🛡️

🧠 Quick-Fire Warm-Up Quiz!

Test your recall before diving into the proper practice exam. No peeking!

Q1: Which NIC module should you use to identify redundant, shadowed, or unused firewall rules?

Firewall Insights. It continuously analyzes firewall logs and configuration metrics to flag rules that can be safely deleted or tightened.

Q2: True or False: Connectivity Tests send actual live synthetic traffic across your network to test paths.

False! Connectivity Tests perform a logical configuration simulation of the packet path, evaluating routes, firewalls, and perimeters in the control plane. It does not send live data packets.

Q3: What BGP feature should you enable to detect hybrid link failures in sub-seconds and trigger instant routing failover?

BFD (Bidirectional Forwarding Detection). It provides sub-second link failure detection for BGP sessions.

Q4: What error code should you search for in your Cloud NAT logs to confirm SNAT port exhaustion?

OUT_OF_RESOURCES. This error indicates that a VM has exhausted its allocated NAT ports and cannot open new outbound connections.

📝 Theme 7 Practice Exam Questions

Let's see if you can handle these actual exam-style operations and troubleshooting brain busters!

Question 1: You are troubleshooting a connectivity issue where VMs in a newly created GKE node pool cannot pull container images from an external registry. The GKE nodes are private and use Cloud NAT for internet access. You suspect SNAT port exhaustion. What is the most efficient way to verify this using Google Cloud Observability?

  • A) Configure VPC Flow Logs on the private GKE subnet and write a custom SQL query in BigQuery to parse packet payload sizes and count local MTU fragmentation drops.
  • B) Search Cloud Logging for the OUT_OF_RESOURCES error in the dropped_sent_packets_count metric or Cloud NAT gateway logs.
  • C) Run a manual Connectivity Test in Network Intelligence Center from the GKE node interfaces to the external registry's public IP using dynamic protocol packet simulation.
  • D) Delete and recreate the Cloud NAT gateway with the static port range mapping parameter changed to support dual-stack IPv4/IPv6 egress.

Correct Answer: B

Why Option B is correct: Cloud NAT automatically logs OUT_OF_RESOURCES errors in Cloud Logging whenever a VM attempts to open an outbound connection but has exhausted its allocated SNAT ports. This is the fastest, most direct way to confirm port exhaustion! 🚨

  • A: VPC Flow Logs capture traffic samples but do not explicitly flag Cloud NAT port allocation failures or resource exhaustion errors, nor do they help count MTU drops.
  • C: Connectivity Tests check routing and firewall rules, but they do not simulate dynamic port allocation limits or SNAT port exhaustion under live load.
  • D: Recreating the gateway is a highly disruptive action that does not diagnose the root cause and will not prevent the issue from reoccurring under load.

Question 2: You have configured a custom static route in your VPC that routes traffic destined for a secure partner network through a specific third-party firewall VM (NVA). The partner reports that they are no longer receiving data. You suspect the firewall VM has crashed or been deleted. Which Network Intelligence Center module will automatically detect this suboptimal configuration and alert you?

  • A) Connectivity Tests configured to run as a recurring scheduled task targeting the secure partner network's external endpoints.
  • B) Network Topology dashboard set to alert on active traffic path changes and node latency anomalies across peered VPC networks.
  • D) Performance Dashboard configured to monitor packet loss and round-trip times (RTT) for the specific firewall VM interface.

Correct Answer: C

Why Option C is correct: Network Analyzer continuously and automatically monitors your VPC configurations. It detects invalid next hops (such as a static route pointing to a VM that is stopped, crashed, or deleted) and publishes an active configuration error insight, allowing you to proactively fix the route! 🚨

  • A: Connectivity Tests can diagnose the issue, but they must be run manually and do not continuously monitor configurations in the background.
  • B: Network Topology visualizes traffic flows but does not automatically flag invalid route next hops or generate configuration alerts.
  • D: Performance Dashboard monitors latency and packet loss but does not analyze routing configurations or next-hop VM states.

Question 3: You have established a Dedicated Interconnect connection to Google Cloud. The physical link is up, but your Cloud Router BGP session is stuck in the Active state, and no routes are being learned. You want to troubleshoot this. What is the most likely cause?

  • A) The VPC dynamic routing mode is set to Global, which prevents BGP sessions from establishing across regions unless a regional route reflector is configured.
  • B) A VPC firewall rule is blocking TCP port 179 (BGP) on the Cloud Router's link-local IP addresses, or there is an ASN/IP mismatch in the BGP peer configuration.
  • C) The physical Dedicated Interconnect port is not using MACsec encryption, causing Cloud Router to drop unencrypted packets at the physical Layer 1 layer.
  • D) Connection Draining is enabled on the Cloud Router's dynamic routing protocol layer, which blocks new routes from being exchanged with the peer.

Correct Answer: B

Why Option B is correct: When a BGP session is stuck in Active or Idle, the routers are failing to establish the TCP connection (BGP uses TCP port 179). This is almost always caused by a firewall blocking port 179, a mismatch in the configured BGP peer IPs (link-local IPs), or a mismatched Autonomous System Number (ASN) configuration! 🔌

  • A: Global dynamic routing does not prevent BGP sessions from establishing; it only controls how learned routes are propagated.
  • C: MACsec provides Layer 2 encryption but is optional; its absence does not prevent standard BGP sessions from establishing.
  • D: Connection Draining is a load balancer feature, not a Cloud Router feature.

Question 4: You need to perform a rolling upgrade on a pool of Compute Engine VMs behind an external Application Load Balancer. You want to ensure that active users currently in the middle of checking out do not experience any connection drops or error pages when their target VM is shut down. What should you configure?

  • A) Configure a Geolocation DNS routing policy in Cloud DNS to route all new traffic to a separate temporary regional load balancer while you upgrade the VMs.
  • B) Enable Connection Draining (Deregistration Delay) on the backend service, allowing active sessions to complete before VMs are terminated.
  • C) Delete the VMs immediately during the maintenance window to force the Application Load Balancer to perform sub-second failover to the healthy instances.
  • D) Enable Cloud Armor rate-limiting on the backend security policy to block new connection attempts, forcing active users to retry their checkouts.

Correct Answer: B

Why Option B is correct: Connection Draining (Deregistration Delay) tells the load balancer to stop sending new requests to a VM marked for removal, while keeping existing connections open to complete their transactions gracefully within a timeout window. This ensures zero disruption for active checkout sessions! 🛒

  • A: DNS caching makes DNS-based redirection far too slow (minutes or hours) to protect active web sessions during a rolling VM upgrade.
  • C: Deleting VMs immediately will instantly sever active user connections, causing failed checkouts and HTTP 502/503 error pages.
  • D: Cloud Armor rate limiting blocks malicious traffic but does not manage graceful backend VM lifecycle transitions.
