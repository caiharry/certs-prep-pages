Section 1.1

Managing Cloud Identity

Centralise identity: bring users under Cloud Identity, federate to your IdP for SSO, automate the user lifecycle, and lock down the super admin.

WeightSection 1: ~25%
Cloud Identity GCDS SSO / SAML Super admin Workforce Identity Federation

Official guide bullets

  • Configuring Google Cloud Directory Sync and implementing SSO with a third-party IdP.
  • Managing a super administrator account.
  • Automating the user lifecycle management process.
  • Administering user accounts and groups programmatically.
  • Configuring Workforce Identity Federation.

What to know

  • Cloud Identity is the IdP/identity source of truth for Google Cloud — the fix for developers using personal Gmail accounts (centralised management + enforceable security).
  • GCDS (Google Cloud Directory Sync) one-way syncs users/groups from AD/LDAP into Cloud Identity (it never writes back to your directory).
  • SSO: Cloud Identity can act as SAML IdP for third-party SaaS, or as a SAML SP federating to your existing IdP (Okta, ADFS, Ping) so users authenticate against your directory.
  • Super admin: very few, hardware-key 2SV, no day-to-day use, break-glass only; don't use it as a project owner.
  • Workforce Identity Federation: let external workforce identities access Google Cloud without provisioning them into Cloud Identity — short-lived access via your existing IdP.
Exam discriminator: "developers using consumer Gmail for company work" → bring them under Cloud Identity (centralised control), not just adding 2SV to the Gmail accounts. "Keep using our existing directory as source of truth" → federate (Cloud Identity as SAML SP / Workforce Identity Federation), don't migrate users away from it.

Section 1.2

Managing service accounts

Go keyless wherever possible. Prefer Workload Identity Federation and impersonation over downloaded keys, and protect the default service accounts.

WeightSection 1: ~25%
Workload Identity Federation Impersonation Short-lived credentials Key hygiene Default SAs

Official guide bullets

  • Securing and protecting service accounts (including default service accounts).
  • Identifying scenarios requiring service accounts.
  • Creating, disabling, and authorizing service accounts.
  • Securing, auditing, and mitigating the usage of service account keys.
  • Managing and creating short-lived credentials.
  • Configuring Workload Identity Federation.
  • Managing service account impersonation.

What to know

  • Keyless first: attach a SA to the resource (VM/GKE/Cloud Run) or use Workload Identity Federation for external/CI workloads (GitHub Actions, other clouds, on-prem). Avoid downloaded JSON keys.
  • Impersonation + short-lived credentials (generateAccessToken) let a user/SA act as another SA for a bounded time — better than handing out keys.
  • If keys are unavoidable: rotate centrally, distribute from a controlled location (never commit to source), restrict who can create them, and use the disable SA key creation org policy.
  • Default SAs: the Compute/App Engine defaults are over-privileged (Editor). Disable automatic role grants, replace with least-privilege user-managed SAs.
  • Use SA Insights / Policy Intelligence to find unused SAs and keys.

Section 1.3

Managing authentication

Strong, enforced authentication: password and session policies, SAML/OAuth federation, and mandatory 2-step verification (ideally phishing-resistant security keys).

WeightSection 1: ~25%
Password policy Session management SAML / OAuth 2-Step Verification

Official guide bullets

  • Creating a password and session management policy for user accounts.
  • Setting up SAML and OAuth.
  • Configuring and enforcing 2-step verification.

What to know

  • Password policy: enforce length/strength and reuse rules in the Admin console; Google Cloud session length controls reauthentication frequency for the Console/gcloud.
  • SAML = federated SSO (authentication). OAuth 2.0 / OIDC = delegated authorization to apps/APIs. Restrict which third-party OAuth apps can access org data (app access control).
  • 2SV: enforce org-wide; security keys / passkeys are phishing-resistant and the strongest factor — preferred for admins and sensitive roles.
  • Context-Aware Access can layer device/location conditions on top of authentication.
Exam signal: "phishing-resistant MFA" → security keys (FIDO) / passkeys, not SMS/OTP. "SSO across SaaS" → SAML; "let an app act on a user's behalf to an API" → OAuth.

Further reading

  • Session length
  • Enforce 2SV
  • Context-Aware Access

Section 1.4

Authorization controls

Least privilege done with the full IAM toolbox: predefined/custom roles, conditions, deny policies, groups, Policy Intelligence, Access Context Manager, and PAM for just-in-time elevation.

WeightSection 1: ~25%
Least privilege IAM conditions / deny Policy Intelligence Access Context Manager Privileged Access Manager

Official guide bullets

  • Managing privileged roles and separation of duties with IAM roles and permissions.
  • Managing IAM and ACL permissions.
  • Granting permissions using IAM conditions and IAM deny policies.
  • Defining access control at org/folder/project/resource level using least privilege.
  • Configuring Access Context Manager.
  • Applying Policy Intelligence.
  • Managing permissions through groups.
  • Identifying use cases and configuring Privileged Access Manager.

What to know

  • Roles: prefer predefined over basic (Owner/Editor/Viewer are too broad); build custom roles when predefined is too wide. Grant to groups, not individuals.
  • IAM conditions: attribute-based grants (time, resource name, request). IAM deny policies: explicit denials that override allows — guardrails even if someone grants a role.
  • Policy Intelligence: Recommender trims over-granted roles, Policy Analyzer answers "who can do what", Policy Troubleshooter explains an access decision.
  • Access Context Manager: defines access levels (IP, device, region) used by VPC Service Controls and Context-Aware Access.
  • PAM: just-in-time, time-bound, approval-gated elevation (entitlements → grants) instead of standing privilege. JIT access
  • Separation of duties: split sensitive abilities (e.g. key admin vs key user; deployer vs approver).

Section 1.5

Defining the resource hierarchy

Org → folders → projects → resources. IAM and org policies inherit downward — set guardrails high, and use org policies to prevent (not just detect) bad configurations.

WeightSection 1: ~25%
Org / folders / projects Inheritance Org Policy Service Custom org policies

Official guide bullets

  • Managing folders and projects at scale.
  • Managing pre-built or custom organization policies for the org, folders, and projects.
  • Using the resource hierarchy for access control and permissions inheritance.

What to know

  • Inheritance: IAM and org policies flow downward; apply at the highest level that avoids repetition (org/folder), with project-level overrides where allowed.
  • Org Policy Service sets constraints that block non-compliant creation up front: Resource Location Restriction (residency), domain-restricted sharing, disable SA key creation, restrict external IPs, require OS Login, etc.
  • Custom org policies (custom constraints) extend the catalogue to your own rules.
  • Org policy is preventive; SCC / Security Health Analytics is detective — the exam loves this distinction.

Exam discriminator: "ensure resources can only be created in the EU" → Resource Location Restriction org policy (prevents creation), not an SCC finding or a log alert (those detect after the resource already exists — already a breach).

Section 2.1

Perimeter security

Layered edge defence: Cloud NGFW for L3-L7 filtering, Cloud Armor for WAF/DDoS, IAP for app-level access, plus DNS security and CA Service for certs.

WeightSection 2: ~22%
Cloud NGFW Cloud Armor (WAF) IAP Secure Web Proxy Cloud DNS / DNSSEC CA Service

Official guide bullets

  • Configuring network perimeter controls (Cloud NGFW rules/policies, IAP, load balancers, Certificate Authority Service).
  • Setting up application-layer (L7) inspection on Cloud NGFW.
  • Differentiating private and public IP addressing.
  • Configuring web application firewalls (Google Cloud Armor).
  • Deploying Secure Web Proxy.
  • Configuring Cloud DNS security settings.
  • Continually monitoring and restricting configured APIs.

What to know

  • Cloud NGFW: hierarchical/global/network firewall policies, FQDN & geo & threat-intel objects, and L7 inspection / intrusion prevention. Prefer service-account-targeted rules over network tags (tags can be self-assigned by anyone who can edit a VM).
  • Cloud Armor: edge WAF + DDoS on external load balancers — OWASP rules, rate limiting, geo/IP allow/deny, bot management.
  • IAP: identity-based access to apps/VMs/SSH without a VPN — the answer for "differentiated access by user identity over HTTPS".
  • Secure Web Proxy: secured, policy-controlled egress to the internet (URL/domain allow-lists, TLS inspection).
  • Cloud DNS: enable DNSSEC; use DNS policies / response policy zones to control resolution.
  • CA Service: managed private CA for issuing internal TLS certs.
Exam discriminator: "differentiated access to an app by user identity over the web" → IAP, not VPC firewall rules (which only allow/deny traffic, not who). "Restrict use of a firewall rule so only authorised engineers can rely on it" → target a service account, not a network tag.

Further reading

  • Cloud NGFW
  • Cloud Armor
  • Identity-Aware Proxy
  • Secure Web Proxy
  • CA Service

Section 2.2

Boundary segmentation

Segment with VPC design, firewall rules, and Shared VPC; then wrap data services in a VPC Service Controls perimeter to stop exfiltration even by authorised identities.

WeightSection 2: ~22%
VPC / Shared VPC / peering Firewall rules N-tier isolation VPC Service Controls

Official guide bullets

  • Configuring security properties of a VPC, VPC peering, Shared VPC, and firewall rules.
  • Configuring network isolation and data encapsulation for N-tier applications.
  • Identifying use cases and configuring VPC Service Controls.

What to know

  • Shared VPC: central host project owns the network; service projects attach — one team owns networking/security for many projects.
  • VPC peering: private RFC 1918 connectivity between two VPCs, non-transitive; peers stay administratively separate.
  • Firewall rules: default = deny ingress / allow egress. Segment N-tier apps (web/app/db) with SA-targeted rules and separate subnets; deny egress by default for sensitive tiers.
  • VPC Service Controls: a service perimeter around managed APIs (Cloud Storage, BigQuery, etc.) to prevent data exfiltration — even a stolen credential can't move data outside the perimeter. Use access levels + ingress/egress rules; dry-run first.
Exam discriminator: "stop data being exfiltrated from BigQuery/Cloud Storage even with valid creds" → VPC Service Controls perimeter (it's not a tool to find sensitive data — that's SDP). "Many projects share one network centrally" → Shared VPC; "connect two existing VPCs privately" → peering.

Section 2.3

Establishing private connectivity

Reach Google APIs and on-prem privately: Private Google Access and PSC for APIs, HA VPN / Interconnect for data centres, Cloud NAT for controlled egress.

WeightSection 2: ~22%
Private Google Access Private Service Connect HA VPN / Interconnect Cloud NAT

Official guide bullets

  • Private connectivity between VPCs and projects (Shared VPC, peering, Private Google Access for on-prem).
  • Private connectivity and encryption between data centers and VPC (HA VPN, Cloud Interconnect).
  • Private connectivity between VPC and Google APIs (Private Google Access, restricted Google access, Private Service Connect).
  • Using Cloud NAT to enable outbound traffic.

What to know

  • Private Google Access: VMs with no external IP reach Google APIs over internal IPs. restricted.googleapis.com VIP keeps API access inside a VPC-SC perimeter.
  • Private Service Connect: private endpoints to Google APIs or to producer services — no peering, your own internal IP.
  • HA VPN (IPsec over internet, 99.99%) vs Cloud Interconnect (Dedicated/Partner, private, high bandwidth, low latency) for data-centre links. Interconnect alone isn't encrypted — add MACsec/IPsec if required.
  • Cloud NAT: managed, scalable egress for instances without external IPs — outbound only, nothing inbound.
Exam discriminator: "VMs with no public IP must call Google APIs" → Private Google Access. "Outbound internet for private VMs without exposing them" → Cloud NAT. "Private, high-bandwidth, consistent link to a data centre" → Dedicated Interconnect (HA VPN if over the internet is acceptable).

Section 3.1

Sensitive data & secrets

Discover and de-identify PII with Sensitive Data Protection, lock data services down (VPC-SC, IAM, public-access prevention), and keep credentials in Secret Manager.

WeightSection 3: ~23%
Sensitive Data Protection De-identify / redact Secret Manager Restrict data services VM metadata

Official guide bullets

  • Configuring Sensitive Data Protection (discover/redact PII, pseudonymization, format-preserving encryption).
  • Restricting access to data services (BigQuery, Cloud Storage, Cloud SQL).
  • Securing secrets with Secret Manager.
  • Protecting and managing compute instance metadata.

What to know

  • Sensitive Data Protection (formerly Cloud DLP): discover, inspect, and de-identify PII in storage and streams. Techniques: redaction, masking, tokenization/pseudonymization, format-preserving encryption, date shifting. SDP
  • infoTypes: built-in detectors + custom regex / dictionary / stored infoTypes — use a regex custom infoType to match a pattern (e.g. an internal email domain) reliably rather than a hand-maintained list.
  • Restrict data services: bucket-level least-privilege IAM, public access prevention + uniform bucket-level access, VPC-SC perimeter, and signed URLs for scoped access.
  • Secret Manager: versioned secrets, IAM-scoped, CMEK-encryptable, with rotation — never hardcode creds or commit them.
  • VM metadata: protect/lock metadata, disable legacy endpoints, enforce OS Login, and don't store secrets in startup scripts/metadata.
Exam discriminator: "find where PII lives / redact it" → Sensitive Data Protection (VPC-SC and Bucket Lock do not locate PII). "Redact some emails but keep an internal domain" → regex custom infoType, not a static dictionary that drifts and misses entries.

Section 3.2

Encryption & key management

Everything is encrypted at rest by default. The exam is about choosing default vs CMEK vs CSEK vs EKM, software vs HSM keys, and managing rotation/revocation. Full matrix on the synthesis slide.

WeightSection 3: ~23%
Default / CMEK / CSEK / EKM Cloud KMS / HSM Key rotation / revocation Object lifecycle Confidential Computing

Official guide bullets

  • Identifying use cases for Google default encryption, CMEK, and Cloud EKM.
  • Determining when to use software and hardware keys.
  • Creating and managing encryption keys for CMEK and EKM (rotation, revocation, import).
  • Applying encryption methods to various use cases.
  • Configuring object lifecycle policies for Cloud Storage.
  • Enabling Confidential Computing.

What to know

  • Default: Google-managed keys, AES-256, zero config — encryption at rest is always on.
  • CMEK: you control keys in Cloud KMS (software or HSM level, or imported) — rotation, disable/destroy, IAM separation of duties.
  • CSEK: customer-supplied raw AES-256 key provided per request; Google never stores it (Cloud Storage / Compute).
  • EKM: keys live in an external key manager (or Key Access Justifications) — for "hold your own keys outside Google / off-cloud control".
  • Confidential Computing: encrypts data in use in memory (Confidential VMs/GKE).
  • Object lifecycle: transition storage classes / delete by age — a cost/retention tool, not an immutability control (that's Bucket Lock retention).
Exam discriminator: "use our existing on-prem keys with Cloud Storage" → CSEK (you supply the raw key) — CMEK uses Google-hosted KMS keys, not your on-prem KMS. "Keys must stay outside Google" → Cloud EKM. Lifecycle rules don't prevent manual deletion — use Bucket Lock for immutable retention.

Further reading

  • Default encryption
  • CMEK
  • Cloud EKM
  • CSEK
  • Confidential Computing

Section 3.3

Securing AI workloads

Treat data, models, and endpoints as assets: least-privilege IAM, private networking and VPC-SC around Vertex AI, CMEK on data/models, and guardrails against data/model exploitation.

WeightSection 3: ~23%
Vertex AI controls VPC-SC / private endpoints CMEK on models/data Model Armor / safety IaaS vs PaaS training

Official guide bullets

  • Implementing security and privacy controls for AI/ML systems to protect against unintentional exploitation of data or models.
  • Determining security requirements for IaaS-hosted and PaaS-hosted training models.
  • Implementing security controls for Vertex AI.

What to know

  • Vertex AI controls: least-privilege IAM on datasets/models/endpoints, VPC-SC perimeter and private endpoints to keep training data and inference traffic off the internet, and CMEK on data and model artifacts.
  • Protect against exploitation: sanitise training data (no secrets/PII — run it through SDP), control who can read/export models (model exfiltration), and guard prompts/responses (Model Armor / safety filters) against prompt injection and sensitive-data leakage.
  • PaaS vs IaaS: PaaS (Vertex AI) inherits Google's managed security — you focus on IAM/network/keys. IaaS (self-managed training on GCE/GKE) means you also own OS hardening, patching, and image security.
  • Supply chain: scan training containers, pin trusted base images, and apply Binary Authorization to model-serving deployments.
Exam signal: AI questions reduce to the same primitives — IAM least privilege + private networking/VPC-SC + CMEK + clean data (SDP). "Stop the model leaking sensitive data / prompt-injection" → Model Armor / safety filters plus de-identified training data.

Section 4.1

Automating security

Shift security left: scan for CVEs in CI/CD, gate deploys with Binary Authorization, harden and patch images automatically, and manage posture/drift at scale.

WeightSection 4: ~19%
Artifact Analysis Binary Authorization Image hardening / VM Manager SHA custom modules CSPM / drift

Official guide bullets

  • Automating CVE scanning through a CI/CD pipeline.
  • Configuring Binary Authorization to secure GKE or Cloud Run.
  • Automating VM and container image creation (hardening, maintenance, patch management).
  • Managing policy and drift detection at scale (CSPM, custom org policies, custom modules for Security Health Analytics).

What to know

  • Artifact Analysis (a.k.a. Container Analysis): automatic vulnerability scanning of images in Artifact Registry; surfaces CVEs but does not patch.
  • Binary Authorization: deploy-time policy — only images with required attestations run on GKE/Cloud Run. Pair with scanning; use dry-run before enforce.
  • Keep images current: build on Google-managed base images (patched upstream) rather than startup-time update scripts; use VM Manager for OS patch management and Shielded/Confidential VMs for integrity.
  • Posture at scale: Security Health Analytics (in SCC) detects misconfigurations; write custom modules for org-specific checks, and custom org policies to prevent drift.

Section 4.2

Logging, monitoring & detection

Capture the right logs (audit, network, data access), route them securely with sinks, watch threats in Security Command Center, and export to your SIEM.

WeightSection 4: ~19%
Cloud Audit Logs VPC Flow / Cloud IDS / Packet Mirroring Aggregated sinks Security Command Center SIEM export

Official guide bullets

  • Configuring and analyzing network logs (Cloud NGFW, VPC Flow Logs, Packet Mirroring, Cloud IDS, Log Analytics).
  • Designing an effective logging strategy and secure access to logs.
  • Logging, monitoring, responding to, and remediating security incidents.
  • Exporting logs to external security systems.
  • Configuring and analyzing Cloud Audit Logs and data access logs.
  • Configuring log exports (log sinks and aggregated sinks).
  • Configuring and monitoring Security Command Center.

What to know

  • Cloud Audit Logs: Admin Activity (always on, free), Data Access (must enable; logs reads/writes), System Event, Policy Denied. Enable Data Access logs where the question demands "who read the data".
  • Network logs: VPC Flow Logs (flow metadata), Cloud IDS (managed intrusion detection via Packet Mirroring), Packet Mirroring (full packet capture to your tools).
  • Routing: aggregated sinks at org/folder level export logs from all projects to a central, access-controlled bucket / BigQuery / Pub/Sub (to a SIEM). Lock down the log bucket separately from the projects.
  • Security Command Center: central posture + threat view — Security Health Analytics, Event Threat Detection, Container Threat Detection, Web Security Scanner, attack path / exposure. Premium/Enterprise tiers add more.
Exam discriminator: "centralise security logs across all projects, tamper-resistant" → aggregated sink at the org/folder to a separate, locked-down destination (BigQuery/GCS/Pub/Sub) — not per-project copies. "Who accessed the data?" requires Data Access audit logs (off by default). Threat findings live in SCC.

Section 5.1

Compliance & regulatory standards

Know the shared-responsibility line, map a regime to GCP controls, scope what's in/out, and use Assured Workloads + Access Transparency/Approval for regulated estates.

WeightSection 5: ~11%
Shared responsibility Assured Workloads Access Transparency / Approval Data regionalization Control mapping

Official guide bullets

  • Determining technical needs relative to compute, data, network, and storage.
  • Evaluating the shared responsibility model.
  • Configuring security controls to support compliance (Assured Workloads, org policies, Access Transparency, Access Approval, regionalization).
  • Determining the environment in scope for regulatory compliance.
  • Mapping compliance requirements to Google Cloud services and controls (segmentation, audit log coverage).

What to know

  • Shared responsibility: Google secures the infrastructure; you secure IAM, data, configuration, and access. The line shifts toward Google for PaaS/SaaS vs IaaS.
  • Map regime → workload: PCI DSS = payment-card / e-commerce / POS; HIPAA = health data; FedRAMP = US government; SOX = financial reporting; GDPR = EU personal data.
  • Assured Workloads: enforces data residency, personnel controls, and compliance guardrails (e.g. regulated regions) for in-scope folders.
  • Access Transparency logs Google staff access to your data; Access Approval requires your explicit approval first.
  • Scope reduction: segment in-scope workloads (separate projects/VPC-SC perimeters), keep audit-log coverage complete, and regionalize data/services.
Exam discriminator: match the industry to the regime — retail/payments → PCI DSS; healthcare → HIPAA; US gov → FedRAMP. "Approve/limit Google support access to data" → Access Approval + Access Transparency. "Enforce residency + personnel controls" → Assured Workloads.
