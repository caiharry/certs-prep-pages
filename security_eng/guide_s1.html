Section 1~25%

Configuring Access

Identity, authentication, and least-privilege authorization — the biggest single slice of the exam.

Identity & federation

Cloud Identity is the IdP for Google Cloud users and groups. Sync from on-prem directories with Google Cloud Directory Sync (GCDS) and enable SSO with a third-party IdP. Use Workforce Identity Federation for human external identities and Workload Identity Federation for external workloads/CI — both avoid long-lived service-account keys. Protect the super-admin account with hardware 2SV.

Service accounts & keys

Prefer short-lived credentials and service-account impersonation over keys. Disable key creation org-wide with the iam.disableServiceAccountKeyCreation org policy. Lock down default service accounts. Use Workload Identity Federation so external systems get tokens without keys.

⚠️ Exam Trap: Service-account keys are the riskiest credential — the recommended answer is almost always impersonation, short-lived creds, or WIF, with key creation disabled by org policy.

IAM authorization & hierarchy

Grant least privilege using predefined/custom roles, refine with IAM Conditions, and explicitly block with IAM deny policies. Privileged Access Manager grants just-in-time elevated access. Apply policy across the resource hierarchy (org → folders → projects), where IAM and org policies inherit downward; manage permissions through groups, not individuals.

Order An IAM deny policy always overrides an allow. Deny is evaluated first.

IAM Conditions & context-aware access

Plain role bindings are blunt; IAM Conditions add attribute logic so a grant only applies when certain things are true — a request before an expiry date, traffic from a specific resource tag, or access during business hours. For human access to apps and the console, layer on context-aware access: access levels built in Access Context Manager evaluate device posture, IP range, and geo, then Identity-Aware Proxy (and Chrome Enterprise Premium / BeyondCorp) enforces them. The result is zero-trust: identity plus context, no VPN required.

  • IAM Condition — refines a single role binding (e.g. request.time < expiry or resource.matchTag).
  • Access level — a reusable device/network condition (managed device, corporate IP, country).
  • IAP — gates the app/SSH/RDP tunnel; honours the access level for the request.
WHEN vs WHO IAM roles answer who can do what; IAM Conditions and access levels answer when and from where they may do it.

Resource hierarchy & org policy guardrails

The hierarchy is organization → folders → projects → resources. Both IAM allow policies and Organization Policy constraints inherit downward — but they are different tools. IAM grants permissions; org policies set guardrails on configuration (which regions, whether external IPs are allowed, whether SA keys can be made). Manage access through Google Groups so leavers and joiners are handled in one place, and reserve IAM deny policies for hard blocks that must beat any allow.

  • IAM = who gets which permissions on a resource.
  • Org Policy = what configurations are even permitted (e.g. constraints/gcp.resourceLocations).
  • Deny policy = explicit prohibition evaluated before allows.
⚠️ Exam Trap: If a stem says "regardless of project-level permissions" or "even if an admin grants a role," the answer is an org policy or a deny policy — not another role binding, which an admin could simply re-add.

🧠 Warm-Up Quiz

Q1. Stop anyone from creating service-account keys across the whole org. What do you use?

Reveal answer

The iam.disableServiceAccountKeyCreation organization policy.

Org-policy guardrail inherited by all projects regardless of project IAM.

Q2. Grant a contractor elevated access only for a 2-hour change window. Which feature?

Reveal answer

Privileged Access Manager (just-in-time access).

Time-bound, audited privilege elevation.

Q3. A CI pipeline on GitHub Actions needs to deploy to Google Cloud without a stored secret. What do you configure?

Reveal answer

Workload Identity Federation.

The external OIDC token is exchanged for short-lived Google credentials, so no service-account key is ever stored.

Q4. Allow console access to a sensitive project only from corporate-managed devices on the company IP range. Which mechanism defines that condition?

Reveal answer

An access level in Access Context Manager (enforced via context-aware access / IAP).

Access levels encode device posture, IP, and geo for zero-trust enforcement.

📝 Exam-Style Practice

Q1. A data-science team and a separate platform team both work in one project. You must ensure the platform admins, who hold roles/owner, can never read objects in a specific Cloud Storage bucket holding regulated PII — even though they can edit IAM. What is the most reliable control?

  • A) Remove the Storage Object Viewer role from the platform admins on that bucket
  • B) Create an IAM deny policy denying storage object read permissions to the platform group on that bucket
  • C) Apply a bucket IAM condition that only grants access during business hours to the platform group
  • D) Move the bucket into a folder where the platform admins have no role bindings at all
Reveal answer & explanation

Correct: B. An IAM deny policy is evaluated before allow policies and cannot be overridden by a role binding, so even an owner who can edit IAM cannot grant themselves read access in the face of an explicit deny.

A: Owners can edit IAM, so they would simply re-add the role — removal is not durable against the very people you are restricting. C: A condition that allows access during business hours still permits reads (just within a window); it does not prohibit them. D: A bucket lives in a project, not a folder, so it cannot be moved into a separate folder away from the project's owners.

Q2. An external SaaS analytics vendor needs read access to one BigQuery dataset for the next 30 days only, after which access must lapse automatically with no manual cleanup. You want to avoid issuing any long-lived credential. Which approach best fits?

  • A) Create a service account, generate a JSON key, and email it to the vendor with a calendar reminder to revoke it
  • B) Add the vendor's Google identity to a group and grant the group BigQuery Data Viewer with no expiry
  • C) Grant the vendor's Google identity BigQuery Data Viewer with an IAM condition that expires after 30 days
  • D) Share a service-account impersonation grant and rely on the vendor to stop using it after a month
Reveal answer & explanation

Correct: C. An IAM Condition with a request.time expiry makes the grant lapse automatically at the deadline, needs no stored key, and requires no manual revocation.

A: A JSON key is a long-lived credential and manual revocation is exactly the failure mode you are told to avoid. B: A group grant with no expiry persists indefinitely and still relies on someone remembering to remove it. D: Impersonation here has no time bound and trusts the vendor to self-restrict, which is not enforceable.

Q3. Security wants to guarantee that no project anywhere in the organization can attach external IP addresses to VMs, and the guarantee must hold even if a project owner tries to enable them. Where do you enforce this?

  • A) A firewall rule at each VPC denying ingress from the public internet to the VMs
  • B) The compute.vmExternalIpAccess organization policy constraint set at the org node
  • C) A custom IAM role for project owners that omits the compute network update permission
  • D) A VPC Service Controls perimeter wrapping all the projects in the organization
Reveal answer & explanation

Correct: B. The compute.vmExternalIpAccess org policy constraint set at the organization is a guardrail that inherits to every project and cannot be overridden by a project owner.

A: A firewall controls traffic flow but does not stop an external IP being assigned, and it must be set per-VPC. C: A project owner can grant themselves a different role, so a custom role is not a durable guardrail. D: VPC-SC stops API data exfiltration across a service perimeter; it does not govern whether a VM may carry an external IP.
