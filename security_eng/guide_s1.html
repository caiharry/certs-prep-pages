Section 1: Configuring Access

Identity, authentication, and least-privilege authorization — the single biggest slice of the exam (~25%). Master who gets in, what they can touch, and how to slam the door on everyone else.

Section 1 🟣 ~25%

Welcome to the heaviest-weighted domain on the whole paper. If you nail nothing else, nail this. Roughly one in four questions lives here, and almost every other domain assumes you already understand the IAM hierarchy. Think of access control as the bouncer, the guest list, and the velvet rope all rolled into one — get any of them wrong and the data walks straight out the front door. 🕴️

1.1 Identity: The Front Door

Before anyone can be authorized to do anything, they must first be authenticated — proven to be who they claim. In Google Cloud, identities come in two flavours: human (people logging in) and workload (code, VMs, and pipelines acting on their own).

👤 Cloud Identity — Your Identity Provider (IdP)

Cloud Identity is Google's free IdP that holds your users and groups and underpins your Google Cloud Organization. If you already run an external IdP (Active Directory, Okta, Azure AD/Entra), you have two jobs:

  • Provision the accounts — sync users and groups in from on-prem AD with Google Cloud Directory Sync (GCDS). GCDS is one-way: AD is the source of truth, Cloud Identity is the mirror.
  • Authenticate them — wire up SSO (SAML/OIDC) so the password check happens at your IdP, not Google.

Manage people through Google Groups, never one-by-one. Grant roles to the group; joiners and leavers are then handled in exactly one place.

Don't confuse GCDS with SSO! GCDS provisions accounts (it copies the user/group objects across). SSO authenticates them (it federates the login). The exam loves a stem like "users exist in AD and we want them to sign in with their AD password" — that's SSO, while "we need the AD groups to appear in Google Cloud" is GCDS. You usually need both.

🔑 Protect the Super Admin Like It's the Nuclear Football

The Organization's super-admin account can do anything, including bypassing many controls. Lock it down hard:

  • Enforce hardware security keys (2SV) — phishing-resistant FIDO keys, not SMS codes.
  • Use a dedicated, named-but-not-daily account; do not browse the web or read email signed in as super admin.
  • Keep a small number of them (2–3) for break-glass, and store recovery codes offline.
"I told my super-admin account to take a break. Now it's on permanent 2-Step Vacation." 🏖️🔐

1.2 Federation Without Keys

The golden rule of modern GCP security: stop minting long-lived credentials. Identity Federation lets an external identity exchange a token it already has for a short-lived Google credential — no service-account key ever stored. There are two federations, and the exam tests which is which constantly.

Federation Type Who/What It's For Classic Scenario
Workforce Identity Federation Human external identities from an external IdP (Okta, Entra, etc.) accessing Google Cloud (console, gcloud, APIs). "Let our Okta users use the Google Cloud console without creating Cloud Identity accounts for them."
Workload Identity Federation Machine external workloads (CI/CD, AWS, Azure, on-prem apps) acting as themselves. "GitHub Actions / an AWS Lambda must deploy to GCP with no stored service-account key."

💡 Workforce = Workers, Workload = Work-machines

Both start with Work, so anchor on the second word. Workforce = a force of people (humans). Workload = a running load of compute (machines). If a person is logging in, it's Workforce; if code is calling an API, it's Workload.

1.3 Service Accounts & Key Hygiene

A service account (SA) is a non-human identity that a workload uses to call Google Cloud APIs. The number-one cause of real-world GCP breaches is a leaked service-account key (the downloadable JSON file). So Google's whole strategy is: avoid creating keys at all.

The credential ladder (best to worst)

  • Attached service account — a VM/GKE/Cloud Run resource runs as an SA; the platform supplies short-lived tokens automatically. No key exists. ✅
  • Service-account impersonation — a trusted principal generates a short-lived token for an SA on demand (roles/iam.serviceAccountTokenCreator). ✅
  • Workload Identity Federation — external workloads swap their native token for a short-lived Google one. ✅
  • Service-account key (JSON) — a long-lived secret you must store, rotate, and pray never leaks. ❌ Last resort only.

🛡️ Guardrails Every Org Should Set

  • iam.disableServiceAccountKeyCreation — org policy that bans new keys everywhere, regardless of project IAM.
  • iam.automaticIamGrantsForDefaultServiceAccounts — stop the default SAs from getting broad Editor on creation.
  • Lock down or remove the default Compute/App Engine SAs — they're over-privileged out of the box.
"Most secure" almost never means a key. If you see a JSON service-account key in the options for a CI/CD or cross-cloud scenario, it is nearly always a distractor. The keyed answer is impersonation, an attached SA, or Workload Identity Federation — with key creation disabled by org policy.

1.4 The Resource Hierarchy & IAM Roles

Google Cloud arranges everything in a tree: Organization → Folders → Projects → Resources. Both IAM allow policies and Org Policy constraints flow downward and are additive — a role granted at the folder applies to every project beneath it. You cannot subtract an inherited allow with another allow lower down (that's what deny policies are for).

💡 "Only Foolish People Resign" — Org, Folder, Project, Resource

Top to bottom: Org → Folder → Project → Resource. Permissions granted higher up rain down on everything below. Grant broadly at the top only when you really mean it.

The three kinds of roles

Role Type What It Is When To Use
Basic (Owner/Editor/Viewer) Legacy, project-wide, enormously broad. Almost never in production — a classic wrong answer for "least privilege".
Predefined Google-curated bundles scoped to a service (e.g. roles/storage.objectViewer). The default right answer — least privilege without the maintenance burden.
Custom Your own hand-picked set of permissions. When no predefined role fits tightly enough and you need to trim further.

⏱️ Privileged Access Manager (PAM) — Just-In-Time Elevation

Standing admin access is a liability. Privileged Access Manager grants elevated roles temporarily through an approval workflow: a user requests, an approver signs off, the role is granted for a bounded window, and it auto-revokes. Perfect for "give the contractor admin for a 2-hour change window, fully audited."

Owner can edit IAM. If a stem says a control must hold "even if a project owner tries to override it" or "regardless of project-level permissions," then removing a role binding is not durable — the owner just re-adds it. The answer is an org policy (config guardrail) or an IAM deny policy (explicit prohibition), which an owner cannot simply allow their way around.

1.5 Conditions & Context-Aware Access (Zero Trust)

Plain role bindings answer who can do what. But modern security also asks when and from where. That's the zero-trust idea: trust identity plus context, drop the VPN.

Mechanism Question It Answers Example
IAM Role WHO can do what Grant roles/bigquery.dataViewer to a group.
IAM Condition WHEN a single binding applies request.time < expiry, or only on resources with a given tag.
Access Level (Access Context Manager) WHERE / WHAT DEVICE Reusable rule: managed device + corporate IP + allowed country.
Identity-Aware Proxy (IAP) ENFORCES the above on an app/SSH/RDP Gate an internal web app; honour the access level; no VPN.

💡 WHO · WHEN · WHERE

Roles say WHO. Conditions say WHEN. Access levels say WHERE/what device. IAP is the bouncer that actually checks all three at the door. Together that's BeyondCorp / Chrome Enterprise Premium zero-trust access.

1.6 Org Policy Guardrails & Deny Policies

IAM and Organization Policy are different tools that solve different problems, and the exam deliberately blurs them.

Tool Controls Example
IAM allow policy Who has which permissions on a resource. Grant a group Storage Object Viewer on a bucket.
Organization Policy What configurations are even permitted (a guardrail). compute.vmExternalIpAccess, gcp.resourceLocations, disable SA keys.
IAM deny policy An explicit prohibition, evaluated before allows. Deny storage.objects.get to a group even if they're Owner.

💡 DENY beats ALLOW, every time

IAM evaluates deny policies first. If a deny matches, the request stops dead — no allow, not even Owner, can rescue it. When a question needs a hard, un-overridable block on a permission, reach for a deny policy. When it needs to block a configuration (regions, external IPs, key creation), reach for an org policy.

1.7 IAM Recommender & Enforcing Least Privilege Automatically

Granting the right roles is hard enough. Keeping them right over time, as people change jobs, projects end, and service accounts accumulate, is genuinely painful. IAM Recommender is Google's machine-learning-backed tool that watches actual API usage and flags where permissions exceed what a principal actually uses. Think of it as a very patient security auditor who never sleeps and never misses a log entry.

What IAM Recommender actually does

  • Analyses Cloud Audit Log data from the past 90 days of activity for each principal.
  • Produces role recommendations: either replace an over-broad role with a tighter one, or remove an unused binding entirely.
  • Also produces policy insights: data points about how roles are being used (or not) — the raw evidence behind the recommendation.
  • Covers users, groups, and service accounts — particularly useful for detecting idle or abandoned SAs that still hold powerful roles.

🔍 Seeing Recommendations Requires a Role

Recommendations are not visible to everyone by default. A principal needs at least the Recommender Viewer role (roles/recommender.iamViewer) to read them, or the Recommender Admin role (roles/recommender.iamAdmin) to apply them. The individual permission behind it is recommender.iamRecommendations.get. Security teams that want a central review dashboard must be explicitly granted access — don't assume org-wide visibility just because you're an admin.

Recommendations are also available programmatically via the Recommender API and surfaced inside the IAM console under each principal's role bindings.

Role recommendations vs Policy insights — know the difference

Artefact What It Is What You Do With It
Policy Insight An observation: "this principal has not used permission X in 90 days." Evidence to inform a decision. Doesn't suggest a specific change.
Role Recommendation A concrete suggestion: "replace roles/editor with roles/storage.objectCreator". Apply directly via the console, gcloud, or the Recommender API. Can be accepted, dismissed, or snoozed.
SA Idle Recommendation Flags a service account that has not used any API in 90 days (all bindings unused). Investigate and delete or disable the SA if it's truly orphaned.

Principal Access Boundary (PAB) policies

Principal Access Boundary policies are a newer IAM feature that lets you define a boundary around which resources a principal can even be granted access to. Rather than reviewing every individual role binding, a PAB policy says "this service account's permissions can only ever apply within projects tagged env:dev." Any role binding that would give the SA access to a resource outside that boundary is simply ignored — even if the binding was legitimately granted. It's a defence-in-depth control that sits on top of the normal IAM allow/deny model.

IAM Recommender is not the same as Audit Logs — and the exam will try to swap them. Cloud Audit Log records every API call as it happens — it's your compliance trail, your forensic record, your immutable event stream. IAM Recommender consumes that audit data, summarises it over 90 days, and produces actionable suggestions to tighten permissions. A stem asking "how do you detect that a service account hasn't called any APIs for three months" is fishing for Recommender (idle SA detection), not Audit Logs (Audit Logs won't automatically surface that summary for you). Conversely, "provide an immutable audit trail of who called storage.objects.delete" is Audit Logs, not Recommender. Don't mix them up.

Manual review vs Recommender vs other cleanup tools

Approach How It Works Strengths Limitations
Manual IAM review A human reads through role bindings per-project and decides what to remove. Full context, can account for business reasons (e.g. emergency break-glass). Doesn't scale; misses subtle excess permissions; depends on reviewer knowledge.
IAM Recommender ML model analyses 90 days of audit data; generates per-principal recommendations. Scales org-wide; data-driven; surfaces idle SAs and over-broad roles automatically. Only as good as the audit log coverage; doesn't know future access needs; 90-day window can miss seasonal use.
GCDS cleanup Syncs AD group membership changes back to Google Groups, automatically removing leavers. Lifecycle management for human identities; no orphaned accounts after offboarding. Only applies to human identities synced from AD; does nothing about over-broad roles or service accounts.
The IAM Recommender looked at my service account's permissions and said "you granted Editor to something that's been running a cron job for 90 days and only ever called storage.objects.create." Fair point, honestly. 😬

1.8 Groups, Domain Restriction & Workforce Federation Controls

IAM bindings tell you what a principal can do. But there's a prior question the exam loves: which principals are even allowed to appear in your IAM policies at all? Domain restriction org policies and group-based access management are the controls that answer it.

Domain restriction — keep strangers off the guest list

The org policy constraint iam.allowedPolicyMemberDomains is one of the most important guardrails in a production GCP organisation. It restricts which identities can be added to any IAM allow policy anywhere in the org (or folder or project where the constraint is applied). If a domain isn't on the allowlist, you cannot bind that identity to any role — not even Viewer — regardless of who is trying to do it.

  • The allowlist accepts Cloud Identity or Workspace customer IDs (the C-prefixed string), not domain names directly.
  • Apply it at the organisation node so it inherits everywhere; override for specific folders (e.g. sandbox environments) only when absolutely necessary.
  • Also prevents granting access to allUsers or allAuthenticatedUsers — the org policy blocks those members too unless they're in an allowed domain (which they can't be, being pseudo-members).

💡 "Allowed Domains = Approved Arrivals" — think customs control

Constraint: iam.allowedPolicyMemberDomains. Trick to remember: the word "allowed" is in the constraint name itself — it's your allowlist of who's even allowed through customs. If your org's Customer ID isn't on the list, no IAM binding will let the identity in, full stop. Not even an Owner can add a gmail.com personal account if gmail.com's customer ID is absent from the allowlist.

Group-based access management — why it matters

The standing best practice is: bind roles to groups, not to individuals. This is not just tidy — it's operationally critical:

  • Joiners: add a person to the group; they inherit all the bound roles instantly.
  • Leavers: remove them from the group; all access evaporates in one operation.
  • Movers: change group membership; no need to audit every project's policy.
  • Paired with GCDS, group membership is driven automatically from Active Directory — zero-touch IAM for routine HR events.
  • Group nesting is supported, but keep hierarchies shallow: deep nesting makes effective permissions hard to reason about.

IAM conditions for resource-level patterns

When you need to grant a role only on resources that match a naming pattern or carry a specific tag, use IAM Conditions with a resource.name attribute or tag-based expressions. Common patterns:

  • resource.name.startsWith("projects/_/buckets/prod-") — bind a Storage role only on buckets whose name starts with prod-.
  • resource.matchTag("env", "production") — grant an SA elevated rights only on resources tagged as production (combined with resource manager tags).

Tag-based conditions are powerful but require resources to be consistently tagged. Missing a tag is a silent failure — the condition doesn't match and access is silently denied, which can look like a bug.

allUsers vs allAuthenticatedUsers — the classic trap

Member Who It Covers Common Legitimate Use
allUsers Literally everyone on the public internet — authenticated or not, bot or human, any origin. Truly public static assets (e.g. a marketing website's images in Cloud Storage). Virtually never appropriate for APIs or data.
allAuthenticatedUsers Anyone with any Google Account — that's 3 billion+ personal gmail.com accounts, every Google Workspace tenant, every Cloud Identity tenant. It does not mean "authenticated by your company." Almost never. Sounds reassuring ("they're authenticated!") but is nearly as open as allUsers.
allAuthenticatedUsers does NOT mean "authenticated to your org." This is one of the most reliable exam traps in the IAM domain. A stem will present a scenario like "allow only authenticated users to access the API" and offer allAuthenticatedUsers as an option. In the real world this means anyone with a Gmail account can call your API. The correct control is a proper IAM binding to your specific domain's group (or enforced via iam.allowedPolicyMemberDomains). If the scenario mentions Security Command Center flagging a bucket as publicly readable, suspect allUsers or allAuthenticatedUsers as the culprit.

🏋️ Workforce Identity Federation — Access Controls on External Humans

When using Workforce Identity Federation to give external IdP users (Okta, Entra) access to Google Cloud, remember you're not creating Cloud Identity accounts — so iam.allowedPolicyMemberDomains doesn't apply directly. Instead, workforce pool principals look like principal://iam.googleapis.com/locations/global/workforcePools/POOL_ID/subject/USER_ID in IAM bindings. Control which external users get in by:

  • Setting attribute conditions on the workforce pool provider (e.g. only users with a specific Okta group claim).
  • Using attribute mapping to translate IdP attributes (like group membership) into Google attributes that conditions can evaluate.
  • Scoping IAM bindings to principalSet (a group of workforce principals matching an attribute) rather than individual subjects where possible.

1.9 Workload Identity in GKE & Cloud Run

Storing a JSON service account key in a Kubernetes Secret is the classic "works but dangerous" pattern — the key ends up in etcd (often unencrypted), in CI logs, and in every backup. Workload Identity eliminates the key entirely by letting a Kubernetes ServiceAccount (KSA) impersonate a Google Service Account (GSA) via the GKE metadata server. No credentials ever leave Google Cloud.

🏋️ Minimal Workload Identity Setup — Four Steps

  1. Create the KSA in the relevant namespace (kubectl create serviceaccount my-ksa -n my-ns).
  2. Annotate the KSA so GKE knows which GSA it should impersonate:
    kubectl annotate serviceaccount my-ksa -n my-ns iam.gke.io/gcp-service-account=my-gsa@PROJECT.iam.gserviceaccount.com
  3. Bind roles/iam.workloadIdentityUser on the GSA, granting the KSA permission to impersonate it:
    gcloud iam service-accounts add-iam-policy-binding my-gsa@PROJECT.iam.gserviceaccount.com --role roles/iam.workloadIdentityUser --member "serviceAccount:PROJECT.svc.id.goog[my-ns/my-ksa]"
  4. Annotate the Pod spec to use the KSA (serviceAccountName: my-ksa). The GKE metadata server handles token exchange transparently at runtime.

After step 3, calls from the pod hit the GKE metadata server, which exchanges the Kubernetes-signed token for a short-lived Google OAuth token. The app code calls GCP APIs exactly as it would with ADC — no code change needed.

How the Token Exchange Works

When a workload pod calls the GCP metadata endpoint (http://metadata.google.internal/), the GKE metadata server intercepts the request, validates the pod's projected service account token against the Kubernetes API, and calls the GCP Security Token Service (STS) to exchange it for a short-lived Google OAuth 2.0 access token scoped to the GSA. This is transparent to the app — standard Application Default Credentials (ADC) picks it up automatically.

Workload Identity on Cloud Run

Cloud Run simplifies this further: each revision runs as a specific service account that you attach at deploy time with --service-account. There is no KSA/GSA annotation dance — the revision is the identity. Because Cloud Run is serverless, there is no persistent VM, no long-lived metadata server to intercept, and no default over-permissive SA to forget about.

🚨 Why Cloud Run Is a More Secure Default Than GCE

A Compute Engine instance launched without specifying a service account gets the Compute Engine default SA — which has the Editor role on the project by default. This is an enormous blast radius. Cloud Run does not have this problem: a Cloud Run service with no SA attached gets no identity (it can only access public resources). You must explicitly grant it an SA, forcing least-privilege by design.

"A pod needs to call the BigQuery API securely — no keys should be stored anywhere." The answer is Workload Identity, full stop. The most common wrong answer is "mount a JSON key file as a Kubernetes Secret." A Secret does encrypt at rest if etcd encryption is configured, but it still stores a long-lived credential inside the cluster. Workload Identity is the only option where no key material is ever created or stored. A second trap: annotating only the KSA but forgetting the workloadIdentityUser IAM binding on the GSA — the metadata server call will fail silently with a permission denied from STS.

💡 "KSA Impersonates GSA — K before G, just like the alphabet"

The Kubernetes ServiceAccount is always the caller; the Google ServiceAccount is always the impersonated identity. K impersonates G, and the annotation iam.gke.io/gcp-service-account lives on the KSA (Kubernetes side). The IAM binding workloadIdentityUser lives on the GSA (Google side). Two places, two sides — one lets the other in.

🧠 Quick-Fire Warm-Up Quiz!

Retrieve before you review. No peeking at the answers first!

Q1: You need to stop anyone from creating service-account keys across the whole organization, even project owners. What do you use?

The iam.disableServiceAccountKeyCreation organization policy. It's a config guardrail that inherits to every current and future project, regardless of project-level IAM.

Q2: A contractor needs elevated admin access for a single 2-hour change window, fully audited, auto-revoked afterwards. Which feature?

Privileged Access Manager (PAM) — just-in-time, time-bound, approval-gated privilege elevation.

Q3: A GitHub Actions pipeline must deploy to Google Cloud with no stored secret. What do you configure?

Workload Identity Federation. GitHub's OIDC token is exchanged for short-lived Google credentials, so no service-account key is ever stored.

Q4: You must allow console access to a sensitive project only from corporate-managed devices on the company IP range. Which mechanism defines that condition?

An access level in Access Context Manager (enforced via context-aware access / IAP). Access levels encode device posture, IP, and geo for zero-trust enforcement.

📝 Section 1 Practice Exam Questions

Question 1: A data-science team and a separate platform team both work in one project. You must ensure the platform admins, who hold roles/owner, can never read objects in a specific Cloud Storage bucket holding regulated PII — even though they can edit IAM. What is the most reliable control?

  • A) Remove the Storage Object Viewer role from the platform admins on that bucket.
  • B) Create an IAM deny policy denying storage object read permissions to the platform group on that bucket.
  • C) Apply a bucket IAM condition that only grants access during business hours to the platform group.
  • D) Move the bucket into a folder where the platform admins have no role bindings at all.

Correct Answer: B

Why B is correct: An IAM deny policy is evaluated before allow policies and cannot be overridden by a role binding, so even an owner who can edit IAM cannot grant themselves read access in the face of an explicit deny.

Why the others are wrong:
A: Owners can edit IAM, so they'd simply re-add the role — removal isn't durable against the very people you're restricting.
C: A condition that allows access during business hours still permits reads (just in a window); it doesn't prohibit them.
D: A bucket lives in a project, not a folder, so it can't be moved into a separate folder away from the project's owners.

Question 2: An external SaaS analytics vendor needs read access to one BigQuery dataset for the next 30 days only, after which access must lapse automatically with no manual cleanup. You want to avoid issuing any long-lived credential. Which approach best fits?

  • A) Create a service account, generate a JSON key, and email it to the vendor with a calendar reminder to revoke it.
  • B) Add the vendor's Google identity to a group and grant the group BigQuery Data Viewer with no expiry.
  • C) Grant the vendor's Google identity BigQuery Data Viewer with an IAM condition that expires after 30 days.
  • D) Share a service-account impersonation grant and rely on the vendor to stop using it after a month.

Correct Answer: C

Why C is correct: An IAM Condition with a request.time expiry makes the grant lapse automatically at the deadline, needs no stored key, and requires no manual revocation.

Why the others are wrong:
A: A JSON key is a long-lived credential and manual revocation is exactly the failure mode you're told to avoid.
B: A group grant with no expiry persists indefinitely and still relies on someone remembering to remove it.
D: Impersonation here has no time bound and trusts the vendor to self-restrict, which isn't enforceable.

Question 3: Security wants to guarantee that no project anywhere in the organization can attach external IP addresses to VMs, and the guarantee must hold even if a project owner tries to enable them. Where do you enforce this?

  • A) A firewall rule at each VPC denying ingress from the public internet to the VMs.
  • B) The compute.vmExternalIpAccess organization policy constraint set at the org node.
  • C) A custom IAM role for project owners that omits the compute network update permission.
  • D) A VPC Service Controls perimeter wrapping all the projects in the organization.

Correct Answer: B

Why B is correct: The compute.vmExternalIpAccess org policy constraint set at the organization is a guardrail that inherits to every project and cannot be overridden by a project owner.

Why the others are wrong:
A: A firewall controls traffic flow but doesn't stop an external IP being assigned, and it must be set per-VPC.
C: A project owner can grant themselves a different role, so a custom role isn't a durable guardrail.
D: VPC-SC stops API data exfiltration across a service perimeter; it doesn't govern whether a VM may carry an external IP.
