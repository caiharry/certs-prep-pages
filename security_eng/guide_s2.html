Section 2: Communications & Boundary Protection

Perimeter controls, network segmentation, and private connectivity (~22%). Keep the good traffic flowing and the bad traffic — and your data — firmly on the right side of the fence.

Section 2 🔵 ~22%

Section 1 decided who gets in. Section 2 decides where they can go once inside and how data is allowed to move. This domain is a tale of two perimeters: the classic network perimeter (firewalls, load balancers, private links) and the modern API data perimeter (VPC Service Controls). Mix those two up and you'll lose easy marks — so let's draw the fence lines clearly. 🚧

2.1 The Edge Defenders

Several services guard the front door of your applications. Each owns a distinct job, and the exam tests the boundaries between them.

Defender Operates At Its One Job
Cloud Armor L7, in front of the external load balancer WAF rules, DDoS protection, IP/geo allow/deny, rate limiting. Inbound web traffic only.
Identity-Aware Proxy (IAP) L7, identity layer Identity- and context-aware access to apps/SSH/RDP — no VPN.
Cloud NGFW L3–L7 network firewall Stateful rules, FQDN/geo objects, and (Enterprise) L7 intrusion prevention.
Secure Web Proxy Egress, L7 Filters and logs outbound internet traffic by URL/host.
Certificate Authority Service PKI Run a private CA to issue and manage internal certificates.
Cloud Armor is inbound; Secure Web Proxy is outbound. If the stem says "filter what our VMs can reach on the internet and log every request," that's Secure Web Proxy egress filtering — not Cloud Armor, which only protects inbound HTTP(S) at the load-balancer edge.

2.2 The Three Firewall Layers

GCP firewalling comes in three layers, and a favourite exam discriminator is which one can a project owner not delete.

Layer Attaches To Who Controls It
VPC firewall rules A single VPC network; scoped by tags/service accounts Anyone with network-admin on that project — editable, not a guardrail.
Hierarchical firewall policies Organization or folder node; inherit downward Central security team. Project owners cannot remove them. Use goto_next to delegate the rest.
Cloud NGFW Enterprise Network firewall policies + L7 inspection Adds intrusion prevention, TLS inspection, and threat intelligence.

💡 Hierarchy = Handcuffs

A hierarchical firewall policy at the org/folder is a handcuff the project owner can't pick. A plain VPC firewall rule is a sticky note — anyone with network-admin can peel it off. "Owners cannot remove it" ⇒ go hierarchical.

2.3 VPC Service Controls — The Anti-Exfiltration Perimeter

This is the single most-tested service in the whole domain, so internalise it. VPC Service Controls (VPC-SC) draws a perimeter around Google-managed, API-based services (Cloud Storage, BigQuery, Pub/Sub, and friends). Inside the perimeter, data can move freely; crossing the boundary is blocked — even for an identity with perfectly valid IAM permissions.

🛡️ Why VPC-SC Is Special

IAM answers "are you allowed?" VPC-SC answers "are you inside the box?" A stolen credential with legitimate Storage access still can't copy a bucket to a project outside the perimeter, because the request fails at the API layer, not the IAM layer. That's the defence against the classic exfiltration attack: valid creds, wrong destination.

Firewalls don't stop serverless exfiltration. BigQuery and Cloud Storage calls go to Google's API frontends, not arbitrary internet egress, so a VPC firewall or subnet egress rule won't reliably block an authorised API-to-API copy. To stop data leaving a serverless / API product, you need VPC Service Controls. Full stop.
"Our data tried to escape the perimeter. VPC Service Controls gave it a stern look and said: 'You're not on the list.'" 📋🚪

2.4 Private Connectivity & Controlled Egress

These options all keep traffic off the public internet, but they solve different problems. The exam rewards you for telling the pipes apart.

Option Problem It Solves Tell-Tale Phrase
Private Google Access A no-external-IP VM reaching Google APIs over internal routes. "VM has no public IP but must call the Storage/BigQuery API."
Private Service Connect (PSC) Consuming a Google or published service via a private endpoint IP in your VPC. "Reach a managed service through a private endpoint inside our network."
HA VPN / Cloud Interconnect Private connectivity between on-prem and GCP. "Connect the data centre to the VPC privately / with an SLA."
Cloud NAT Outbound internet for private VMs with no inbound exposure. "Private VMs need to download updates but accept no inbound."
Secure Web Proxy Policy-based, logged egress filtered by hostname/URL. "Allow only approved domains and log every outbound request."

💡 PERIMETER vs PIPE

VPC Service Controls is a perimeter around API data. PSC, Private Google Access, VPN, and NAT are private pipes for connectivity. When the question is about exfiltration, reach for the perimeter; when it's about reaching something privately, reach for a pipe. Don't plumb a pipe at an exfiltration problem.

2.5 VPC Service Controls: Ingress Rules, Egress Rules & Dry-Run Mode

VPC-SC's core concept is the perimeter, but the real exam depth lives in ingress rules, egress rules, and the dry-run mode you'll use before you lock things down. Get these three wrong and you'll block your own pipelines on day one.

Beyond the Perimeter: Ingress & Egress Rules

A plain perimeter is binary — requests either originate inside or they don't. Ingress and egress rules punch precise, auditable holes.

  • Ingress rules let specific external identities (service accounts, users) and/or specific source networks call in-perimeter APIs. Think: an on-prem pipeline service account that must write to a perimeter-protected BigQuery dataset.
  • Egress rules let in-perimeter resources call specific external APIs or projects. Think: a perimeter-protected Cloud Run job that must read from a shared GCS bucket in another project.
  • Both rules are additive exceptions on top of the deny-by-default perimeter. They do not replace the perimeter; they extend it surgically.

Service Perimeter Bridges

When two separate perimeters need to exchange data, you add a perimeter bridge. A bridge grants bidirectional access between exactly two perimeters without merging them. Each perimeter retains its own policies; only the shared resources listed in the bridge are reachable across the boundary. If you need one-way flow only, use an ingress/egress rule pair instead — a bridge is always bidirectional.

Access Levels Inside VPC-SC

Access levels (defined in Access Context Manager) add a context check on top of identity. You can combine an access level with an IAM condition so that an external service account is only granted ingress when it is also coming from a specific IP range or a corporate device. The combination looks like: ingress from serviceAccount X when accessLevel Y is satisfied. This is the same access-level object used by IAP and BeyondCorp — one definition, reusable across products.

📐 Ingress / Egress Rule Structure

Every ingress or egress rule has two halves that must both be satisfied:

  • "From" / "To" (source or destination) — identifies the external side: a specific identity (serviceAccount:…, user:…), an access level, a VPC network by resource name, or a project number.
  • "Operations" (what is allowed) — the API service and optionally specific methods. For example: storage.googleapis.com with method google.storage.objects.get. Omitting methods means all methods on that service are allowed.

Both halves must match for the exception to apply. A rule that specifies a service account but wildcard operations still only affects that one identity.

Dry-Run (Audit) Mode vs Enforced Mode

Before enforcing a perimeter in production, enable dry-run mode (also called audit mode). In dry-run mode, VPC-SC evaluates every request against the perimeter policy and writes a VIOLATION log entry to Cloud Audit Logs whenever a request would have been denied — but it does not actually block the request. This lets you measure blast radius and tune your ingress/egress rules against real traffic before flipping to enforced.

Promotion path: dry-run → observe violations in Logs Explorer → add ingress/egress rules for legitimate traffic → iterate until violations drop to noise → promote perimeter to enforced.

Dry-run mode logs violations but does NOT block traffic. A common question frames a scenario where a security team "enabled a VPC-SC perimeter in audit mode" and asks whether unauthorised API calls are being denied. They are not denied — only logged. Enforced mode is required to actually block. Don't conflate "violation detected" with "request rejected."
"Dry-run mode is like a velvet rope with a clipboard. It writes your name down every time you skip the queue — but it still lets you in. Enforced mode is the actual bouncer." 📋🚫
Control What It Protects Applies To Blocks IAM-Authorized Requests?
VPC Firewall Rules Network-level traffic between resources (IP, port, protocol) VM-to-VM, VM-to-internet, internet-to-VM N/A — operates below the API layer
VPC Service Controls Google API data plane (Cloud Storage, BigQuery, Pub/Sub, etc.) API calls crossing a perimeter boundary Yes — the defining characteristic of VPC-SC
Org Policy Constraints Resource configuration and what can be created Resource creation, sharing settings, allowed regions Yes, for resource management actions (not data-plane reads/writes)

2.6 Shared VPC Security & Network Isolation Patterns

Multi-team GCP environments need a network ownership model. Shared VPC is the GCP-native answer: one central team owns the network, everyone else borrows subnets from it. The exam tests who controls what and why that matters for security.

How Shared VPC Works

In a Shared VPC setup, one project is designated the host project. It owns the VPC networks, subnets, firewall rules, and routing. Other projects — service projects — attach to the host and deploy workloads (VMs, GKE nodes, Cloud Run services) into the host's subnets. Service projects do not have their own VPCs; they borrow network capacity from the host.

  • Resource ownership: network resources live in the host; compute and application resources live in service projects.
  • Billing for network egress: charged to the project that owns the resource generating traffic, not the host.
  • Enabling Shared VPC requires the Shared VPC Admin role at the organisation or folder level; the host-project owner alone cannot enable it.

Who Controls Firewall Rules in Shared VPC?

This is the most-tested detail about Shared VPC security. Firewall rules are attached to the VPC in the host project. To create or modify firewall rules in a Shared VPC, you need the compute.networkAdmin or compute.securityAdmin role on the host project, not on your service project. Service project owners see the subnets they are allocated but cannot write firewall rules that govern them. This is the entire point: centralised, tamper-resistant network security.

VPC peering does NOT share firewall rules. When two VPCs are connected via VPC Network Peering, each VPC keeps its own independent firewall rules. Firewall rules in VPC A do not propagate to or affect traffic in VPC B — you must manage them separately in each project. A common distractor asks whether peering "extends" one VPC's security controls to the peer; it does not. If you need shared, centrally managed firewall rules, that is Shared VPC, not peering.

💡 Shared VPC = The Landlord Model

Think of Shared VPC as a landlord (host project) and tenants (service projects). The landlord owns the building's plumbing, wiring, and front door locks (VPC, subnets, firewall rules). Tenants furnish their own rooms (VMs, apps) but cannot rewire the building. If a tenant wants a new door lock, they ask the landlord. Centralised control = security landlord.

Why Shared VPC Is a Security Pattern

  • Prevents VPC sprawl: without Shared VPC, each team creates its own VPC, its own firewall rules, and its own peering mesh. Security posture diverges quickly.
  • Centralises egress control: Cloud NAT, Secure Web Proxy, and VPC Service Controls can be deployed once in the host and apply to all service projects automatically.
  • Supports least-privilege networking: service projects can be allocated specific subnets and no others; they cannot create routes or firewall rules that escape their allocation.
  • Audit trail consolidation: network-level audit logs (flow logs, firewall logs) live in the host project, giving the security team a single pane of glass.

VPC Peering vs Shared VPC

Both keep traffic private (no internet transit), but they solve different problems:

Attribute VPC Network Peering Shared VPC
Network ownership Each VPC owned and managed separately One host VPC; service projects are tenants
Firewall rules Independent in each VPC; do not propagate Centralised in host project; apply to all service projects
Subnet sharing No — peers can reach each other's IPs, but subnets remain in their own VPC Yes — host subnets are directly allocated to service projects
Transitive routing Not supported (A-B and B-C peered does not give A-C connectivity) All service projects share one host network; all can route to each other via the host
Typical use case Connecting two independently managed VPCs (different orgs, or vendor services) Multi-team org with centralised network security

Network Segmentation Patterns

Even within Shared VPC, good practice isolates workloads by tier. Common patterns you should recognise on the exam:

  • Three-tier subnet segmentation: separate subnets (or even separate VPCs) for data (databases, warehouses), application (APIs, microservices), and management (bastion hosts, monitoring). Firewall rules allow only the minimum flows between tiers.
  • Separate VPC per sensitivity tier: when regulatory requirements mandate hard network isolation, workloads that process PCI or PHI data run in a dedicated VPC (or dedicated host project) with its own VPC-SC perimeter.
  • Bastion host / IAP tunnelling: the management subnet hosts a bastion (or you replace it entirely with IAP for TCP forwarding) so that SSH/RDP never traverses the public internet and is never exposed in the data or app subnets.
  • Private Google Access per subnet: enable it on subnets where VMs need to reach Google APIs, and disable internet routes — this locks down egress to Google's backbone only.

2.7 Transport Security: TLS, mTLS & Certificate Management

Encryption in transit is non-negotiable on the exam. GCP handles the easy cases automatically, but east-west workload-to-workload traffic inside a cluster requires deliberate configuration. Knowing which scenario maps to which control is the discriminator.

TLS — The Free Lunch

Google enforces TLS 1.2+ on all public API endpoints automatically. Every call to a Google Cloud API (Cloud Storage, BigQuery, Pub/Sub, etc.) travels over TLS — you do not configure this, you cannot disable it, and there is no certificate to manage on your side. The same applies to HTTPS load balancers: traffic from the internet to a Google-managed HTTPS load balancer is always TLS-terminated at the edge. This is a deliberate GCP design choice: security by default, not by opt-in.

Managed SSL Certificates vs Self-Managed

When you configure an HTTPS load balancer, you choose between two certificate modes:

Mode Who Provisions? Who Rotates? Use When…
Google-managed SSL certificates Google (via Let's Encrypt or Google Trust Services) Google (auto-rotates before expiry) Public HTTPS endpoints where you own the domain; easiest path, no rotation toil.
Self-managed certificates You (your own CA, or a commercial CA) You (manual or scripted) Internal CAs, custom CAs, wildcard certs, or compliance that mandates your own CA chain.

Google-managed certs are attached to the load balancer's target HTTPS proxy. They require a domain verified via Cloud DNS or Cloud Domains — Google provisions the cert only after DNS resolves to the load balancer IP. For *.internal or private hostnames, self-managed is the only option.

mTLS — East-West Workload-to-Workload

Standard TLS is one-way: the server proves its identity to the client. Mutual TLS (mTLS) is two-way: both the client and server present certificates, proving identity in both directions. This is the pattern for zero-trust service-to-service communication inside a GKE cluster or Anthos environment.

  • Istio / Anthos Service Mesh (ASM): the preferred GCP-native approach. ASM deploys Envoy sidecar proxies alongside every pod. The proxies transparently negotiate mTLS for all pod-to-pod traffic — application code does not need to implement TLS. The control plane automatically rotates certificates via SPIFFE/SPIRE workload identity.
  • Certificate Authority Service (CAS): a managed, private CA hierarchy on GCP. Use CAS when you need to issue certificates from your own CA chain for mTLS, internal services, or device identity. CAS integrates with Workload Identity Federation and ASM so that ASM can draw certificates from your CA rather than Google's default issuer.
  • GKE Workload Identity + mTLS: even without ASM, GKE workloads can use SPIFFE certificates via the GKE Certificate Controller; however ASM/Istio is the exam-recommended approach for full mesh mTLS.
"Encrypt data in transit between GKE pods" = mTLS / Anthos Service Mesh, NOT "enable HTTPS on the load balancer." HTTPS on the load balancer encrypts north-south traffic (internet → load balancer → service). It does nothing for east-west pod-to-pod traffic inside the cluster. If a question asks how to ensure encrypted communication between microservices inside GKE, the correct answer is Anthos Service Mesh / Istio with mTLS (or Certificate Authority Service for certificate issuance). Any option involving "enable HTTPS on the frontend" or "add an SSL certificate to the load balancer" is a distractor — it solves the wrong layer.

Cloud Domains & Cloud DNS Integration

For public HTTPS endpoints backed by Google-managed SSL certificates, the provisioning flow ties Cloud DNS and load balancers together:

  • Register or transfer your domain via Cloud Domains (or point your registrar's nameservers to Cloud DNS).
  • Create a Cloud DNS A record pointing your domain at the load balancer's reserved external IP.
  • Attach a Google-managed SSL certificate to the target HTTPS proxy referencing that domain.
  • Google validates DNS propagation, provisions the certificate, and begins auto-rotating it — typically within 60 minutes of DNS propagation completing.

Private domains (e.g. service.corp.internal) cannot use Google-managed certs because Google cannot complete a public DNS challenge. Use Certificate Authority Service to issue internal certificates and load them as self-managed certs on an internal load balancer.

💡 North vs East: Two Different Problems

Draw a compass rose. North-south = internet traffic coming in through the load balancer — HTTPS + Google-managed cert handles it. East-west = pod chatting to pod inside the cluster — Anthos Service Mesh + mTLS handles it. Any question about pod-to-pod, service mesh, microservice communication, or sidecars is an east-west question. Don't reach for the load balancer cert — reach for ASM.

"The engineer enabled HTTPS on the load balancer and declared 'all traffic is now encrypted!' Meanwhile, across the cluster, pods were chatting in the clear like old friends at a coffee shop. ☕ ASM watched in silence." 🕸️🔒

Certificate Authority Service Deep-Dive

Certificate Authority Service (CAS) is a managed, private CA infrastructure on GCP. It is not just a certificate issuer — it is a full hierarchical CA you control, with no need to run your own CA software on VMs.

  • Two CA tiers: DevOps CA (high-velocity, short-lived certs for workloads, billed per certificate) and Enterprise CA (lower volume, longer-lived certs for devices and services, more control over policy). Choose DevOps CA for Kubernetes workload certificates; Enterprise CA for internal TLS between long-running services.
  • Integration with GKE: the GKE Certificate Controller can be configured to use a CAS CA to issue node and workload certificates, replacing the self-signed default CA. This puts certificate issuance under your KMS-backed CA rather than a cluster-generated key.
  • Certificate templates: define allowed key usages, subject attributes, and validity windows centrally. Callers can only request certificates that conform to the template — preventing engineers from issuing over-privileged or excessively long-lived certs.
  • Revocation: CAS supports CRL and OCSP for certificate revocation. If a workload identity is compromised, revoke its certificate immediately — downstream mTLS peers will reject it on the next handshake (OCSP) or CRL fetch.

🔑 When the Exam Says "Internal CA" — Think CAS

Any question involving: issuing certificates from your own CA chain, private hostnames that Google cannot validate, workload identity certificates at scale, or OCSP/CRL revocation — the answer involves Certificate Authority Service. Google-managed SSL certificates are for public domains validated by Google Trust Services. CAS is for everything internal, custom, or regulated.

TLS Version & Cipher Policy

For HTTPS load balancers, GCP lets you configure an SSL policy that controls the minimum TLS version and the permitted cipher suites. Three managed profiles exist:

  • COMPATIBLE: broadest browser compatibility, supports older cipher suites and TLS 1.0/1.1. Use only when you must support legacy clients that cannot be updated.
  • MODERN: TLS 1.2+ with a reduced set of cipher suites. The recommended default for most web applications.
  • RESTRICTED: TLS 1.2 only, with the most secure cipher suites (ECDHE + AESGCM). Required by PCI DSS and similar frameworks that mandate disabling TLS 1.0/1.1.

You can also create a CUSTOM SSL policy if you need to specify exact cipher suites. Attach the SSL policy to the target HTTPS or target SSL proxy — not to individual backend services. The exam occasionally presents a scenario where "the compliance team requires TLS 1.2 minimum" and the answer is to configure an SSL policy with the RESTRICTED profile, not to patch the backend servers.

SSL policies apply to the load balancer, not the backends. A question asking "how do you enforce TLS 1.2+ for all traffic to your web application behind a Global HTTPS LB?" has a one-component answer: configure a RESTRICTED SSL policy on the target HTTPS proxy. Configuring TLS on the backend VMs independently is redundant for this exam question and not the GCP-native answer. The load balancer terminates TLS from the client; traffic between the LB and backends is on Google's private network (or can be re-encrypted separately with backend HTTPS).

Section 2.7 Quick Reference

  • TLS on Google APIs: automatic, always-on — nothing to configure.
  • Google-managed SSL cert: attach to HTTPS load balancer for public domains; Google provisions and rotates automatically via Cloud DNS validation.
  • Self-managed cert: use for private CAs, wildcard certs, or when your org mandates its own CA chain.
  • mTLS (east-west): Anthos Service Mesh / Istio with Envoy sidecars — the pod-to-pod encryption answer. Application code is unaware.
  • Certificate Authority Service: managed private CA for issuing certs from your own CA hierarchy (internal services, GKE workload identity, OCSP revocation).
  • SSL policy (RESTRICTED profile): enforce TLS 1.2 minimum and strong ciphers on HTTPS load balancers for PCI DSS compliance.
  • Cloud DNS + Cloud Domains: prerequisite for Google-managed SSL cert provisioning on public endpoints.
  • North-south vs east-west: load balancer cert = north-south (internet → LB). ASM mTLS = east-west (pod → pod). Never confuse them on the exam.
  • CAS DevOps vs Enterprise tier: DevOps CA for high-volume short-lived workload certs; Enterprise CA for lower-volume, longer-lived service or device certificates.
  • SPIFFE/SPIRE: the open standard for workload identity certificates that ASM uses internally. You don't need to configure it directly — ASM handles it — but knowing the acronym helps decode scenario question wording.
  • mTLS ≠ encryption at rest: mTLS encrypts data in transit between workloads. It has no effect on how data is stored — CMEK and CSEK handle encryption at rest.
  • Google-managed cert provisioning delay: certs can take up to 60 minutes to provision after DNS propagation. Plan for this in deployment runbooks — the LB will serve a certificate error until provisioning completes.
  • Wildcard certificates: not supported by Google-managed SSL certs — use self-managed certs issued by CAS or a commercial CA for *.example.com coverage.

🧠 Quick-Fire Warm-Up Quiz!

Retrieve before you review. Cover the answers!

Q1: Prevent a user who has legitimate BigQuery access from exporting a dataset to an outside project. Which control?

A VPC Service Controls perimeter. It blocks data egress from in-scope services across the perimeter boundary regardless of IAM.

Q2: Give employees identity-based access to an internal web app from anywhere, with no VPN. What do you use?

Identity-Aware Proxy (IAP) — context-aware, identity-based access to apps without a VPN.

Q3: Security must enforce a deny-all baseline firewall rule across every project that owners cannot delete. Which feature?

A hierarchical firewall policy at the organization or folder node. It inherits downward and is managed centrally, so project network admins cannot remove it.

Q4: A VM with no external IP must reach the Cloud Storage API over Google's network. What do you enable on its subnet?

Private Google Access. It routes requests to Google APIs internally, so no public IP is required.

📝 Section 2 Practice Exam Questions

Left-click to select, right-click to strike out a distractor, then Submit for the full diagnostic.

Question 1: A compromised service account inside a project has valid IAM permissions on a Cloud Storage bucket and is attempting to copy the bucket's contents to a personal project in another organization. You must stop the copy even though the IAM grant is legitimate. What blocks it?

  • A) A VPC firewall egress rule denying outbound traffic to the internet from the affected subnet.
  • B) Removing the Storage Object Viewer role from the service account on the bucket.
  • C) A VPC Service Controls perimeter around the project that restricts Cloud Storage egress.
  • D) A Cloud Armor security policy applied in front of the storage endpoint.

Correct Answer: C

Why C is correct: VPC Service Controls enforces a perimeter at the API layer, so a copy from in-perimeter Cloud Storage to a project outside the perimeter is denied regardless of the identity's valid IAM permissions.

Why the others are wrong:
A: Storage API calls go to Google's API frontends, not arbitrary internet egress, so a subnet egress rule doesn't reliably stop an authorised API-to-API copy.
B: The grant is described as legitimate and needed; revoking it breaks normal use and an attacker with IAM-edit could restore it.
D: Cloud Armor protects inbound HTTP(S) load-balanced applications, not outbound storage API exfiltration.

Question 2: An internal HR application runs on Compute Engine and must be reachable by employees from anywhere, but only after their identity and device posture are verified, and you want to retire the corporate VPN. Which design meets this?

  • A) Put the app behind Identity-Aware Proxy with an access level that checks device posture and group membership.
  • B) Publish the app on an external load balancer protected by a Cloud Armor IP allowlist of office ranges.
  • C) Keep the VPN but add a hierarchical firewall policy restricting the app's port to internal ranges.
  • D) Expose the app through Private Service Connect so only endpoints inside the VPC can reach it.

Correct Answer: A

Why A is correct: IAP combined with an Access Context Manager access level gives identity-aware, device-aware, VPN-less access from any network — the BeyondCorp model the question describes.

Why the others are wrong:
B: An IP allowlist binds access to office networks, which fails the "from anywhere" requirement and checks neither identity nor device.
C: This keeps the VPN you were asked to retire and does no identity/device verification.
D: PSC restricts reachability to inside-VPC endpoints, the opposite of letting verified employees connect from anywhere.

Question 3: VMs in a locked-down subnet have no external IPs but must download OS patches from a specific set of vendor domains on the public internet, and security wants every outbound request logged and filtered by hostname. Which control fits best?

  • A) Cloud NAT on the subnet so the VMs can reach the patch servers via a managed egress IP.
  • B) Private Google Access so the VMs can resolve and reach the external vendor endpoints.
  • C) Secure Web Proxy with a policy allowing only the approved vendor hostnames and logging all egress.
  • D) A VPC Service Controls perimeter listing the vendor domains as allowed egress destinations.

Correct Answer: C

Why C is correct: Secure Web Proxy applies URL/host-based egress policy and logs every request, exactly matching "filtered by hostname and logged."

Why the others are wrong:
A: Cloud NAT provides egress but applies no hostname filtering or per-request application logging.
B: Private Google Access only reaches Google APIs, not arbitrary external vendor domains.
D: VPC-SC governs access to Google-managed API services, not generic internet hostnames for patch downloads.
