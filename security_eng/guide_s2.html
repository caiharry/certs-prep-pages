Section 2~22%

Communications & Boundary Protection

Perimeter controls, network segmentation, and private connectivity.

Perimeter controls

Defend the edge with Cloud NGFW (incl. L7 inspection), Cloud Armor (WAF, DDoS, rate limiting), Identity-Aware Proxy (identity-based access, no VPN), Secure Web Proxy (egress filtering), and Certificate Authority Service for private CAs.

Segmentation & VPC Service Controls

Segment with VPC design, Shared VPC, and firewall rules. Crucially, VPC Service Controls create a perimeter that prevents data exfiltration from serverless, API-based services (BigQuery, Cloud Storage, Pub/Sub) even when an identity has valid IAM permissions.

⚠️ Exam Trap: To stop exfiltration from a serverless product, use VPC Service Controls — firewall rules only govern VM/network traffic.

Private connectivity

Reach Google APIs without public IPs via Private Google Access and Private Service Connect. Connect on-prem privately with HA VPN or Cloud Interconnect, and allow controlled egress with Cloud NAT.

Firewalls: VPC rules vs hierarchical policies vs Cloud NGFW

There are three layers and the exam loves to test which one fits. VPC firewall rules live in a single network and act on tags/service accounts. Hierarchical firewall policies attach at the organization or folder node and inherit downward, so security teams can enforce baseline allow/deny across many projects centrally — and use goto_next to delegate the rest to project owners. Cloud NGFW (the Cloud Firewall stack) adds network firewall policies, FQDN and geo objects, and L7 intrusion prevention via the Enterprise tier. Evaluation runs most-specific to least: a higher-level deny in a hierarchical policy beats a permissive project rule.

  • VPC firewall rule — per-network, tag/SA-scoped, classic.
  • Hierarchical firewall policy — org/folder-wide guardrail, inherited.
  • Cloud NGFW Enterprise — L7 IPS, TLS inspection, threat intel.
⚠️ Exam Trap: "Enforce a baseline firewall rule across every project, owners cannot remove it" = hierarchical firewall policy at org/folder. A plain VPC rule is editable by anyone with network-admin on that project.

Private access paths & egress control

Distinguish the private-connectivity options, because they solve different problems. Private Google Access lets a VM with no external IP reach Google APIs over internal routes. Private Service Connect exposes a Google or published service behind a private endpoint IP inside your VPC, and is the modern way to consume managed services privately. For controlled outbound traffic, Cloud NAT gives egress without inbound exposure, and Secure Web Proxy filters and logs egress to the internet by URL/host. Pair these with VPC Service Controls to box in the data-exfiltration risk at the API layer.

  • Private Google Access — no-external-IP VM reaching Google APIs.
  • Private Service Connect — private endpoint to a producer service.
  • Secure Web Proxy — policy-based, logged internet egress.
PERIMETER vs PIPE VPC Service Controls is a perimeter around API data; PSC and Private Google Access are private pipes for connectivity. Don't reach for a pipe when the question is about exfiltration.

🧠 Warm-Up Quiz

Q1. Prevent a user with BigQuery access from exporting data to an outside project. Which control?

Reveal answer

VPC Service Controls perimeter.

Blocks data egress from in-scope services across the perimeter regardless of IAM.

Q2. Give employees identity-based access to an internal web app without a VPN. What do you use?

Reveal answer

Identity-Aware Proxy (IAP).

Context-aware, identity-based access to apps.

Q3. Security must enforce a deny-all baseline firewall rule across every project that owners cannot delete. Which feature?

Reveal answer

A hierarchical firewall policy at the organization or folder node.

It inherits downward and is managed centrally, so project network admins cannot remove it.

Q4. A VM with no external IP must reach the Cloud Storage API over Google's network. What do you enable on its subnet?

Reveal answer

Private Google Access.

It routes requests to Google APIs internally so no public IP is required.

📝 Exam-Style Practice

Q1. A compromised service account inside a project has valid IAM permissions on a Cloud Storage bucket and is attempting to copy the bucket's contents to a personal project in another organization. You must stop the copy even though the IAM grant is legitimate. What blocks it?

  • A) A VPC firewall egress rule denying outbound traffic to the internet from the affected subnet
  • B) Removing the Storage Object Viewer role from the service account on the bucket
  • C) A VPC Service Controls perimeter around the project that restricts Cloud Storage egress
  • D) A Cloud Armor security policy applied in front of the storage endpoint
Reveal answer & explanation

Correct: C. VPC Service Controls enforces a perimeter at the API layer, so a copy from in-perimeter Cloud Storage to a project outside the perimeter is denied regardless of the identity's valid IAM permissions.

A: Storage API calls go to Google's API frontends, not arbitrary internet egress, so a subnet egress rule does not reliably stop an authorised API-to-API copy. B: The grant is described as legitimate and needed; revoking it breaks normal use and an attacker with IAM-edit could restore it. D: Cloud Armor protects inbound HTTP(S) load-balanced applications, not outbound storage API exfiltration.

Q2. An internal HR application runs on Compute Engine and must be reachable by employees from anywhere, but only after their identity and device posture are verified, and you want to retire the corporate VPN. Which design meets this?

  • A) Put the app behind Identity-Aware Proxy with an access level that checks device posture and group membership
  • B) Publish the app on an external load balancer protected by a Cloud Armor IP allowlist of office ranges
  • C) Keep the VPN but add a hierarchical firewall policy restricting the app's port to internal ranges
  • D) Expose the app through Private Service Connect so only endpoints inside the VPC can reach it
Reveal answer & explanation

Correct: A. IAP combined with an Access Context Manager access level gives identity-aware, device-aware, VPN-less access from any network — the BeyondCorp model the question describes.

B: An IP allowlist binds access to office networks, which fails the "from anywhere" requirement and checks neither identity nor device. C: This keeps the VPN you were asked to retire and does no identity/device verification. D: PSC restricts reachability to inside-VPC endpoints, the opposite of letting verified employees connect from anywhere.

Q3. VMs in a locked-down subnet have no external IPs but must download OS patches from a specific set of vendor domains on the public internet, and security wants every outbound request logged and filtered by hostname. Which control fits best?

  • A) Cloud NAT on the subnet so the VMs can reach the patch servers via a managed egress IP
  • B) Private Google Access so the VMs can resolve and reach the external vendor endpoints
  • C) Secure Web Proxy with a policy allowing only the approved vendor hostnames and logging all egress
  • D) A VPC Service Controls perimeter listing the vendor domains as allowed egress destinations
Reveal answer & explanation

Correct: C. Secure Web Proxy applies URL/host-based egress policy and logs every request, exactly matching "filtered by hostname and logged."

A: Cloud NAT provides egress but applies no hostname filtering or per-request application logging. B: Private Google Access only reaches Google APIs, not arbitrary external vendor domains. D: VPC-SC governs access to Google-managed API services, not generic internet hostnames for patch downloads.
