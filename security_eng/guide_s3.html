Section 3~23%

Ensuring Data Protection

Protect data at rest, in transit, and in use; secure secrets and AI workloads.

Encryption key models

  • Google-managed — default.
  • CMEK (Cloud KMS) — you control rotation/revocation; supports import.
  • Cloud EKM — keys remain in an external/on-prem HSM, never in Google Cloud.
  • Confidential Computing — encrypts data in use (memory).
⚠️ Exam Trap: "Keys must never leave our HSM" = Cloud EKM. "Generate then import" = Cloud KMS import.

Sensitive data & secrets

Sensitive Data Protection (Cloud DLP) discovers, classifies, redacts, and pseudonymises PII. Secret Manager stores versioned secrets with IAM control and rotation. Apply Cloud Storage object lifecycle policies and restrict access to data services (BigQuery, Cloud SQL, GCS).

Securing AI workloads

Apply security and privacy controls to ML systems: protect training data and models from exfiltration, secure Vertex AI access, and consider Model Armor / Sensitive Data Protection for generative AI inputs and outputs.

Cloud KMS in practice: CMEK, rotation & separation of duties

When you choose CMEK, you own the key lifecycle in Cloud KMS: create key rings, set automatic rotation schedules, disable or destroy versions, and import your own key material if a regulator demands "generate-then-import." Each Google service uses your CMEK through a per-service service agent that needs the roles/cloudkms.cryptoKeyEncrypterDecrypter role on the key — revoke it and the data becomes inaccessible (a deliberate kill-switch). Keep separation of duties: the key admin should not also be the data admin, so no single person can both grant data access and control the keys.

  • CMEK — your key in KMS; you control rotation, disable, destroy.
  • CMEK import — bring your own key material into KMS.
  • Cloud EKM — key stays in an external HSM/KMS, never in Google Cloud.
  • Confidential Computing — encrypts data in use, in memory.
⚠️ Exam Trap: "Revoke access to encrypted data instantly without touching the data" = disable/destroy the CMEK key version (or revoke the service agent's key role). Re-running DLP or changing bucket IAM does not achieve a cryptographic kill-switch.

Sensitive Data Protection & Secret Manager

Sensitive Data Protection (the service formerly called Cloud DLP) does three distinct jobs the exam separates: discovery (profiling where sensitive data lives across BigQuery, Cloud Storage, and Cloud SQL), inspection (matching infoTypes like credit cards or national IDs), and de-identification (redaction, masking, or format-preserving tokenisation/pseudonymisation so analytics still work). For application credentials, use Secret Manager — versioned secrets, IAM-scoped access, CMEK encryption, and rotation — never hard-coded strings or config files baked into images.

  • Discovery / profiling — find sensitive data at scale.
  • De-identification — redact, mask, or tokenise while keeping utility.
  • Secret Manager — central, versioned, IAM-controlled secrets.
FIND-then-FUZZ Sensitive Data Protection first finds the PII, then fuzzes it (redact/mask/tokenise). If the stem keeps analytics usable, the answer is tokenisation, not deletion.

🧠 Warm-Up Quiz

Q1. Discover and redact credit-card numbers in a Cloud Storage dataset before analysis. Which service?

Reveal answer

Sensitive Data Protection (Cloud DLP).

Detects, classifies, and de-identifies sensitive data.

Q2. Regulator requires keys held only in your on-prem HSM. Which option?

Reveal answer

Cloud EKM.

External keys never persist in Google Cloud.

Q3. An auditor asks you to instantly render a CMEK-encrypted dataset unreadable without deleting the data. What do you do?

Reveal answer

Disable (or destroy) the CMEK key version in Cloud KMS.

Without the key, the ciphertext cannot be decrypted, giving an immediate cryptographic kill-switch.

Q4. A database password is currently baked into a container image. Where should it live instead?

Reveal answer

Secret Manager.

It provides versioned, IAM-controlled, encrypted storage with rotation, retrieved at runtime.

📝 Exam-Style Practice

Q1. A financial regulator requires that the encryption keys protecting your BigQuery datasets are generated and stored exclusively in your own on-premises HSM and never reside in Google Cloud, even briefly. Which key option satisfies this?

  • A) CMEK with key material imported into Cloud KMS from the on-prem HSM
  • B) Google-managed encryption keys with Access Approval enabled on the project
  • C) Cloud External Key Manager backed by your on-prem HSM through a supported partner
  • D) CMEK in Cloud HSM with a strict automatic rotation policy and audit logging
Reveal answer & explanation

Correct: C. Cloud EKM keeps the key in your external HSM and Google only receives a short-lived wrapped reference, so the key material never resides in Google Cloud — exactly the regulator's constraint.

A: Importing key material places that material into Cloud KMS, violating "never resides in Google Cloud." B: Google-managed keys live entirely in Google's infrastructure; Access Approval gates support access but not key location. D: Cloud HSM keys are generated and held in Google's HSMs, not your on-prem HSM.

Q2. Analysts need to run aggregate queries over a customer table that contains national ID numbers. The IDs must stay non-readable, yet rows for the same customer must still join consistently across tables. What is the right transformation?

  • A) Apply Sensitive Data Protection redaction so the ID values are removed from the output entirely
  • B) Apply Sensitive Data Protection deterministic tokenisation (format-preserving encryption) to the ID column
  • C) Encrypt the whole table with a CMEK key and grant analysts the key decrypt role
  • D) Move the table behind a VPC Service Controls perimeter and grant analysts BigQuery access
Reveal answer & explanation

Correct: B. Deterministic tokenisation replaces each ID with a stable surrogate, so the value is unreadable but the same input always yields the same token, preserving joins across tables.

A: Redaction removes the values, breaking the cross-table joins the analysts require. C: Granting decrypt access makes the raw IDs readable, defeating the requirement. D: A perimeter controls where data can go but does nothing to de-identify the ID values for the analysts who are allowed in.

Q3. Your security team must ensure that the engineer who manages encryption keys cannot also grant themselves read access to the data those keys protect, satisfying a separation-of-duties audit finding. How do you structure this?

  • A) Give the key engineer Owner on the project so they can manage both keys and data consistently
  • B) Store all keys as Google-managed so no human holds key permissions at all
  • C) Put the keys in Secret Manager and grant the key engineer the Secret Accessor role
  • D) Grant the key engineer Cloud KMS Admin on the key project and ensure they hold no data-read role on the data project
Reveal answer & explanation

Correct: D. Splitting duties means the key admin holds only KMS roles in the key project and has no data-read role on the data resources, so they can manage keys but cannot read the protected data.

A: Owner grants both key and data control to one person, which is the exact concentration the audit flagged. B: Google-managed keys remove the requirement to manage keys at all, but the question requires a human key admin who is fenced off from data. C: Secret Manager stores secrets, not the KMS keys that encrypt the datasets, so it does not implement key/data separation.
