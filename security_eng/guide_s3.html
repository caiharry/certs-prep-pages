Section 3: Ensuring Data Protection

Protect data at rest, in transit, and in use; tame secrets, keys, and modern AI workloads (~23%). The crown jewels live here — encrypt them, hide them, and keep a kill-switch handy.

Section 3 🟢 ~23%

This domain is all about the data itself — not who reaches it (Section 1) or how it travels (Section 2), but how it's encrypted, de-identified, and secret-managed. The exam leans hard on the four encryption-key models and on Sensitive Data Protection, so we'll spend most of our time there. Grab your cryptographic hard hat. ⛏️🔐

3.1 The Four Encryption Key Models

Everything in Google Cloud is encrypted at rest by default. The exam question is always who holds and controls the key. Memorise these four rungs — the distinctions win you marks.

Model Where The Key Lives You Reach For It When...
Google-managed (default) Entirely in Google's infrastructure; you never see it. No special key-control requirement exists.
CMEK (Cloud KMS) In Cloud KMS; you own rotation, disable, destroy. Supports import. You must control the key lifecycle and have a cryptographic kill-switch.
Cloud EKM In an external/on-prem HSM or KMS — never persists in Google Cloud. A regulator says "keys must never reside in the cloud."
Confidential Computing N/A — encrypts data in use, in memory, via the CPU. Data must stay encrypted even while being processed.
EKM vs CMEK-import is the classic trap. "Keys must never leave / never reside in Google Cloud" = Cloud EKM. "We generate the key ourselves then bring it in" = CMEK import into Cloud KMS (the material now lives in KMS). "Encrypted while being processed in memory" = Confidential Computing, not a key model at all.

💡 The 3-in-a-row: REST / TRANSIT / USE

Data at rest → CMEK/EKM. Data in transit → TLS (automatic on Google's network). Data in use → Confidential Computing. If a stem stresses "while being processed," the answer is almost always Confidential VMs/GKE Nodes.

3.2 CMEK in Practice: Rotation, Kill-Switch & Separation of Duties

Choose CMEK and you own the whole key lifecycle in Cloud KMS: create key rings, set automatic rotation schedules, and disable, destroy, or import key material. Each Google service reaches your key through a per-service service agent that needs roles/cloudkms.cryptoKeyEncrypterDecrypter on it.

🔌 The Cryptographic Kill-Switch

Need to make encrypted data instantly unreadable without touching the data itself? Disable (or destroy) the CMEK key version, or revoke the service agent's key role. With no key, the ciphertext can't be decrypted — an immediate, surgical kill-switch. Re-running DLP or editing bucket IAM does not achieve this.

🧑‍⚖️ Separation of Duties

The person who manages keys should not also be able to read the data those keys protect. Split it: the key admin holds only KMS roles in the key project and holds no data-read role in the data project. Now no single human can both unlock and read — satisfying the audit finding examiners love to set.

"Instantly revoke access to encrypted data" = disable the key version. Don't pick "re-run DLP," "change bucket IAM," or "delete the data." Only the cryptographic move gives an instant, reversible-by-re-enabling kill-switch without destroying the underlying objects.

3.3 Sensitive Data Protection (formerly Cloud DLP)

Sensitive Data Protection does three distinct jobs the exam carefully separates:

Job What It Does Trigger Phrase
Discovery / Profiling Finds where sensitive data lives across BigQuery, Cloud Storage, Cloud SQL at scale. "Find out which tables contain PII."
Inspection Matches infoTypes (credit cards, national IDs, emails) in content. "Detect credit-card numbers in this dataset."
De-identification Redaction, masking, or format-preserving tokenisation/pseudonymisation. "Hide the IDs but keep analytics working."

💡 FIND, then FUZZ

Sensitive Data Protection first finds the PII (discovery + inspection), then fuzzes it (redact / mask / tokenise). If the stem says analytics or joins must still work, the answer is deterministic tokenisation (the same input always maps to the same token) — not redaction, which deletes the value and breaks joins.

"I asked Cloud KMS to keep a secret. It said: 'Sure — but if you ever want it back, you'll need the key.' Brutal, but honest." 🔑🤐

3.4 Secret Manager

Passwords, API keys, and certificates do not belong in code, config files, or container images. Secret Manager gives you central, versioned secrets with IAM-scoped access, CMEK encryption, rotation, and audit logging — fetched at runtime.

Secret Manager stores secrets, not KMS keys. If a question is about separation of key/data duties, the answer involves Cloud KMS roles — not granting Secret Manager Accessor. Secret Manager holds the database password; Cloud KMS holds the encryption key. Don't swap them.

3.5 Securing AI / ML Workloads

The current exam adds security controls for ML systems. Protect training data and models from exfiltration (wrap the project in VPC Service Controls, use CMEK on the buckets and Vertex AI resources), lock down Vertex AI access with least-privilege IAM and private endpoints, and screen generative AI inputs/outputs with Model Armor and Sensitive Data Protection (catch prompt injection, jailbreaks, and PII leakage).

🤖 Model Armor in One Line

Think of Model Armor as a WAF for LLM prompts and responses: it inspects generative AI traffic for prompt-injection, harmful content, and sensitive-data leakage, independent of which model you run.

3.6 Cloud HSM & Certificate Authority Service

Cloud HSM is not a separate product — it is a key protection level within Cloud KMS. When you create a Cloud HSM key, your key material is generated in and never leaves a FIPS 140-2 Level 3 certified hardware security module that Google operates. You still manage the key through the same KMS API; the only difference is the physical guarantee.

Cloud HSM keys live in Google's infrastructure — they are NOT the same as Cloud EKM. With Cloud HSM, the key material is physically protected inside Google-operated HSMs and never leaves them. With Cloud EKM, the key material never enters Google infrastructure at all — it stays in your external HSM. The exam loves this distinction: "FIPS Level 3 hardware protection" = Cloud HSM (Google infra); "key never enters Google Cloud" = Cloud EKM (your infra).

Software KMS vs Cloud HSM vs Cloud EKM vs Confidential Computing

Option Key Location FIPS Level Protects Data… Reach For When…
Software KMS (CMEK) Google infrastructure, software-protected 140-2 Level 1 At rest You need key lifecycle control without hardware requirements.
Cloud HSM Google-operated HSM (hardware-backed, still Google infra) 140-2 Level 3 At rest Compliance mandates hardware-backed keys; key can stay in Google Cloud.
Cloud EKM Your external HSM / KMS (never enters Google infra) Depends on your HSM At rest Regulatory requirement that keys never reside in the cloud provider's environment.
Confidential Computing N/A — CPU-level memory encryption (AMD SEV / Intel TDX) N/A In use (memory) Data must remain encrypted while being actively processed — e.g., secure multi-party computation.

Certificate Authority Service (CAS)

Certificate Authority Service lets you run a fully managed private CA hierarchy inside Google Cloud without operating any CA server yourself. It supports a standard two-tier model: a Root CA (rarely online; signs subordinate CA certificates) and one or more Subordinate CAs (do the day-to-day signing). Every issued X.509 certificate carries audit trails back to your root, making it ideal for internal services, mutual TLS (mTLS), and workload identity.

🏢 CAS Tier: DevOps vs Enterprise CA

  • DevOps CA — fast issuance (sub-second), lower cost, no per-certificate audit log. Use it for ephemeral workload identities, short-lived service mesh certs, or Kubernetes pod identities where you issue thousands of certs per minute.
  • Enterprise CA — full per-certificate audit logging, certificate revocation (CRL + OCSP), longer-lived certs. Use it for anything subject to compliance audit — TLS certs for internal APIs, user client certs, code-signing certs.

Certificate-based Access Control extends CAS beyond encryption into authorisation: services present a CAS-issued X.509 certificate at the application layer (mTLS), and the service validates the certificate's Subject or SAN fields as proof of identity. Combined with BeyondCorp Enterprise or Anthos Service Mesh, this removes reliance on network location for trust — only a cryptographically signed identity wins access.

⚠️ CAS vs Public Certificate Managers

CAS issues private certificates — they are trusted only within hierarchies you control. For public TLS (web-facing HTTPS trusted by browsers), use Google-managed certificates or a public CA. CAS is not a Let's Encrypt replacement; it's a private PKI factory for internal use.

"The Root CA signed the Subordinate CA. The Subordinate CA signed the leaf cert. The leaf cert signed into the Kubernetes pod. It's certificates all the way down — turtles would be jealous." 🐢🔏

3.7 De-identification Techniques & Tokenization Deep-Dive

Sensitive Data Protection offers five de-identification transforms. Choosing the wrong one in the exam costs marks — they all "hide" data, but differ critically in reversibility and analytical utility.

Transform What It Does Reversible? Use When…
Redaction Removes the value entirely (replaces with nothing or a placeholder like [REDACTED]). No Value must never appear again — e.g., free-text medical notes published externally.
Masking Replaces characters with a fixed mask character (e.g., ****-****-****-1234). No Audit logs, support interfaces — show partial info to humans but never reconstruct the original.
Date Shifting Shifts date values by a randomised (but consistent-per-entity) offset, preserving relative time spans. Yes (with the shift key) Clinical trial data, age-sensitive analytics — time relationships matter but absolute dates must be obscured.
Bucketing (Generalisation) Replaces precise values with a range or category (e.g., age 34 → 30–40). No Statistical reporting where groupings are sufficient — k-anonymity requirements.
Pseudonymisation / Tokenisation Replaces a value with a surrogate token generated by encryption; the same input always yields the same token (deterministic). Yes (with the crypto key) Analytics pipelines, cross-table joins — must preserve referential integrity while hiding the raw value.

💡 RMDBT — "Rather Many Data Bits Transformed"

The five transforms in order: Redact, Mask, Date-shift, Bucket, Tokenise. The first two wipe information; the last one is the only reversible option that preserves join keys. Think FIND then FUZZ with RMDBT: first Sensitive Data Protection finds PII (inspect/profile), then it fuzzes it with one of these five transforms.

Masking is NOT reversible — tokenisation IS (with the key). Masking replaces characters with * or a mask character; there is no key and the original cannot be recovered. Tokenisation (deterministic / format-preserving encryption) uses a cryptographic key, meaning the operation can be reversed to recover the original value, and the same input always produces the same token (enabling joins). The exam regularly describes a scenario where "you need to re-identify data later" and the only correct transform is tokenisation — never masking.

Deterministic vs Random Tokenisation

Deterministic (format-preserving) encryption always maps the same plaintext to the same ciphertext under the same key. This is the right choice when you need stable surrogates for analytics or joins across tables. Random tokenisation generates a different token each time, breaking the ability to correlate records — use it when maximum unlinkability matters more than analytical utility (e.g., one-time payment nonces).

Inspect Jobs vs De-ID Jobs

The Sensitive Data Protection API has two distinct entry points:

  • DlpService.InspectContent / CreateInspectJob — read-only: identifies where PII exists and returns findings. Used for discovery and profiling.
  • DlpService.DeidentifyContent / CreateDlpJob with a deidentify template — transforms the data: runs redact/mask/tokenise on the content and returns transformed output.

The exam may ask which API call to use; remember these are different jobs and different request bodies, not the same call with an optional transform flag.

TINK & Format-Preserving Encryption

Under the hood, Sensitive Data Protection uses Google's open-source TINK cryptography library for format-preserving encryption (FPE) when performing deterministic tokenisation. TINK provides AES-SIV and AES-FFX primitives that keep the ciphertext in the same character space as the plaintext (so a 16-digit card number tokenises to a 16-digit surrogate). You can use TINK directly in application code to apply the same cryptographic transforms outside the DLP API — useful for client-side de-identification before data even reaches Google Cloud.

Hybrid Inspection: DLP in Streaming Pipelines

When data flows through a streaming pipeline (Pub/Sub → Dataflow), you can run hybrid inspection: the Dataflow job calls the DLP API inline, inspecting and de-identifying messages in flight before they land in BigQuery or Cloud Storage. This means PII is scrubbed at the pipeline layer without ever touching a persistent store. The exam presents this as the pattern for "inspect streaming data before it lands."

🔄 Re-identification Risk & Data Profiling Workflow

Even after de-identification, combining several quasi-identifiers (age range, gender, postcode) can re-identify individuals. Sensitive Data Protection's data profiling (discovery) workflow automates the scan: it profiles BigQuery datasets across an organisation, assigns a data risk score and sensitivity level, and can export findings to Security Command Center. The exam asks: "How do you continuously monitor for new PII in BigQuery?" — answer: configure an automatic discovery scan at the organisation level, not a one-time inspect job.

3.8 Data Lifecycle & Retention Controls

Knowing how long data lives and who can delete it is as important as knowing how to protect it. The exam tests whether you can wire up compliant retention without confusing "retention policy" with "access control."

Cloud Storage Object Lifecycle Management

Object Lifecycle Management lets you define rules that automatically act on objects as they age. Rules are evaluated daily and trigger one of three actions:

  • Delete: permanently deletes objects after a specified age (days since creation, or days since becoming non-current for versioned buckets).
  • Change storage class: automatically transitions objects from Standard → Nearline → Coldline → Archive as they age — the typical cost-optimisation pattern for compliance data that must be retained but rarely accessed.
  • Abort incomplete multipart uploads: cleans up incomplete uploads after a threshold, preventing orphaned storage charges.

Lifecycle rules are attached at the bucket level and evaluated independently for each object. They cannot be used to prevent deletion — that requires a retention policy or Bucket Lock (below).

Bucket Lock & Retention Policies: WORM Compliance

Retention policies set a minimum duration before objects in a bucket can be deleted or replaced. Bucket Lock makes that policy permanent. This is the Cloud Storage answer for WORM (Write Once, Read Many) compliance requirements mandated by regulations such as SEC Rule 17a-4 and FINRA.

  • Set a retention policy on a bucket (e.g. 7 years). Objects cannot be deleted or overwritten until the retention period has elapsed.
  • Lock the retention policy by calling lockRetentionPolicy(). Once locked, the policy cannot be reduced, removed, or modified — not even by a project owner or the Google Cloud support team.
  • Objects can still be read at any time during retention; they simply cannot be deleted before the period expires.
  • Versioning + retention: if you overwrite an object in a versioned bucket, the non-current version still carries its own retention clock and cannot be deleted early.
Bucket Lock is PERMANENT. Once a retention policy is locked, it cannot be shortened or removed, and objects cannot be deleted before expiry — by anyone, under any circumstances. This is the WORM answer. A common distractor offers "disable the retention policy to delete the object early" — this is impossible once locked. If you need flexibility to change the retention period later, do NOT lock; if regulatory compliance requires WORM immutability, locking is mandatory.

BigQuery Table & Partition Expiration

BigQuery supports expiration at two granularities:

  • Table expiration: set --time_to_live on a dataset (default expiry for new tables) or on individual tables. BigQuery automatically deletes expired tables. Useful for staging datasets, temporary experiment results, or ephemeral analytics outputs that must not accumulate indefinitely.
  • Partition expiration: for partitioned tables, set partitionExpirationMs. Partitions older than the threshold are deleted automatically, while newer partitions remain — ideal for time-series data with rolling retention windows (e.g. keep 90 days of raw events, always).

Unlike Cloud Storage retention policies, BigQuery expiration is a soft delete (deletion is automatic but not WORM). There is no equivalent of Bucket Lock in BigQuery — for immutable audit records, export to Cloud Storage with a locked retention policy.

Cloud Logging Retention

Cloud Logging has two special built-in log buckets with fixed retention:

  • _Required bucket: holds Admin Activity and System Event audit logs. Retention is 400 days, non-configurable. You cannot reduce or delete this — Google mandates it for accountability.
  • _Default bucket: holds all other logs (Data Access, Access Transparency, user-defined). Default retention is 30 days. You can increase this up to 3,650 days (10 years) for a storage cost, or decrease it as low as 1 day.

For retention beyond what Cloud Logging supports (or for cheaper long-term archival), create a log sink that exports to Cloud Storage. Cloud Storage + Bucket Lock is the standard pattern for 7-year audit log retention required by SOX, PCI DSS, and similar frameworks.

Preventing Public Exposure: iam.allowPublicObjectBehavior

Accidental public access to Cloud Storage objects is a leading cause of data breaches. The organisation policy constraint storage.publicAccessPrevention enforces that no bucket in the org (or a specified folder/project) can grant allUsers or allAuthenticatedUsers access — regardless of bucket-level ACLs or IAM bindings. Setting this constraint at the org level prevents any team from accidentally making a bucket public, even if they have storage admin rights. This is the preferred data loss prevention mechanism over relying on individual bucket configuration audits.

💡 Lock, Expire, Sink — The Lifecycle Trio

Lock it (Bucket Lock) when regulators demand WORM immutability. Expire it (lifecycle rules / BQ expiration) when data should auto-clean after its useful life. Sink it (log sink → Cloud Storage) when you need cheaper long-term retention than Cloud Logging provides. Three tools, three distinct problems — pick the wrong one on the exam and you're choosing between "cannot delete" and "deleted too soon."

"The compliance officer asked: 'Can you delete this locked object?' The bucket replied: 'Have you tried turning off WORM compliance?' Nobody laughed. The retention policy remained. For seven years." 🔒😅

Legal Holds

Bucket Lock enforces time-based retention. A legal hold enforces event-based retention: an object under legal hold cannot be deleted regardless of the retention policy expiry — the hold takes precedence until explicitly released. Use legal holds when litigation or regulatory investigation requires preserving specific objects beyond their standard retention period. Holds are applied per-object (not per-bucket) and can be set via the Cloud Storage API or gcloud.

Retention Policy vs Legal Hold: A Comparison

Feature Retention Policy Legal Hold
Scope Bucket-level (applies to all objects) Object-level (applied selectively)
Release mechanism Time-based (expiry after duration) Event-based (released explicitly by an authorised user)
Interaction if both set Object cannot be deleted until both the retention period has elapsed and all legal holds have been released.
Lockable? Yes — Bucket Lock makes it permanent No — holds are always releasable by authorised users
Exam use case WORM compliance, regulatory retention windows Litigation hold ("preserve all emails from 1 Jan – 31 Mar")

Org Policy for Data Loss Prevention

Beyond per-bucket controls, organisation policies provide fleet-wide enforcement. Key constraints for data lifecycle and exposure prevention:

  • storage.publicAccessPrevention — blocks allUsers / allAuthenticatedUsers on any bucket in scope. The single most impactful prevention control for Cloud Storage data exposure.
  • storage.uniformBucketLevelAccess — enforces Uniform Bucket-Level Access on all buckets, disabling legacy ACLs and ensuring IAM is the sole access-control mechanism. Without this, ACL loopholes can bypass IAM bindings.
  • storage.retentionPolicySeconds — enforce a minimum retention period on all buckets (useful when you need to prevent teams from setting zero-day expiry on regulated data).

🏛️ The Hierarchy: Policy > Lock > Hold

Think of data lifecycle protection as three layers: the org policy prevents accidental public access fleet-wide (the outer wall). The retention policy + Bucket Lock prevents premature deletion of regulated data (the vault door). The legal hold freezes specific objects when litigation demands it (the evidence bag). Each layer operates independently — a locked retention policy does not release a legal hold, and an expired legal hold does not override a valid retention period.

Cloud Logging Retention Comparison: In-Product vs Sink

The decision between extending Cloud Logging retention and sinking logs to Cloud Storage comes down to cost, searchability, and compliance requirements:

Approach Retention Limit Search/Query Cost Model Best For
Extended _Default bucket Up to 3,650 days Log Explorer (full query) Per-GiB-per-month storage charge Frequently queried logs (security investigations, compliance spot-checks)
Log sink → Cloud Storage Unlimited (bucket lifecycle controls) None natively; query via BigQuery or Dataflow Nearline/Coldline/Archive storage rates Long-term archival (> 3 years), WORM compliance via Bucket Lock
Log sink → BigQuery Table expiration-controlled SQL queries over log data BigQuery storage + query costs Analytics on logs, trend queries, custom dashboards

For compliance frameworks that mandate 7-year log retention (SOX Section 802, HIPAA, PCI DSS), the standard pattern is: sink to Cloud Storage → Nearline storage class (30–90 day access) → lifecycle rule transitions to Coldline at 1 year, Archive at 3 years → Bucket Lock with 7-year retention policy. This minimises cost while meeting WORM requirements.

🧠 Quick-Fire Warm-Up Quiz!

Retrieve before you review. No peeking!

Q1: Discover and redact credit-card numbers in a Cloud Storage dataset before analysis. Which service?

Sensitive Data Protection (Cloud DLP). It detects, classifies, and de-identifies sensitive data.

Q2: A regulator requires that encryption keys are held only in your on-prem HSM and never reside in Google Cloud. Which option?

Cloud EKM. External keys never persist in Google Cloud; Google receives only a short-lived wrapped reference.

Q3: An auditor asks you to instantly render a CMEK-encrypted dataset unreadable without deleting the data. What do you do?

Disable (or destroy) the CMEK key version in Cloud KMS. Without the key, the ciphertext can't be decrypted — an immediate cryptographic kill-switch.

Q4: A database password is currently baked into a container image. Where should it live instead?

Secret Manager. Versioned, IAM-controlled, encrypted storage with rotation, retrieved at runtime.

Question 1: A financial regulator requires that the encryption keys protecting your BigQuery datasets are generated and stored exclusively in your own on-premises HSM and never reside in Google Cloud, even briefly. Which key option satisfies this?

  • A) CMEK with key material imported into Cloud KMS from the on-prem HSM.
  • B) Google-managed encryption keys with Access Approval enabled on the project.
  • C) Cloud External Key Manager backed by your on-prem HSM through a supported partner.
  • D) CMEK in Cloud HSM with a strict automatic rotation policy and audit logging.

Correct Answer: C

Why C is correct: Cloud EKM keeps the key in your external HSM and Google only receives a short-lived wrapped reference, so the key material never resides in Google Cloud — exactly the regulator's constraint.

Why the others are wrong:
A: Importing key material places that material into Cloud KMS, violating "never resides in Google Cloud."
B: Google-managed keys live entirely in Google's infrastructure; Access Approval gates support access but not key location.
D: Cloud HSM keys are generated and held in Google's HSMs, not your on-prem HSM.

Question 2: Analysts need to run aggregate queries over a customer table that contains national ID numbers. The IDs must stay non-readable, yet rows for the same customer must still join consistently across tables. What is the right transformation?

  • A) Apply Sensitive Data Protection redaction so the ID values are removed from the output entirely.
  • B) Apply Sensitive Data Protection deterministic tokenisation (format-preserving encryption) to the ID column.
  • C) Encrypt the whole table with a CMEK key and grant analysts the key decrypt role.
  • D) Move the table behind a VPC Service Controls perimeter and grant analysts BigQuery access.

Correct Answer: B

Why B is correct: Deterministic tokenisation replaces each ID with a stable surrogate, so the value is unreadable but the same input always yields the same token, preserving joins across tables.

Why the others are wrong:
A: Redaction removes the values, breaking the cross-table joins the analysts require.
C: Granting decrypt access makes the raw IDs readable, defeating the requirement.
D: A perimeter controls where data can go but does nothing to de-identify the ID values for the analysts who are allowed in.

Question 3: Your security team must ensure that the engineer who manages encryption keys cannot also grant themselves read access to the data those keys protect, satisfying a separation-of-duties audit finding. How do you structure this?

  • A) Give the key engineer Owner on the project so they can manage both keys and data consistently.
  • B) Store all keys as Google-managed so no human holds key permissions at all.
  • C) Put the keys in Secret Manager and grant the key engineer the Secret Accessor role.
  • D) Grant the key engineer Cloud KMS Admin on the key project and ensure they hold no data-read role on the data project.

Correct Answer: D

Why D is correct: Splitting duties means the key admin holds only KMS roles in the key project and has no data-read role on the data resources, so they can manage keys but cannot read the protected data.

Why the others are wrong:
A: Owner grants both key and data control to one person, which is the exact concentration the audit flagged.
B: Google-managed keys remove the need to manage keys at all, but the question requires a human key admin who is fenced off from data.
C: Secret Manager stores secrets, not the KMS keys that encrypt the datasets, so it doesn't implement key/data separation.
