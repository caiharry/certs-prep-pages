Section 4: Managing Operations

Automate security, detect threats, and respond — the SecOps backbone of the exam (~19%). When the alarm goes off, this is the domain that tells you how to see it, prove it, and contain it.

Section 4 🟠 ~19%

Sections 1–3 build the walls. Section 4 is the security operations centre: hardening the software supply chain, capturing the right logs, spotting threats, and responding without trampling the evidence. Two things get tested to death here — which audit logs are on by default and what Security Command Center actually does. Let's wire up the alarm system. 🚨

4.1 Supply-Chain & Deploy-Time Security

Stop bad software before it ever runs. Binary Authorization is a deploy-time gate for GKE and Cloud Run that admits only container images signed by an approved attestor; everything else is rejected. Feed it by scanning images for CVEs in CI/CD (Artifact Registry / Artifact Analysis), automating image hardening and patching, and catching configuration drift with custom organization policies and Security Health Analytics custom modules.

✍️ Binary Authorization in One Line

"Only signed, attested images may deploy" ⇒ Binary Authorization. It's the bouncer at the cluster door checking each image's signature against your policy before admission.

4.2 The Four Cloud Audit Log Types

This is the most reliably tested fact in the whole domain. Cloud Audit Logs has four streams, and the killer detail is which are on by default.

Log Type Records Default State
Admin Activity Configuration changes (who changed what). Always on, free, cannot disable.
System Event Google-initiated system actions. Always on, free.
Data Access Reads/writes of data (who read which object/row). Off by default (except BigQuery) — enable per service in the IAM audit config.
Policy Denied Requests denied by a security policy (e.g. VPC-SC). On when the relevant service is used.
"We had no record of who read the data." This almost always means Data Access audit logs were never enabled — they're off by default. Don't blame the sink or buy a SIEM first; fix the IAM audit config so the events exist, then route them. A sink can only ship logs that are already being generated.

💡 "Admins are Always Audited; Data is a Decision"

Admin Activity = Always on. Data Access = a Decision you must switch on. If the missing evidence is a read of data, the cause is unenabled Data Access logs.

4.3 Routing Logs: Sinks & Aggregated Sinks

Logs land in the _Default and _Required buckets, but for long retention, analysis, or a SIEM you create log sinks that export to BigQuery, Pub/Sub, Cloud Storage, or an external destination.

🧲 Aggregated Sinks = One Sink to Rule Them All

An aggregated sink at the organization or folder level captures logs from every current and future child project in one place, with no per-project setup. When a question says "central SIEM, least ongoing maintenance, all projects," that's an aggregated sink.

4.4 Threat Detection: Security Command Center

Security Command Center (SCC) is the single pane of glass for posture and threats. Know which built-in service does which job.

SCC Service Detects Example Finding
Security Health Analytics (SHA) Misconfigurations & posture drift. Public bucket, open firewall, no MFA.
Event Threat Detection (ETD) Threats in logs. Malware C2 callout, brute force, anomalous IAM grant.
Container / VM Threat Detection Runtime threats in workloads. Crypto-mining binary, reverse shell, added bad library.

💡 SCC = "SEE": SHA sees config, ETD sees threats

Security Command Center is where you SEE it all. SHA = misconfig. ETD = log threats. CTD/VMTD = workload threats. Posture, vulnerabilities, and active attacks in one console.

4.5 Network Forensics

Three network tools, three depths of visibility — a favourite discriminator.

Tool What You See Use When...
VPC Flow Logs Connection metadata (5-tuple, bytes) — not payloads. You need who-talked-to-whom and volume, cheaply.
Packet Mirroring Full packet payloads copied to a collector. You need deep inspection of actual traffic content.
Cloud IDS Signature-based intrusion detection on mirrored packets. You want managed (Palo Alto) threat signatures, no rule-writing.
Flow Logs ≠ payload inspection. If the stem wants signature-based, full-payload intrusion detection on east-west traffic with no rule authoring, that's Packet Mirroring feeding Cloud IDS. VPC Flow Logs only carry metadata, and Cloud Armor only guards inbound HTTP(S) at the edge — neither inspects east-west payloads.

4.6 Incident Response Without Destroying Evidence

When a host is compromised, the order of operations matters. Contain, then preserve, then investigate:

  • Isolate the VM with a deny-all firewall rule via a network tag — cuts the command-and-control channel immediately.
  • Snapshot the disks for forensic evidence before any further change.
  • Rotate / disable compromised credentials and keys.
  • Then analyse — never delete the instance first, or you destroy the evidence.
"First rule of incident response: don't rm the crime scene. The forensics team gets very grumpy." 🕵️‍♀️💾

4.7 Software Supply Chain Security: SLSA & Artifact Analysis

Modern attacks don't punch through your firewall — they hitch a ride inside a package you trusted. SLSA (Supply-chain Levels for Software Artifacts, pronounced "salsa") is an open framework that grades the integrity of your build process across four levels. Artifact Analysis (formerly Container Analysis) is the GCP service that scans your images and packages for CVEs and stores the results as notes and occurrences in the Grafeas store backing Artifact Registry.

SLSA Levels — What Each Requires

SLSA Level Key Requirements Practical Bar
Level 1 Build process exists and provenance is generated. Easy — many CI systems do this automatically.
Level 2 Hosted, tamper-resistant build service generates signed provenance. Cloud Build natively produces SLSA 2 provenance.
Level 3 Hardened build platform, no persistent worker credentials, hermetic builds. Practically achievable on GCP with Cloud Build + Artifact Registry + Binary Authorization policy.
Level 4 Two-party review, reproducible builds, hardware-attested environment. Rarely required; aspirational for most orgs.

Artifact Analysis: Scanning Modes

Artifact Analysis hooks into Artifact Registry and can scan container images as well as Maven and NPM package metadata.

  • On-demand scanning — triggered explicitly via the gcloud artifacts docker images scan command or the API; useful inside CI pipelines before pushing.
  • Continuous scanning — enabled at the Artifact Registry repository level; re-scans images when new CVE metadata is published, so vulnerabilities discovered after the push still surface.
Binary Authorization does NOT scan images. It only checks whether a valid attestation signature exists on the image before admitting it. You are responsible for running the scan (via Artifact Analysis), creating the attestation (via KRITIS or Sigstore / cosign), and wiring Binary Authorization to require that attestation. If the scan step is missing, Binary Authorization will happily admit an unscanned image that has been signed — it checks signatures, not vulnerability reports.

Attestors & Attestations

The pipeline from scan to gate works in three steps:

  1. Artifact Analysis scans the image and finds no blocking CVEs (or all CVEs are below the threshold).
  2. A signer service (e.g. KRITIS signer, or cosign backed by Cloud KMS) creates a signed attestation on the image digest and stores it in Artifact Registry.
  3. Binary Authorization evaluates the pod admission request: if the policy requires the attestation from a named attestor and the signature is valid, the workload deploys; otherwise it's rejected.

Software Bill of Materials (SBOM)

An SBOM is a machine-readable list of every dependency in a software artefact — libraries, versions, licences, known CVEs. Regulators (US Executive Order 14028, EU Cyber Resilience Act) increasingly mandate SBOMs for software sold to government. Artifact Analysis can generate SBOMs in SPDX or CycloneDX format and store them as attestations, giving you a signed, tamper-resistant bill of materials for every image.

Cloud Deploy Integration

Cloud Deploy adds deploy-time controls on top of the binary: approval gates require a human sign-off before a release advances to production, and rollout release controls (canary, blue/green) limit blast radius. Pair Cloud Deploy with Binary Authorization to ensure that a human approved and the image is attested before anything touches production.

🔗 The End-to-End Supply Chain on GCP

Artifact Analysis scans → KRITIS / cosign attests → Binary Authorization gates admission. Every step is audited: scan results in Artifact Registry, attestation signed with a Cloud KMS key, Binary Authorization policy violations surfaced in Cloud Audit Logs and SCC. Add Cloud Deploy approval gates for the human checkpoint before production.

Service What It Does What It Does NOT Do
Artifact Analysis Scans container images and packages for CVEs; stores results as Grafeas notes and occurrences; generates SBOMs. Does not block deployments; does not sign attestations automatically.
Binary Authorization Admission controller for GKE, Cloud Run, GKE Autopilot; blocks workloads unless named attestations are present. Does not scan images; does not create attestations; does not run CVE checks.
Web Security Scanner Crawls deployed web apps for OWASP Top-10 issues (XSS, mixed content, outdated libraries in the browser). Does not inspect container images or supply-chain dependencies; does not block deployments.
The SBOM told us we were shipping 47 transitive dependencies we'd never heard of. The good news: we finally have a list. The bad news: we now have to care about 47 more things. 📦

4.8 Chronicle SIEM & Threat Intelligence

Chronicle is Google's cloud-native Security Information and Event Management (SIEM) and Security Orchestration, Automation, and Response (SOAR) platform. It is not a feature of Cloud Logging or SCC — it is a separate product built on Google's own petabyte-scale infrastructure, originally developed to protect Google internally and now available as a managed service. The elevator pitch: ingest everything, store it for 12 months by default, and let analysts hunt across the full corpus in seconds.

Unified Data Model (UDM)

Chronicle normalises ingested logs from any source (GCP, AWS, on-prem, SaaS, network devices) into the Unified Data Model. UDM events share a common schema regardless of origin, which means a single detection rule can match a Windows login event, a GCP IAM grant, and a Fortinet firewall alert without bespoke parsing per source.

YARA-L: Chronicle's Detection Language

YARA-L is Chronicle's rule language for writing detections over the UDM event stream. It extends the well-known YARA pattern language with a temporal window construct so rules can express multi-event sequences — for example: "if the same user creates an IAM binding and then exfiltrates data from Cloud Storage within 10 minutes, raise an alert." YARA-L rules are evaluated continuously against the live stream of incoming events.

Threat Intelligence Integration

  • VirusTotal enrichment — Chronicle natively enriches IOCs (IPs, domains, file hashes) against VirusTotal reputation data, surfacing threat context without leaving the investigation UI.
  • Curated Threat Intelligence (CTITM) — Google-managed threat intelligence feeds built into both Chronicle and SCC, covering malware families, attacker infrastructure, and CVE-to-exploit mappings updated continuously by Google's Threat Intelligence team (formerly Mandiant).
  • Third-party feeds — Chronicle can ingest STIX/TAXII feeds from external providers via the Threat Intelligence management interface.

Chronicle vs Cloud Logging vs SCC: Pick the Right Tool

Dimension Cloud Logging Security Command Center Chronicle
Primary role Collect and route logs; short-term storage and alerting. Posture management; aggregated security findings. SIEM/SOAR; large-scale investigation and hunting.
Default retention 30 days (_Default bucket); configurable up to 3650 days. Findings retained; not a log store. 12 months (petabyte-scale, included in price).
Query language Log Explorer filter syntax / SQL in BigQuery. SCC filter syntax; no raw log queries. UDM Search + YARA-L rules.
Multi-tenant / cross-org Aggregated sinks help; log buckets are per-project. Yes, org-level dashboard. Yes — designed for MSSPs and large enterprises.
Threat Intel enrichment None built-in. CTITM via SCC Premium. VirusTotal + CTITM native.

🔍 Chronicle vs Cloud Logging: Know the Boundary

Cloud Logging is where logs start — they land in the _Default bucket and you route them via sinks. Chronicle is where analysts investigate at scale — it ingests logs from Cloud Logging (and everywhere else), normalises them into UDM, and lets you hunt across 12 months of data in seconds. The two are complementary, not competing: logs flow from Cloud Logging into Chronicle.

Chronicle is not Cloud Logging and not SCC. All three appear in Security Engineer questions and distractors love to conflate them. Chronicle has its own ingestion pipeline, its own data model (UDM), its own query language (YARA-L / UDM Search), and 12-month retention included. Cloud Logging has 30-day default retention and Log Explorer filter syntax. SCC surfaces security findings (misconfigs, threats) but is not a log store. If a question mentions "SIEM," "detection rules across 12 months of logs," or "SOC investigation at scale" — that's Chronicle.

💡 "LFC: Logs Flow to Chronicle"

Think of it as a pipeline:
Logging → collects and routes raw logs (the pipe).
Findings → SCC surfaces security findings from those logs and GCP APIs (the dashboard).
Chronicle → ingests, normalises, and lets analysts hunt at scale (the SIEM).
Each layer consumes the one before it. "Logs Flow to Chronicle" — LFC.

SOC Workflow in Chronicle

Chronicle maps to the classic SOC workflow: Ingest (pull logs from GCP, AWS, on-prem, SaaS via forwarders and ingestion APIs) → Detect (YARA-L rules match events in real time; curated detections from Google Threat Intelligence fire automatically) → Investigate (analysts pivot from an alert through 12 months of UDM-normalised telemetry using graph search and IOC enrichment) → Respond (SOAR playbooks in Chronicle trigger actions: isolate a VM via the Compute API, revoke an IAM token, open a ticket in Jira). This end-to-end loop is what "cloud-native SIEM/SOAR" means in practice.

4.9 Vulnerability Management & Security Health Analytics

Security Command Center is not just an alert dashboard — it is an active posture scanner. The two most exam-relevant built-in detectors are Security Health Analytics (SHA) for configuration misconfigurations and Web Security Scanner (WSS) for application-layer vulnerabilities. Both surface findings in SCC; the key skill is knowing which detector fires when.

Security Health Analytics (SHA)

SHA continuously scans the configuration of GCP resources against a library of security best-practice checks. It does not read traffic or logs — it inspects resource metadata via GCP APIs. Common findings it produces:

  • PUBLIC_BUCKET_ACL — Cloud Storage bucket has allUsers or allAuthenticatedUsers on an ACL.
  • OPEN_FIREWALL — a VPC firewall rule allows ingress from 0.0.0.0/0 on port 22 (SSH) or 3389 (RDP).
  • OS_LOGIN_DISABLED — OS Login not enforced on a Compute Engine instance.
  • AUDIT_LOGGING_DISABLED — data access audit logs not enabled for a service.
  • MFA_NOT_ENFORCED — Cloud Identity tenant has accounts without 2-step verification.

SHA findings are classified by category (VULNERABILITY, MISCONFIGURATION, OBSERVATION, THREAT) and by severity (CRITICAL, HIGH, MEDIUM, LOW). In the SCC console you can filter findings by any combination of these fields, mute individual findings, or create mute rules that suppress whole classes of findings (e.g., suppress all OPEN_FIREWALL findings on a project used for load-testing).

🏋️ Custom SHA Modules

SHA ships with ~130 built-in detectors, but you can extend it with custom modules — CEL (Common Expression Language) expressions that SHA evaluates against resource configurations. A custom module can, for example, flag any Cloud Run service without a VPC connector, or any BigQuery dataset shared with a principal outside your org domain. Custom modules surface findings in SCC under the same finding-type framework as built-in detectors, so your existing SIEM exports and notification channels pick them up automatically.

Web Security Scanner (WSS)

WSS scans external-facing web applications (App Engine, GKE Ingress, Cloud Run) for common application-layer vulnerabilities from the OWASP Top 10: XSS, SQL injection, mixed content, outdated libraries with known CVEs, and more. Two modes:

  • Managed scan — SCC runs WSS automatically on supported resources, no configuration required. Available in Security Command Center Standard and Premium tiers.
  • Custom scan — you define the target URL, authentication credentials (if the app requires login), and scan schedule. Gives coverage for apps behind auth that managed scans can't reach.

WSS findings appear as WEB_SECURITY_SCANNER-type findings in SCC. Because WSS actually sends HTTP requests, it is only appropriate for external endpoints — running it against an internal service requires network reachability from GCP's scanner infrastructure.

🚨 WSS Is Not a Substitute for a Proper DAST Tool

Web Security Scanner covers the most common OWASP Top 10 patterns, but it is not a full dynamic application security testing (DAST) suite. It will miss business-logic flaws, complex authentication bypass chains, and vulnerabilities that only appear after multiple sequential interactions. For the exam, WSS is the right answer when the scenario is specifically about basic OWASP vulnerability detection on a GCP-hosted web app with low setup cost. If the scenario mentions "comprehensive penetration testing" or "DAST" as a requirement, the answer is likely a third-party tool integrated via SCC's partner findings API, not WSS alone.

OS Config & Patch Management

For fleet-wide OS-level vulnerability management, OS Config (VM Manager) provides: patch compliance baselines showing which VMs are missing OS patches, on-demand patching (patch jobs that target VMs by label or zone), and OS inventory data (installed packages and versions ingested into SCC as OS_VULNERABILITY findings when known CVEs are matched). This is the GCP answer to "how do I know which VMs are missing the latest kernel patch?"

OS Config requires the OS Config agent to be installed on each VM (pre-installed on public GCP images from mid-2020 onward; must be manually installed on custom images). Key exam scenario: a question asks how to apply a critical kernel patch to 200 Compute Engine VMs with minimal manual effort — the answer is a VM Manager patch job scoped by label, not SSHing into each instance. Patch jobs support rollout controls (zone-by-zone, percentage-based) to avoid taking down your whole fleet at once. The compliance report in VM Manager shows a per-VM breakdown of COMPLIANT / NON_COMPLIANT / PENDING against the patch baseline, and these findings also flow into SCC so you have a single pane of glass for both posture and patch state.

SCC Finding Lifecycle: States, Muting, and Tiers

Every SCC finding has a state — ACTIVE, INACTIVE, or MUTED. Understanding these is important for the exam because they affect what a security operations team actually sees:

  • ACTIVE — the finding is open and unresolved. Appears in the default SCC dashboard view.
  • INACTIVE — the underlying condition has been remediated (e.g., the public bucket ACL was removed). SHA automatically sets findings to INACTIVE on the next scan cycle when the misconfiguration is gone.
  • MUTED — the finding still exists but is suppressed from view. Muting is appropriate for accepted risk or known false positives. You can create mute rules (CEL-based) to auto-mute whole categories of findings on specific resources.

On SCC tier: SHA is available at both Standard and Premium. The distinction matters for the exam — ETD, Web Security Scanner (managed), and Security Posture are Premium-only. If a question asks about the lowest-cost way to detect active threats from log signals, the answer requires SCC Premium.

One timing detail that catches people out: SHA findings do not move to INACTIVE immediately after remediation. SHA runs scans on a schedule (roughly every 12–24 hours depending on the check type). If you fix a public bucket ACL and check SCC five minutes later, the finding will still show ACTIVE until the next SHA scan cycle completes and confirms the condition is gone. This matters in incident-response scenarios where teams assume the finding disappearing means the fix worked — always verify directly on the resource, not via finding state alone.

🏋️ SCC Standard vs Premium — What's Available Where

A reliable exam shorthand:

  • Standard (free): Security Health Analytics built-in detectors, Web Security Scanner basic managed scan, IAM Recommender findings.
  • Premium (paid): Event Threat Detection (ETD), Container Threat Detection (CTD), Virtual Machine Threat Detection (VMTD), Web Security Scanner with custom scans, Security Posture management, Rapid Vulnerability Detection, compliance reports (CIS, PCI-DSS, NIST).
  • Enterprise (paid + SIEM/SOAR): Everything in Premium, plus Chronicle integration, SOAR playbooks, attack path simulation, and multi-cloud (AWS/Azure) posture.

If the stem involves compliance reporting or runtime threat detection, Premium (or Enterprise) is the minimum tier. Standard alone cannot do it.

SHA detects misconfigurations — not active threats. SHA sees that port 22 is open; it does not see a brute-force SSH attempt in progress. If the exam stem says "detect a crypto-miner running on a VM" or "identify a lateral-movement pattern in API logs," the answer is Event Threat Detection (ETD) or Container Threat Detection (CTD) — not SHA. SHA would only fire if the underlying misconfiguration (e.g., public bucket that the miner is reading from) were itself a policy violation. Read the stem carefully: misconfiguration = SHA; active/runtime threat = ETD/CTD.

Rapid Vulnerability Detection (RVD)

Rapid Vulnerability Detection is a SCC Premium feature that performs active network scanning — it sends test probes to GCP-hosted endpoints to find exploitable vulnerabilities without requiring agent installation. RVD currently targets Compute Engine and GKE workloads and detects things like exposed admin UIs (unauthenticated Jupyter Notebooks, Hadoop dashboards), weak credentials on databases, and unpatched software reachable from the internet. Key distinctions for the exam:

  • RVD vs WSS: RVD scans network services at the OS/application layer for exploitable conditions; WSS scans web application logic for OWASP-class bugs (XSS, SQLi). They complement each other.
  • RVD vs SHA: SHA is configuration-based (no network traffic sent); RVD actively probes. SHA runs on any SCC tier; RVD is Premium-only.
  • RVD vs Vulnerability Scanning in Artifact Registry: Artifact Registry scanning finds CVEs in container image layers; RVD finds CVEs in running workloads reachable on the network. Different surface, same goal.

RVD findings surface in SCC under the VULNERABILITY category with finding class VULNERABILITY and source RAPID_VULNERABILITY_DETECTION. Remediation guidance is included in each finding.

💡 SHA = Static Health; ETD = Event-based Threat Detection

SHA is a static health check — it looks at how your resources are configured right now. Think of it as an MOT inspection: it tells you the brakes are worn, not that you just had an accident.
ETD is event-based — it watches log streams and API calls in real time for signs of compromise. Think of it as a dashcam that flags dangerous driving as it happens.
RVD is active probing — it actually knocks on the door to see if it opens. Think of it as a penetration tester on a schedule: it doesn't wait for misconfiguration metadata or log signals, it tries the attack itself (safely).
Same car, three very different instruments. On the exam, the word "detect" in the stem is not enough — look for whether the scenario is about configuration posture (SHA), runtime log signals (ETD/CTD), or actively exploitable network exposure (RVD/WSS).

Question 1: A compliance team needs a tamper-resistant record of every read against objects in a set of regulated Cloud Storage buckets, retained for seven years in an external SIEM. Currently nothing is recorded. What is the correct first step before any export will help?

  • A) Create an aggregated log sink to the SIEM and assume the read events will start flowing.
  • B) Enable Data Access audit logs for Cloud Storage in the IAM audit configuration.
  • C) Turn on VPC Flow Logs on the subnets that host the storage clients.
  • D) Add a Security Health Analytics custom module that watches the buckets for reads.

Correct Answer: B

Why B is correct: Object reads are Data Access events, which are off by default; until you enable Data Access audit logs for Cloud Storage there is nothing for any sink to export.

Why the others are wrong:
A: A sink only routes logs that already exist — with Data Access logging off there's nothing to route.
C: Flow Logs record network connection metadata, not who read which storage object.
D: Security Health Analytics reports misconfigurations, not per-object read events.

Question 2: Event Threat Detection raises a finding that a Compute Engine instance is communicating with a known crypto-mining command-and-control IP. You must contain the host immediately while preserving evidence for later analysis. What is the best response?

  • A) Delete the instance right away so the mining process can no longer run anywhere.
  • B) Remove the service account from the instance and wait to see if the traffic stops on its own.
  • C) Apply a deny-all firewall rule via a network tag to isolate the VM, then snapshot its disks.
  • D) Lower the instance's machine type to starve the miner of CPU while you investigate.

Correct Answer: C

Why C is correct: Isolating the VM with a deny-all firewall tag cuts the C2 channel immediately, and snapshotting the disks preserves forensic evidence before any further change.

Why the others are wrong:
A: Deleting the instance destroys exactly the evidence you're told to preserve.
B: Removing the service account doesn't stop outbound network traffic, and "wait and see" isn't containment.
D: Resizing slows the miner but leaves the C2 connection open, so the host isn't contained.

Question 3: Security wants signature-based intrusion detection on east-west traffic between application tiers in a VPC, with full-payload inspection and managed threat signatures, without writing detection rules themselves. Which combination fits?

  • A) Packet Mirroring feeding Cloud IDS, which applies Google-managed threat signatures.
  • B) VPC Flow Logs exported to BigQuery with scheduled queries flagging suspicious patterns.
  • C) Cloud Armor rules inspecting the traffic between the application tiers.
  • D) A custom Security Health Analytics module scanning the tier configurations for drift.

Correct Answer: A

Why A is correct: Cloud IDS consumes mirrored packets and applies Google-managed (Palo Alto) threat signatures for full-payload, signature-based detection with no rule authoring required.

Why the others are wrong:
B: Flow Logs carry only connection metadata, not payloads, and require you to write the detection queries yourself.
C: Cloud Armor protects inbound HTTP(S) at the load balancer edge, not east-west VPC traffic.
D: Security Health Analytics checks configuration posture, not live payload-level intrusions.
