Section 3~17.5%

Designing for Security & Compliance

Apply least-privilege IAM over the resource hierarchy, choose the right encryption model, and map regulations to Google Cloud controls.

Resource hierarchy & guardrails

Organization → Folders → Projects → Resources. IAM and organization policies inherit downward, so set guardrails high: restrict resource locations for residency, disable service-account key creation, enforce OS Login. Apply separation of duties (e.g., split key admin from key user).

Encryption: choosing the key model

  • Google-managed — default, zero effort.
  • CMEK (Cloud KMS) — you control rotation/revocation; can import key material.
  • Cloud EKM — keys remain in an external/on-prem HSM, never persist in Google Cloud.
  • Confidential Computing — encrypts data in use (in memory).
⚠️ Exam Trap: "Keys must never leave our HSM" = Cloud EKM. "We generate keys then import them" = Cloud KMS import. Don't confuse the two.

Boundaries & secure access

VPC Service Controls build a perimeter that stops data exfiltration from serverless APIs (BigQuery, GCS, Pub/Sub) even with valid credentials. IAP / Chrome Enterprise Premium give context-aware, identity-based access with no VPN. Workload Identity Federation lets external/CI workloads call Google APIs without long-lived keys. Compliance: know the shared-responsibility model, data-residency via org policy, and Assured Workloads for regulated estates.

⚠️ Exam Trap: To stop exfiltration from a serverless product, reach for VPC Service Controls — not firewall rules (which only govern VM traffic).

IAM design & least privilege

Grant roles to groups, not individuals, and prefer predefined roles over broad basic roles (Owner/Editor/Viewer); craft custom roles only when predefined ones are too broad. Give workloads their own service accounts with minimal roles, use service-account impersonation instead of keys, and add IAM Conditions for time- or resource-bound access. Use the IAM Recommender to trim unused permissions.

⚠️ Exam Trap: Assigning Editor/Owner "to get it working" is always wrong — the keyed answer grants the narrowest predefined role that satisfies the task.
GRoup Groups get roles, Roles stay predefined-and-minimal, accounts impersonate (no keys).

🧠 Warm-Up Quiz

Q1. Prevent data in a BigQuery dataset from being copied to projects outside an approved boundary, even by authorised users. What control?

Reveal answer

VPC Service Controls perimeter.

It blocks data movement from in-scope managed services across the perimeter regardless of IAM.

Q2. Regulator requires encryption keys to be generated and held only in your on-prem HSM. Which option?

Reveal answer

Cloud EKM.

EKM uses externally held keys that never persist inside Google Cloud.

Q3. You must enforce data residency so resources can only be created in EU regions, org-wide. What do you use?

Reveal answer

An organization policy restricting resource locations.

Org policies inherit down the hierarchy and constrain where resources may be created.

Q4. A team asks for Editor on the prod project "to deploy". What's the right response?

Reveal answer

Grant the narrowest predefined deploy role (e.g. Cloud Run Admin) to their group.

Least privilege: Editor is far too broad; scope to exactly what deployment needs.

📝 Exam-Style Practice

Q1. Analysts hold valid BigQuery access, but a new policy says dataset contents must never be exfiltrated to projects outside an approved boundary — even by those authorised users. What enforces this?

  • A) Tighten IAM so analysts have only the Viewer role.
  • B) A VPC Service Controls perimeter around the projects holding the data.
  • C) VPC firewall rules blocking egress from the dataset.
  • D) CMEK encryption on the BigQuery dataset.
Reveal answer & explanation

Correct: B. VPC Service Controls draws a perimeter around managed services so data can't move across it regardless of valid IAM — the only control that stops authorised-user exfiltration.

A: Viewer still lets data be read and copied out. C: Firewall rules govern VM network traffic, not BigQuery's managed API. D: CMEK controls keys, not where data can be copied.

Q2. A regulator mandates that encryption keys be generated and stored only in the customer's on-premises HSM and never persist inside Google Cloud, while Google services still encrypt the data. Which model meets this?

  • A) Google-managed default encryption keys.
  • B) CMEK with key material imported into Cloud KMS.
  • C) Cloud External Key Manager (Cloud EKM).
  • D) Confidential Computing for encryption in use.
Reveal answer & explanation

Correct: C. Cloud EKM keeps the key in the external/on-prem manager; Google uses it for crypto operations but the key never persists in Google Cloud.

A: Default keys are Google-held. B: Imported CMEK material does persist in Cloud KMS, violating the rule. D: Confidential Computing protects data in memory, not key custody.

Q3. A developer requests the Editor role on the production project so their service can write to one Cloud Storage bucket. What should the architect grant?

  • A) Editor on the project, since it's simplest and unblocks them.
  • B) A dedicated service account with Storage Object Admin scoped to that bucket.
  • C) Owner on the project so they can self-serve future needs.
  • D) A long-lived service-account key emailed to the developer.
Reveal answer & explanation

Correct: B. Least privilege: a workload service account with the narrowest predefined role on just the target bucket meets the need without over-granting.

A/C: Editor/Owner are far broader than required. D: Distributing long-lived keys is an anti-pattern; use the attached service account / impersonation.
