Section 5~18%

Maintaining & Automating Workloads

Optimise cost, automate repeatably, and design for failure.

Cost & capacity

Choose BigQuery Editions / reservations for predictable analytics spend and decide interactive vs batch jobs. For Dataproc, decide persistent vs job-based (ephemeral) clusters — ephemeral per-job clusters cut idle cost.

Automation & resilience

Build repeatable Cloud Composer DAGs and schedule jobs idempotently. Observe pipelines with Cloud Monitoring/Logging and the BigQuery admin panel; troubleshoot errors, quotas, and billing. Design for fault tolerance: multi-region/zone jobs, restart handling, and replication/failover.

Orchestration choice & idempotency

Pick the orchestrator by need: Cloud Composer (managed Airflow) for complex, dependency-rich DAGs across many systems; Workflows for lightweight serverless service-to-service sequencing; Cloud Scheduler for simple cron triggers; and Dataform for SQL-based BigQuery transformation pipelines. Make every step idempotent (e.g. WRITE_TRUNCATE partitions, MERGE upserts) so a retried run can't double-write.

⚠️ Exam Trap: Heavy cross-system DAG with dependencies and retries = Cloud Composer. A simple "run this at 2am" trigger = Cloud Scheduler, not a full Airflow environment.

🧠 Warm-Up Quiz

Q1. Cut idle cost for sporadic Spark jobs on Dataproc. What do you do?

Reveal answer

Use job-based (ephemeral) clusters that spin up per job.

Avoids paying for a persistent idle cluster.

Q2. Author repeatable, scheduled pipeline orchestration on Google Cloud. Which service?

Reveal answer

Cloud Composer (Airflow DAGs).

Managed Airflow for repeatable orchestration.

Q3. You only need to trigger one job every night at 02:00 with no dependencies. Composer or Scheduler?

Reveal answer

Cloud Scheduler.

A simple cron trigger doesn't justify a full managed Airflow environment.

Q4. Why design a daily load to overwrite the day's partition rather than append?

Reveal answer

Idempotency — a retried run won't duplicate rows.

Truncating/replacing the target partition makes re-runs safe after failures.

📝 Exam-Style Practice

Q1. A team runs Spark jobs only a few times a day but pays for a Dataproc cluster around the clock. How do they cut cost without losing the ability to run jobs on demand?

  • A) Keep the cluster running but on smaller machines.
  • B) Use ephemeral, job-scoped Dataproc clusters that spin up per job and tear down after.
  • C) Migrate the Spark jobs to BigQuery SQL.
  • D) Switch the cluster to preemptible nodes only and keep it always on.
Reveal answer & explanation

Correct: B. Job-scoped ephemeral clusters exist only while a job runs, eliminating idle cost while preserving on-demand Spark.

A: A smaller always-on cluster still pays for idle time. C: Rewriting to SQL is a large change, not a cost toggle. D: An always-on preemptible cluster still incurs idle cost and risks preemption mid-job.

Q2. A nightly pipeline coordinates ingestion, transformation, and load across several services with inter-step dependencies and automatic retries. Which orchestrator best fits?

  • A) Cloud Scheduler firing each step at fixed times.
  • B) Cloud Composer (Airflow) DAGs modelling the dependencies and retries.
  • C) A cron job on a Compute Engine VM running a shell script.
  • D) Pub/Sub messages chained between Cloud Functions.
Reveal answer & explanation

Correct: B. Composer/Airflow models complex DAGs with dependencies, retries, and monitoring — exactly this multi-step, cross-service workflow.

A: Fixed-time Scheduler triggers don't express dependencies. C: A VM cron script is brittle and unmonitored. D: Hand-chained functions reinvent orchestration without DAG visibility or retry control.

Q3. A daily BigQuery load job occasionally retries after a transient failure, and you must guarantee the target table never ends up with duplicate rows. What design makes retries safe?

  • A) Append each run's rows and de-duplicate manually later.
  • B) Write each day's data to its date partition with WRITE_TRUNCATE (or MERGE upserts).
  • C) Disable retries on the load job entirely.
  • D) Add a unique timestamp column to every row.
Reveal answer & explanation

Correct: B. Truncating/replacing the day's partition (or using MERGE) makes the load idempotent, so a retried run produces the same result with no duplicates.

A: Append-then-clean leaves a window of duplicates and manual toil. C: Disabling retries sacrifices reliability. D: A timestamp column doesn't prevent duplicate rows from a re-run.
