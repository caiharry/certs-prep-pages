Section 1.1

Capacity & usage planning

Size compute and storage from real workload metrics plus growth, and trade cost against performance deliberately — don't eyeball a machine type.

WeightSection 1: ~32%
Solution sizing Machine & storage types Cost vs performance Recommenders

Official guide bullets

  • Perform solution sizing based on current environment workload metrics and future requirements.
  • Evaluate performance and cost tradeoffs of different database configurations (e.g., machine types, storage types).
  • Size database compute and storage based on performance requirements.

What to know

  • Size from evidence: gather peak/avg CPU, memory, IOPS, throughput, connections, and storage growth over a representative window (Cloud Monitoring), then add headroom for growth — not a single point-in-time snapshot.
  • Storage type: SSD for low-latency OLTP; HDD only for large, throughput-oriented, cost-sensitive data. IOPS/throughput scale with disk size on Cloud SQL, so undersized disks throttle.
  • Compute: more vCPU also raises the network throughput and connection ceilings. Cloud SQL Enterprise Plus edition adds a data cache and higher limits for performance-sensitive workloads.
  • Right-sizing tools: the Cloud SQL over-provisioned instance recommender flags instances that are too large for their actual workload — the deliberate way to resize, vs guessing from raw dashboards.
  • Spanner / Bigtable: scale by nodes/processing units; provision for the busiest workload and keep CPU below the recommended utilisation ceiling (e.g. Spanner ~65% regional) to preserve headroom.
Exam discriminator: "plan the right instance size after months of real use" → lean on the over-provisioned recommender, which analyses sustained utilisation. Raw monitoring dashboards or logs show a momentary window, not a sizing recommendation; query tuning tools tell you about queries, not machine size.

Section 1.2

High availability & disaster recovery

Match the deployment footprint (zonal → regional HA → multi-region) to the availability and RPO/RTO the business actually needs, and control maintenance disruption.

WeightSection 1: ~32%
Zonal / regional / multi-region Failover Maintenance windows Notifications

Official guide bullets

  • Evaluate tradeoffs between multi-regional, regional, and zonal database deployment strategies.
  • Define maintenance windows and notifications based on application availability requirements.

What to know

  • Cloud SQL HA = a synchronously-replicated standby in another zone within one region; automatic failover survives a zone outage. It is not a read scaler and does not protect against a region loss.
  • Cross-region read replica adds region-failure protection and can be promoted for DR; promotion is manual, so expect some RPO/RTO.
  • Spanner: multi-region configs give 99.999% and survive a region; regional gives 99.99%. AlloyDB/Cloud SQL regional HA gives 99.99%.
  • Maintenance: set a window (preferred low-traffic time), an order (earlier/later relative to the fleet), opt into notifications, and use deny / maintenance windows to block changes during peak periods.
  • To minimise maintenance impact: use a connection pooler (app-to-pooler connections survive) and exponential backoff retries in the app — removing the window doesn't stop maintenance.
Availability ladder — pick the rung the SLA needs
Zonal single instance cheapest, no failover← dev / non-critical
Regional HA standby in 2nd zone, auto-failover, 99.99%← survives a zone
Cross-region replica / multi-region promote for DR; Spanner multi-region 99.999%← survives a region
Exam discriminator: separate HA (zone-level, automatic, same region) from DR (region-level, often a promotable cross-region replica) from read scaling (replicas for read load). Self-managing two instances with dual writes is never the "reliable, Google-recommended" answer — use built-in HA/replication.

Section 1.3

How applications connect

Private by default, encrypted in transit, least-privilege auth, pooled connections, and audit logging on — the secure connectivity baseline for a managed database.

WeightSection 1: ~32%
Private IP / PSA / PSC CMEK Auth Proxy & IAM auth Connection pooling Audit logging

Official guide bullets

  • Configure networking, key management, encryption, and security.
  • Justify the use of session pooler services.
  • Assess auditing policies for managed services.

What to know

  • Private connectivity: keep the DB off the public internet. Use Private Services Access (PSA) (VPC peering range for Google services) or Private Service Connect (PSC) endpoints.
  • Hybrid networking routing traps (On-Premises):
    • PSA Peering Trap: VPC Peering is non-transitive. On-prem VPN/Interconnect traffic cannot automatically reach a PSA-provisioned database. The Fix: Update the peering connection config to "Export Custom Routes" (and import on Google's side), and configure Cloud Router to add a Custom Route Advertisement for the private allocated database CIDR range via BGP.
    • PSC Routing Advantage: PSC places a local endpoint Private IP directly inside your consumer VPC subnet. It is natively routable over VPN or Interconnect. To propagate, simply advertise its subnet.
  • Global Dynamic Routing: Must be set to Global (not Regional) to allow resources in Region A (e.g., VMs or Cloud Run VPC connectors) to route traffic to a private database in Region B.
  • Cloud SQL Auth Proxy / connectors: give encrypted, IAM-authorised connections without managing SSL certs or allow-lists. Combine with IAM database authentication so users/SAs log in with IAM identities, not stored DB passwords. Supports SOCKS5 proxy chaining (ALL_PROXY=socks5://...) to bypass strict boundary network blocks.
  • Encryption: data is encrypted at rest by default; use CMEK (Cloud KMS) when you must own the key lifecycle/rotation.
  • Session poolers: Managed Connection Pooling (transaction mode by default, session mode available) reuses server connections, absorbs connection spikes, stays under connection limits, and keeps app↔pooler connections alive across backend blips. Managed Connection Pooling
  • Auditing: enable Cloud Audit Logs — Admin Activity is always on; turn on Data Access logs and engine-level audit (e.g. pgAudit, SQL Server audit) where the policy requires query-level trails.

Exam discriminator: "reach a managed DB privately, no public IP" → Private Services Access / private IP (or PSC), not a firewall rule or an authorized-networks CIDR (those still assume a public IP path). "Survive dropped connections / spikes" → connection pooling + backoff.

Section 1.4

Evaluate the right database

Match data model, scale, consistency and operational ownership to the workload. This is the heart of the exam — the full picker lives on the "Which database?" synthesis slide.

WeightSection 1: ~32%
Managed vs unmanaged SQL vs NoSQL Cost analysis Compliance / org policy Multi-tech & federation GenAI / vector

Official guide bullets

  • Differentiate managed vs unmanaged services (self-managed, bare metal, Google-managed, Google Cloud-native and partner offerings).
  • Distinguish SQL vs NoSQL requirements (structured, semi-structured, unstructured, vector).
  • Analyze the cost of running database solutions (comparative analysis).
  • Assess application and database dependencies.
  • Identify solutions for regulatory and compliance requirements.
  • Understand implications of organizational policies on database strategy.
  • Consider solutions spanning multiple technologies (federation, exports, hybrid).
  • Leverage database technologies for generative AI and LLM use cases.

What to know

  • Managed vs unmanaged: prefer Google-managed (Cloud SQL, AlloyDB, Spanner, Bigtable, Firestore, Memorystore) to offload patching/backups/HA. Bare Metal Solution runs Oracle near Google Cloud; self-managed-on-GCE only when you need full engine control.
  • Relational SQL: Cloud SQL (regional OLTP), AlloyDB (Postgres, HTAP + AI), Spanner (global, horizontal scale, strong consistency).
  • Non-relational & Firestore Modes:
    • Firestore mode choice is irreversible at creation. Fixing requires new DB/project and data migration (Dataflow / GCS export/import).
    • Native Mode (N.O.R.S.): Web/Mobile client direct access, Offline client cache, Real-time listeners (onSnapshot), and client Security rules. Document model (Collections/Documents). Max ~1 write/sec per document.
    • Datastore Mode (D.S.H.): Legacy Datastore API, Server-side backend access only (no client SDKs, real-time, or offline caching), High-volume write throughput (millions/sec). Entity data model (Kinds/Entities).
  • Compliance / org policy: data-residency needs → pick regional configs and Resource Location Restriction org policy; CMEK/EKM for key control.
  • Multi-tech: BigQuery federated queries read Cloud SQL/Spanner live; exports and hybrid deployments bridge engines.
  • GenAI / vector: store embeddings + similarity search in AlloyDB / Cloud SQL (pgvector, ScaNN), Spanner, BigQuery, or Firestore vector search — keep vectors next to operational data for RAG.
Exam signal: flexible schema + real-time push to clients → Firestore Native mode (Datastore mode lacks real-time). Global, strongly-consistent, horizontally-scalable relational → Spanner. Massive low-latency key/time-series → Bigtable. Existing Oracle that must stay Oracle → Bare Metal Solution.

  • Vector databases on GCP
  • BigQuery federated queries
  • Bare Metal Solution

Section 2.1

Connectivity & access management

Two layers of access: IAM governs who can administer/connect to the instance; database users/roles govern what they can do once inside. Keep both least-privilege.

WeightSection 2: ~25%
IAM roles IAM DB auth DB users & roles Least privilege

Official guide bullets

  • Determine IAM and policies for database connectivity and access control.
  • Manage database users including authentication and access.

What to know

  • Two access planes: IAM (cloudsql.client, cloudsql.admin, cloudsql.viewer, Spanner/Bigtable/Firestore equivalents) controls the control plane and connection; engine-level GRANTs control data access.
  • IAM database authentication: log in as an IAM user or service account — no long-lived DB passwords to rotate or leak. Prefer SAs for applications.
  • Least privilege: grant the narrowest role at the lowest scope (instance/database), use groups not individuals, and split duties (admins vs read-only analysts).
  • Privileged Access Manager (PAM): grant just-in-time, time-bound, approval-gated elevation for DBA maintenance instead of standing admin rights — with an audit trail. JIT access
Exam discriminator: "app on Compute Engine needs to connect" → a service account with the minimal IAM DB role + IAM auth, not a shared human user with a stored password. Need temporary elevated DBA access → PAM entitlement, not a permanent owner grant.

Section 2.2

Monitoring & troubleshooting

Find slow queries and missing indexes, watch the vitals (CPU/RAM/IO/storage/connections), and alert proactively on errors — before users open a ticket.

WeightSection 2: ~25%
Query Insights Slow query log Locking / contention Vitals & quotas Log-based metrics & alerts

Official guide bullets

  • Assess slow running queries, database locking — identify missing indexes.
  • Monitor and investigate vitals — RAM, CPU, storage, I/O, and audit logging.
  • Monitor and update quotas.
  • Investigate database resource contention.
  • Set up alerts for errors and performance metrics.

What to know

  • Query Insights (Cloud SQL, AlloyDB): visualises top queries, latency, wait events, and tags by tag/end-user — the first stop for "queries are slow" on a managed instance.
  • Slow query log: enable engine flags (e.g. slow_query_log, log_min_duration_statement) and read entries in Logs Explorer — the quick way to surface slow statements, especially on self-managed engines.
  • Missing indexes / locking: read EXPLAIN plans; watch for full scans, deadlocks, lock waits; add/rebuild indexes.
  • Vitals & quotas: Cloud Monitoring tracks CPU, memory, disk utilisation/IOPS, active connections; raise quotas before they throttle.
  • Proactive alerting: turn a recurring log signal (e.g. deadlock detected) into a log-based metric + alerting policy for near-real-time notification — don't rely on people watching dashboards or hourly batch scans.
Exam discriminator: "be proactive about a recurring error" → log-based metric + alert fires the moment it happens; daily BigQuery exports or hourly polling are too slow, and staring at a dashboard isn't a control. "Slow queries on a managed instance" → Query Insights; on a self-managed VM → slow query log.

  • Log-based metrics

Section 2.3

Backup & recovery

Design to explicit RPO/RTO. Automated backups + binary logging enable point-in-time recovery; exports cover portability and long-term retention.

WeightSection 2: ~25%
Automated backups PITR Export / import RTO / RPO Retention

Official guide bullets

  • Recommend backup and recovery options (automatic scheduled backups).
  • Configure export and import data for databases.
  • Design for RTO, RPO, and PITR.
  • Manage data retention.

What to know

  • RPO = max acceptable data loss (how far back); RTO = max acceptable downtime (how long to restore).
  • PITR = automated backups plus write-ahead/binary logs, letting you restore to a specific timestamp (near-zero RPO). Enable it for business-critical data — daily backups alone can lose up to a day.
  • Export / import: SQL dump or CSV to Cloud Storage for portability, migrations, and long-term archive. Use serverless export to offload the export to a temporary instance so production performance is unaffected.
  • Retention: configure backup count/retention windows to meet compliance; archive exports to Cloud Storage with lifecycle/retention as needed.
Recovery options vs data loss (RPO) — shorter is better
Daily backup onlyup to ~24h loss
Backups + frequent exportshours
PITR (backups + WAL)seconds–minutes
Large RPO Medium RPO Near-zero RPO
Exam discriminator: "recover to just before an error with minimal data loss" → PITR, not automated backups (coarse, daily) and not failover replication (HA, not a backup). "Export a busy production DB without hurting performance" → serverless export, not Datastream (that's CDC) or DMS (that's migration).

Section 2.4

Optimize cost & performance

Scale up vs out, add read replicas for read load, tune queries, and continuously trim waste with recommenders, schedules, and the right discount model.

WeightSection 2: ~25%
Scale up vs out Read replicas Replication strategy Query optimization CUDs / recommenders

Official guide bullets

  • Assess scaling up and scaling out options.
  • Scale database instances based on current and upcoming workload.
  • Define replication strategies.
  • Continuously assess and optimize the cost of running a database solution.
  • Optimize queries for cost and performance.

What to know

  • Scale up (bigger machine) for write-bound/single-node engines like Cloud SQL; scale out (add nodes/replicas) for read load or for horizontally-scalable engines (Spanner, Bigtable).
  • Read replicas offload reads and analytics from the primary; route users to the nearest replica for low latency. Cross-region replicas double as DR.
  • Query & Schema Optimization (Spanner & Bigtable):
    • Key Randomization: Eliminate sequential/monotonic keys to avoid hotspotting using UUIDv4, prefix sharding, or bit-reversal.
    • Index Hotspotting: Monotonic/low-cardinality index columns cause split hotspotting. Fix with Null-Filtered Indexes (CREATE NULL_FILTERED INDEX) to ignore nulls.
    • Table Interleaving: Co-locates parent and child rows physically for zero-network-hop SQL joins. Specs: Child PK must start with Parent PK. Supports up to 7 levels of nested interleaving. Configurable cascading deletes (ON DELETE CASCADE / ON DELETE PREVENT).
    • Secondary Index Write Overhead: treated as a separate table under the hood (MVCC); each write requires a distributed transaction across splits, adding latency. Avoid back-joins (network hops) using the STORING clause to duplicate frequently queried columns.
  • Cost levers: over-provisioned recommender, committed use discounts (CUDs) for steady baselines, stop/schedule non-prod instances, right-size storage, and watch egress.
Exam discriminator: "let analysts query all the data without slowing production, minimal effort" → read replicas, not Memorystore (partial/hot data) or per-analyst views (high maintenance). "Reduce non-prod cost with minimal effort" → schedule start/stop (Cloud Scheduler + a small function), not a self-managed cron VM or adding HA.

Section 2.5

Automate common tasks

Scheduled exports, index maintenance, controlled upgrades, and SLA/SLO monitoring — automated, not manual and not ad-hoc cron on a VM you have to babysit.

WeightSection 2: ~25%
Scheduled exports Index rebuilds Managed upgrades SLA / SLO monitoring

Official guide bullets

  • Perform database maintenance (e.g., rebuilding indexes, data exports).
  • Schedule database exports.
  • Manage upgrades for Google Cloud-managed databases.
  • Monitor database SLA/SLOs.

What to know

  • Scheduled exports: Cloud Scheduler → Cloud Functions/Run (or workflows) calling the export API on a cron — the low-ops pattern vs a self-managed VM cron.
  • Maintenance: rebuild/REINDEX to fix bloat and fragmentation; automate routine jobs.
  • Upgrades: managed engines support in-place major-version upgrades (e.g. Cloud SQL) and maintenance windows; test on a clone/replica first. AlloyDB/Spanner handle minor maintenance transparently.
  • SLA/SLO monitoring: define availability/latency SLOs in Cloud Monitoring and alert on burn; the provider SLA is the external contract, your SLO is the stricter internal target.
Exam signal: any "schedule a recurring DB task with minimal effort/cost" → Cloud Scheduler + serverless function, never a long-running Compute Engine cron host (more cost, more to manage).

Section 3.1

Migration & replication

Choose the migration tool and strategy by source, downtime tolerance, and whether the engine changes. Continuous CDC enables near-zero-downtime cutovers with a fallback path.

WeightSection 3: ~23%
DMS Datastream (CDC) Zero / near-zero downtime DDL/DML conversion Fallback / reverse replication

Official guide bullets

  • Develop and execute migration strategies including zero/near-zero downtime, extended outage, and fallback.
  • Reverse replication from Google Cloud to source.
  • Plan and perform migration, including fallback plans and DDL/DML conversion.
  • Determine the correct migration tools for a given scenario (including databases outside Google Cloud).

What to know

  • Database Migration Service (DMS): managed migrations with a continuous (CDC) phase → minimal-downtime cutover. Homogeneous (MySQL→Cloud SQL, PostgreSQL→Cloud SQL/AlloyDB) and heterogeneous (Oracle/SQL Server→PostgreSQL) with conversion.
  • Heterogeneous = schema/code conversion: convert DDL and rewrite DML/PL-SQL; DMS conversion workspaces (and the Database Migration Assessment) help.
  • Datastream: serverless change data capture — stream changes to BigQuery/Cloud Storage for analytics or replication, not a full instance migrator.
  • Strategy by downtime: extended outage = simple dump/restore; near-zero = initial load + continuous CDC, then cut over at a quiet window.
  • Fallback / reverse replication: after cutover, replicate back to the source so you can roll back if the new system misbehaves — a safety net for risky migrations.

Section 4.1

Deploy scalable & HA databases

Provision HA the managed way, scale reads with replicas, automate provisioning as code, and actually test failover — an untested DR plan is a hope, not a plan.

WeightSection 4: ~20%
Provision HA Test failover / DR Multi-region replication Read replicas IaC automation

Official guide bullets

  • Provision highly available database solutions.
  • Test high availability and disaster recovery strategies.
  • Set up multi-regional replication for databases.
  • Deploy and scale read replicas.
  • Automate database instance provisioning.
  • Configure monitoring for highly available databases.

What to know

  • Provision HA: enable Cloud SQL/AlloyDB regional HA (automatic standby + failover); choose Spanner multi-region for the highest tier.
  • Test it: trigger a manual failover and a replica promotion in a game-day to validate RTO/RPO and app reconnection behaviour — don't wait for a real outage.
  • Read replicas: add in-region for read scale, cross-region for latency + DR; promote a cross-region replica if the primary region fails.
  • Automate provisioning: define instances with Terraform (or Deployment scripts) for repeatable, identical environments — not console clicks.
  • Monitor HA: alert on replication lag, failover events, standby health, and connection errors.
Exam discriminator: "ensure HA for a single-zone instance" → turn on the built-in HA configuration (auto-failover to a standby), not Cloud Scheduler restoring from backup (slow) and not two hand-synced instances (error-prone). HA protects a zone; for a region you need cross-region replication / multi-region.
