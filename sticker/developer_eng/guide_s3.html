Section 3~24%

Configuring Apps for Deployment

Deploy to Cloud Run and GKE with the right triggers, traffic control, and health checks.

Cloud Run deployment & traffic

Deploy from source (buildpacks) or a container. Each deploy creates an immutable revision; split traffic across revisions for canary releases (e.g., deploy with --no-traffic, then route 10%). Trigger services with Eventarc or Pub/Sub push, and expose/secure APIs with Apigee.

⚠️ Exam Trap: Cloud Run gives native percentage traffic splitting across revisions — don't reach for DNS round-robin or a second service.

GKE health & autoscaling

Use liveness probes to restart hung containers, readiness probes to gate traffic until a pod is ready, and startup probes for slow-booting apps so liveness doesn't kill them prematurely. The Horizontal Pod Autoscaler scales replicas on CPU or custom metrics; set resource requests/limits so the scheduler and HPA behave.

⚠️ Exam Trap: A failing readiness probe removes a pod from the Service (no restart); only a failing liveness probe restarts the container.
Probes Readiness = Receive traffic; Liveness = restart if Lifeless; Startup = wait while booting.

Configuration & secrets

Externalise config so the same image runs in every environment (12-factor). Pass non-secret settings as environment variables; keep secrets in Secret Manager and mount/reference them at runtime — never bake credentials into the image or commit them. On GKE use ConfigMaps for config and Secret Manager (via the CSI driver / Workload Identity) for secrets.

⚠️ Exam Trap: Hardcoding API keys or DB passwords in the container image (or env vars committed to source) is always wrong — use Secret Manager.

Deployment strategies

Roll out safely: a rolling update replaces pods gradually (GKE default), a canary sends a small traffic slice to the new version first, and blue-green keeps two full environments and flips traffic for instant rollback. On Cloud Run, revision traffic splitting gives canary and blue-green for free.

SAFE Small slice first (canary), keep the old one warm (blue-green), and you can always roll back fast.

🧠 Warm-Up Quiz

Q1. Send 10% of traffic to a new version, then ramp up with instant rollback, on Cloud Run. How?

Reveal answer

Deploy a new revision with --no-traffic and split traffic 10/90.

Revision-based traffic splitting is native to Cloud Run.

Q2. A pod process deadlocks but keeps running. Which probe recovers it?

Reveal answer

A liveness probe (restarts the container).

Readiness only de-registers; liveness restarts.

Q3. Where should a database password used by a Cloud Run service live?

Reveal answer

Secret Manager, referenced at runtime.

Keeps the secret out of the image and source, with rotation and access control.

Q4. An app takes 90s to initialise; liveness probes keep killing it during boot. What fixes this?

Reveal answer

Add a startup probe.

It defers liveness checks until startup completes, so slow boots aren't mistaken for failures.

📝 Exam-Style Practice

Q1. You want to release a new Cloud Run version to 5% of users, watch error rates, then ramp to 100% — with the ability to roll back instantly. What's the simplest approach?

  • A) Deploy a second Cloud Run service and switch DNS when confident.
  • B) Deploy a new revision with no traffic, then shift 5% to it and ramp using the revision traffic split.
  • C) Replace the running revision in place and monitor logs afterward.
  • D) Put both versions behind an external load balancer with weighted backends.
Reveal answer & explanation

Correct: B. Cloud Run's built-in revision traffic splitting gives canary and instant rollback by simply shifting percentages back.

A: DNS switching is coarse and slow to revert. C: In-place replacement has full blast radius. D: An external LB is unnecessary extra plumbing when Cloud Run does this natively.

Q2. A GKE pod's process occasionally deadlocks but the container keeps running, so it stops serving without being replaced. Which probe configuration addresses this?

  • A) A readiness probe only, so the pod is removed from the Service.
  • B) A liveness probe that fails and triggers a container restart.
  • C) Increasing CPU requests so the process never deadlocks.
  • D) A startup probe that runs once at boot.
Reveal answer & explanation

Correct: B. A liveness probe detects the hung process and restarts the container, restoring service automatically.

A: Readiness only de-registers the pod; it never restarts it. C: More CPU doesn't fix a logical deadlock. D: Startup probes only cover the boot window.

Q3. The same container image must run in dev, staging, and prod with different database credentials, and security forbids credentials in the image or source. How should the app get its secrets?

  • A) Bake per-environment credentials into separate images at build time.
  • B) Commit an encrypted .env file and decrypt it in the entrypoint.
  • C) Reference Secret Manager secrets at runtime, scoped per environment.
  • D) Pass the credentials as plaintext build arguments.
Reveal answer & explanation

Correct: C. Secret Manager keeps credentials out of the image and source, supports per-environment access and rotation, and is read at runtime.

A: Per-environment images break "one image everywhere" and embed secrets. B: Committing secrets, even encrypted, risks exposure and key sprawl. D: Build args end up in image history.
