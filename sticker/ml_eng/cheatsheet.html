GCP ML Engineer

High-yield revision deck ordered by the official exam guide. Vertex AI map, build-vs-buy ladder, and the skew-vs-drift trap.

🤖 The big picture: Climb the build-vs-buy ladder — pretrained ML APIs / Model Garden → BigQuery ML → AutoML → custom training — and stop at the first rung that meets the requirement and team skill. Know the Vertex AI service map end to end, and never confuse training-serving skew with drift.
Section 1 ~13%

Architecting Low-Code AI Solutions

Fastest paths to a working model.

Build-vs-buy ladder

  • Pretrained ML APIs / Model Garden: standard tasks, no training.
  • BigQuery ML: SQL users, data already in BQ.
  • AutoML: custom model, little ML code.
  • Custom training: full control, framework freedom.

GenAI building blocks

  • Model Garden: foundation + open models catalog.
  • Vertex AI Agent Builder: search, RAG, agents.
  • Document AI / Retail API: industry-specific.

⚠️ Exam Trap: If the data is already in BigQuery and the team writes SQL, BigQuery ML beats spinning up custom training.
⚠️ Exam Trap: "No labeled data / standard task (OCR, translation, sentiment)" → a pretrained ML API, not AutoML.
Section 2 ~14%

Managing Data & Models Across Teams

Data prep, notebooks, and experiment tracking.

Feature management

  • Vertex AI Feature Store: central, reusable features; online + offline serving; reduces skew.
  • Preprocess at scale with Dataflow / TFX.

Notebook backends

  • Vertex AI Workbench: managed JupyterLab, secure.
  • Colab Enterprise: collaborative, managed.
  • Dataproc notebooks: Spark kernels for big data.
⚠️ Exam Trap: Using the same Feature Store for training and serving is the canonical way to prevent training-serving skew.
Section 3 ~18%

Scaling Prototypes into ML Models

Train at scale and choose accelerators wisely.

Training options

  • Vertex AI custom training: containers, any framework.
  • Distributed training for large data; reliable pipelines.
  • Hyperparameter tuning service for sweeps.
  • Fine-tune foundation models in Vertex AI / Model Garden.

Accelerator choice

  • CPU: small/classical models.
  • GPU: most deep learning, flexible.
  • TPU: very large TensorFlow/JAX matrix workloads.
  • Reduction Server: faster multi-GPU all-reduce.
TPU TPU = Tensor-heavy, Predictable, hUge matmuls (large TF/JAX). For mixed or smaller models, reach for GPUs first.
⚠️ Exam Trap: TPUs shine for large matrix-multiply DL graphs; for custom ops or smaller models, GPUs are usually the right call.
Section 4 ~20%

Serving & Scaling Models

Inference patterns and production endpoints.

Inference patterns

  • Batch prediction: large offline scoring, latency-tolerant.
  • Online endpoint: low-latency, per-request.
  • Private endpoints for VPC-internal traffic.

Production tuning

  • Model Registry + versioning; A/B test via traffic split.
  • Autoscale serving on throughput.
  • Optimise latency/memory (quantisation, distillation).
⚠️ Exam Trap: "Score the whole table once a night" = batch prediction; "respond per user request in ms" = online endpoint.
Section 5 ~22%

Automating & Orchestrating Pipelines

The biggest slice: reproducible, automated MLOps.

Orchestration

  • Vertex AI Pipelines (KFP): serverless ML DAGs — default choice.
  • Cloud Composer (Airflow): broader data orchestration.
  • TFX components for TensorFlow pipelines.

Retraining & lineage

  • Trigger retraining on schedule or on drift/skew signals.
  • CI/CD model deploy with Cloud Build.
  • Vertex ML Metadata: artifact versions & lineage.
⚠️ Exam Trap: For an ML-specific, serverless, reproducible pipeline pick Vertex AI Pipelines; Cloud Composer is the answer only when you need general workflow orchestration across many systems.
Section 6 ~13%

Monitoring AI Solutions

Risk, Responsible AI, and the skew/drift distinction.

Skew vs drift

  • Training-serving skew: training and serving data/preprocessing differ.
  • Prediction drift: serving input distribution shifts over time.
  • Feature attribution drift: feature importance changes.

Responsible AI

  • Monitor for bias / fairness; assess readiness.
  • Explainable AI for feature attributions.
  • Protect models/data from exploitation.
⚠️ Exam Trap: Fix skew by sharing preprocessing/Feature Store between train and serve; detect drift with Vertex AI Model Monitoring over time.
