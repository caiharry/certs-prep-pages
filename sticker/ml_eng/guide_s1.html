Section 1~13%

Architecting Low-Code AI Solutions

The fastest paths to a working model: BigQuery ML, pretrained ML APIs and foundation models, and AutoML.

The build-vs-buy ladder

The single most-tested instinct in this domain: climb only as high as you need. Each rung up costs more data, more skill, and more time to maintain.

  • Pretrained ML APIs / Model Garden — standard tasks (OCR, translation, sentiment, transcription) with no training data.
  • BigQuery ML — the data already lives in BigQuery and the team writes SQL.
  • AutoML — you need a custom model but want minimal code and have some labelled data.
  • Custom training (Vertex AI) — full control over framework, architecture, and hardware.

Pick the lowest rung that satisfies the requirement and the team's skill. The exam loves to dangle a fancy option when a humbler one is correct.

⚠️ Exam Trap: "No labelled data + a standard task" points to a pretrained ML API — never AutoML or custom training, both of which need labelled examples.
LADDER Lowest rung wins. If two options both work, the cheaper / less-custom one is almost always the keyed answer.

Pretrained ML APIs at a glance

Know which API owns which task — the exam tests the boundaries, not the SDK calls:

  • Cloud Vision — labels, objects, OCR, faces, safe-search on images.
  • Video Intelligence — shot/label/object tracking and explicit-content detection in video.
  • Natural Language — entities, sentiment, syntax, classification of free text.
  • Speech-to-Text / Text-to-Speech — transcription and voice synthesis.
  • Translation — language detection and translation.
  • Document AI — structured extraction from forms, invoices, and contracts (key-value pairs, tables).
⚠️ Exam Trap: "Extract fields from scanned invoices" is Document AI, not the raw Vision OCR — Document AI understands document structure, Vision just returns text.

BigQuery ML model picker

BigQuery ML lets analysts train and serve models with CREATE MODEL SQL — no data leaves the warehouse. Match the goal to the model type:

  • Predict a number → linear regression.
  • Yes/no or a class → logistic / multiclass classification.
  • Forecast a time series → ARIMA_PLUS.
  • Recommendations → matrix factorization.
  • Maximum tabular accuracy → boosted trees (XGBoost).
  • Anomaly detection / grouping → autoencoder / k-means.
  • Reuse a TensorFlow model → imported TF model for in-warehouse inference.
BQML Data in BigQuery + a SQL-speaking team = BigQuery ML beats standing up custom training every time.

AutoML — the low-code custom rung

AutoML (now folded into Vertex AI) trains a custom model on your labelled data with almost no code, handling feature engineering, architecture search, and tuning for you. Use it when a pretrained API can't capture your domain (e.g. classifying your defect types) but you lack the team or time for custom training.

It spans tabular, image, text, and video. The trade-off versus custom training is control: you don't choose the architecture or framework, and very large or exotic models still want custom training.

⚠️ Exam Trap: AutoML still needs a labelled dataset. If the scenario says "we have no labels", AutoML is wrong — consider a pretrained API or an unsupervised approach.

Generative AI building blocks

Model Garden is the catalogue of foundation and open models (Gemini, plus third-party and open-weight models). Vertex AI Agent Builder provides managed retrieval-augmented generation (RAG), search, and agents — ground an LLM's answers in your own documents with citations and minimal infrastructure.

Choose the lightest customization that works: prompt design first, then grounding / RAG to inject fresh private knowledge, then tuning (e.g. supervised fine-tuning) only when style or format must be baked in. Full pre-training is essentially never the exam answer.

⚠️ Exam Trap: "Answer from a document corpus, with citations, kept current" = RAG via Agent Builder. Fine-tuning memorizes, doesn't cite, and goes stale the moment the docs change.
PGT Prompt → Ground (RAG) → Tune. Walk up the GenAI ladder one step at a time, just like build-vs-buy.

Responsible AI & data residency

Even low-code solutions carry obligations. Watch for scenarios that hinge on data residency (keep training/serving in a specific region), sensitive data (run Cloud DLP / Sensitive Data Protection to de-identify before training), or fairness/explainability (Vertex Explainable AI feature attributions). Pretrained APIs process data in Google's managed service — if a constraint forbids that, you may be pushed up to a model you host in-region.

⚠️ Exam Trap: A strict "PII must never leave our region / be exposed to a shared API" constraint can rule out a pretrained API even when the task is standard.

🧠 Warm-Up Quiz

Q1. SQL-fluent analysts must train a churn classifier on data already in BigQuery, fast. What do you use?

Reveal answer

BigQuery ML (logistic regression).

In-warehouse training/serving with SQL, no data movement.

Q2. Chatbot must answer from thousands of internal PDFs with citations, minimal infra. Which approach?

Reveal answer

RAG with Vertex AI Agent Builder.

Managed retrieval-augmented generation grounds answers and cites sources.

Q3. You must pull line items and totals out of scanned supplier invoices. Vision OCR or Document AI?

Reveal answer

Document AI.

It returns structured key-value pairs and tables; Vision OCR only gives you raw text you'd have to parse yourself.

Q4. A standard sentiment task, but the team has zero labelled data and no ML engineers. Which rung?

Reveal answer

Pretrained Natural Language API.

No labels and a standard task = lowest rung. AutoML and custom training both need labelled examples.

📝 Exam-Style Practice

Q1. A retailer wants automatic object and label tagging on catalogue photos. There is no ML team and no labelled training data, and the budget is tight. What is the best approach?

  • A) Train an AutoML Vision model on a labelled subset of the catalogue and deploy it to a Vertex endpoint.
  • B) Use the pretrained Cloud Vision API label and object-detection features from the application.
  • C) Fine-tune a Model Garden vision foundation model on the product images.
  • D) Build a custom CNN with Vertex AI custom training on a managed image dataset.
Reveal answer & explanation

Correct: B. Object and label detection is a standard task with no labelled data, so the pretrained Cloud Vision API is the lowest, cheapest rung that works.

A: AutoML needs labelled data and effort that a standard task doesn't justify. C: Fine-tuning needs data and skill for no benefit here. D: Custom training is the most expensive rung — massive overkill.

Q2. SQL analysts must ship a customer-churn classifier this week, on data already in BigQuery, with minimal new infrastructure. What should they use?

  • A) Export the data to Cloud Storage and train an AutoML Tabular model.
  • B) Stand up Vertex AI custom training with a TensorFlow model.
  • C) Train a logistic-regression model in place using BigQuery ML and SQL.
  • D) Call the Vertex AI Gemini API to classify each customer row.
Reveal answer & explanation

Correct: C. BigQuery ML trains and serves with SQL where the data already lives — no movement, no new infrastructure, and it fits a SQL team and a tight deadline.

A: Exporting adds data movement and setup for no gain. B: Custom training is heavy for a SQL team on a one-week timeline. D: An LLM is the wrong, costly tool for structured tabular classification.

Q3. A support assistant must answer questions from 10,000 internal PDFs, cite its sources, and stay accurate as documents are updated, with low engineering effort. What fits best?

  • A) Fine-tune a foundation model on the PDFs so it memorizes their content.
  • B) Use Vertex AI Agent Builder RAG over the indexed PDFs to ground answers and cite sources.
  • C) Paste the full set of PDFs into the prompt context on every request.
  • D) Train an AutoML text-classification model on the PDF corpus.
Reveal answer & explanation

Correct: B. RAG retrieves the relevant passages at query time, so answers stay grounded with citations and update automatically as the corpus changes — with minimal infrastructure.

A: Fine-tuning is costly, can't cite, and goes stale on every document change. C: The corpus far exceeds the context window and is expensive per call. D: Classification assigns labels; it doesn't answer questions with citations.
