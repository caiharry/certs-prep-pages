Section 4~20%

Serving & Scaling Models

Choose inference patterns, manage versions, and tune production serving.

Batch vs online inference

Batch prediction scores large datasets offline — the cost-efficient choice when latency doesn't matter (e.g. nightly scoring of a whole table). Online endpoints serve low-latency, per-request predictions for interactive apps.

⚠️ Exam Trap: "Score the whole table once nightly" = batch prediction. "Respond per request in milliseconds" = online endpoint. Don't pay for an always-on endpoint for an offline job.

Endpoint types & private serving

Online prediction can run on a shared public endpoint (simple) or a private endpoint / Private Service Connect for VPC-internal, low-latency traffic that must not traverse the internet. Choose dedicated resources when one model needs guaranteed capacity, and attach GPUs to the endpoint for heavy models.

⚠️ Exam Trap: "Predictions must stay on the private network / meet data-exfiltration controls" points to a private endpoint or PSC, not the default public endpoint.

Versioning & safe rollout

Register every model in the Model Registry (versioned, with aliases). Deploy multiple versions behind one endpoint and use a traffic split for A/B tests and canary rollouts — send 5% to the new version, watch metrics, then ramp. Roll back instantly by shifting traffic back.

CANARY Small slice first, watch the metrics, then ramp. The endpoint's traffic split is your dial.

Autoscaling & latency/cost tuning

Endpoints autoscale node count on traffic; set min/max replicas to balance cost against tail latency, and keep a warm minimum if cold starts hurt. Shrink the model itself with quantization and distillation, and batch requests where the app tolerates it, to cut cost and latency.

⚠️ Exam Trap: "Cut serving cost without retraining" leans on quantization/distillation and right-sizing replicas — not a bigger machine type.

🧠 Warm-Up Quiz

Q1. Score 500M rows once per night, latency irrelevant, cost matters. Which serving mode?

Reveal answer

Vertex AI batch prediction.

Designed for large offline scoring; cheapest for the job.

Q2. Compare two model versions on a slice of live traffic, then promote. How?

Reveal answer

Traffic split across versions on a Vertex AI endpoint.

Native A/B testing and gradual promotion.

Q3. Predictions must never leave the VPC for compliance. Which endpoint do you use?

Reveal answer

A private endpoint / Private Service Connect.

Keeps online prediction traffic on the private network instead of the public internet.

Q4. Serving cost is too high but you can't retrain. Two levers?

Reveal answer

Quantize/distill the model and right-size endpoint replicas.

Both cut compute per request and idle capacity without changing model quality much.

📝 Exam-Style Practice

Q1. A risk model scores the entire customer table once every night to populate a dashboard. There is no interactive latency requirement and cost should be minimal. How should it be served?

  • A) A dedicated online endpoint with autoscaling and a warm minimum of replicas.
  • B) A Vertex AI batch prediction job that scores the table and writes results out.
  • C) A Cloud Function calling the model per row in a nightly loop.
  • D) A private online endpoint behind Private Service Connect.
Reveal answer & explanation

Correct: B. A one-shot, latency-insensitive scoring of a whole table is the canonical batch-prediction job — cheapest and simplest.

A/D: Always-on endpoints cost money to idle for a nightly job. C: Per-row function calls are slow and expensive at table scale.

Q2. You want to release a new model version with minimal blast radius: route a small share of live traffic to it, watch error and latency metrics, then ramp up or roll back fast. What should you configure?

  • A) Deploy the new version to a separate endpoint and switch DNS when ready.
  • B) Replace the old version in place and monitor afterward.
  • C) Deploy both versions to one endpoint and use a traffic split, starting at ~5% on the new one.
  • D) Run a batch prediction with the new model and compare offline only.
Reveal answer & explanation

Correct: C. A traffic split on a single endpoint gives a true canary on live traffic with instant rollback by shifting the split back.

A: DNS switching is coarse and slow to roll back. B: In-place replacement has full blast radius. D: Offline comparison doesn't test live serving behaviour.

Q3. An online endpoint meets latency targets but costs too much, and retraining is off the table this quarter. Which combination best reduces cost while protecting latency?

  • A) Switch to batch prediction for the interactive app.
  • B) Quantize/distill the model and tune the min/max replica counts.
  • C) Move the endpoint to a much larger machine type.
  • D) Disable autoscaling and run a single fixed large replica.
Reveal answer & explanation

Correct: B. Quantization/distillation cut compute per request, and right-sizing replicas removes idle cost without hurting latency — all without retraining the original model.

A: Batch can't serve an interactive app. C: A bigger machine usually raises cost. D: A fixed large replica wastes money at low traffic and risks latency at peaks.
