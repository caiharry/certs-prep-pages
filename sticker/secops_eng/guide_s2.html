Section 2~14%

Data Management

Ingest, parse, and normalise logs into the Unified Data Model, and build entity context.

Ingestion & the UDM

All ingested logs are normalised into the Unified Data Model (UDM) — one schema that detections and searches reference. Parsers map raw logs to UDM fields; build or extend a parser when a custom source's fields don't map. Assess which logs you actually need and manage ingestion cost by prioritising high-value sources.

⚠️ Exam Trap: If custom-app fields aren't queryable in detections, create/extend a parser to normalise them into UDM — don't write rules against raw text.

Event vs entity context

Distinguish event data (activity, e.g., Cloud Audit Logs) from entity data (context about users/assets, e.g., Active Directory org context). Enrich events with entity data using aliasing fields so detections and investigations have context.

Parsers, parser extensions & the data pipeline

Most common sources have a default (Google-managed) parser that maps raw logs to UDM automatically. When fields go missing or a source is bespoke, you have three escalating moves — and the exam wants the least invasive one that works:

  • Parser extension — a small overlay that adds or remaps a handful of fields on top of the default parser without forking it. Reach for this first.
  • Custom parser — a full parser you author when there is no default (a truly bespoke app log format).
  • Data normalisation fields — choose UDM field types deliberately (noun vs metadata, principal vs target) so detections match correctly.

Validate parser changes against sample logs before promoting them, and remember mappings are forward-looking — only newly ingested logs pick up the change.

⚠️ Exam Trap: A parser edit does not retroactively reparse old data. If old logs are mis-mapped, you must reingest/backfill — editing the parser only fixes events ingested afterward.

Cost, retention & data tiers

Ingestion is priced and capacity is finite, so data management is partly an economics exercise. Triage every prospective source by detection value versus volume:

  • Prioritise high-signal logs (auth, audit, EDR, DNS) over chatty low-value noise (verbose debug/app logs).
  • Use data tiers to keep hot, frequently-searched data fast while pushing rarely-touched logs to cheaper retention.
  • Tune at the source — drop or filter fields you will never detect on rather than paying to store them.
VALUE/VOLUME Before adding a feed, ask: does its detection value justify its ingest volume? If not, filter at source.

🧠 Warm-Up Quiz

Q1. A custom app's log fields don't map to the schema so detections can't reference them. What do you do?

Reveal answer

Create or extend a parser to normalise into UDM.

Parsers map raw fields to UDM so rules/searches can use them.

Q2. What is the single normalised schema all SecOps logs are mapped into?

Reveal answer

The Unified Data Model (UDM).

Detections and searches operate over UDM fields.

Q3. A Google-managed parser maps most of a source correctly but drops two fields you need. What's the least invasive fix?

Reveal answer

A parser extension overlaying the default parser.

Extensions add/remap a few fields without forking the whole default parser.

Q4. You fixed a parser mapping, but yesterday's already-ingested logs are still wrong. Why?

Reveal answer

Parser changes are forward-looking; they don't reparse historical data.

Only events ingested after the change pick up the new mapping — old data needs reingestion.

📝 Exam-Style Practice

Q1. A new SaaS source ingests via its default parser, but the username lands in a generic text field instead of the principal, so your user-based detections never fire on it. You want the fix scoped to this one source without affecting other tenants of the same parser. What do you do?

  • A) Author a brand-new custom parser for the source and disable the Google-managed default
  • B) Rewrite each affected YARA-L rule to read the generic text field instead of the principal
  • C) Create a parser extension that maps the username into the correct UDM principal field
  • D) Add the usernames to a reference list so detections can resolve them at match time
Reveal answer & explanation

Correct: C. A parser extension overlays just the field mapping you need onto the existing default parser, fixing the principal mapping for this source without forking or replacing the managed parser.

A: A full custom parser is heavy, throws away the working default mappings, and is more than this single-field gap requires. B: Rewriting rules to chase a mis-mapped field scatters the workaround across every detection and breaks portability of shared rules. D: Reference lists hold reusable values for matching; they cannot relocate a value into the correct UDM field.

Q2. Your SecOps ingest bill is dominated by verbose application debug logs that no detection or hunt has ever referenced, while you want to onboard high-value EDR telemetry within the same budget. What is the right data-management move?

  • A) Shorten retention on every source uniformly so total stored volume drops across the board
  • B) Filter the unused debug logs at the source so they are not ingested, freeing budget for EDR
  • C) Move all logs into a cold data tier to cut cost, then onboard EDR into the same cold tier
  • D) Route the debug logs to Security Command Center instead so they no longer count against SIEM ingest
Reveal answer & explanation

Correct: B. Filtering low-value, never-referenced logs at the source removes their ingest cost directly, which is the textbook value-versus-volume trade that frees budget for high-signal EDR data.

A: Blanket retention cuts also shorten your valuable logs' searchable history and don't target the actual cost driver. C: A cold tier slows the EDR telemetry you most want fast and still pays to ingest the useless debug logs. D: SCC is a posture tool, not a dumping ground for application logs, and this misroutes data rather than reducing it.

Q3. During an investigation you must correlate a sign-in event with the user's department and manager — context that isn't in the auth log itself. Which kind of data do you bring in, and from where?

  • A) More event data, by lowering the auth source's filtering so additional fields are ingested
  • B) A retrohunt over historical auth logs to reconstruct the department from past activity
  • C) A parser extension on the auth log to synthesise department and manager fields locally
  • D) Entity context data, such as identity/org records, enriched onto events via aliasing fields
Reveal answer & explanation

Correct: D. Department and manager describe who the user is, which is entity context (from identity/org sources); SecOps joins it onto events through aliasing so investigations and detections gain that context.

A: The auth log is event data about an action; it does not contain HR org attributes no matter how much filtering you relax. B: A retrohunt finds past matching events but cannot manufacture org context that was never logged. C: A parser maps fields that exist in the raw log; it cannot invent department/manager that the source never sent.
