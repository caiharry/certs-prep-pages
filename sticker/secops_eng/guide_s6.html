Section 6~10%

Observability

Dashboards and reports for insight, plus health monitoring of the SecOps platform itself.

Dashboards & reports

Build dashboards that visualise ingestion volume, detections, alerts, open cases, and top IOCs (SecOps SOAR/SIEM, optionally Looker Studio), and generate scheduled, exportable reports for leadership. Track key security analytics — metrics, KPIs, and trends.

Health monitoring & silent sources

Create threshold alerts on key metrics and route notifications via Cloud Monitoring. Crucially, configure silent source detection to alert when an expected log feed stops sending data — a blind spot your content rules cannot catch (they simply see nothing).

⚠️ Exam Trap: A feed that quietly stops sending logs is caught by silent source detection, not by your detection rules.

Metrics that matter & audience-fit dashboards

Observability is only useful if the right person sees the right number. Match the metric to the audience:

  • SOC-operational — alert volume, MTTD/MTTR, case backlog, false-positive rate; these tune day-to-day operations.
  • Detection-coverage — rules firing, ATT&CK coverage, top noisy rules; these guide detection engineering.
  • Executive/leadership — trend lines, risk posture, SLA adherence in scheduled, exportable reports — not raw event firehoses.
  • Ingestion-health — per-source volume, latency, and parser error rates; an early-warning layer for the pipeline itself.
RIGHT EYES Operators want live ops metrics; execs want trends & SLAs. Don't hand leadership a raw alert feed or analysts a quarterly PDF.

Pipeline health beyond silent sources

A detection can only fire on data that arrives, on time, correctly parsed. So monitor the pipeline itself as a first-class concern:

  • Silent / missing sources — alert when an expected feed stops; content rules see nothing and stay quiet, so they cannot catch this.
  • Ingestion latency — data arriving hours late means detections fire late; alert on lag, not just absence.
  • Parser errors / drops — a vendor format change can silently break a parser, so events ingest but land in the wrong fields and dodge rules.
  • Routing & thresholds — push pipeline-health alerts through Cloud Monitoring to the on-call channel, separate from security alerts.
⚠️ Exam Trap: "Logs still arrive but a vendor changed their format and detections went quiet" is a parser error problem, not a silent-source one — the feed isn't silent, it's mis-parsed.

🧠 Warm-Up Quiz

Q1. A critical log source stopped 3 days ago and nobody noticed because detections produced no alerts. What prevents this?

Reveal answer

Silent source detection / ingestion-health monitoring.

Alerts when an expected feed goes quiet.

Q2. Leadership wants a single visual view of ingestion, detections, cases, and IOCs plus weekly exports. What do you build?

Reveal answer

SecOps SOAR/SIEM dashboards with scheduled reports (optionally Looker Studio).

Centralised observability with exportable reporting.

Q3. A feed still sends data, but after a vendor format change detections on it stopped firing. What's the likely cause?

Reveal answer

A parser error — events land in the wrong UDM fields.

The source isn't silent; mis-parsed fields slip past rules. Not a silent-source case.

Q4. Analysts need live alert-volume and MTTR; the CISO wants quarterly trends. How do you serve both?

Reveal answer

Audience-fit views: live operational dashboards for the SOC, scheduled trend reports for the CISO.

Match the metric and cadence to the audience rather than one shared view.

📝 Exam-Style Practice

Q1. A firewall feed keeps streaming events at its normal volume, but after the vendor pushed a firmware update, none of your firewall detections fire anymore. Ingestion-volume dashboards look completely healthy. What is the most likely root cause?

  • A) The source went silent, so silent-source detection should have already raised an alert
  • B) The vendor format changed and the parser now mis-maps fields, so events dodge the rules
  • C) Ingestion latency spiked, so the events arrive too late for the detections to evaluate
  • D) The detections were accidentally disabled during the firmware change-management window
Reveal answer & explanation

Correct: B. Steady volume plus dead detections after a vendor format change is the signature of a broken parser: events still ingest but land in the wrong UDM fields, so rules no longer match.

A: Silent-source detection only triggers when a feed goes quiet, and volume is normal here, so it is not the cause. C: Latency would show as a lag or backlog, not normal-volume ingestion with healthy dashboards. D: A disable would be unrelated to the format change and is speculative; the format-change timing points squarely at parsing.

Q2. The CISO wants a monthly, exportable summary of detection trends, SLA adherence, and overall risk posture to share with the board, while the SOC floor needs constantly-updating alert and case metrics. How should you deliver this?

  • A) Give both groups the same live SIEM dashboard and let each filter it to whatever they need
  • B) Send the CISO the raw alert feed export weekly and have the SOC read the same export
  • C) Build a live operational dashboard for the SOC and scheduled exportable trend reports for the CISO
  • D) Configure Cloud Monitoring threshold alerts to email both the board and the SOC on every spike
Reveal answer & explanation

Correct: C. Matching cadence and content to audience — a live operational dashboard for the SOC and scheduled, exportable trend reports for leadership — is exactly the audience-fit observability the scenario needs.

A: A single live dashboard buries the board in operational noise and assumes execs will self-serve filters they don't want. B: A raw alert feed is unreadable for a board summary and too coarse for live SOC work. D: Threshold alerts are for pipeline/operational anomalies, not board-level trend reporting or SOC dashboards.

Q3. You want early warning before detections degrade: notification when any expected source stops sending, when ingestion lags by hours, or when parser error rates climb. Where should these pipeline-health alerts be wired so they reach on-call without polluting the security alert queue?

  • A) As high-severity YARA-L rules in the SIEM so they surface alongside threat detections
  • B) Through Cloud Monitoring thresholds routed to the on-call channel, separate from security alerts
  • C) Into the SOAR case queue so each ingestion hiccup opens a case for an analyst to triage
  • D) Onto the executive trend report so leadership sees ingestion health each reporting cycle
Reveal answer & explanation

Correct: B. Pipeline-health signals (silent sources, latency, parser errors) belong in Cloud Monitoring with notifications routed to on-call, keeping operational health distinct from the security alert queue.

A: YARA-L watches ingested UDM events for threats; it cannot natively alert on the ingestion pipeline's own health and would mix the two queues. C: Opening a SOAR case per ingestion hiccup floods case management with operational, non-incident work. D: A monthly trend report is far too slow to act as early warning before detections degrade.
